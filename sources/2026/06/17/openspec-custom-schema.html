01 · 核心概念

什么是 OpenSpec Schema

OpenSpec 是一个面向 AI Coding Agent 的规范驱动开发（SDD）框架。它解决的核心问题是：AI 编码助手功能强大但行为难以预测，当需求活在对话历史里时，输出就会飘移。OpenSpec 的解法是：在编码之前，先锁定意图文档。

Schema 是 OpenSpec 的工作流引擎。它定义了"从一个想法到最终代码合并，中间需要 AI 生成哪些文档、按什么顺序生成、依据什么规则生成"。Schema 决定了整个变更周期的形状。

◈
核心心智模型 Schema 不是"流程图"，而是一个有向无环图（DAG）的声明。你声明节点（artifact）和边（requires 依赖），OpenSpec CLI 负责遍历这个图，Agent 负责在每个节点填充内容。

OpenSpec 内置一个默认 schema 叫 spec-driven，产出四个 artifact：

图 1 · 默认 spec-driven schema 的 artifact 流

但这个默认流程并不适合所有团队。数据仓库开发需要血缘校验；事件驱动系统需要 AsyncAPI 规约；快速迭代项目可能不需要 design 步骤。Custom Schema 就是把这个流程变成你自己的。

02 · 体系架构

三层定制体系

OpenSpec 提供三个颗粒度不同的定制层次，从轻到重：

图 2 · 三层定制体系与优先级

Layer 1：Project Config（config.yaml）

最轻量的定制方式，不修改任何工作流结构，只向 AI 注入项目上下文和规则约束：

openspec/config.yaml
# 指定使用哪个 schema（Layer 2 定义的，或内置的）
schema: spec-driven

# context 注入所有 artifact 的 Prompt
context: |
  Tech stack: Hive/Spark, Java, TypeScript
  Data warehouse: 四层 YAML Spec 架构
  Testing: 单元测试 + 集成测试
  所有 DDL 必须包含 COMMENT 字段描述

# rules 只注入匹配的 artifact
rules:
  proposal:
    - 明确上下游血缘影响范围
    - 包含回滚方案
  tasks:
    - 每个任务必须标注对应的 Spec 层级
    - HQL 文件必须放在 sql/ 目录下
ℹ
注入结构 生成任意 artifact 时，OpenSpec 按此结构组装 Prompt：<context>...</context> + <rules>...</rules>（仅匹配 artifact）+ <template>模板内容</template>。context 全局有效，rules 按 artifact id 精准注入。

Layer 2：Custom Schema（核心）

定义 artifact 的类型、顺序、依赖关系和 AI 生成指令。存放于 openspec/schemas/<name>/，与代码一起版本控制。这是最有价值的定制层。

Layer 3：Global Override

放在 ~/.openspec/schemas/，在本机所有项目中共享。适合个人习惯性工作流，但因为不在项目仓库里，团队协作时每人都需要单独配置，通常不推荐作为团队方案。

Schema 解析优先级

优先级来源典型场景
1（最高）CLI flag --schema <name>临时切换 schema 测试
2变更文件夹 .openspec.yaml单次变更使用特定 schema
3openspec/config.yaml 中的 schema:项目默认 schema
4（最低）内置默认 spec-driven无任何配置时的兜底
03 · 字段详解

schema.yaml 完整字段详解

一个完整的 schema.yaml 由元信息、artifacts 列表和 apply 块三部分组成。以下是所有可用字段：

schema.yaml · 完整结构
name: dw-spec-driven          # 唯一标识符，用于 --schema 参数
version: 1                     # 整数版本号
description: 数据仓库 SDD 工作流  # 人读的描述

artifacts:
  - id: business-spec           # artifact 唯一 ID（用于 requires 引用）
    generates: specs/business/spec.md  # 输出路径，支持 glob
    description: 业务层规约      # CLI status 显示用
    template: business-spec.md   # templates/ 下的模板文件
    instruction: |              # 注入 AI Prompt 的生成指令
      按四层 YAML Spec 的 Business Layer 格式，
      描述业务需求、数据血缘和验收标准。
    requires: []               # 依赖的前置 artifact id 列表

  - id: implementation-spec
    generates: specs/impl/spec.md
    template: impl-spec.md
    instruction: |
      根据 business-spec，生成 Implementation Layer，
      包含 HQL 模板、分区策略、调度依赖。
    requires:
      - business-spec           # 必须先有 business-spec 才能创建

  - id: tasks
    generates: tasks.md
    template: tasks.md
    instruction: |
      根据 business-spec 和 implementation-spec 生成任务列表。
      最后一组任务必须包含写入 verify/.apply-done 的步骤。
    requires:
      - business-spec
      - implementation-spec

apply:
  requires:                    # /opsx:apply 前必须存在的 artifact
    - tasks
  tracks: tasks.md             # Agent 读哪个文件追踪任务进度

artifacts 子字段

id
string · 必填
artifact 的唯一标识符。用于 requires 引用、openspec status 输出、config.yaml 的 rules 键名。
generates
string · 必填 · 支持 glob
输出文件路径，相对于变更文件夹。支持 specs/**/*.md 这样的 glob，此时只要目录下有任意匹配文件即视为完成。
template
string · 必填
templates/ 目录下的 Markdown 文件路径。内容注入 AI Prompt 作为结构骨架，HTML 注释可向 AI 提供填写指导。
instruction
string · 推荐
生成该 artifact 时注入 AI 的核心指令。描述内容要求、格式规范、关注点。这是定制 AI 行为最重要的字段。
requires
string[] · 可选
依赖的前置 artifact id 列表。OpenSpec CLI 基于此构建 DAG，只有所有依赖都已存在（文件存在），该 artifact 才进入 ready 状态。
description
string · 可选
openspec status 输出中显示的描述文字，帮助人类快速理解这个 artifact 的用途。不影响 AI 行为。

apply 块字段

apply.requires
string[] · 必填
执行 /opsx:apply 前必须已存在的 artifact 列表。相当于"实现门控"——缺少这些文档就无法开始写代码。
apply.tracks
string · 必填
Agent 执行 apply 时读取和更新的文件，通常是 tasks.md。Agent 逐条读取任务，完成后将 [ ] 改为 [x]。
⚠
config.yaml 不是 schema.yaml 的一部分 context、rules、默认 schema 名这三个字段在 openspec/config.yaml 里，不在 schema.yaml 里。两者职责分离：schema.yaml 定义工作流结构，config.yaml 注入项目上下文。
04 · 命令映射

命令与 Schema 的对应关系

OpenSpec 的斜杠命令和 schema 字段有明确的映射关系，理解这个映射是使用自定义 schema 的关键：

/opsx:propose
驱动 artifacts 列表的完整执行。循环调用 openspec status，按 DAG 顺序依次生成所有 ready 状态的 artifact，直到 apply.requires 全部满足为止。
对应 schema.artifacts
/opsx:apply
使用 apply.tracks 指定的文件（通常 tasks.md）逐条执行任务，写代码、创建文件，并标记任务完成。前置条件由 apply.requires 控制。
对应 schema.apply
/opsx:verify
完全独立于 schema。Agent 读取变更文件夹内所有 artifact + 搜索代码库，做语义层面的一致性校验（完整性、正确性、连贯性）。
独立于 schema
/opsx:archive
独立于 schema。将变更文件夹中的 delta specs（ADDED/MODIFIED/REMOVED 章节）合并到 openspec/specs/ 主目录，再将整个变更文件夹移入 archive/。
独立于 schema

05 · 状态机制

openspec status 的状态判断机制

openspec status 是 propose 循环的控制器，它的判断逻辑完全基于文件是否存在，不做任何内容语义解析：

图 4 · artifact 状态转换逻辑

openspec status --json 返回结构化数据，这是 Agent 真正读取的格式：

openspec status --change add-lineage --json
{
  "changeName": "add-lineage",
  "schemaName": "dw-spec-driven",
  "isComplete": false,
  "applyRequires": ["tasks"],
  "artifacts": [
    {
      "id": "business-spec",
      "outputPath": "specs/business/spec.md",
      "status": "done"        // 文件存在
    },
    {
      "id": "implementation-spec",
      "outputPath": "specs/impl/spec.md",
      "status": "ready"       // 依赖满足，文件不存在 → Agent 下一个目标
    },
    {
      "id": "tasks",
      "outputPath": "tasks.md",
      "status": "blocked",
      "missingDeps": ["implementation-spec"]
    }
  ]
}

与 status 配合使用的还有 openspec instructions，它是 Agent 真正用来"拿到下一个 artifact 完整生成上下文"的命令：

openspec instructions implementation-spec --change add-lineage --json
{
  "artifact": "implementation-spec",
  "template": "...模板文件内容...",
  "context": "...config.yaml 的 context...",
  "rules": ["..."],
  "dependencyContent": {
    "business-spec": "...business spec 文件内容..."
  }
}
✦
一次调用，全部上下文 openspec instructions 一次返回 template + context + rules + 所有依赖 artifact 的内容，Agent 不需要多次读文件。这个命令是 propose 循环内层的核心，相当于 Agent 的"配料包"。
06 · 创建方式

创建自定义 Schema

方式一：Fork 现有 Schema（推荐）

最快的方式，把内置 schema 复制到项目目录，再按需修改：

# Fork 内置 spec-driven
openspec schema fork spec-driven my-workflow

# 指定目标名称
openspec schema fork spec-driven dw-spec-driven

执行后会在 openspec/schemas/dw-spec-driven/ 创建完整结构：

openspec/schemas/dw-spec-driven/
├── schema.yaml          # 工作流定义，可自由编辑
└── templates/
    ├── proposal.md      # 各 artifact 的模板骨架
    ├── specs.md
    ├── design.md
    └── tasks.md

方式二：从零创建

# 交互式创建
openspec schema init dw-spec-driven

# 非交互式，指定 artifacts 列表
openspec schema init dw-spec-driven \
  --description "数据仓库 SDD 工作流" \
  --artifacts "business-spec,implementation-spec,tasks" \
  --default

方式三：从社区仓库安装

社区维护了一个 schema 集合仓库 openspec-schemas，可以让 Agent 直接安装：

# 告诉 Agent 安装指定 schema
Read this file:
https://raw.githubusercontent.com/intent-driven-dev/openspec-schemas/refs/heads/main/README.md
and install schema event-driven

验证与调试

# 验证 schema 合法性（语法、模板存在、无循环依赖）
openspec schema validate dw-spec-driven

# 查看 schema 解析来源
openspec schema which dw-spec-driven

# 列出所有可用 schema
openspec schemas

设为项目默认

openspec/config.yaml
schema: dw-spec-driven   # 此后 /opsx:propose 自动使用这个 schema
07 · 模板设计

Template 文件设计

Template 是 Markdown 文件，注入 AI Prompt 作为结构骨架。HTML 注释 <!-- ... --> 用于向 AI 提供填写指导，不会出现在最终输出中。

好的 template 设计原则：

  • Section 标题明确，AI 知道这里写什么
  • HTML 注释提供格式样例或约束说明
  • 不在 template 里硬编码具体内容——那是 instruction 的工作
templates/implementation-spec.md · 数据仓库场景示例
# Implementation Spec: {change-name}

## Target Table
<!-- 目标表名，格式：database.table_name -->

## Source Tables
<!--
列出所有上游来源表，格式：
- database.table_name: 字段描述
每个来源表都必须在 Hive 中实际存在
-->

## Field Mapping
<!--
字段映射表，格式：
| 目标字段 | 来源表.字段 | 转换逻辑 | 类型 |
每行对应一个输出字段
-->

## Partition Strategy
<!-- 分区字段、分区粒度、动态/静态分区 -->

## Schedule Dependencies
<!-- 调度依赖的上游任务，格式：task_id: 描述 -->

## HQL Template
<!--
提供关键 HQL 逻辑骨架，包含：
- INSERT OVERWRITE 语句
- JOIN 条件
- WHERE 过滤逻辑
不需要完整可运行的 HQL，只需关键结构
-->
```sql
INSERT OVERWRITE TABLE {target_table}
PARTITION (dt = '${dt}')
SELECT
  ...
FROM {main_source}
...
```
ℹ
template vs instruction 的分工 template 决定结构形状（有哪些章节、每节放什么）；instruction 决定生成策略（关注什么、遵循什么规范、不要做什么）。两者都注入 Prompt，合起来才是 AI 的完整指导。
08 · SKILL 机制

SKILL 文件：Agent 的操作手册

Schema 定义了工作流的结构，但 SKILL 文件定义了 Agent 的行为方式——触发条件、执行步骤、输出格式。两者协同工作：

图 5 · SKILL 文件的生成与加载机制

SKILL.md 的内容结构大致如下（简化示意）：

.claude/skills/openspec-propose/SKILL.md（示意）
# SKILL: OpenSpec Propose

## Trigger
Use when: user runs /opsx:propose [change-name]
Also use when: user says "start a new change", "create change"

## Steps
1. Run `openspec new change $ARGUMENTS`
2. Loop until all apply.requires artifacts are done:
   a. Run `openspec status --change {name} --json`
   b. Find first artifact with status "ready"
   c. Run `openspec instructions {artifact-id} --change {name} --json`
   d. Generate artifact content based on instructions
   e. Write file to the path in artifact.outputPath
3. When all apply.requires done:
   Report completion, suggest /opsx:apply

这就是为什么说"Agent 按 SKILL 行动"——SKILL.md 是一段结构化的自然语言 + 伪代码，Claude 把它当成执行手册，逐步推进。

09 · Verify 扩展

扩展自定义 verify 能力

内置的 /opsx:verify 只做通用语义校验，无法覆盖领域特定的结构化检查（如血缘路径合法性、HQL 字段对齐）。有三种扩展方式：

方案 A：在 Schema 中加 verify-report artifact（格式校验）

适合：在 apply 之前就能做的静态格式校验（无需代码已生成）。把校验本身变成一个 artifact，用文件产出（report.md）作为 tasks 的强制前置。

schema.yaml · 加入 lineage-check artifact
artifacts:
  # ... business-spec, implementation-spec ...

  - id: lineage-check
    generates: verify/lineage-check.md
    template: lineage-check.md
    instruction: |
      Read implementation-spec.md and validate:
      1. source_tables 字段格式是否为 db.table 格式
      2. field_list 中是否有重复字段名
      3. partition_fields 是否在 field_list 中有对应项
      4. target_table 名称是否符合命名规范（dws_/ads_ 前缀）
      Output a structured PASS/FAIL report.
      If any check FAIL, still write the report but mark overall: FAILED
    requires:
      - implementation-spec

  - id: tasks
    generates: tasks.md
    requires:
      - business-spec
      - implementation-spec
      - lineage-check   # lineage 校验通过才能生成 tasks

方案 B：独立 verify SKILL 文件（动态内容校验）

适合：apply 完成后需要做的校验（代码已生成，验证代码与 spec 的一致性）。写一个独立的 SKILL 文件，不依赖 schema 结构：

.claude/skills/dw-verify/SKILL.md
# SKILL: Data Warehouse Verify

## When to use
After /opsx:apply completes, before /opsx:archive.
Trigger: "verify", "run dw verify", "check lineage consistency"

## Steps

### 1. Task completion check
Read tasks.md → confirm all [ ] are now [x]

### 2. Spec vs HQL alignment
Read specs/impl/spec.md → extract:
  - source_tables, target_table, field_list, partition_fields

For each generated .hql file in sql/ directory:
  - Check source_tables appear in FROM / JOIN clauses
  - Check field_list matches SELECT column names
  - Check partition_fields match PARTITIONED BY clause

### 3. Business spec cross-check
Read specs/business/spec.md:
  - Verify partition convention matches business layer agreement
  - Verify target_table naming matches domain prefix rules

### 4. OpenSpec structural validation
Run: `openspec validate --change {change-name} --json`

### 5. Write report
Write to verify/report.md:
  - Overall: PASS / FAIL
  - Task Completion: [x] N/N tasks done
  - HQL Alignment: per-check results
  - Partition Check: result

### 6. Gate
If any check FAILED:
  - Show specific failures with file locations
  - Do NOT suggest /opsx:archive
  - Ask user to fix and re-run verify

If all PASSED:
  - "Verification passed. Ready for /opsx:archive"

方案 C：apply.instruction 内嵌验证

最简单但最不灵活，适合简单场景：

schema.yaml · apply 块内嵌验证步骤
apply:
  requires: [tasks]
  tracks: tasks.md
  instruction: |
    完成所有 tasks.md 中的任务。
    
    所有任务完成后，执行以下验证：
    1. 读 specs/impl/spec.md 中的 field_list
    2. 对每个生成的 HQL 文件，检查字段名是否匹配
    3. 在 tasks.md 末尾追加 ## Verification 章节，
       记录每项检查结果（PASS/FAIL）
    4. 如有 FAIL，输出具体位置并停止

三种方案对比

方案 检查时机 适合场景 可重跑 复杂度
A · artifact propose 阶段（apply 前） 格式合规、必填校验 删文件重跑 低
B · SKILL apply 完成后手动触发 代码与 spec 一致性 随时 中
C · apply.instruction apply 结束时自动 简单检查、快速场景 需重跑 apply 极低
✦
推荐组合 方案 A（schema artifact）处理静态格式校验，方案 B（独立 SKILL）处理动态内容校验。两者职责清晰，互不干扰，覆盖 propose → apply → verify 的完整链路。
10 · 完整实战

完整实战：数据仓库 SDD 场景

以下是一个为数据仓库开发量身定制的完整 schema 设计，对应四层 YAML Spec 架构（Business → Design → Implementation → Runtime）：

1

Fork 内置 schema

从 spec-driven 出发，避免从零开始。

openspec schema fork spec-driven dw-spec-driven
2

定义 schema.yaml

四个 artifact 对应四个关键文档层次，lineage-check 作为 tasks 的强制前置。

openspec/schemas/dw-spec-driven/schema.yaml
name: dw-spec-driven
version: 1
description: 数据仓库 Spec 驱动开发工作流

artifacts:
  - id: business-spec
    generates: specs/business/spec.md
    description: 业务需求与血缘来源
    template: business-spec.md
    instruction: |
      按 Business Layer 格式描述：
      - 业务背景与数据需求
      - 上游血缘来源（source system / source table）
      - 核心指标定义与计算口径
      - 数据质量要求与验收标准
    requires: []

  - id: implementation-spec
    generates: specs/impl/spec.md
    description: Implementation Layer：HQL 规约
    template: impl-spec.md
    instruction: |
      根据 business-spec 生成实现规约：
      - target_table: 目标表完整名称
      - source_tables: 所有来源表列表
      - field_list: 字段映射关系
      - partition_fields: 分区策略
      - schedule_deps: 调度依赖
      - hql_skeleton: 关键 HQL 逻辑骨架
    requires:
      - business-spec

  - id: lineage-check
    generates: verify/lineage-check.md
    description: 血缘格式静态校验报告
    template: lineage-check.md
    instruction: |
      Read implementation-spec.md and validate:
      1. source_tables 格式：每项必须是 db.table_name
      2. field_list 无重复字段名
      3. partition_fields 的每个字段必须在 field_list 中
      4. target_table 必须有 dws_ / ads_ / dim_ 前缀
      5. 命名必须是 snake_case
      Write structured PASS/FAIL report. 
      Mark overall FAILED if any check fails.
    requires:
      - implementation-spec

  - id: tasks
    generates: tasks.md
    description: 实现任务清单
    template: tasks.md
    instruction: |
      根据 implementation-spec 和 lineage-check 生成任务：
      1. DDL 创建任务（建表 SQL）
      2. HQL 开发任务（按字段组分组）
      3. 数据质量检查任务
      4. 调度配置任务
      最后添加任务：写入 verify/.apply-done 标记文件
    requires:
      - business-spec
      - implementation-spec
      - lineage-check

apply:
  requires: [tasks]
  tracks: tasks.md
3

配置 config.yaml 注入项目上下文

openspec/config.yaml
schema: dw-spec-driven

context: |
  数据仓库技术栈：Hive 3.x / Spark 3.x / YARN
  命名规范：snake_case，层级前缀 ods_/dwd_/dws_/ads_/dim_
  分区约定：日期分区字段统一为 dt，格式 yyyyMMdd
  所有 DDL 必须包含 COMMENT 字段描述
  HQL 文件统一放置在 sql/ 目录，文件名与 target_table 一致

  business-spec:
    - 必须明确指定上游血缘来源的系统名和表名
    - 核心指标需注明计算口径和去重逻辑

# SKILL: DW Verify

## When to use
After /opsx:apply, before /opsx:archive.
Trigger: "run dw verify", "verify the change", "check alignment"

## Steps
1. Confirm tasks.md: all [ ] changed to [x]
2. Read specs/impl/spec.md: extract source_tables, field_list
3. For each .hql file in sql/:
   - source_tables → appear in FROM/JOIN
   - field_list → match SELECT columns
   - partition fields → match PARTITIONED BY
4. Run: openspec validate --change {name} --json
5. Write verify/report.md with PASS/FAIL per check
6. If FAILED: show specific locations, block archive
7. If PASSED: "Ready for /opsx:archive"

核心要点回顾

概念关键理解
schema.yaml 定义 artifact DAG 和 apply 门控，不包含 context/rules（那在 config.yaml）
状态判断 openspec status 完全基于文件是否存在，与内容无关
propose 循环 status 检查 → instructions 拿上下文 → Agent 生成文件 → 循环，直到 apply.requires 满足
verify 内置 verify 独立于 schema，做通用语义校验；领域校验需通过 artifact 或 SKILL 自行扩展
archive 完全独立于 schema，负责 delta spec 合并和历史归档
SKILL vs schema schema 控制"生成什么文档"，SKILL 控制"Agent 如何行动"，两者互补不重叠

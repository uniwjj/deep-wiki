# 让 DataAgent 真正用起来: DataAgent 在企业信用的优化和实践

## 导读

Data Agent 要真正可用，必须同时解决"取数正确"和"业务认知"两大难题。面对企业信用场景下用户、场景、资产的三重复杂度，蚂蚁企信团队通过增强元信息、混合语义层和可演进的认知系统筑牢基座，并借助 Harness Engineering 驱动持续迭代 Data Agent。这一实践为 AI 时代构建 Agent 友好、去中心化且能智能治理的数据架构提供了切实路径。

主要内容包括：
1. 观察与判断
2. Data Agent 在企业信用的落地
3. 未来展望
4. Q&A

## 01 观察与判断

最近半年，Coding Agent 的进展非常显著。它的变化已经不只是"代码生成更强了"，而是逐步证明了 Agent 这套形态可以稳定进入复杂工程的主流程。在编码场景里，Coding Agent 不仅能够完成代码编写，在很多任务上的生成质量也已经接近甚至超过经验丰富的工程师；更重要的是，它开始覆盖更完整的工程链路，从任务理解与计划生成、代码修改、测试执行，到上线后的问题定位和运维处理，都展现出了更强的可用性。与此同时，我们也观察到一个很有代表性的现象：当这类 Agent 被放到数据分析、文档处理、工作流执行等更通用的环境中时，它同样可以保持较稳定的表现。这意味着，Coding Agent 的价值已经超出了"写代码"这个垂直任务本身，它更像是率先验证了一类通用的 Agent 工程模式。

这个变化背后，大致有两类因素在共同推动。一类来自模型本身的进步，包括更强的推理能力、更稳定的长链路任务规划能力，以及更可靠的工具调用能力。另一类则来自框架层的成熟。实践里很容易看到，同样的模型放在不同的 Agent 框架中，最终效果差异会很明显。像 Claude Code 这类优秀框架，之所以能显著提升任务完成质量，并不只是因为它"集成了更多工具"，而是因为它把任务拆解、环境交互、上下文管理、反馈修正这些环节组织成了一个稳定可用的工程闭环。也就是说，Coding Agent 的突破，并不只是模型能力提升的自然外溢，而是模型能力与工程系统第一次更完整地协同。

从这个角度看，Data Agent 当前最值得讨论的问题，并不是"它能不能做数据分析"，而是如何把 Coding Agent 已经验证过的闭环能力迁移到数据场景中，同时补齐数据领域自己的约束条件。数据分析任务和编码任务有相似之处，都属于长链路、强上下文依赖、需要多轮执行和修正的复杂任务；但两者也有非常明显的差异。比如，在长周期分析任务里，上下文管理比代码场景更敏感。分析过程中会持续产生中间结果、中间假设和临时结论，如果这些内容全部累积在上下文中，不仅会显著增加推理噪声，也容易引发幻觉和准确率下降。因此，Data Agent 不仅要具备上下文动态加载能力，还必须具备上下文动态卸载能力。换句话说，它需要管理的不是一个静态 Prompt，而是一组随任务推进不断被引入、替换和淘汰的工作上下文。

另一个决定性因素，是工具系统本身。Coding Agent 的能力边界，很大程度上取决于它所拥有的环境接口和工具链；Data Agent 也是一样。一个只能执行 SQL 的 Agent，本质上只是一个更自动化的问数工具。只有当它同时具备取数、语法校验、资产业务语义召回、结果校验、知识加载等能力时，才有可能进入更稳定的分析状态。也正因为如此，我们更倾向于把 Data Agent 理解为一个"能力系统"而不是一个"回答系统"。因此，第一个判断并不是简单地说"Data Agent 会像 Coding Agent 一样变强"，而是更准确地说：Coding Agent 率先验证了 Agent 稳定可用的工程闭环，而 Data Agent 正在进入承接这套闭环红利、并完成领域化补齐的阶段。

第二个判断是，企业内部真正需要的，并不是一个孤立存在的 Data Agent 产品，而是一组可集成、可复用、可组合的 Data Agent 能力。这个判断背后，对应的是企业内部能力供给方式的变化。过去很多能力天然以"独立产品"的方式被交付，用户需要主动进入某个系统、打开某个页面、发起某个操作。但在 Agent 与数字助手逐渐进入企业工作流之后，这种交付方式正在改变。现在越来越多能力可以被封装成 Skill，被数字助手、工作流引擎或其他业务系统直接调用，而不再需要维持一个单独的入口形态。这里的变化不只是产品形态变化，更是能力范式变化：企业开始从"构建独立入口"转向"沉淀可被调度的能力单元"。

这一点在 Data Agent 的实际使用中表现得非常明显。真实业务里，同学们并不是孤立地"取一次数"或者"做一段分析"就结束了。分析通常只是工作流中的一环，后面往往还会接着运营动作、消息通知、报告生成等等。所以，Data Agent 在企业内部真正需要适配的，不是一个静态问答界面，而是一条完整工作链路。它必须能够嵌入工作流，能够和其他 Skill 协同，能够被数字助手或业务系统作为能力节点调用。从这个角度看，"独立 Agent 产品"并不是唯一形态，很多时候甚至不是长期最优形态。

如果从能力体系的角度拆分，我们更倾向于把这套能力看成几层结构。最外层是交互层，这一层正在从传统的人机界面，逐渐演化为同时面向人和 Agent 的交互接口，比如 CLI、MCP、API 等形式。再往里是 Agent 编排与框架层，它承载了任务拆解、路由、调度、工作流组织等逻辑。这一层很重要，但它本身也是通用框架快速演进的区域。真正值得长期沉淀的，是更内核的两层：一层是可复用能力层，包括数据分析框架能力、校验能力、知识召回能力、路由能力等；再往内一层，是数据资产和业务认知。只有这一层，才真正对应企业最独特、最难以被通用模型和框架吸收的价值。因此，第二个判断的核心不是简单地说"不要做产品"，而是更精确地说：企业级 Data Agent 的差异化价值，不应主要沉淀在外围入口和外壳层，而应主要沉淀在可复用能力层以及其背后的数据资产与业务认知之中。

第三个判断是，Data Agent 的最终效果，本质上取决于两个基础面：数据资产和业务认知。这个判断之所以重要，是因为很多关于 Data Agent 的讨论容易停留在"模型会不会生成 SQL"或者"回答像不像人"这样的表层问题上，但真正决定系统可靠性的，往往不是语言表达，而是它能不能在正确的数据基础上，沿着合理的业务路径持续完成分析。Data Agent 面对的并不是一句自然语言到一句自然语言的转换，而是一个完整的分析链路：从用户问题开始，到取对数据、理解业务，再到形成洞察和行动建议。在这条链路里，数据资产和业务认知分别决定了两个不同维度的边界。

其中，数据资产决定的是效果下限。所谓"下限"，是指系统至少不能在基础事实层面出错。如果取数错误但没有被识别，风险往往会比"不回答"更大，因为错误结果一旦进入后续分析和决策链路，带来的偏差会被进一步放大。要做到取数正确，依赖的并不是单点的 SQL 能力，而是一整套数据资产能力：能不能找到对应的资产表，数据口径能不能被清晰定义，每次取数能不能保持一致的口径约束，整条链路能不能稳定运行。如果这些基础条件不成立，那么模型再强，也只是在更快地产生不可靠结果。所以数据资产的重要性，并不只在于"有没有数据"，而在于这些数据是否具备清晰的语义边界、稳定的口径定义和可持续消费的能力。

业务认知决定的则是效果上限。只有取数能力，没有业务认知，Data Agent 本质上仍然只是一个问数工具。它可以帮助用户更方便地拿到结果，但很难真正承担分析任务。只有当 Agent 具备业务认知之后，它才可能理解业务概念本身，识别问题所处的场景，掌握对应的分析思路，并沿着长链条分析路径持续推进，最终形成更有价值的洞察。也就是说，业务认知决定了 Data Agent 能否从"回答问题"进一步走向"参与分析"。很多时候，一个系统之所以让人觉得"会说，但不够懂业务"，并不是因为它没有语言能力，而是因为它缺少足够深的领域认知和场景理解能力。因此，更准确的表述应该是：数据资产决定系统的取数可靠性下限，业务认知决定系统的分析与洞察上限。

## 02 Data Agent 在企业信用的落地

### Data Agent 在芝麻企业信用的落地挑战

我们在做 Data Agent 的最初设想，其实很朴素：希望把数据分析能力尽可能产品化、通用化，让更多业务角色都能更低门槛地使用数据，真正接近"数据平权"的目标。也就是说，不只是让少数分析师会用数据，而是让业务、运营、产品、技术等不同角色，都能在自己的工作场景中直接获得数据支持。

但真正开始落地之后，我们很快发现，企业内部的 Data Agent 并不是一个简单的"自然语言问数"问题。它面对的不是单一任务，也不是单一用户，而是一个由用户空间、场景空间和资产空间共同作用形成的复杂分析环境。正是这三类复杂度叠加在一起，决定了企业级 Data Agent 很难靠一个通用问答壳子就真正跑通。

### 核心解决方案

正因为存在这样一个三维复杂环境，我们后来逐渐明确，Data Agent 的建设不能从"问一句、答一句"的思路出发，而必须围绕真实分析过程来组织。换句话说，我们不是要做一个更聪明的问答机器人，而是要构建一个能够参与分析、理解场景、稳定取数并在使用中持续进化的分析系统。

在这个判断基础上，我们把整体方案概括成两个核心能力和一个闭环流程。两个核心能力分别是认知能力和数据能力，这是整个系统必须优先做好的基础设施。认知能力解决的是"懂不懂业务"的问题，数据能力解决的是"能不能取对数据"的问题。只有这两部分同时成立，后面的分析过程才有意义。在这两类能力之上，我们把 Agent 的运行过程组织成从规划到分析，再到反思迭代的闭环。也就是说，用户问题进来后，不是直接生成一个结果，而是先做意图识别和任务拆解，再进入分析阶段，在过程中持续加载知识、发现资产、生成洞察，最后再对结果进行验证，并把识别出来的知识缺口、用户修正和资产问题继续沉淀下来，反哺到认知和数据能力本身。这种设计，本质上是把 Data Agent 从一个静态响应系统，推进成一个带有反馈回流和持续增强能力的分析系统。

从技术实现上看，我们把整个架构分成四层。最底层是语义层，这一层的任务是解决"能不能取对数据"的问题，包括元信息增强、语义模型、指标管理和标签管理。语义层往上是认知层，主要解决业务认知和用户认知的问题，也就是 Agent 如何理解业务术语、分析场景、角色差异和用户偏好。再往上是 Agent 的核心流程层，负责把规划、分析和反思迭代这些关键动作组织成闭环。最上层是渠道层，这一层不是决定 Agent"有多聪明"，而是决定 Agent 能以什么形态被消费。我们同时提供两类方式，一类是 Skill 形态，把能力集成到龙虾或 Coding 工具中；另一类是独立入口形态，通过封装 MCP 能力对外提供服务。虽然入口形式不同，但底层共享的是同一套核心内核。

在这样的整体架构下，我分享下四个主要实践。

### 四个关键实践

#### 第一类实践：增强元信息

这个实践的重要性，来自企业真实数据环境的复杂度。在很多业务场景里，一个主题域下有几千张甚至上万张离线资产表是非常正常的。到了这个量级，如果让 Agent 只依赖表名、字段名和简单备注来独立推理，失败几乎是必然的。因为资产中存在大量有歧义的表，也存在很多看起来相似但口径定义、过滤逻辑完全不同的字段。如果把这些内容混用，表面上结果可能"看起来合理"，但业务上却可能是致命错误。

增强元信息的核心价值，就是在离资产最近的地方补充语义，使 Agent 能够真正理解它所面对的数据对象。它可以帮助澄清歧义，补充缺失的业务语义和值域信息，让资产从"可浏览、可检索"走向"可理解、可推理、可解释"。更重要的是，元信息增强并不完全依赖额外人工维护，它可以在数据研发和治理链路里自然"长"出来。我们通过数据采样、统计 Profiling、歧义消解、置信度打标和专家审核等方式，让知识持续以增强元信息的形式保鲜，最后再通过 MCP 对外暴露，成为 Agent 可直接消费的语义服务。

#### 第二类实践：混合语义方案

我们没有尝试用一条路径解决所有问题，而是根据不同场景采用不同的数据语义链路。

- **第一条路径：NL to DAL to SQL**。面向 Chat BI 的方案，建立在 BI 报表数据集之上，优势是效率高、准确度高、对用户友好，但覆盖范围相对有限，自由度也较低。
- **第二条路径：NL to Model to SQL**。这是面向核心业务分析洞察的主力方案。基于 Data Model 来完成自然语言到分析查询的映射，参考了 MetricFlow、Cube.dev 这一类建模思路，通过度量、指标、维度和视图组织统一语义。优势是口径一致、支持版本化管理，也是长期希望沉淀成统一语义层的方向，但代价是初期建设成本较高。
- **第三条路径：NL to SQL**。主要依赖 Table Metadata，再结合离线表元信息和知识增强，用于探索式分析和长尾问题。优势是覆盖面广、冷启动快、灵活性强，但缺点也很明显，就是准确度不稳定、口径难统一。

我们最终采用的是一套路由策略，而不是单一方案。系统会先判断问题类型，再选择最适合的路径。

#### 第三类实践：建立业务认知

对我们来说，业务认知从来不是一次性写进 Context 的一段文本，而是一套可构建、可组织、可加载、可演进的系统。

它的来源主要有三类：
1. **显性知识抽取**：从文档、代码库、SQL 逻辑中提取业务术语、业务规则和分析思路
2. **人工整理和校正**：真正高价值的知识往往需要人来做归纳、消歧和补充
3. **Agent 的反馈沉淀**：包括成功案例、失败案例和用户修正，这些内容会持续暴露认知缺口，也反过来推动知识演进

在组织方式上，我们把知识分成两类：
- **召回知识库**：适合定义、规则和短知识片段，通过 Chunking 和向量化进行组织
- **文件系统**：用来维护更成体系的领域文档、长分析路径和结构化 Know-How，支持分层级组织、可搜索、可渐进式加载

用户提问之后，系统并不会一次性注入全部知识，而是先经过意图理解与路由，再选择合适的知识形式动态加载到分析流程中。业务认知不是静态存在的，而是随着提问、理解、分析和修正不断被激活和增强的。

#### 第四类实践：用 Coding Agent 来构建 Data Agent

这一部分实际上体现的是我们对 Agent 自迭代能力的探索。我们把 Agent 日志分析、Bug 收集、功能需求等统一纳入需求管理系统，再通过 MCP 暴露给 Coding Agent，让它参与 Data Agent 的持续优化。具体来说，它已经可以参与提示词优化、技能优化、元信息增强和知识加工等工作。

与此同时，我们不是让一个 Agent 独自完成所有工作，而是让多个 Agent 协同：
- **Main Agent**：负责整体规划和分析
- **Reviewer Agent**：负责持续 Review
- **Executor Agent**：负责具体研发动作
- **Eval Agent**：负责评测和效果验证

今天这套系统整体上仍然处在 Human in the loop 的阶段，很多关键节点还需要人直接参与，但目标是逐步走向 Human on the loop，也就是让人从"深度参与执行"转向"负责监督、审核和关键决策"，而把更多结构化、可验证的工作交给 Agent 自主完成。

## 03 未来展望

展望未来，我们认为 AI 时代需要新的数据架构。

第一，平台要从"给人用"走向"给 Agent 也能直接操作"。这意味着数据平台不仅要有人机交互界面，也要天然支持 MCP、Client、API 等 Agent 可消费接口。

第二，资产管理方式会从"大一统平台"走向"分域、去中心化"的管理模式，让知识和资产尽可能在靠近业务域的地方沉淀，再由统一规范连接起来。

第三，治理方式会从"人工治理为主"走向"智能发现 + 人机协同治理"，让 Agent 能够在治理链路中承担更多自动发现、自动归因、自动回流的职责。

因此，未来会持续投入三个方向：
1. 持续建设数据资产语义，让资产更适合被 Agent 理解、调用和解释
2. 继续加深业务认知，让 Agent 不只是会取数，而是真的更懂业务、更懂分析
3. 持续迭代能力本身，进一步深挖垂直场景的业务深度，同时持续探索 Agent 在复杂企业系统中的自演进能力

## 04 Q&A

### Q1：关于记忆管理与用户日志的处理

这个问题我们目前主要分两种场景来处理，对应渠道层里两套不同的方案。

第一种，是我们有独立 Agent 入口的场景。在这类场景下，系统天然可以沉淀用户和 Agent 的完整交互过程，包括对话日志，以及中间的一些推理链路信息。如果用户在使用过程中对结果进行了人工修正，这些修正信息也能够被记录下来，进一步帮助我们识别知识缺口，并对缺失的认知进行补强。

第二种，是像龙虾这类数字助手场景。这类场景的特点是，交互发生在本地系统中，我们并不能像独立 Agent 一样天然、完整地收集用户的交互信息。所以目前我们的做法是，在对话结束之后，会增加一个明确的用户确认环节，询问用户是否愿意将本次对话中形成的认知和修正信息反哺给平台。如果用户同意，这部分信息会被上传到我们维护的知识库里，并进一步进入后续的认知构建和优化流程。

所以整体上，我们在记忆管理和用户日志处理这件事上，并不是采用单一机制，而是会根据不同渠道形态，设计不同的信息回流方案。

### Q2：关于知识迭代的准确性与保鲜机制

这个问题也非常关键。坦率地讲，这部分我们还没有完全解决，尤其是你提到的认知知识如何持续保鲜、如何保证沉淀质量，这确实是当前一个比较现实的挑战。

目前我们的做法，还是以人工审核为主。具体来说，用户反馈的问题会先回流到我们的知识构建环节，在这一轮构建完成之后，还会进入后续的审核流程。现在这个审核环节主要还是由产品经理和数仓同学共同参与，他们会对反馈内容进行校验，判断它是否足够稳定、是否适合作为通用认知沉淀下来，而不是直接把所有用户反馈原样写回知识体系。

所以在现阶段，我们对这件事的基本原则是：反馈可以快速进入构建流程，但认知沉淀必须经过审核。也正因为如此，目前在线上阶段我们还不支持这部分知识的热更新。也就是说，中间必须经过人工干预和审核调整，整体上仍然属于一个 Human-in-the-loop 的模式，而不是完全自动演进。

从长期看，这部分会继续优化，但在当前阶段，我们更优先保证知识沉淀的准确性和稳定性。

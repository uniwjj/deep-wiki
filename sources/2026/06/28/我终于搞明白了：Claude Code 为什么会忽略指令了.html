我终于搞明白了：Claude Code 为什么会忽略指令了

架构未来，你来不来？



用 Claude Code 久了，很容易养成一个习惯。

它哪里做错了，就往 CLAUDE.md 里补一句。

没跑测试，就补“修改后要跑测试”；动了无关文件，就补“只改当前任务相关文件”；把本地构建成功说成已经上线，就补“代码修改、本地验证、部署状态和生产证据要分开说”。

刚开始，这个办法很灵。Claude Code 确实少犯了不少同类错误。

但文件越写越长以后，另一个问题也出来了：有些指令明明还在，Claude Code 还是会忽略。

它不是完全没读。很多时候，它前面还遵守得好好的，任务跑到后面，工具输出多了、文件读多了、需求补了几轮，某条规则就像被挤到了角落里。

后来慢慢想明白，这件事不能只怪模型。

很多指令失效，是因为我们把太多不同性质的东西，都塞进了同一个入口文件。

CLAUDE.md 很重要，但它不是总控台。

在《CLAUDE.md 拆解：Agent 进仓库前的上下文入口》里，它被放在 Agent 进仓库前的第一眼上下文里看：减少错误假设，少走弯路。现在再看 Anthropic 这篇 steering 文档，还要往后补一层：入口写对了还不够，规则还得放到合适的位置。

先把问题说小

这里不想写成一份 Claude Code 配置大全。

更值得聊的是一个具体问题：

为什么我们明明写了指令，Claude Code 还是会忽略？

从实际使用经验看，常见原因大概有几类。

有些指令并没有出现在当前上下文里。有些指令写得太空，模型知道“要注意”，但不知道具体动作。还有一些指令其实是流程，应该做成可调用的步骤；有些更进一步，本来就该由系统拦住，不该只靠模型记住。

这几类问题看起来都像“Claude 没听话”，处理方式却完全不同。

如果每次都继续往 CLAUDE.md 里加一句，文件会越来越胖，效果反而越来越不稳定。

CLAUDE.md 只适合做入口

更稳的理解是：CLAUDE.md 像一张“进仓库前的工作卡”。

它应该告诉 Claude Code 几件事：这个项目做什么，不做什么；最容易猜错的技术栈和目录边界是什么；常用命令在哪里；什么证据才能算完成；哪些地方过去反复出过问题。

这些内容越具体越好。

比如：

- API handlers validate input and shape responses only.
- Services own business orchestration and transactions.
- External HTTP and LLM calls must stay outside DB transactions.
- Separate code changes, local checks, deployment status, and production evidence in summaries.

这类规则有对象、有动作，也能验证。

反过来，下面这些话听起来没错，实际帮助有限：

- Write clean code.
- Follow best practices.
- Be careful with security.
- Keep the project organized.

Claude Code 看完以后，仍然不知道哪个目录不能碰，哪个命令必须跑，哪种情况要停下来确认。

所以 CLAUDE.md 不是越完整越好。它越像百科，越容易把关键内容淹掉。

Anthropic 官方文档也建议让 CLAUDE.md 保持简洁，并且要有人维护。官方给的目标是控制在 200 行以内。这个建议很实际：每一行常驻规则，都会占上下文预算，也会增加维护成本。

这也能解释为什么不太建议把 CLAUDE.md 写成项目知识库。知识库可以放在 docs 里，让 Claude Code 需要时自己读；入口文件更像门口那张提醒卡，只放最容易误判、最常影响结果的东西。

指令为什么会被忽略

遇到“明明写了但没执行”，可以先看四件事。

第一，规则有没有真的加载。

有些规则放在子目录 CLAUDE.md 或带路径的 rules 里。Claude Code 只有读到对应目录或文件时，才会把它们放进上下文。任务刚开始、新建文件、压缩后继续写，都可能出现“规则还没进来”的情况。

这时要先查加载状态，别急着改措辞。比如用 /memory、/context 看看当前会话到底加载了哪些指令。

第二，规则是不是太空。

“注意安全”“保持简洁”“写高质量代码”更像会议提醒，不像 Agent 的执行规则。Agent 更需要可执行的边界：什么输入必须校验，哪些调用不能放在事务里，哪类文件不能改，改完以后跑哪个命令。

第三，规则是不是被当前任务盖住了。

长任务里，Claude Code 会不断读文件、跑命令、看报错、接收新的补充。上下文越来越厚，最近的信息往往更容易影响下一步动作。这也是很多人会有的体感：刚开会话时很听规则，跑久了就开始“飘”。

第四，这条规则是不是本来就不该交给模型记。

比如：

- Never modify `.env`.
- Never run destructive commands.
- Always run tests before commit.
- Never deploy without approval.

这些句子可以作为提醒，但不能当最后一道防线。

如果一条规则漏掉以后只是返工，可以先写进上下文。

如果漏掉以后会删数据、泄露密钥、绕过发布流程，那就应该交给权限、Hook、CI、脚本或仓库保护。

软提醒和硬边界要分开。

规则可以分成五层

整理 Claude Code 项目规则时，可以先分五层。

把它画出来，大概是这样：

这张表不是为了显得整齐。

它真正有用的地方，是把一个模糊问题拆开：这条规则到底是给模型看的，给流程调用的，还是给系统执行的。

放错层以后，表现出来就是 Claude Code 忽略指令。

真正落到一条新规则时，可以再用一个简单分流判断：先看它的性质，再决定机制。

局部规则别放全局

很多团队的 CLAUDE.md 变胖，是从局部规则开始的。

比如 API handler 必须做输入校验，migration 文件必须写回滚说明，mobile 目录不能引入 Web-only 包，某个团队目录有自己的测试命令。

这些规则都重要，但不该让每个任务都加载。

更合适的位置，是子目录 CLAUDE.md 或 .claude/rules/。

例如：

---
paths:
  - "src/api/**/*.ts"
  - "**/*.handler.ts"
---

# API rules

- Validate request input before business logic.
- Return the standard error response shape.
- Keep external calls outside database transactions.

这样，Claude Code 处理 API 文件时才会看到这段规则。改文档、补脚本、整理 README 时，就不用为它付上下文成本。

对大仓库来说，这点尤其重要。不同团队、不同包、不同技术栈，往往都有自己的约定。如果都堆在根 CLAUDE.md，最后会变成所有任务加载所有规则。

规则不是越多越稳。

很多时候，少一点、准一点，反而更可靠。

流程应该离开入口文件

还有一类内容，经常被误塞进 CLAUDE.md：流程。

比如发布流程。检查分支、跑测试、看环境变量、打包、部署、smoke、记录证据、写 changelog，这已经不是一条规则了。

它是一套过程。

过程放在入口文件里，会有几个问题：不发布的时候也会加载；步骤越补越长；每个团队都想加自己的特殊情况；最后没人知道哪一步还有效。

这类内容更适合做成 Skill。

CLAUDE.md 里只留一句路由就够了：

- For release work, use the release-check skill and report evidence separately.

真正的步骤放到 Skill 里维护。

这件事对团队有价值。Skill 不只是提示词，它更像一份可复用的工作流程：有输入、有步骤、有验收、有失败处理。

放到 Loop 和长任务目标控制面里看，这个问题会更清楚。Agent 要做长任务，不能只靠“记住更多”。更稳的办法，是把反复发生的过程沉淀成可调用、可检查的流程资产。

Subagent 不是多一个人

Subagent 很容易被理解成“多找一个助手”。

这个说法能入门，但不够准确。

这里更值得关注的，是它对上下文的隔离作用。

真实工程任务里，有很多工作会产生大量中间材料。比如全仓搜索、日志分析、依赖版本对比、历史提交调查、安全审计、批量读文档。

这些材料如果都进主会话，主线很快就会变胖。更麻烦的是，很多中间过程只对探索有用，后面并不需要继续占着上下文。

Subagent 更像一块临时白板。

它在旁边把搜索、分析、归纳做完，最后只把结论、证据和建议带回来。主会话继续保留决策线，不被探索过程拖住。

所以 Subagent 不只是多 Agent 玩法。

它更像一种上下文卫生工具。

高风险规则要有硬边界

最容易出事故的，往往是这类规则：

- Never edit secrets.
- Never run destructive commands.
- Always run tests before commit.
- Never deploy without approval.

这些话可以保留，但不能只靠它们。

软提醒和硬边界的差别，可以简单看成这样：

权限、密钥、删除、发布、付款、生产数据，一旦出错，代价不是“再提醒 Claude 一次”能解决的。

更稳的做法，是把它们下沉到系统边界里：密钥和生产配置用权限规则或仓库保护拦住；危险命令用 PreToolUse Hook 或 deny rule；格式化、测试、发布前 smoke，用 Hook、CI、脚本或 release checklist 留痕。

一句话：

能让系统兜住的，就不要只写成自然语言。

自然语言适合提醒模型。

硬约束负责兜底。

这点放到真实工程里很关键。对“完成”这两个字保持谨慎，也是同一个逻辑：模型可以给出总结，但真正能证明完成的，还是测试输出、diff、部署记录、线上 smoke 这些外部证据。

一份变胖的 CLAUDE.md 怎么收

如果今天接手一份几百行的 CLAUDE.md，不必一上来就删。

更稳的做法，是先做一次归类。

每次会话都该知道的内容，留在入口文件。比如项目定位、技术栈、常用命令、目录边界、完成证据和高频错误假设。

只对某些目录、文件类型、团队包生效的内容，拆到子目录 CLAUDE.md 或 .claude/rules/。

超过 5 步的流程，先考虑 Skill。发布、review、迁移、验收、排障、归档，都更像过程资产。

会产生大量中间材料的工作，优先交给 Subagent。主会话只拿摘要、证据和结论。

看到 always、never、must、forbidden、production、secret、deploy、payment、delete 这些词，要多停一下。如果它们关系到事故成本，就不要只留在自然语言里。

写在最后

现在再看 CLAUDE.md，态度会比一开始克制很多。

它仍然很有用。

但它只适合做入口：把项目里最容易被猜错、最常影响结果、最需要反复提醒的东西，放到 Claude Code 第一眼能看到的位置。

再往后，局部规则交给路径触发，流程交给 Skills，旁支探索交给 Subagents，高风险动作交给 Hooks、permissions、CI 和脚本。

这些东西合在一起，才是一个更稳的 Claude Code 工作现场。

所以下次 Claude Code 忽略某条指令时，先别急着往 CLAUDE.md 里再加一句。

可以先问：它现在真的看到了这条规则吗？这条规则够具体吗？它是不是只在某些目录下才相关？它是不是一个流程？它失败以后有没有事故成本？

这些问题问清楚以后，很多“为什么又忽略了”的问题，就有了更稳的解法。

上下文窗口就那么大。

每一行常驻指令，都在花预算。

# Loop Engineering详解：把反馈循环放进工程现场

作者：若飞 | 2026-06-11 | 来源：架构师（JiaGouX）微信公众号

架构师（JiaGouX）我们都是架构师！架构未来，你来不来？

这两天，Loop Engineering 突然热了起来。

Peter Steinberger 在 X 上说，重点会从一轮一轮提示 coding agent，转向设计能提示 agent 的 loop。Boris Cherny 也提到，他现在更多是在写 loop，让 loop 去驱动 Claude 判断下一步。

这两句话很适合被写成一个刺激标题：提示词工程要过时了。

但这几天读下来，我反而不太关心这个口号。

Gergely Orosz 的担心很接地气：很多团队没有无限 token，也没到“手工提示已经卡住工作”的阶段。Garry Tan 也提醒过，不要把 Agent 做成机械重复的工厂。Graham Neubig 给的日常 loop 例子里，还保留了一个普通但关键的动作：人先过一遍任务清单，再决定派给哪个 Agent。

这些讨论放在一起，问题反而更清楚了。

Loop Engineering 更值得看的地方，不是“人不再写提示词”。更准确地说，提示词开始进入一个反馈系统：发现工作、分配任务、执行、验证、记录状态，然后决定继续、停止，还是交给人。

放回我们最近这条线里看，它其实并不突兀。

写 Harness 时，我们聊的是状态边界和失败闭环：Agent 不能只会生成，还要知道哪些只是候选、哪些已经验证、哪些真的提交了。

写 Harness Engineering，我们换了一个说法：模型外面需要一张工作台。上面有工具、上下文、状态、测试、权限，也有将来可以删掉的旧约束。

今天的 Loop Engineering，正好再往前走半步：

如果 Harness 是让 Agent 跑在一个可检查的工作现场里，Loop 接着问的是：工作现场能不能定期醒来，继续发现问题、处理问题、留下证据。

这篇就不围着“提示词死没死”转了。

我更关心一个工程问题：如果明天就在团队里小范围试 loop，哪些地方能先跑，哪些地方要先收住。图 1：Loop Engineering 的控制循环判断
- • Loop 不只是 cron，也不只是让 Agent 一直重复“继续优化”。它更接近一个带反馈的控制循环。
- • Prompt 没有消失，只是从聊天框里的手工输入，变成了循环系统里的一个组件。
- • 一个可用的 loop 至少要有五样东西：自动触发、隔离工作区、过程资产、外部连接、独立验证。再加一个状态记忆。
- • 我会先从事实核验、CI 分流、文档检查、重复故障归类、陈旧配置清理这类场景试，不急着碰核心业务代码。
- • Loop 的价值取决于验证是否便宜。如果验证很贵、边界很模糊、后果很重，loop 会把问题放大。
- • 不是所有团队现在都需要 loop。任务重复、验证自动化、预算撑得住、Agent 有真实工具时，再考虑把它放进循环。
- • 对架构师来说，新的工作不是退出现场。目标、边界、预算、证据和停止条件，会被放到更靠前的位置。

一句话说：

提示词解决的是“下一句话怎么说”，loop 解决的是“这件事怎么持续做、怎么知道做对、什么时候停”。提示词

“从提示 Agent 转向设计 loop”，这句话很有传播力。

但在真实工程里，我不太愿意把它理解成二选一。

更贴近现场的说法是：手工提示仍然有效，只是它不再适合承担所有控制责任。

过去我们和 coding agent 协作，常见模式是：
- 1. 人写一个 prompt；
- 2. Agent 改一轮；
- 3. 人读结果；
- 4. 人再补一轮 prompt；
- 5. 反复到自己觉得差不多。

这种协作适合小任务，也适合探索。任务一长，问题就出来了：
- • 人会忘记上一轮边界；
- • Agent 会提前收工；
- • 同一个上下文里自写自审，标准容易松；
- • 验证结果散在聊天记录、终端、截图和脑子里；
- • 第二天接着做时，又要重新解释一遍。

Loop 处理的是另一个问题：这些来回动作能不能变成一个有状态的系统。

比如：
- • 每天早上扫描昨天的 CI 失败和新 issue；
- • 把能独立处理的问题拆成任务卡；
- • 每个任务进独立 worktree；
- • 一个 Agent 尝试修复，另一个 Agent 按测试、日志、需求文档复核；
- • 通过就生成 PR 和证据；
- • 不确定就进入人工队列；
- • 所有尝试、失败和下一步写回状态文件。

这时候 prompt 还在。

只是它不再只是人临时写下的一句话，还会被放进 Skills、任务模板、验证脚本和自动化流程里。控制层

前面写 Harness Engineering 时，我们一直在讲一个判断：

Agent 不只是模型，它还包括围绕目标持续行动的一整套系统。

Model 负责推理，Tool 负责接触外部世界，Skill 负责沉淀做事方法，Sub-agent 负责分工，Harness 负责让整套东西跑起来、跑得住、跑完还能检查。

Loop 再往上走一层。

如果说 Harness 管的是“这一次任务怎么跑”，Loop 管的是“这类任务怎么持续发生”。

可以粗略看成这样：
图 2：从 Prompt 到 Loop 的层次关系

这也解释了为什么这次话题会同时指向 Claude Code 和 Codex。

Addy Osmani 在原文里把 loop 拆成几块：自动化、worktree、skills、plugins/connectors、sub-agents，再加状态记忆。这套拆法有启发。放到团队现场，我会先把它翻译成六个问题：
问题对应能力解决的风险什么时候启动自动化、定时任务、/goal、/loop靠人想起来才做在哪里改worktree、隔离环境、临时分支并行互相覆盖按什么规则做Skills、项目规则、计划模板每轮重新解释业务能连到哪里MCP、插件、连接器、CLI只能看本地文件谁来复核sub-agent、reviewer、测试自写自审过于宽松怎么接上下一轮状态文件、issue、看板、日志上下文中断和目标漂移

从架构上看，我会先把 loop 拆成四个口：
- • 触发入口：谁能启动，多久启动一次，输入从哪里来。
- • 执行沙箱：在哪个 worktree、分支或临时环境里跑，能不能回滚。
- • 验收出口：用哪些测试、日志、规则和人工复核来判定结果。
- • 状态账本：这轮尝试、证据、失败原因和下一步写到哪里。

这些问题都不玄。

难点在组合：这些东西放在一起后，系统还能不能停得住、看得清、交得出去。闭环

几份资料都提到“闭环”和“开环”。这组区分值得留下来。

闭环像一个边界明确的小控制系统。目标写清楚，动作范围受限，反馈足够客观，停止条件能验证。比如 CI 分流、链接检查、事实核验、依赖升级预检查。

开环复杂得多。Agent 可以自己继续发现任务、扩展路径、决定下一步，看起来更“自主”，风险也更高：预算更难控，目标更容易漂移，验证也更依赖人工。图 3：闭环和开环的差异

我会把 loop 的第一站放在闭环，开环先放一放。

这不是排斥新东西。工程里有个老经验：先让一个小闭环跑稳，再谈更大的自动化。验证

我不会一上来就把核心业务开发交给无人值守 loop。

它更适合“目标清楚、验证便宜、失败可回滚”的任务。

这里我挺认同 AlphaSignal 那个保守判断：大多数开发者还不急着把 Agent 放进 loop。能不能放进去，可以先看四个条件：这件事会不会重复，能不能自动验证，token 预算能不能承受重试，Agent 有没有真实工具可用。

如果验证很便宜，loop 会放大收益。

如果验证很贵，loop 会放大幻觉、成本和维护压力。

先用一张小表做准入判断：检查项可以试暂缓输入日志、issue、测试报告、文档这类输入稳定输入靠口头描述，边界每天变输出分类表、候选 PR、证据清单、风险说明直接改核心逻辑，影响难评估验证有测试、lint、链接检查、可复现命令只能靠人读一遍感觉对不对权限默认只读，写入走隔离分支直接改主分支或生产配置停止有预算、时间、证据不足等停止条件只写“继续优化”，没有退出点

这张表不复杂，但很管用。五项里只要有两项落在右边，我一般会先补测试、补状态、补边界，再考虑自动 loop。

一个直观例子是技术稿事实核验。

发布前，让 Agent 先抽取文章里的事实断言，再分头去官方文档、代码仓库、论文、issue、commit 里核对，最后把每条断言标成：
- • 已确认；
- • 来源不足；
- • 时间可能过期；
- • 只是推测；
- • 建议删除。

这类任务适合先放进 loop 里试。

它不直接改生产系统，验证标准也清楚。即便 Agent 做错了，通常也只是多花一点检查时间，不会马上变成线上故障。

再比如 CI 分流。

每天固定读取失败日志，把失败分成环境问题、测试不稳定、最近提交引入、已知旧问题，再生成一个小表。能定位到具体 commit 的，再开一个隔离 worktree 尝试最小修复。定位不到的，进人工队列。

这类 loop 的价值不在“自动把所有 CI 修好”。

它先把每天重复的第一轮排查做干净，让人把注意力留给需要判断的部分。

如果让我挑试点，我会先看这几类：场景为什么适合技术稿事实核验读多、改少、证据可列CI 失败分流输入结构化，验证路径明确文档链接和配置漂移检查可自动扫描，可人工确认重复故障归类需要并行阅读大量日志和 issue陈旧 feature flag / 实验配置清理天然可拆，结果可先只读输出依赖升级预检查可以先生成影响范围和测试建议

这些任务有一个共同点：

Agent 可以跑得久一点，但最后的结论能被证据压住。预算

网上有个质疑很实在：你们有无限 token，我没有。

这不是扫兴，这是工程约束。

Loop 一旦跑起来，每一轮都可能重新读上下文、调用工具、生成计划、做验证。如果再叠加 sub-agent，成本会很快从“多几次对话”变成“一个小型后台系统”。

预算不能等到账单出来再看。

它一开始就放进任务卡。

比如：

循环名称：每日 CI 分流
触发频率：每天 09:30
输入范围：过去 24 小时失败 job、最近 20 个 commit
最大运行：30 分钟
最大分支：一次最多处理 5 个失败簇
权限：默认只读；修复必须进入独立 worktree
验证：只运行相关测试，不跑全量回归
停止条件：无新失败、证据不足、预算到达、需要人工决策
交付物：失败分类表、可复现命令、候选 PR、剩余风险

这张任务卡看着普通，但它把几件事说清楚了：
- • loop 能看什么；
- • 能改什么；
- • 最多跑多久；
- • 哪些情况要停；
- • 什么算完成；
- • 人什么时候接手。

这些边界如果不写，loop 会很快变成一个乐观的 while 循环。

它会一直尝试，一直解释，一直“看起来有进展”。

但 momentum 不是 progress。

动静很大，不代表系统真的变好了。状态

几篇资料里都提到了一个极简例子：Ralph loop。

大意就是把一份 PROMPT.md 反复喂给 coding agent：

while :; do cat PROMPT.md | claude-code; done

这行命令粗糙，但它点出了一个关键事实：每一轮都重新读取外部文件和当前仓库状态，而不是指望聊天窗口记住一切。

当然，这样直接跑进真实项目里，多半会烧钱，也容易跑偏。

但它给了一个很好的提醒：

loop 的记忆不能只放在对话里。

Loop 里有个常被低估的部分：状态记忆。

长任务里，麻烦的不是 Agent 不努力，而是每一轮都像新来的同事。

它不知道昨天试过什么，不知道哪个方向已经被否掉，不知道哪个测试因为环境问题可以暂时忽略，也不知道哪条边界绝对不能碰。

这些东西如果只留在聊天上下文里，很脆。

上下文会压缩，会截断，会被新信息冲淡。更现实一点，人第二天也会忘。

loop 需要一个对话之外的状态载体。

可以是 plan.md，可以是 issue、看板、数据库，也可以是一份很普通的 Markdown：

## 当前目标
修复订单导出最近 24 小时的 CI 失败。

## 已尝试
- 读取 job 1324 日志，失败点在 export timeout。
- 检查最近 20 个 commit，疑似与批量查询改动有关。
- 尝试增加超时，不通过，已回滚。

## 已验证
- 单测 `OrderExportTest.test_large_batch` 复现失败。
- 小批量导出正常。

## 禁止事项
- 不改权限模型。
- 不改导出文件格式。

## 下一步
- 对比批量查询前后 SQL。
- 如无法定位，交给人工看慢查询日志。

这份文件不复杂。

但它能让下一轮 loop 接上前一轮，也能让人快速看懂它到底做了什么。

我们前面写 Agentic Engineering 工作台时讲过：计划、上下文、执行、验证和治理要有固定落点。Loop 只是把这件事进一步自动化。

如果没有状态记忆，loop 就会变成一串断开的 prompt。

看起来连续，实际上每轮都在重新开始。人在场

Loop Engineering 常被误读成：人可以从流程里拿掉。

我的体感刚好相反：Loop 越强，人的判断越要提前出现。

以前人是在每一轮 prompt 里临时判断：
- • 下一步做什么；
- • 哪个结果可信；
- • 哪个文件不能碰；
- • 哪个测试失败可以接受；
- • 什么时候停。

现在这些判断会提前变成规则、模板、权限、预算和停止条件。

Graham Neubig 那个例子给我的提醒也在这里：他的 loop 没有把所有待办直接丢给 Agent，而是先整理信息、走一遍优先级，再由人决定哪些任务值得委派。这一步看起来慢，其实是在保护系统不被“自动化的热情”带偏。

人的位置变了，但责任没有消失。

甚至更重。

因为无人值守的 loop 会在无人值守时犯错。它可能开很多分支，写很多代码，生成很多解释，也可能把一个错误假设扩散到十几个子任务里。

放到团队里，我会先保留几条偏保守的原则：
- 1. 先只读，后写入。
- 2. 先低风险，后核心路径。
- 3. 先小频率，后高频率。
- 4. 先人工确认，后自动合并。
- 5. 先写停止条件，再写继续条件。

尤其是第五条。

很多自动化出问题，不是因为不会继续，而是因为不知道什么时候停。

一个成熟的 loop，最好能诚实地说：

我没有足够证据继续。
这次修改超过了授权范围。
预算已经到达。
验证结果不稳定。
需要人做产品判断。

比起“我继续试试”，这种回答更接近工程系统。试点

如果团队想试，我会从一个很小的 7 天试点开始。

不用一上来就做“自动开发平台”。

先做一个“每日可验证循环”。

可以这样拆：

第 1 天：选场景

选一个低风险任务，比如 CI 分流、文档链接检查、技术稿事实核验。目标清楚，输入稳定，结果也容易人工复核。

第 2 天：写任务卡

把输入范围、权限、预算、停止条件、交付物写下来。先不急着自动化，把边界说清楚。

第 3 天：做 Skill

把项目规则、常见误区、验证命令、输出格式写成 Skill。这样 Agent 每轮不用重新猜。

第 4 天：接状态记忆

用 plan.md、issue 或看板记录已尝试、已验证、阻塞和下一步。先简单，能延续就行。

第 5 天：跑一次手动 loop

人手动触发每一步，看哪些地方需要自动化，哪些地方仍然要人工判断。

第 6 天：加自动触发

只给一个固定频率，设置时间上限和预算上限。输出先进入人工 inbox，不自动合并。

第 7 天：复盘

看三件事：省了多少重复排查，误报多少，人工复核是否更轻。这三件事都说得过去，再考虑扩大范围。

复盘时，我会把验收口径写得更具体一点：指标看什么先到什么程度算可继续命中率loop 找到的问题里，有多少确实值得处理人工复核后仍有稳定收益误报率有多少输出会浪费人的时间不影响日常节奏回滚率Agent 写入后有多少需要撤回写入先控制在候选分支成本token、运行时间、工具调用次数账单和收益能对上证据每个结论有没有来源、命令、日志或测试结果人能在 5 分钟内复核一轮

如果这些数说不清，问题通常不在 Agent，而在工作流本身还没有被定义清楚。

这套试点不炫目。

但它能回答一个关键问题：团队到底缺的是更强的 Agent，还是更清楚的工作现场。

很多时候，答案是后者。收束

Loop Engineering 这个词可能会热一阵，也可能很快被下一个词盖过去。

但它指向的变化是真实的。

当 Agent 可以长时间运行、调用工具、拆任务、开 worktree、接插件、找 sub-agent 复核时，工程师的发力点确实会移动。

以前我们把大量精力花在“这一轮怎么说清楚”。

接下来会有更多精力放在“这类事怎么被系统稳定地做完”。

这不是提示词工程的结束。

我的理解是：提示词被放回工程系统里。

我现在对 loop 的态度比较克制：

可以试，值得学，不用神化。

它能放大已经清楚的工作流，但想清楚工作流这件事仍在人这里。

它能持续处理可验证的小闭环，但模糊的业务判断仍要有人负责。

它能让工程师少做重复操作，但不能让工程师少理解系统。

循环越自动，人的判断越要在场。

因为 loop 本身不知道你是在加速理解，还是在逃避理解。

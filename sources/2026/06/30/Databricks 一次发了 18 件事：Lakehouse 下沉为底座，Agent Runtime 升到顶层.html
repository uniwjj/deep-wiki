2. Genie 系列：Databricks 把"AI 同事"做成了产品

全场分量最重的发布，来自 Genie 这条产品线。它从去年的"会话式数据分析助手"，一次性升级成了三件事：一个跨端的 AI 同事、一组可分享的领域 Agent、一张自动生长的企业知识图。

2.1 Genie One：从聊天框走向"在工作的地方工作"

图源：Databricks 官方博客

官方的定位短语是 "data-smart AI coworker"——一个"懂数据"的 AI 同事。它和过去的 Genie 最大的不同，是从一个"独立网页里的聊天框"，变成一个跨 Web、iOS、Android、Slack、Microsoft Teams 的统一身份。

具体能力：

  • • 跨数据源：通过 Lakehouse Federation 与 Lakeflow Connect 直接拉通几十个外部系统
  • • 双向集成：能读，也能写——Gmail 草稿、Slack 消息、Teams 会议、Google Drive / SharePoint 文档全部在覆盖范围内
  • • 可调度、可监控：Genie 可以创建定时任务、设告警、跑监控，而不只是"问一句答一句"
  • • 自定义 MCP：企业既可以挂自家的 MCP 服务器进来，也可以把 Genie 通过 Genie MCP App 暴露给第三方 Agent 调用
  • • 新发布的原生 iOS/Android App
图源：Databricks 官方博客

官方给的两个例子很有代表性：销售主管早上打开 Slack，@Genie 让它生成"今天我所有客户会议的简报"；零售连锁的 GM，让 Genie 用最新库存数据和昨天的会议转写，自动更新一份业务复盘文档。

2.2 Genie Agents：从"Genie Spaces"到"领域 Agent"

去年发布的 Genie Spaces 已经被客户创建超过 100 万个。这次，Databricks 把 Spaces 直接升级为 Genie Agents——一类面向特定业务域、可以自主执行任务、可以分享给团队甚至外部客户的 Agent。

它和"Spaces"最大的区别：

  • • 单 Prompt 创建：在 Genie One 或 Genie Code 里描述一句"我要一个负责供应链异常预警的 Agent"，系统会自动 scope、benchmark、给出可分享版本
  • • 可推理非结构化数据：不仅能查表、跑 SQL，还能直接读文档、PDF、邮件、知识库
  • • MCP 多步执行：通过 MCP 调用外部工具、写入外部系统，多步任务无需人工逐步确认
  • • 可被分享与售卖：这是关键——Genie Agents 可以通过 OpenSharing 协议对外发布，理论上可以做"按问题计费"的商业模式
图源：Databricks 官方博客

Foot Locker 在演讲里给出了一句很 Databricks 风格的客户证言："Genie 已经成为我们组织里自助洞察的引擎。"

2.3 Genie Ontology：这次最容易被低估的发布

如果让我从 18 项发布里挑最被低估的一项，是 Genie Ontology。

它的定义是："a living context graph for your business"——一张活的、自动生长的企业上下文图。

图源：Databricks 官方博客

它的工作机制（这部分非常技术，但很关键）：

  • • 自动抽取：从所有连接的表、查询、Dashboard、Pipeline、Wiki、Jira、Slack、SharePoint、Google Drive 中抽取知识片段——不需要人工编辑词条
  • • 类 PageRank 的权威加权：每个知识片段会被打分，权重由 5 个维度决定——
    • • 来源系统的权威性（一份 Finance 部门认证的 Dashboard 比一条 Slack 消息权威）
    • • 作者的组织权威（CFO 写的口径 > 实习生）
    • • 使用频率（被广泛查询的 metric 更可信）
    • • 是否关联到"已认证资产"
    • • 新鲜度（昨天的口径优先于半年前的）
  • • 强制继承源系统 ACL 与 Unity Catalog 权限：你只能"看见"你本来就有权限看的内容，AI 不会做权限旁路

Databricks 官方放出了一组很大胆的数字：

图源：Databricks 官方博客（厂商自报口径）

内部 28 题真实业务问题集（2026 年 6 月）：Genie 首问命中率 84.5%；最强的竞品 coding agent 52.4%；最弱 25%；速度约为最强对手 2 倍。
——Databricks 官方博客（厂商自报口径）

为什么这件事重要？

这相当于把过去几年分散在 Headless BI 的 Metric Layer、知识图谱、向量索引、Catalog 几条赛道——合并成了一种新的、自动构建的"企业上下文产品"。Databricks 给同行留下的命题是：没有持续生长的上下文图，Agent 在企业里不会真正落地。

Snowflake 的 Cortex Agents、Microsoft Fabric 的 Copilot in Dataflows、Salesforce 的 Data Cloud Agents——所有这些产品在 2026 下半年都会被迫回答一个问题：你的"上下文"是哪里来的？

2.4 Genie Code：开发者侧的 ML 工程 Agent

Genie Code 不是一个新东西，但这一次升级到覆盖完整 ML 生命周期——它现在能跨 MLflow / Feature Store / Model Serving / Inference Tables 直接执行操作，而不只是"生成代码片段"。

具体新增：

  • • MLflow 全链路感知：从特征工程、实验跟踪、模型部署、监控、漂移检测到自动重训
  • • GPU 调试：会话式调 CUDA / cuDNN / NCCL profile（很多 ML 工程师会喜欢这一项）
  • • 并行线程 + 调度任务：Agent 自己开多条 thread 跑实验
  • • BI 迁移工具：可以把 Tableau / Power BI 的旧 Dashboard 直接导入到 Databricks AI/BI

最后一条尤其值得划重点——Databricks 开始正式"挖墙脚"。

3.3 Omnigent：Apache 2.0 的 meta-harness

Omnigent 是这次发布里"最有产业野心"的一项。

它的定位很微妙：不是一个新的 Agent 框架，而是一个站在 LangGraph、CrewAI、Claude Code SDK 之上的"统筹层"——你可以把它想成 Kubernetes 之于容器：你仍然可以选 Docker、containerd 或 CRI-O，但调度它们的是上面那一层。

Databricks 把它以 Apache 2.0 协议开源，同时在自己平台上提供托管版（Beta）。

当 Databricks 把"框架编排器"开源出来，意味着它不打算押注任何一个具体框架——下一阶段，框架本身会被进一步降维成实现细节，竞争上移到 meta-harness 与治理层。

LangChain 的 Harrison Chase 在 Summit 现场出席这件事，多少有点意味深长。

3.4 Governed Vibe Coding 三件套

Databricks Apps 这条线这次也给了三个东西，把"自然语言写企业应用"塞进了 Unity Catalog 的边界里：

Apps 这条线已经在跑——比上一届 Summit 5 倍增长，每周有 5,000+ 账号在生产环境跑 Apps。

4. 数据底座三连发：LTAP、Lakebase、Lakehouse//RT

这是这场 Summit 的工程级里程碑——Databricks 在数据底座上同时挑战 Snowflake（OLAP）、Aurora/Postgres（OLTP）、Pinecone（向量）、Pinot/ClickHouse（实时数仓）四条战线。

如果未来 12~18 个月企业数据架构会被重画，这一节是震源。

4.1 LTAP：把 OLTP 与 OLAP 合并到同一份数据

LTAP（Lake Transactional and Analytical Processing） 是 Ali Ghodsi 在 Keynote 上点名的"新架构范式"——他原话讲："this is the industry's first true LTAP architecture"。

它解决的是过去十年所有数据团队都在打的一场仗：

  • • 你有 Aurora / Postgres 做事务（OLTP）
  • • 你有 Snowflake / BigQuery / Databricks 做分析（OLAP）
  • • 中间架一条 CDC 管线（Debezium / Fivetran / Estuary）
  • • 数据延迟从分钟到小时不等
  • • 一份数据三份副本，每份都要做权限、做 governance、做监控

LTAP 的方案：OLTP 与 OLAP 共用同一份开放格式数据（Delta / Iceberg），不再 CDC，不再复制。

它不是凭空冒出来的——是 Lakebase（Postgres）、Delta UniForm（Delta/Iceberg 互通）、Lakehouse//RT（实时引擎）三件事拼出来的能力。这是 Databricks 用三年时间把 OneHouse 这条 Apache Hudi 路线讲过的故事，用自己的产品矩阵跑通了一遍。

4.2 Lakebase：Serverless Postgres 的全面升级

Lakebase 这次的升级清单堪称密集：

  • • Scale-to-zero：闲置时自动缩容到零，不开机不收钱
  • • Database branching：Git 风格的数据库分支——给开发/测试环境复制一份"指针"，不复制数据
  • • 跨云跨区灾备（DR）
  • • 自治运维：自动 vacuum / index / 调优
  • • Lakebase Search（Beta）：Hybrid Vector + Full-Text 混合检索内置在 Postgres 中

最后一条，是这场 Summit 里"最安静的一颗炸弹"：

当 Postgres 自己就能跑混合检索，"再单独装一个向量库"的工程合理性会被快速质疑。Pinecone、Weaviate、Qdrant 这一类独立向量数据库的存在理由，从今年下半年开始会被 CIO 们重新审视。

运营数字佐证产品成熟度：Lakebase 日均 1,200 万次数据库启动——这个数字甩开了大多数云厂商的 Serverless DB 业务。

4.3 Lakehouse//RT：新引擎 Reyden 与"亚百毫秒"

图源：Databricks 官方博客

Lakehouse//RT 是这场 Summit 工程含量最高的发布。

它的定位是实时数据仓库——直接跑在 Delta / Iceberg 之上，不再需要把数据复制到 Pinot / Druid / ClickHouse / 内部专用引擎。

底层是一颗全新的引擎：Reyden。

关键性能数字（厂商自报口径）：

  • • 12,000 QPS 下亚 100 毫秒延迟
  • • 较专门实时层最高 16x 提速
  • • 小数据集 10ms 响应
  • • TPCH 1TB 测试稳态运行
  • • 复杂 TPCDS 查询（join、subquery、window function）下，竞品大幅放缓或失败的场景，Reyden 仍稳定

5. 治理与安全：把"边界"做成一线产品

过去几年所有大厂都在讲"治理重要"，但真正把治理放到主舞台、放到 Keynote 第二段讲的，本届是 Databricks。

5.1 Unity AI Gateway：从治数据到治 Agent 运行时

图源：Databricks 官方博客

治理对象的扩张是这条产品线最大的变化：

  • • 过去：Unity Catalog 治理"数据资产 + AI 资产（模型、向量索引、特征）"
  • • 现在：Unity AI Gateway 治理"运行时——Model + Agent + MCP + Tool"

它做的事情非常实际：

  • • 策略管控：哪些 Agent 能调哪些工具、哪些数据
  • • 成本控制：按 Agent / 团队 / 项目设 token 预算
  • • 智能路由：同一个语义请求路由到合适的 model（成本/延迟/质量平衡）

8. 产业格局：这场发布会真正改变了什么

抬一下视角。这一节，我们把 18 项发布揉成 5 条产业指引 + 1 张竞争格局表。

8.1 五条产业指引

指引 1 ——"模型决胜论"被"Agent 跑得稳"取代

2026 年的故事，不是"哪个模型 benchmark 最高"，而是"哪个平台能让 Fortune 500 把 Agent 跑到生产"。Genie One 的 84.5% 命中率、Agent Bricks 的 100,000+ Agent、1+ quadrillion tokens——这些数字共同说明，企业 AI 的"评价体系"正在从模型能力分迁移到部署可靠性。

这条指引最直接的受影响方，是那些把全部估值押在"模型代差"上的玩家。

指引 2 ——"上下文图谱"成为新一类企业产品

Genie Ontology 是这次最深远的发布。它把过去几年分散的 Headless BI / Knowledge Graph / Vector Index / Catalog 几条赛道合并为一类新产品：自动构建、持续生长、权限继承的"企业上下文图"。

未来 12 个月，所有平台厂商都会被迫给出自己的"上下文产品"——dbt Labs、Atlan、Collibra、Alation、Cube.dev 这一类公司的边界会被严重压缩。

指引 3 —— OLTP / OLAP / 实时 / 向量 终于走向"合一"

LTAP + Lakebase + Lakehouse//RT + Lakebase Search 的组合拳，意在让客户不再需要在 Snowflake、Aurora、Pinot、ClickHouse、Pinecone 之间拼图。

这是过去十年 Lakehouse 叙事的"最后一公里"——也是 Databricks 这次最值得未来 18 个月密切跟踪的赌注。

指引 4 —— 治理被前置成一线产品

Unity AI Gateway 把 MCP / Tool Call / Prompt Injection / 越狱 / 模型路由 / 成本治理统一到平台层。"Agent 之前先治理"的口径在 Keynote 上被反复强调。

这条指引会强迫整个行业——把过去拼图状态的 Agent Observability / Guardrail / 身份能力，2026 年合并到平台层。AgentOps 这类初创公司的窗口期正在收窄。

指引 5 —— 全面去席位化定价 + Agent token 成本将冲击企业 FinOps

Genie 转 Pay-as-you-go、每用户月送 150 DBU、Free Edition 升级、Lakehouse//RT 引导期 30% 折扣——配合 IPO 节奏（市场盛传 H2 递交 S-1，估值 165~175B / run-rate 5.4B ≈ 32x 营收倍数），这是"低进入价 + 用量回收"做营收曲线的组合拳。

但 Ali Ghodsi 那句"会变得极其昂贵"也在提醒企业：token、tool call、agent action 必须进入 2026 年的 FinOps 模型。CIO 的预算口径要重做。

9. 对开发者与 CIO 意味着什么

讲完格局，回到读者侧——你明天能做什么？

9.1 对开发者：四件事要重新认识

  1. 1. 模型选型不再是核心命题
    2024~2025 工程师常做的"挑模型"工作，会迅速被平台默认配置接管。真正消耗时间的，会是 harness、上下文、Agent 调试。
  2. 2. "上下文工程"成为工程师的新工作面
    Genie Ontology / Glossary / Domains / Metrics——这一整套"上下文产品"是工程师未来的核心战场之一。一个能写好"语义层"、把企业知识沉淀进 Catalog 的工程师，会比一个调 prompt 的工程师值钱。
  3. 3. 数据库技能要扩到"Postgres + 向量 + Iceberg"
    Lakebase Search 把"是不是要单独装一个向量库"这个问题彻底变成存量问题。新项目里，能用 Postgres 做混合检索的，没有必要再引一个独立向量库。
  4. 4. Agent 调试与 FinOps 化
    token、tool call、agent action 进入工程师日常视野。学会读 Unity AI Gateway 的 Trace、看 Agent Bricks 的成本面板——这是 2026 年下半年的"基础工程素养"。

9.2 对 CIO 与企业架构师：5 条务实建议

  1. 1. 优先在内部高价值流程试点 Genie One / Genie Agents
    销售简报、运营复盘、合规问询、数据自助查询——这是最快见效的四个场景。
  2. 2. 如果重度依赖 Tableau / Power BI，开始评估 AI/BI 迁移路径
    Genie Code 已经把"导入旧 Dashboard"工程化。不需要立刻全量迁移，但需要把"迁移的可行性、成本、收益"写进 2026 年下半年的预算文件。
  3. 3. 治理先行
    Unity Catalog + Unity AI Gateway 必须早于大规模 Agent 化；否则 PII / 身份 / 成本会在 6 个月内同时失控。
  4. 4. 重新建模 Token 与 Tool Call 成本
    这次 Summit 的最大隐形信号，是 Ali Ghodsi 那句"会变得极其昂贵"。FinOps 模型必须把 Agent 化的成本科目独立列出——按团队、按 Agent、按 use case 分层。
  5. 5. 盯紧 Lakebase 与 LTAP
    如果 LTAP + Lakebase 真的证伪了"OLTP/OLAP 数据要分库"的旧定律，未来 12~18 个月，企业数据架构会被重画。这是这次发布里对架构师而言最重要的一项观察作业。

9.3 收束：一个运行时，几个未解之问

这是 Databricks 用一场 Summit 给出的产业答案——它把"模型、上下文、数据、治理、应用"全部压进同一个平台，并给它取了个新名字。OpenAI 的 Brockman 与 Microsoft 的 Nadella 都来了现场，这件事本身就在说明，2026 年的企业 AI 不再缺模型，缺的是把 Agent 真正交给一个能负责的运行时。

至于这个运行时是不是必须叫 Databricks——还有一些问题没有答案：

  • • Lakehouse//RT 的 16x 是不是只在 Databricks 选过的 benchmark 上成立？
  • • Lakebase 的 1,200 万次/日启动，是不是包含了大量短生命周期的 dev/test 实例？
  • • Genie Ontology 的 84.5%，在客户自己的真实业务问题集上是否还能保持？
  • • Agent 化带来的 token 通胀，会不会让企业最后选择 self-host 开源模型？
  • • Microsoft Fabric 的反扑，会以什么形态在 Q3/Q4 发生？

这些问题都要等 GA 之后、客户真实生产数据出来，才能给出答案。

但有一件事是确定的——2026 年下半年开始，企业数据与 AI 平台的竞争，主战场不再在"模型 benchmark"，而是在"Agent 能不能上岗"。

而 Databricks 这一周做的事情，是给整个行业上了一堂示范课。

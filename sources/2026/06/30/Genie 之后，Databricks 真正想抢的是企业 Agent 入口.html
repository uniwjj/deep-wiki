Genie 之后，Databricks 真正想抢的是企业 Agent 入口

作者：数智技术落地 发表：2026年6月26日 11:35 来源：微信公众号 下载：2026-06-30

Databricks 今年 Data + AI Summit 的主题很直接：build apps and agents that work。

这句话听起来像会场标语，但放到它最近一组产品动作里看，含义更具体。Databricks 不再满足于把 lakehouse 包装成 AI 数据底座，而是在把业务入口、语义上下文、模型运行、工具调用、实时数据、行业应用和安全治理串到同一套平台里。

它想抢的不是单一模型，也不是一个 BI 问答框，而是企业 Agent 真正运行时需要的控制点：Agent 到底相信什么上下文，能调用哪些工具，按谁的权限行动，怎样接入业务流程，出问题时如何治理和追踪。

这轮发布不是一组 AI 功能，而是一张控制面拼图

Databricks 在 Summit 前后连续放出了几条线。

第一条是面向业务用户的 Genie One、Genie Agents 和 Genie Ontology。Genie 从原来的自然语言分析入口，继续往企业 AI coworker 方向走：可以进入 Slack、Teams 和移动端，可以设置提醒、生成内容、创建可复用 Agent，也可以通过 MCP 或业务系统执行动作。

真正关键的是 Genie Ontology。公开资料里，Databricks 把它描述为从表、查询、仪表盘、pipeline、文档和业务工具中抽取知识的上下文层，还会处理指标定义、业务术语、计算逻辑和资产关系。媒体报道也注意到，Databricks 正在把 Genie 从数据团队工具推向 finance、marketing、sales 和高管场景。

第二条是 Unity AI Gateway。Databricks 把 Unity Catalog 的治理边界，从数据和模型资产扩展到 AI 运行时：模型、Agent、MCP server、skills、工具调用、prompt、response、agent action、非人类身份和成本控制。它还拉进 CrowdStrike、Okta、Palo Alto、Zscaler、Cyera、Openlayer、Obsidian 等安全、身份和可观测性伙伴。

第三条是 AI Platform 和数据工程侧的 Agent 化。Genie Code 被放进 ML 工程、数据工程、分析工作流里，结合 MLflow、Unity Catalog、Feature Store、Model Serving、Lakeflow Jobs 和 Databricks Asset Bundles。这里的信号是：Databricks 不只想让业务用户问数，也想让 Agent 进入开发、训练、部署、监控和排障流程。

第四条是 CustomerLake。它把 CDP 能力放进 Databricks lakehouse，强调 Customer 360、identity resolution、audience building、campaign automation、activation 和 personalization 都可以在受 Unity Catalog 治理的数据基础上完成。CustomerLake 的 Profile Agents 和 Campaign Agents，本质上是在把“客户数据平台”改写成“营销执行 Agent 的上下文和行动平台”。

这些东西分开看像多个产品发布，放在一起看更像一张控制面拼图：业务用户入口在 Genie，语义和上下文在 Ontology，权限和资产治理在 Unity Catalog，运行时安全和成本在 Unity AI Gateway，工程执行在 Genie Code/Lakeflow/MLflow，行业行动入口在 CustomerLake。

Databricks 想补的短板，是业务入口和运行时治理

Databricks 的原始主场很清楚：Spark、Delta Lake、lakehouse、数据工程、机器学习、MLflow，以及后来围绕 Unity Catalog 做的统一治理。

这些能力足以支撑“企业 AI 需要干净、可治理的数据”这个叙事。但 Agent 时代的问题不止是数据在哪里。更难的是业务用户是否愿意在 Databricks 里工作，Agent 能否理解企业内部相互冲突的指标定义，工具调用是否可控，AI 成本是否可管，执行动作是否能被追踪。

所以 Genie One 的价值不在于多了一个聊天入口，而在于 Databricks 终于开始正面争业务工作入口。过去 lakehouse 的主要用户是数据工程师、数据科学家和分析师；Genie 想进入销售、市场、财务、高管和一线业务流程。这是 Databricks 必须补的一块短板。

Unity AI Gateway 则补另一块短板：Agent 运行时治理。

传统数据治理主要管表、字段、模型、权限、血缘和审计。Agent 一旦开始调用工具，风险会移动到运行时：prompt injection、数据泄露、危险工具调用、MCP tool poisoning、非人类账户权限扩张、模型调用成本失控。Databricks 这次把安全、身份、可观测性伙伴拉进 Unity AI Gateway，说明它已经不把治理只理解成 catalog，而是想把治理推进到 Agent 执行动作发生的地方。

Axios 关于 Unity AI Gateway 成本控制的报道也说明了这一点：Agent 越自主，AI 支出越不可预测，企业需要在跨模型、跨 provider 的使用上设限和监控。成本控制听起来偏 FinOps，但对 Agent 落地很关键。没有成本边界，很多自动化工作流很难放进生产。

Genie Ontology 是这轮最值得跟踪的产品假设

Databricks 这轮最重要的概念不是 Genie One，而是 Genie Ontology。

原因很简单：如果没有稳定的企业上下文，Agent 入口越多，错误传播越快。业务用户问“毛利为什么变了”，背后需要的不只是 SQL 生成，而是指标口径、时间口径、组织维度、认证 dashboard、数据质量、业务解释、权限边界和可追溯来源。

Databricks 的思路是把这些上下文自动从企业已有资产里抽出来：表、查询、仪表盘、pipeline、文档、业务工具、认证资产、使用频率、作者权威和新鲜度。它甚至强调 authority ranking，用类似图排序的方式处理企业内部多个定义之间的可信度。

这条路线很有野心，也最需要验证。

它的优势是 Databricks 本身掌握数据工程、数据资产、MLflow、Unity Catalog 和 AI/BI 入口，可以看到很多上下文来源。它比单纯 BI 工具更接近数据生产链路，也比单纯治理工具更接近业务消费入口。

但风险也明显。企业里真正麻烦的业务语义，很多并不天然存在于表、SQL、dashboard 和文档里，而是在流程、组织、审批、Excel、邮件、会议和行业规则里。自动抽取可以加速起步，但能否长期维护一个可信 ontology，仍然要看企业愿不愿意把业务定义、认证资产、owner、质量规则和变更流程持续纳入 Databricks。

所以对 Genie Ontology 的判断要降一档：它已经清楚表明 Databricks 要抢业务语义入口，但还不能证明它已经解决了企业语义治理。真正要看的是后续有没有更多产品文档、API 边界、客户案例和跨系统治理实践。

CustomerLake 说明 Databricks 想从平台走进业务流程

CustomerLake 是这轮发布里很容易被低估的一条线。

表面上，它是在进入 CDP 市场；更深一层，它是在验证 Databricks 的 Agent 路线能不能落到具体业务流程里。

Databricks 对 CustomerLake 的表述很明确：传统 CDP 坐在核心 Data and AI platform 外面，会造成新的集成、治理和数据复制问题；agentic marketing 需要在同一个地方拿到 customer identity、predictive models、business logic、activation endpoints 和 real-time performance signals。CustomerLake 于是把 Customer 360、身份解析、受众构建、活动自动化、激活和个性化放回 lakehouse。

这背后是一个更大的判断：Agent 不会永远停在“回答问题”，它最终要进入营销、供应链、客服、财务、风控等业务流程。Databricks 选择先打 CDP，是因为客户数据天然跨系统、强治理、强实时、强业务动作，也最容易暴露“数据平台离业务行动太远”的问题。

Profile Agents 和 Campaign Agents 的意义就在这里。前者把原始客户数据整理成可信 Customer 360，后者把受众构建、下一步行动、跨渠道激活和持续优化包装成 Agent 工作流。它不只是把营销工具接进 lakehouse，而是把湖仓里的数据、模型、权限和业务行动放进一个闭环。

这也解释了为什么 CustomerLake 要强调开放 martech/adtech 生态、Lakehouse Federation 和 Unity Catalog。Databricks 不可能替代所有营销工具，但它可以把自己放在客户上下文、模型和激活动作的中间层。

与 Snowflake 的差别：一个从应用切入，一个从上下文切入

Databricks 和 Snowflake 都在讲企业 Agent，也都想把业务 Agent 拉到自己的数据平台上。但它们的切入口不一样。

Snowflake 更像从“业务应用怎么跑在数据云上”切入。它围绕 Data Cloud、Horizon、Cortex、应用开发、行业伙伴和 Accenture Context Graph，讲的是应用、行业方案和合作伙伴能力如何在 Data Cloud 上组合起来，进入企业业务流程。

Databricks 更像从“Agent 怎么可信地使用数据平台里的上下文和工具”切入。它的起点仍然是 lakehouse、Unity Catalog、MLflow、Lakeflow、Feature Store 和开放数据生态，然后用 Genie/Ontology 让 Agent 读懂业务语义，用 Unity AI Gateway 管工具调用和运行风险，用 CustomerLake 验证这些能力能不能进入营销和客户运营流程。

如果站在企业用户的选型视角，差别会更清楚。

如果问题是“我们已经有数据云，想更快做业务应用、行业方案和流程 Agent”，Snowflake 的路线更顺。它更适合从应用开发、行业知识、合作伙伴交付和业务流程自动化开始，把数据云变成业务应用的承载环境。

如果问题是“我们要让 Agent 可靠地理解数据、继承权限、调用模型，并进入数据工程和 ML 工程流程”，Databricks 的路线更顺。它更适合从 lakehouse、catalog、ontology、pipeline、feature、model、job 这些数据与 AI 工程对象开始，把这些上下文和工具推到业务用户面前。

所以区别不是“业务应用在谁的平台上”。两家公司都想让业务应用靠近自己的数据平台。真正区别是：Snowflake 更像从业务应用落地倒推数据云能力；Databricks 更像从数据与 AI 工程能力上推到业务入口。

这也是它们竞争会越来越直接的原因。以前 Databricks 和 Snowflake 可以分别被理解成 lakehouse vs cloud data warehouse。现在两家公司都在向上走：谁能让 Agent 读懂企业上下文、遵守权限、调用工具、进入业务流程，谁就更接近下一代数据平台入口。

与 OpenMetadata、Collibra 的差别：Databricks 抢的是运行闭环

OpenMetadata、Collibra、DataHub 这类治理平台也在抢 Agent 上下文，但它们的位置和 Databricks 不同。

OpenMetadata 1.13 更像是在把 glossary、ontology、RDF graph、hybrid search、data marketplace 和 MCP Services 组织成跨系统治理上下文层。Collibra 则强调中立企业本体和跨平台治理，防止上下文被云厂商吸进各自平台。

Databricks 的强项不是中立，而是闭环。它可以把数据生产、模型开发、业务问答、工具调用、运行时安全、成本控制、行业 Agent 和客户数据激活放在一个平台叙事里。它不一定适合成为所有企业的中立语义中枢，但它会努力证明：如果数据、模型、语义、权限和执行都在 Databricks 里，Agent 落地会更快、更可控。

这也是国内平台要注意的地方。治理平台如果只做目录，会被运行时平台绕开；运行时平台如果没有治理上下文，又会被安全和合规挡住。Agent 时代的控制点不在单一模块，而在“上下文能不能进入执行闭环”。

对国内数据平台的参考：别只做问数，要补三层能力

Databricks 这轮发布对国内数据平台的参考，不是“也做一个 Genie”。

更具体地说，国内平台如果想支撑 Agent 落地，至少要补三层能力。

第一层是业务上下文产品化。指标、术语、owner、认证报表、血缘、质量、权限、文档和业务规则，不能只散在几个页面里。它们要能被 Agent 稳定读取、排序、解释和追溯。否则问数入口越智能，越容易把错误口径放大。

第二层是工具调用治理。OpenAPI、MCP、SDK、CLI 只是入口。真正难的是 Agent 身份、用户代入权限、工具调用日志、审批、成本控制、失败回滚和安全拦截。Databricks 把 Unity AI Gateway 放到运行时，就是因为生产 Agent 的风险发生在执行阶段，而不是 PPT 里的架构图里。

第三层是数据平台自身的 Agent 化。数据开发、指标开发、质量检查、模型训练、特征工程、运维排障和补数，都是比“业务用户问一句话”更容易落地的场景。平台要把 job、pipeline、bundle、run history、log、lineage、quality result 和 deployment 这些对象做成连续工具面，而不是只开放零散 API。

如果只做一个聊天框，平台很快会被大模型应用层吸走入口。如果只做底层存算，平台又会被业务系统和 Agent workflow 平台绕开。Databricks 这轮动作真正值得看的地方，是它试图同时抓住底层数据、业务语义、运行治理和业务行动。

接下来要验证什么

Databricks 的路线已经很清楚，但还没有全部被证明。

后面最值得看四件事。

第一，Genie Ontology 能不能从自动抽取走向可治理维护。业务语义不是一次抽取就结束的资产，真正难的是变更、冲突、认证、owner 和跨部门协作。

第二，Unity AI Gateway 能不能成为跨模型、跨框架、跨 MCP 工具的真实运行时控制面。它现在有生态伙伴和清晰方向，但生产采用要看策略粒度、审计深度、性能开销和开发者体验。

第三，CustomerLake 能不能证明 lakehouse 内嵌 CDP 比独立 CDP 更高效。营销技术栈很复杂，Databricks 要说服企业把客户上下文、模型和激活动作放回数据平台，需要真实案例和迁移路径。

第四，Genie Code、Lakeflow、MLflow 和 AI Runtime 能不能把数据工程与 ML 工程 Agent 化。企业内部最先落地的 Agent，很可能不是高管问数，而是升级、修复、测试、训练、监控和成本优化。

从当前公开资料看，Databricks 的战略正在从“统一数据和 AI”推进到“统一 Agent 所需的上下文和执行”。这一步如果成立，湖仓就不再只是存算架构，而会变成企业 Agent 的工作入口。

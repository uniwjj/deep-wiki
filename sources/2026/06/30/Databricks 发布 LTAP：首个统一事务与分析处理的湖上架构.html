📌 核心要点

LTAP（Lake Transactional/Analytical Processing）
 是 Databricks 推出的新一代数据处理架构，在存储层统一事务处理（OLTP）、分析处理（OLAP）、流处理与运营数据，实现单一数据副本、零 ETL 管道、统一治理。

🧩 传统架构之痛

过去二十年，企业数据架构几乎是一样的模式：

应用数据库
    ↓ CDC 管道
    ↓ 落地层
    ↓ 转换管道
    ↓ Lakehouse / 数仓
    ↓ 仪表盘、ML 模型、AI Agents
    ↓ 反向 ETL / API / 应用同步

这种架构带来了七个核心问题：

  • 多副本
     — 同一业务实体在多个系统中复制
  • 延迟
     — 应用操作到分析可用之间存在时间差
  • CDC 失败
     — 管道断裂、Schema 漂移频繁发生
  • 对账困难
     — 运营视图与分析视图反复对账
  • 反向 ETL 复杂
     — 需要额外的 API 或同步层
  • 治理碎片化
     — 不同系统有不同安全模型
  • AI 数据陈旧
     — AI Agent 基于过时或不完整的上下文运行

  • 此前业界尝试过两种方案：

  • HTAP
    （混合事务/分析处理）— 试图在同一引擎中运行两种负载，但牺牲了工作负载隔离，两方面性能都打折扣
  • 零 ETL
     — 只是隐藏了 CDC 管道，而非真正消除

  • LTAP 走的是第三条路。

🏗 ️ LTAP 架构解析

核心思想

"LTAP 不是在同一个引擎内运行 OLTP 和 OLAP，而是在存储层统一数据。"
  • Lakebase
     作为事务层——托管型 Postgres 兼容数据库，与 Databricks 平台深度集成
  • Lakehouse
     作为分析层——现有的湖仓一体引擎
  • Unity Catalog
     作为统一治理层——元数据、权限、血缘全链路统一

  • 关键架构特性：

    特性
    说明
    存储层统一
    Lakebase 直接向 Unity Catalog 写入开放格式（Iceberg / Delta）
    消除 CDC
    运营数据即时可查询，无需管道
    工作负载隔离
    事务与分析负载独立扩展，互不影响
    Postgres 原生事务
    在对象存储上运行 Postgres-compatible 事务
    开放标准
    支持任何 Postgres 应用 + 任何 Iceberg/Delta 读取器

    架构对比

    传统模式：

    运营 DB → CDC → Lakehouse → 反向 ETL → 应用 / AI Agent

    LTAP 模式：

    应用 + 运营数据 + 分析 + AI 上下文
        ↓
       共享治理数据基座
        ↓
    分析 / 风险模型 / AI Agent 上下文
        ↓
    人工或应用操作

    新增 Lakebase 能力

    Databricks 同时为 Lakebase 发布了三项重大增强：

  • 跨云、跨地域灾备
     — 为 Agent 驱动的关键业务提供弹性架构
  • Git 风格分支与快照
     — 基于生产数据安全实验，Git 化工作流
  • 自治数据库运维
     — AI Agent 自动监控、检测瓶颈、建议索引、辅助恢复

🔄 实际应用场景

场景一：客户支持与风控审查

传统架构下，同一个客户的数据分散在：

  • 应用数据库（订单）
  • CRM 系统（客服记录）
  • 数据仓库（分析报表）
  • 特征存储（ML 模型）
  • 向量数据库（AI 上下文）
  • 反向 ETL 工具 + 若干 API

  • LTAP 架构下，所有数据在同一共享治理基座中，AI Agent、风控分析师、客服代表看到的是同一份一致的数据。

    场景二：医疗收入周期管理

    "对于我们所服务的医疗系统，收入周期的速度和准确性直接影响其提供护理的能力……Lakebase 和 LTAP 将运营和分析工作负载统一在单一层上，让我们的 RCM-native AI 能够实时访问运营数据。这为医疗服务商带来更强的财务表现，让更多回收收入回流到急诊室、NICU 和其他关键护理服务。"

    >

    — Grant Veazey，Ensemble 公司 CTO

    Ensemble 在 Databricks 上管理 2PB+ 的医疗收入周期数据。

🧠 对数据工程师意味着什么

Databricks 社区专家 Amit Kumar Singh 指出，LTAP 不会消灭数据工程，而是让数据工程决策更加关键。

1. 工作负载边界设计

并非所有数据都需要实时。月末财务对账可以继续异步；实时欺诈检测则受益于 LTAP。数据工程师需要判断：哪些走 LTAP，哪些保持传统路径。

2. 数据契约（Schema Discipline）

当运营和分析共享同一存储层，应用侧的小 Schema 变更会直接影响：

  • 分析报表
  • ML 特征
  • AI Agent 上下文
  • 监管报告

  • Schema 纪律前所未有地重要。

    3. 治理与访问控制

    单一数据基座的价值取决于访问控制是否得当：

  • 谁可以读/写事务数据？
  • 哪些数据暴露给 AI Agent？
  • 敏感字段如何脱敏？
  • 访问如何审计与保留？

  • 4. 数据质量

    源头脏数据仍然是脏数据。

    靠近运营与分析交汇点的质量检查变得更加重要。

    5. AI 操作的人为审批

    当 AI Agent 从"回答问题"进化为"执行操作"：

    可信数据 → 验证上下文 → AI 建议 → 人工审查或策略检查 → 批准操作

🔭 展望

LTAP 并非万能替代方案。企业仍将拥有 SaaS 应用、主机系统、第三方平台、遗留系统等无法迁移的数据源。

"LTAP 消除的是一类仅仅因为运营和分析环境分离而存在的管道。"

对于新建的 AI 应用和数据密集型企业，LTAP 提供了一条摆脱传统 CDC → ETL → 反向 ETL 复杂性的路径。它为下一代 AI Agent 架构提供了真正统一的数据基座。

Ali Ghodsi 在发布会上的结语或许最能概括这一时刻：

"过去十年，我们将现代数据栈的主要工作负载统一到了一个开放基础之上……现在，LTAP 消除了最后一道瓶颈。"

>

— Ali Ghodsi，Databricks 联合创始人兼 CEO

# Anthropic 用 Claude 做自助数据分析的四层体系

---

## 一句话说清楚

我们用 Claude 把公司里 95% 的"帮我看个数据"类问题自动化了，整体准确率约 95%——这是两个独立的 95%，不要混为一谈（一个是"覆盖了多少问题"，一个是"答对的比例"）。核心发现是：**让 AI 做数据分析，难点不在写 SQL，而在"你说的那个数到底是什么数"——这是个数据歧义问题，不是代码生成问题。**

---

## 背景：传统自助分析为什么做得那么累

做数据的都知道，业务方天天来问数，数据团队疲于应付。以前有两种解法：

1. **把表做宽、做扁**，让不会 SQL 的人也能拖拽——结果是一大堆定义打架的视图，指标越搞越乱。
2. **给每个团队圈一块独立分析环境**——结果是看板膨胀、重复建设，长尾问题照样覆盖不了。

LLM 出来后大家就想了：让 AI 直接查数仓，不就不用搞那些了吗？

然后你就发现：**刚爽了两天就翻车了。** AI 确实给了答案，但没有上下文的外行用户根本判断不了对不对。以前数据团队还会把关，现在 AI 直接把答案端到了不懂数据的人面前——错了都没人知道。

---

## 核心矛盾：数据分析 ≠ 写代码

| | 写代码 | 查数据 |
|---|---|---|
| 答案 | 开放，多种写法都对 | 只一个正确答案 |
| 幻觉的防御 | 文档、测试天然兜底 | 没有确定性方式证明对错 |
| 难点 | 逻辑设计 | **数据歧义** |

用大白话说：让 AI 写代码，写得对不对跑一下测试就知道了。让 AI 查数据，它告诉你"上个月收入 3000 万"，你怎么验证？你甚至不知道它用的是哪张表、哪个口径。

所以整个问题的本质是：**能不能把用户嘴里的"我想看一下 XX 数"，精准映射到数仓里那张对的表、那个对的字段、那个对的口径？能映射过去，后面写 SQL 根本不是事。**

---

## 三类最常见的翻车场景

| 翻车类型 | 大白话解释 | 举例 |
|---|---|---|
| **概念对不上实体** | 你说"活跃用户"，数仓里有一百万个字段，AI 不知道用哪个 | 活跃怎么定义？算不算刷量的？回看几天？ |
| **知识过时** | 数仓天天在变，表改了、口径调了，AI 还在用老黄历 | 上个月那条 SQL 能跑对，这个月不行了——字段改名了 |
| **就是搜不到** | 答案明明在数仓里，元数据也写了，但 AI 就是找不到 | 搜索空间太大，检索跑偏了 |

这三类错误占了绝大部分不准确回答。下面的四层架构就是逐个击破它们的。

---

## 四层架构总览

```
验证层    → 验证答案对不对，揪出还在漏的环节
Skills层  → 规定 AI 干活的步骤和歧义处理
真相源层  → 告诉 AI 每个指标去哪查、用哪张表
数据基础层 → 理清数仓，一个概念只留一个权威来源
```

**从下往上一层层叠上去，下面是上面的地基。** 每一层主要对付前面说的某一类翻车：

- **数据基础层**——把数仓本身理清楚。一个概念只留一个权威来源（"收入"只能有一张权威表），靠 CI 强制治理、所有东西放一个仓库、元数据当一等公民。**主要对付"概念对不上实体"，也是防知识过时的第一道防线。**
- **真相源层**——AI 查数仓时的"导航地图"，规定它按什么信任顺序去找指标：语义层 ＞ 血缘 ＞ 历史查询语料 ＞ 业务上下文。**主要对付"概念对不上实体"。**
- **Skills 层**——AI 的操作手册，规定先查哪个源、遇到歧义怎么办、一份合格分析长什么样。**主要对付"就是搜不到"。**
- **验证层**——前三层搭好了，这层负责搞清楚到底有多准、哪里还在漏，并在它腐化时第一时间发现。靠离线评估、消融实验、在线验证三招。**没有它前三层再漂亮也只是"看起来对"。**

> **先分清一个容易混的点**：数据基础层和真相源层不是"数仓建设 vs 别的建设"的并列关系，而是"**建资产 vs 定优先级**"的关系——
> - **数据基础层 = 建资产**：语义层、血缘图、参考文档、元数据这些资产，都是在这一层建出来并治理的（跟建模代码同仓、受 CI 保护、定义由人拍板）。这不是专门为 AI 建的，而是传统数据工程治理按"用户从数据专家变成了 AI"的新标准来要求。
> - **真相源层 = 定优先级**：**这一层一个资产都不建**，只规定 AI 查询时按什么信任顺序去消费数据基础层已经建好的资产。
>
> 所以同一个语义层会**横跨三层、各管一摊**：数据基础层**建它**、真相源层**把它排为查询第一优先级**、Skills 层**把"必须先走语义层"写成 AI 必须遵守的指令**。

每一层的详细做法见下面"第一层~第四层"分节。

---

### 第一层：数据基础——先把家里的东西理清楚

最关键的认知转变：**以前你的数仓用户是数据专家，现在你的数仓用户是 AI Agent，而 AI 的答案直接给到不懂数据的人。那个人没法帮你验证对不对。**

所以数据基础的核心就是：**一个概念只对应一个权威答案。** 比如"收入"这个词，如果数仓里只有一个治理过的数据集，AI 就不可能选错。如果有四十个候选，那一定出错。

几条行之有效的做法：

**① 搞权威数据集（Canonical Datasets）**

最常见的翻车就是 AI 把一个概念映射到了错的表、错的字段、错的口径。解法很简单：**做减法。** 每个指标只保留一个权威来源，其他的近似重复全部干掉。汇总表、缓存层从权威模型派生出来，不许独立存在。

**② 用工具强制执行，别靠自觉**

规矩写在纸上没用，必须靠 CI 卡口。谁敢绕过治理层直接建模型，PR 直接打回。下游团队要么在治理层上建，要么解释为什么不行。没有强执行的治理撑不过一周。

**③ 所有东西放一个仓库**

建模代码、语义层定义、参考文档、看板定义——全放一个仓库。CI 跨层检查：你改了一个模型，下游看板会不会挂？指标口径会不会破？挂了就在同一个 PR 里修好再合。

**④ 元数据按一等公民对待**

代码 Agent 好用的前提是代码库有 README、类型标注、docstring。数仓也一样——列说明、表说明、指标口径、粒度文档、值域、血缘、负责人，这些不是锦上添花，是帮助 AI 选对表的关键上下文。

---

### 第二层：真相源——AI 去哪找答案

数据基础是数仓本身，真相源是 AI 查数仓时看的"导航地图"。按信任度从高到低：

**① 语义层（最优先，强制使用）**

编译好的指标和维度定义。用户问"周活多少"，AI 调个函数直接出数，跟公司 BI 工具出的数一模一样。我们**在 Skill 里硬性要求 AI 必须先走语义层**。试过一个方案：让 LLM 自动从原始表生成指标定义——翻车了。生成的定义看起来挺像回事，但把我们想消除的歧义又编码回去了。评测结果是净负面。结论：**让 Claude 帮你写文档，但定义必须人拍板。**

**② 血缘和转换图（语义层没覆盖时的备选）**

语义层没这个指标怎么办？看血缘和表排名（table ranking）。这张表从哪个上游模型来的、哪些表已经弃用了、哪些表粒度一样、几张候选表里优先信哪张——AI 可以把"我不知道这个指标"变成"我知道从哪个治理模型聚合出来"。血缘也是后面新鲜度信号和溯源信号的基础。

**③ 历史查询语料（别直接喂给 AI，没用）**

直觉上历史 SQL 是最好的参考——以前答对的都在里面。实际呢？**让 AI 直接搜几千条历史 SQL，准确率提升不到 1%。** 我们专门做了消融实验验证这一点（后面会细说）。非结构化检索没法把新问题映射到正确先例。**有效的做法是把语料蒸馏成结构化的参考文档和分析模式**，写在 Skill 里。

**④ 业务上下文（最被低估的一层）**

AI 不懂你的业务，它会回答你**问了什么**，而不是你**想知道的到底是什么**。它不知道"Q2 那次发布"指的是哪个产品，不知道两个团队对同一个词定义不同，不知道这个问题是因为周四要开董事会在准备材料。我们把公司知识图谱接进 Agent——文档、路线图、决策记录、组织架构——这样 AI 能消歧引用、提出更好的澄清问题。

---

### 第三层：Skills——AI 的操作手册

知识 Skill 里装的是"是什么"（声明性知识：表是什么、怎么用、坑在哪），分析手册 Skill 里装的是"怎么做"（程序性知识：按什么顺序查哪些源、遇到歧义怎么处理、一份合格分析长什么样）。两者配成一对，覆盖 AI 干活的全流程。

在 Claude Code 里，Skill 就是一个 AI 按需读取的 Markdown 文件夹——**Skill 文件本身可以很短，真正的内容是它指向的参考文档，按需加载、不一次性塞进上下文**。

**效果有多猛？** 没有 Skill 的时候，Claude 分析准确率不超过 21%。加上 Skill 之后，整体 95%+，某些领域稳定 99%。

几条核心做法：

**① 成对 Skill：一个路由 + 一本手册**

- **知识 Skill（薄，管"去哪找"）**：本身是个很短的索引文件，不含具体表/字段知识，只装"先去哪、再去哪"的指引——"先走语义层，没覆盖就读 `references/` 下这 ~30 个参考文件"。**关键是按需加载**：这 ~30 个文件不一次性塞进上下文（会爆），AI 真用到某张表时才去读对应文件。效果是把搜索空间从百万字段缩到几十个精选文件。
- **分析手册 Skill（厚，管"怎么干"）**：编码资深分析师的工作流：澄清问题 → 定位源 → 跑查询 → 送对抗审查 → 带溯源输出。打包了十几个可复用模式（留存曲线、比率拆解、漏斗分析），别每次都重新发明轮子。

**② 参考文档的写法跟给人看的不一样**（这就是上面知识 Skill 指向的那 ~30 个文件，也是"描述数据模型的文档"的主体）

给人看的文档可以写得松散一点，给 AI 看的必须精确。我们写法是：
- 表粒度（一行代表什么）
- 范围/排除项（什么数据进去什么不进去）
- 常见陷阱（比如"去掉免费邮箱域名，但保留公司自有域名"）
- **显式路由触发条件**（"如果是实验提升度的问题……不要用这张表做原始事件计数"）

不写固定配方，因为容易过时——而这恰恰引出下一条：参考文档描述的表天天在变，文档必须跟着维护。

**③ Skill 维护不跟上，一个月准确率从 95% 掉到 65%**

Skill 里存的是参考文档，参考文档描述的是数仓里的表——而表天天在变（改名、改口径、改过滤规则），文档不跟着改，AI 就会按旧文档查新表，几周就错。我们眼看着离线准确率从发布时的 ~95% 在一个月内漂移到 ~65%，才把它当工程问题对待。

解法：Skill 文件（含参考文档）和转换模型放同一个仓库，**改表的 PR 必须同时更新对应的参考文档**，CI 钩子会标记任何"改了报表模型却没碰 Skill 文件"的 PR——强制提醒"你是不是忘了更新文档"。现在约 90% 的数据模型 PR 都包含 Skill 变更。

模型（AI 模型，指 Claude）能力变强了怎么办？定期清理 Skill 里已经不需要的脚手架。

**④ 一个 Skill，到处输出一致**

同一个 Skill 在 Slack、IDE、看板工具、独立 Agent 会话里，问同样的问题必须给出同样的答案。做法是 Skill 只存一个权威版本，合代码时自动同步到插件市场、云存储、MCP 资源。

### 第四层：验证——你怎么知道这套东西真的管用

#### 离线评估

很多团队搭了花哨的分析环境，但连 Agent 到底有多准都不知道。

我们做两类离线评估：
- **看板评估**：Claude 自动生成最常见问题，人验证答案。
- **长尾评估**：喂给 Claude 业务上下文，让它生成领域里的其他合理问题。
- **持续采集**：每次业务方在群里纠正 Agent，那次纠正就是一条候选评估用例。

关键细节：
- 评估要锚定快照日期，不然数据一变，答案本身就变了，没法判断对错
- 结果当遥测数据存数仓（Skill 版本、git SHA、模型 ID、通过/失败、Token 数、耗时），随时能查
- 按领域设上线门槛（我们起初用 ~90%），通过后才能给业务方用
- 评估数量按业务复杂度校准：通过追踪"离线准确率多大程度能预测线上准确率"来定，每个主题几十条之后边际递减，且每出一代新模型这个上限还会降
- 离线准确率要接近 100%——这不代表线上不出错，但说明没有明显漏洞

#### 消融实验——最值钱的是那几次"没用"的发现

每个架构决策（暴露哪些数据源、要不要用子 Agent、两个 Skill 要不要合并）都通过固定评估集做对比，只变一个变量。

最重要的发现恰恰是负面结果：

**① 直接给 AI 喂历史 SQL 几乎没用——这一条最值钱。** 我们让 AI 可以 grep 所有看板/转换/笔记本的 SQL（几千个文件），还确认它确实读了，结果准确率变化不到 1%。

但我们没停在"没用"这个结论上，而是怀疑是不是有别的因素干扰（混淆变量），于是往下查了两步：

- **第一步：答错的题，正确答案到底在不在那堆 SQL 里？** 查下来约 80% 的题，正确答案明明就在语料里。这一步推翻了一个直觉解释——本来你可能会想"准确率没提升，是因为历史 SQL 里根本没这些问题的答案，AI 查不到当然没用"，但数据显示答案在、AI 也确实读了，可它还是答错。
- **第二步：那答案在的题，AI 是不是更容易答对？** 也没有。原本答错的题，就算正确答案在它读的文件里，基本也没"翻"成答对。换句话说，把正确答案放进它读的文件里，该错还是错。

两步合起来推出结论：**瓶颈不在"AI 能不能看到正确信息"（可及性），而在"AI 能不能把眼前的问题映射到那条正确的 SQL、并判断它的口径适不适用"（结构）。**

举个具体场景帮助理解（**以下为本文举例说明，非原文案例**）：数仓里几千个历史 SQL 中，有一条查"月活"的，限定了 event_type、排除了刷量用户。业务方问 AI"上个月活跃用户多少"，AI grep 到了这条 SQL——信息它接触到了。但它**没法判断这条 SQL 里的"活跃"口径是不是就是业务方此刻想要的**（包不包括只登录没操作的？排不排除刷量？"上个月"是自然月还是过去 30 天？）。它可能没意识到这就是答案、转头自己写了一条口径不对的；也可能读到了另一条长得差不多但口径不同的 SQL，以为是对的，照抄了。**问题不在"看没看到"，在"看到了能不能认出这就是对的那条、能不能正确理解和复用它的口径"。**

那 5000 条历史 SQL 是一堆没组织过的文件（**"没有标签告诉你哪条是权威定义"——此处为推测说明，原文只说"非结构化检索没法把新问题映射到正确先例"**）。所以——喂更多文件没用（可及性已经满了），把那堆非结构化的 SQL **蒸馏成结构化的领域参考文档**（一个概念一份权威定义，写明口径、过滤、和别名的区别）才有用，因为参考文档解决的就是"映射"。这正是前面 Skills 层那 ~30 个参考文件的由来。

这个结论改了我们几个月的方向。

**② 在 PR 粒度做消融。** 每次有意义的 Skill 改动，都在相关评估切片上跑前后对比，把 delta（变化量）写进 PR 描述。这能让"我以为改好了文档"这种说法变得诚实——专治"好心办坏事"：一个本意是改进的文档改动，实际上可能让准确率下降，前后对比一跑就露馅。

**③ 维护一份"什么没用"的清单。** 消融实验的精神不光是记"什么有用"，更要记"什么没用"——写一行字成本很低，但能挡住团队里下一个人（或几个月后的你自己）把同样的实验再跑一遍，白白浪费时间。我们清单上有两条：

- **文档优化超过一定点会越改越差。** 我们想提升参考文档质量，于是一轮一轮追加优化、加内容。结果连着三轮，每轮都是净负面（评估准确率不升反降），具体表现是"文档变长了，但没变好"。记下来后，再有人觉得"这文档还能补补"，一看清单就知道别补。（原文未解释为什么越改越差，只记录了这个负面结果。）
- **用便宜模型替换对抗审查的"杠精"两头不讨好。** 在线验证里有一步对抗审查（见下一节）——用一个 Claude Skill 当"杠精"挑战答案，准确率 +6%，但代价是 Token +32%、延迟 +72%。为了省这个成本，我们试着把"杠精"换成一个更便宜的模型。结果两头条都落空：准确率丢了大部分原本 +6% 的增益，速度也没快多少。记下来后，再有人想"对抗审查太贵了换个便宜模型吧"，一看清单就知道换过、别折腾。（原文未解释为什么便宜模型两头不讨好，只记录了这个负面结果。）

#### 在线验证

- **对抗审查**：每次出答案前，用一个 Claude Skill 当"杠精"，挑战所有假设。准确率 +6%，代价是 Token +32%、延迟 +72%。
- **来源标注**：每个回复末尾附一段固定格式的来源信息——来源层级（语义层/治理引用/原始表）、数据新鲜度、模型负责人。不提高准确率，但帮接收者判断可信度。
- **数据质量检查**：确保 AI 引用的字段是最新的、完整的、无异常的——原文的说法是"AI 可能用对了字段、用对了方式，但数据本身是错的"，加基本检查确保引用字段 up-to-date、complete、no anomalies。**注意：这其实是数仓本身该保证的数据质量，跟"用 Claude 做 agentic 分析"这个主题关系不大，原文也称它"基本的数据卫生"——放在这里属于顺带提一句，不是 agentic 分析特有的机制。**
- **被动监控**：两个核心指标——多少查询走了语义层、多少回复包含纠正性语言（"表用错了""漏了欺诈过滤"）。
- **主动修正闭环**：定时 Agent 扫群里的纠正语言，自动起草文档修正、开 PR、@领域负责人。路径很朴实——改 Markdown → 合 → 自动同步——负责人花不了什么时间。这些修正还会反哺到离线评估集。

**尚未完全解决的问题：静默失败。** 答案错了但看起来合理，没人质疑，就这么用下去了。缓解手段有三个：来源标注、面向高层报告的人工签批、每个领域 Top KPI 每天对照权威看板做 sanity check——但原文明确说没有根本解法。

## 起步建议

从零开始的团队，**几个权威数据集 + 几十个离线评估用例 + 一个薄的知识 Skill，就能抓住大部分收益。** 本文其余内容是这三样建好之后才需要叠加的。

动手前先对齐几个问题：

- **你是更在意今天的准确率，还是未来的？** 现在多搭基础设施，可能半年后模型能力上去、这些设施就白搭了。如果你风险容忍度高，可以等模型自然补齐短板。
- **你的业务复杂度会怎么变？** 数据量不大、用的人不多、模型简单——很多流程可能是过度设计。
- **看答案的人有多懂数据？** 受众是数据科学家，错了他们能发现；受众是完全不懂数据的管理层，容忍度必须低。
- **你愿意为准确率付多少代价？** 对抗审查准是更准，但更慢更贵。
- **你敢让 AI 看多少数据？** 上下文越多 AI 越强，但跟大多数公司的数据治理姿态冲突。

不管走哪条路，我们最大的收益都来自**逐个解决那三类失败模式：把歧义压缩成一个权威答案、让答案容易被找到、任何一个环节腐化了能第一时间发现。**

# 语义层（必须先走这一步）

治理后的语义层是每个数据问题的**强制默认路径**——和 [BI 工具] 出同一个数，
关联/粒度/过滤器已内置。通过下方参考文档写原始 SQL 只是**兜底**，
只在确认语义层覆盖不了时才用。

## 必走流程
1. **加载** — [各运行环境下怎么加载语义层，含回退]
2. **发现** — 按关键词搜度量/维度；**必须检查分段（segments）**——
   分段是命名好的权威人群过滤器，手写 WHERE 子句替代分段是最高频的错误模式
3. **编译 + 执行** — 构建查询规格 → 编译成 SQL → 跑数
4. **兜底** — 只在两种情况下走原始 SQL：(a) 语义层搜不到相关指标 (b) 编译失败
   → 通过 `references/*.md` 走原始 SQL（见下方第三部分）

> - "需要自定义日期过滤/队列" → 时间维度规格已覆盖
> - "需要关联别的表" → 指标层已经封装了关联逻辑

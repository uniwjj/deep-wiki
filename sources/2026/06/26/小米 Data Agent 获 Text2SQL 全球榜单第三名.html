小米DataAgent问答效果全球第三

1.1 Text-to-SQL 榜单BIRD 介绍

BIRD（ Big Bench for Large-scale Database Grounded Text-to-SQL Evaluation）是目前Text2SQL领域公认最难、最权威的评测基准平台 ，被称为Text2SQL 领域的 “ 奥林匹克” ，是国内外企业Data Agent Harness 能力角逐的核心战场。

BIRD 采用 EX（ Execution Accuracy）作为衡量指标之一： SQL 写法不重要 ，执行结果必须和标准答案一致 ，摒弃语句格式约束 ，评测规则与企业落地场景高度对齐； 同时 ，BIRD 数据集全部取自真实业务数据库 ，内嵌脏数据、行业专属业务常识 ，高度还原企业真实数据分析痛点 ，这也是其成为业界通用标准的关键原因。

有了ClaudeCode/ Codex/ 龙虾，为何还需要DataAgent Harness？

Data Agent Harness 的核心在于语义增强 ，也就是业界所公认的NL2Semantic2SQL ，包括OpenAI/Snowflake 等官方博文在讨论Data Agent 时 ，重点就是语义层的建设。

语义层的建设为何如此重要？

2.1 大模型直接连数据库行不行？ BIRD 榜单给出了答案

从BIRD 榜单中可以看到：

  • Claude Opus 4.6 直接提交BIRD → 排名 30-40 名

  • GPT-5.5 直接提交BIRD → 排名同样在 30-40 名

  • 腾讯Data Agent（ Model + Harness）→ 全球第五

  • 小米Data Agent（ Model + Harness） ：DiMi → 全球第三

最顶级的基座模型，裸打榜只在30-40 名。排名的差异，不在模型，在语义层的Harness

这就引出了一个普遍问题：大模型这么强 ，能不能直接连上数据库 ，让它帮我分析数据？我们的经验时： Demo 或简单分析完全可以 ，规模化生产一定不行。

2.2 瓶颈一： 查数幻觉 ，模型不认识你的业务世界

大模型的训练数据里没有你的公司、你的业务规则、你的字段命名。这导致一类系统性的失败：缺失背景知识

  "小米集团有多少人？ "

模型不知道你的这张表存的是小米在职数据 ，会用通用知识瞎猜 ，或者把字段里碰巧含"小米 "的记录全部拉出来。

  "ITP 这个部门今年的预算有多少？ "

模型不知道ITP = 基础技术平台部 ，直接在SQL 里写  WHERE department = 'ITP ' ，查不到任何结果。

缺失业务规则

  "上周的漏损订单量是多少？ "

这道题有严格的业务规则：要用放款完成日期筛时间 ，且交付日期IS NOT NULL、 当前机构是否自营 = 0。这些规则只存在于业务文档里 ，大模型只能靠猜 ，每次猜的不一样 ，指标口径永远对不上。

这不是模型不够聪明 ，是业务知识无法靠训练学到。企业数据里的状态码、缩写、数据质量问题 ，大模型不知道。

2.3 瓶颈二： 深度分析不是代价高 ，就是完全错误

"有了 LLM + Text2SQL 就能做根因分析" ：这是数据分析 Agent 领域最常见的误区 ，可能错了都不知道怎么错的 ，也是体现Harness 沉淀的重要。

加法类指标：能做 ，但代价极高

  "导致 12 月净收入波动的四级团队有哪些？ "

没有专用工具时 ，模型被迫把这个问题拆成N 个SQL 步骤：查12 月各团队收入→ 查 11 月各团队收⼊→ 在上下文里逐行心算差值→ 排序输出。调用次数从1 次骤增到N 次 ，响应时间增加3 倍以上，累积错误率急剧上升。

有根因分析工具时： 1 次调用，底层引擎完成全部计算。

除法类指标：完全无法正确完成

  "导致 12 月利润率（利润/收入）波动的四级团队有哪些 ，贡献度是多少？ "

⚠实测结果：针对除法逻辑指标，若无科学严谨的工具，大模型完全无法完成任务，且会输出一个看起来合理实则严重错误的答案。

模型的做法是：分别查各团队自身的利润率变化→ 用减法算差值→ 把" 自身变化幅度最大 "的团队认定为主因。

这是根本性的逻辑错误。一个极小规模的团队 ，即使利润率从-97% 变为+129% ，对公司整体利润率的贡献也微乎其微。在商业分析里 ，" 自身变化幅度"和"对整体的贡献度"是两个完全不同的概念。

正确答案需要替代法/杜邦分解算法 ，精准计算每个团队对整体17.44% 涨幅的真实贡献度——这类算法大模型靠Text2SQL 是凑不出来的。

海量维度下钻：上下文溢出

  "分析 11 月整体毛利率下降 ，具体下钻到哪些SKU 导致的（ 10 万+ SKU） "

10 万行数据不可能全塞给大模型。模型的应对方式是强加  LIMIT 50 ___分析的是冰山一角 ，彻底遗漏长尾商品引发的根因。

2.4 瓶颈三：表越多 ，关联越乱

真实业务场景：几百张表 ，字段命名各异 ，没有物理外键 ，没有ER 图 ，关系在DBA 脑子里。

选表错误：两张表都与"离职"相关 ，哪张是准确数据 ，区别只有数据owner 知道。

关联错误：A 表的user_id存的是ID_123，B 表的user_id存的是123（无前缀）。模型直接  A.user_id =B.user_id，Join 失败，0条数据，且无报错——静默地给了错误结果。

数据膨胀：一对多关系未去重 ，指标数值翻倍 ，营销预算基于错误数据做决策。

这类问题 ，靠优化提⽰词无法根治 ，需要结构化的关系模型来约束SQL 的生成边界。

2.5 瓶颈四：效果不稳 ，全靠人盯 ，成本高

SQL 执行成功≠ 答案正确。语义是否正确依赖业务知识 ，只有人能判断。这造成了三个生产级难题：

问题
现实
线上准确率怎么量化？
只能人工标注，成本大，无法规模化
模型升级后会不会退化？
无自动回归测试，只能人工逐一验证
发现badcase 怎么修？
依赖研发手工调优Prompt，迭代极慢

没有评测一回归一迭代的闭环 ，上了生产就是玄学驱动开发——今天跑通的 query，明天换个措辞可能就失败 ，没有人知道为什么。

2.6 瓶颈五：权限是合规红线
龙虾和ClaudeCode 的定位是个人效率工具。但企业数据分析面对的是：
  • 表级隔离：普通员工不能查高管专属财务表
  • 行级隔离：A 部门员工不能看B 部门的薪资数据——同一张表 ，每行的可见范围不同
  • Prompt 注入防护：恶意用户通过构造特殊query 绕过权限限制

这些需求靠提⽰词无法可靠保证 ，需要系统级的权限管控 ，且要通过合规验证。企业数据安全不能建立在"大模型理解了权限要求"这个假设上。

好,各位来宾大家下午好,我是来自于阿里云智能集团 DataWorks团队的孙文强。接下来由我给大家介绍一下我们最近发布的产品 DataWorks DataAgent。它不是一个只会陪你聊天的 AI,也不是只在某一个环节给予一点点辅助,
而是真正进入数据工作的主流程,可以理解目标,调用工具完成对应的任务,然后回报你最终的结果。对于我们最新发布的 DataWorks DataAgent来说,它给出的定义就是双引擎驱动的全托管数据智能体。一句话总结:数据链路端到端自动完成。接下来我的分享会主要围绕三点:第一,我们为什么需要这样一个 Agent;第二,它和以往的 Copilot 系列产品之间有什么联系;第三,为什么它在企业数据场景中是可落地的、可扩展的、全自动托管的。在讲概念之前,先给大家讲两个具体的例子。如果做过 ETL 开发,这两个例子应该很常见。第一个是面对开发同学的,在周五即将下班的时候,业务同学来了需求,希望能在下周一上线一张用户活跃度相关的表。往往在这个时候,我们的 ETL 同学或 BI 同学只能去找相关的同学文档对齐口径,然后查找源表,进行开发调试。如果有问题再重新调试,经过漫长的调试之后进入调度配置,补数据完成这个需求。
整个过程可能涉及七八个环节、三四个系统,时间上肯定是以天计的,周末可能就泡汤了。这是以前。在新的 AI时代,我们有自己的 Agent之后,这个就变了——我们可以把这个需求给到 Agent,由它自动理解上下文、检索资料、检索空间的所有 SQL、所有源表,然后生成代码、进行调试,最终交付。这是一个开发的例子。还有一个是运维的例子。在晚上凌晨大家睡得最香的时候,周期调度任务往往故障概率最大。以前值班同学能做的是什么?电话拷起来,上线打开电脑,查找日志、查找血缘、查找上下游,看故障对大家的影响,决定怎么解决,可能是重跑、可能要修改代码。今天不一样了,我们的 Agent 会把整个错误信息直接推送到你使用的 IM 群里。
你无论使用钉钉也好、飞书也好、企微也好,我们都可以推过去。除了推送问题以外,我们还推送原因、推送解决方案,你只需要通过 AI 交互就可以解决。这两大问题,无论是开发还是运维,过去消耗了大量人力资源,
今天绝大多数情况下可以用 AI 自主完成。这是我们对 Agent 的诉求,从来都非常具体。今天 DataAgent 的问世并不是一蹴而就的,它也是经过了一个漫长的过程。
从 2022年底 2023年初,我们就投入了 AI相关的研发。从最早的智能补全,那时更多的是一个工具,由 AI来预测下一步是什么。然后进入智能问答阶段,更像是一个助手,一问一答的方式帮助大家提高工作效率。
然后进入 IDE Copilot 阶段,这时它更主动,可以主动理解上下文,帮助大家完成对应的工作。这个过程更像结对编程,你在干他也在干,他在干你在看。接下来进入 ChatBI 阶段,我们将自然语言服务对象从数据开发工程师扩展到更多业务、运营和分析人员,推出了 ChatBI 的产品,基于自然语言去取数。到今天我们推出了 DataAgent 完整的智能体,可以实现自主的规划与执行。
以前我们说"会看"和"会写",它只有交互能力和写代码能力;到今天"会做",才是一个真正的智能体。
从 Copilot 到 DataAgent,不是一个简单的增强,而是一个能力的跃迁。以前的 IDE Copilot 更多服务于人在整个过程中扮演重要角色,人和机器、AI 之间是协作的过程,所以它的速度更快。而现在的 DataAgent 更擅长于人可以不在场,但它解决的问题会更加复杂。消耗的时间相对更长,用来解决更复杂的任务。两者会长期并存,并不是简单的替代。
这是两个 Agent,分为 CLI 模式和 Claw 模式,但它们有统一的自然语言交互方式。
用一句话总结:两个 Agent、两类场景,共用一颗大脑。第一个是面向开发生态的 CLI 模式的 DataAgent,它是代码 Agent,面向开发场景,可以实现自主交互、自主理解上下文、自主使用工具,生成文件并调试。第二个是 Claw 模式,面向运维场景,自主执行,更擅长诊断治理场景。
这两个 Agent 看起来入口不同,因为解决的问题不同,面对的战场不一样。
但底层共用同一个数据的世界,理解的是同一个数据。我们不是做两个独立的小产品,而是做了一套统一的内核,上面长出不同的产品形态,解决不同场景下不同用户的问题。只有这样,才能实现开发态和运行态之间数据的一致。一个场景下积累的上下文规则,可以跃迁到下一个场景继续使用。
这是 DataWorks Agent 的产品架构。从覆盖范围上,它覆盖了从开发到分析到运维治理等所有环节,覆盖了数据企业工作的全链路。从交互方式上,它提供全新的自然语言交互方式,就是上面看到的 CLI 和 IM 这两种形式。用户不再需要判断具体页面地址,也不需要选择对应工具,只需以自然语言形式把你的需求说出来,交给它,它会自动路由来实现,交付整个过程。
从产品定位上,我们希望它成为企业在大数据场景的数字员工——不只是理解你的问题,而是进入你的整个流程,调度、完成任务,

但真正决定它们与众不同的,不是左边能写代码、右边能调度,而是下边这层共享的数据语义层。它们面对的是相同的数据世界——可以知道这个表属于哪层、元数据是什么、上下游是谁、改动影响什么、任务为什么不能在当前直接重跑。面对整个企业的大数据生态,全都交由 Agent 来访问。还包括开放的 Skill 注册和调用能力、统一的工具网关。最重要的,是共享会话和记忆。所以说双引擎共享后台、全托管,这就是 DataWorks DataAgent 在企业级大数据场景里真正落地的根本原因。

它会自主推进完成整个项目的交付。入口是 CLI。典型交付流程是:用户提出诉求,"请在 DWD层加一张用户活跃的表,原表是什么,加完后跑测试"。可以看到目标是清晰的,

这里两点非常重要:第一,它更懂你的代码,因为我们把代码运行和 Agent 整合了;第二,它懂数据的世界。

大家好,我先自我介绍一下,我是来自淘天集团淘宝直播团队的,主要负责数据研发。今天跟大家分享的主题是我们团队怎么从传统的做 ETL 变成现在完全基本是 AI托管的数据研发模式。
整体上,我们是以 DataWorks Agent 的能力作为底座,接下来逐一讲我们是怎么构建这个流程的。首先,做这件事情花的时间非常长——从 24年 7月份到今天,将近两年的时间,我们一直在做能力构建。在 AI 时代,我的判断是:25年的时候,AI 在消费我们的经验,帮我们干日常的事情;到今天,AI 已经变成了经验的生产者,不断地帮我们做需求、完成开发,在一定程度上能消费历史经验,也能生成新的经验。最终会形成什么模式?我判断未来一定是一个 Spec-driven 或 Harness Agent 的模式——可以自闭环、自进化,甚至完全全托管。今天主要给大家介绍我们的三层架构:第一层应用层,讲多 Agent 协同;第二层 AI 能力层,特别介绍 Skill 共建、SDD 范式的体验和 AI Coding 的方式;最后讲基建层。今天有一个非常重要的 Highlight:AI 一定会有幻觉,不管怎么调教。所以在 AI 没有完全自主之前,我们始终坚持——用工程的确定性,暂时接管大模型的不确定性。这是我们两年过程中的主要指导思想。
再看 AI 时代数据研发的挑战。第一,准确率是红线——数据是一还是零,影响业务决策,99%也不能接受。第二,知识库是标配——经验在个人脑袋里,AI 学不到,所以知识库一定要线上化。第三,消费场景变复杂——以前交付的是报表或分析报告,消费者是人;现在和未来,下游消费者也是 AI,数据给出来之后 AI 也在消费。第四,协同模式重构——以前串行,今天并行,同学同时开三四个 Agent 框干需求。第五,生产关系重塑——生产效率提升后,生产关系一定发生质变。数据研发用 AI 不是让 AI 替代我们,而是让 AI 产生的数据和经验能被 AI 持续消费,这是一个循环、持续迭代的过程。
在 AI Native 方面,通常大家会讲 NL2SQL,也就是自然语言转 SQL 的过程。但基于我们的分析,自然语言转 SQL 挑战的是大模型的基模能力和后链路能力,一定程度上是黑盒的。所以在构建过程中,我们不是做完整的 NL2SQL 过程,而是中间加了一层——NL to DSL to SQL。这个理念前两年提到很多,在我们团队实践得非常直接。有几个好处:第一,所有代码生成过程都是白盒的,每个过程可以分段调试和审计;第二,过程中哪里写错、哪里要调整,非常明确,不会给出自然语言后拿到最终结果,不知道中间发生什么。我们定义 R2C 流程,不是拿到一句话给个 SQL 就完了,而是从需求沟通开始到最终需求交付的全流程:需求沟通、技术方案设计、代码生成、运维挂基线、数据回刷、数据验证、最后上线。这是完整的 R2C 过程。
做这件事有四个步骤:第一,标准化数仓——这是所有 DataAgent 的基础,没有标准化数仓,AI 生成的代码都是概率性的;第二,规范研发范式——一类需求的完成应该遵循什么模式;第三,建工程平台——用工程的方式兜住大模型,分析出大模型有哪些幻觉是偶发性的,就在那里建工程平台;第四,构建多 Agent 协同机制。
总结这一两年的成果:我们把 AI 从工具变成了同事,现在的需求基本由 AI 全流程完成。代码占有率在我们团队 75%以上都是 AI 全生成,而且这个 75%是百分百准确的,因为用工程能力兜住了代码生成。我们和 DataWorks DataAgent 持续共建,在标准化的输入条件下,准确率从最初 50%做到了现在 80%。某些场景下需求基本 24小时内交付——今天提,明天就交付。
演进时间线:24年 Q3是基建期,建数仓模型和工程能力;25年上半年跑通了 R2C 全流程;25年 Q4做了中心化 AI Native;今年随着小龙虾爆火,开始做本地化 AI Native;到现在把自己这一两年的经验全部变成 Skill,包括 verify-data 这个 Skill 也在阿里云公众号发布了。总结起来三条:第一,坚持长期主义——代码生成或 ETL 开发不能大力出奇迹,必须扎实做好基建;第二,资产持续保鲜——数据资产、知识库、Memory 要持续迭代;第三,紧跟前沿。
介绍整体架构图,有两个重要的东西:第一,以 DataWorks DataAgent 作为基座——所有的代码生成、运维、验数都基于它;第二,有很强的理论基础——本体论。本体论是压住 AI 幻觉的符号锚点,定义清楚一个东西的定义、实体、关系;更重要的是 Harness,它是调度 AI 能力、管住 AI 上下文的运行时锚点。我们做整个数仓模型,按 OneData 的理论基础分层——明细层、DWS、ADS,但模型层不只是给表,而是面向 AI Friendly的产出:表对了、数据对了,还得注释对、描述信息、枚举值、代码结构都要非常准确,AI 才能真正消费。知识库分两层:第一层是 LightRAG 结构化召回——面向 AI Friendly的表,用图的方式构建知识,让 AI 找得到表;第二层是业务 Wiki——让 AI 听得懂人话,因为业务方的白话跟数据库里的数据有 Gap。有了这两层再加平台层的语义管理、DSL 生码、研发后链路和 MCP 代理,整个 AI Native 流程就串联起来了。
然后比如说像直播,我们有一个主播 ID,基本上所有的主播相关表都有这个字段。如果直接用主播 ID 做召回,准确率非常低,所以我们会构建图的结构——让 AI 能找到表。第二层业务 Wiki 让 AI 能听懂人话。业务方提需求、我们问 AI,不是按照数据库的方式说"我想 GROUP BY 什么字段",而是说"我的 GMV 是什么,想要的分析是什么"。所以我们做 query 改写和业务知识层,两套引擎缺一不可。
然后我跟大家介绍一下流程。我们在做研发过程中,要去提炼做需求的范式——每个团队、每个同学可能有区别,
但一定有共性。基于提炼出的范式,我们去做需求或完成团队规划,包括技术方案的构建、模型设计、代码生成、关键信息的关联、DDL变更、以及兜底的监控。这是研发的标准范式。然后我们把需求拆解。在做 AI Native 的过程中我们发现,写代码的时间不一定占需求的大头,大头是需求的澄清阶段。我们要跟业务方沟通清楚:口径是什么、数据范围是什么、要不要回刷、这个指标和那个指标的区别。所以把需求拆成两阶段:第一阶段是澄清部分——构建面向 AI 友好的技术方案,目前以人工方式做 check;第二阶段是真正的执行阶段——输入是完整的技术方案,全过程 AI 驱动,人只在过程中做检查。
在多 Agent 协同方面,从需求分析输入后,由规划器(Planner)做完计划,再去调度职能 Agent 干具体的事情,干完后把信息给到 Reporter Agent。核心一点:我们还是用 DSL 在中间做一层 check。人在里面做确认,不是说它是降级方案,而是整体设计的重要目标。因为在变化过程中,需要确认每一步都是正确的,否则方案推导的效率和沟通效率会非常低。
为什么从中心化的 AI Native 变成个性化/本地化的 AI Native?我们发现在用的过程中有几个问题:第一,如果是中心化 AI Native,Agent 的迭代速度会成为单点能力;第二,Memory 的记忆——我在做需求、其他同学也在做需求,记忆天然有冲突,没法像代码合并一样;第三,Skill 出现后不需要构建那么多 Agent,把 Agent 变成 Skill 就能实现 AI 平权;第四,在中心化 AI Native 里,每次召回都是全量的,没法做到个性化召回,整个淘宝直播的 RAG 量非常大。基于这几点,今年年后我们开始基于 OpenClaw 做本地化的 AI Native 流程——就是四个 Agent 在本地的。业务方提需求:"加几个指标和几个维度",主 Agent 去做协调。标准的需求流程就三趴:第一趴需求分析,第二趴建模,第三趴写代码。基于这个流程,能在本地完成需求。好处是:在本地每次做的需求都有记忆,之前做过的所有需求,不管是修改还是新增,都有非常强的记忆。当 OpenClaw 开始干活时,它会先看我之前做了什么需求,拉取记忆和 Spec 文件,再决定怎么干活。所以效率和对需求的准确解读、信任度都非常高。
然后介绍一下怎么构建 AI 能力层。我们目前做的事情:把所有的能力——虽然什么东西能变成 Skill 没有准确的定义——但我们做的几件事:第一,构建语义层的 Skill,这个从去年开始在建;第二,把原生的 Agent 变成一个个 Skill,像 verify-data 这个 Skill 已经在团内很多团队在用了;第三,把以前的范式变成 Skill。好处是:任何用我 Skill 的同学可以零成本、无缝使用我所有的资源和范式,我会给他非常详细的过程和 Spec 文档。整体上我们沉淀不同的 Skill,最终走向 Agentic 的必经之路。
另外就是 SDD。做一个需求不是从 A 到 B 一个点,中间有非常多的节点——10个甚至11个。节点之间怎么交流?不是 Agent 跟 Agent 用 MCP 协议协同,而是以中间的 Spec 方式协同。Spec 模式在我们团队实际使用过程中非常有帮助,能在生产中用。我们的 Spec 可能有十几个流程,核心拆成:第一,现在的需求怎么样;第二,缺什么资源——表够不够、业务方提的指标写得清不清楚;第三,模型设计——CDM 加 DWS 和 ADS 的设计,数据怎么存;第四,给到 DataAgent 的一个非常标准的输入。在每个环节有一个对应的 Spec 文件,写清楚在做什么。SDD 不是降低需求复杂度,而是降低沟通成本。一个需求可能是前面做建模、后面做验数、做运维发布,中间是不同的同学甚至不同的 Agent 在干活。SDD 降低沟通成本——后面的 Agent 能准确阅读前一个流程 Agent 输出的 Spec,直接拿到 Spec 文件里的信息就开始干活。AI 能做的是什么?第一,上下文管理——基于 Spec 模式,不管上下文多长、切换 Session,信息不会丢失;第二,全过程可追溯——不需要重复流程;第三,Review 前移——最担心的是 AI 写一堆代码,最后 CR 压不动;第四,所有需求有模板。这是坚持用 SDD 的最核心原因。
最后讲 AI Coding 两种协议的方式,两条路并行。我们构建非常强的业务语义层——定义指标、表、维度。第一条路:完全的 NL to DSL to SQL 过程,借助工程化平台。输入是结构化的 JSON,从工程化平台到 DSL 的 JSON,最后到代码产生。好处是全自动、可单测。做 DSL 时发现非常有意思的点:给 AI 的 JSON 可以用工程去检测,让 AI 自己修改——如果 AI 生成的 JSON 结构有错误或不符合平台定义的规则,把信息吐回给 AI,AI 可以基于错误自动修改生成结构。这比传统生成 SQL 后去验证的难度小很多。缺点是需要基于语义层,在工程平台做指标、表、维度的录入和维护,是个长周期的事。第二条路:基于 DataWorks DataAgent 做代码生成,不需要那么强的语义层构建,只要每次给 DataAgent 说清楚数据来源、伪代码就行。标准化的复杂需求走 DSL 全自动,次抛需求、短平快的需求走 DataAgent 半自动。
最后讲我们和 DataWorks DataAgent 的深度实践。从去年六七月份开始深度使用 DataWorks DataAgent 的能力。结果:次抛的临时取数、历史任务只需要改几个东西的、模型性的事情,全让 Agent 帮我们干。代码采纳率 72%,生成代码后只需做一些微调就能完成任务。DSL 全自动已经占了 70%多,两者结合基本百分百完成需求。还有一个点:深挖降低平均对话轮次——对话轮次越高,说明给到 DataAgent 的输入越不标准,需要多次澄清。目前平均对话轮次约 2.99 轮。实践里面给 DataAgent 三类东西:第一,研发规范(Rules)——全局规范,表/字段/主题域/业务过程/命名/缩写/指标体系,项目级 Rules 让业务同学也能用 Agent 模式找表+生码;第二,非常标准的交付方式——目标、主要步骤、二次加工字段、特殊要求(防膨胀/排除逻辑)都写清楚;第三,CTE 参考——有些需求不完全在基础数仓里,在代码片段里,告诉它参考这个 CTE 模式去写。三个核心原则:提示精准、代码严格过审、追求实用。

最后三页:数据研发在构建了这么多 Skill、SDD 范式和多 Agent 协同之后,还有什么一定不能少?第一,CDM 工程——不管 AI 模型多厉害,CDM 要持续构建和保鲜。基于两种体系构建 CDM:OneData 体系 + AI Friendly 交付。不是做表交付,是做更 AI Friendly 的交付——命名、枚举、口径不能有二义性,要做持续保鲜。突击搞一个月 AI 能干好,过一两个月可能就不行了。看一下我们的 LightRAG 图——构建完全的 AI Native 且全托管,绝对不是拿几张表出来就行。这个图里每个表实体关系、每个属性、枚举值都展示得非常清楚,这是让 AI 能找得到表的基础。右边是业务 WIKI——在做 NL2SQL 或 NL2DSL2SQL 时,核心是理解业务想要什么。业务方说的是"我的 GMV 是什么、想要什么样的分析",不是"GROUP BY 什么字段"。所以做 query 改写和业务知识层,左右两边缺一不可。
最后一个,工程平台。有了语义管理、DSL 生码、研发后链路这些原子能力,单点看是工具,但结合原子能力做 AI Native 的编排。去年 7月份我们做了 AI Native 编排,像 Coordinator、DataCard 这些是编排器,API、Model、Sync 这些是职能 Agent。把工程平台变成一个个职能子能力,用 AI Native 方式做动态编排和执行。
未来:我们已经跑了一两年,R2C 全流程已经走通。但未来数据研发不只是写代码、交付报表,而是跟业务目标绑定——上个月的 GMV 怎么样、指标为什么变化、变化原因和谁影响了。下一步以 ChatBI 为切入点,做数据消费分析的探索。

## P2 · 一句话主线 · 长期主义

**用工程的确定性，接管（暂时）大模型的不确定性**——AI 不确定 → [工程接管区] → 可用 AI 结果。

- AI 能力层：Skill + SDD + AI Coding（管得住 AI）

### AI 角色演进
01. 经验执行者（早期）→ 02. 经验生产者（当下）→ 03. SuperAgent / HarnessAgent（未来）

## P3 · AI 时代对数据研发的挑战

不只是工具和效率的升级，而是生产力的质变和生产模式的突破。

01. **准确率是红线**：100% 准确才能上线，人 review 跟不上 AI 输出速度，红线必须硬约束
02. **知识库是标配**：传统模式下经验都在员工脑子里，AI 时代必须全部线上化，领域模型的体系化构建异常重要
03. **消费场景复杂**：不只给下游 ADS 同学用，AI 也要吃数据 — 命名 / 枚举 / 口径无二义性成硬约束
04. **协同模式重构**：人干需求 → 多 Agent 并行，人退出执行环节、只守关键决策点
05. **生产关系重塑**：生产力的质变必将带来生产模式和生产关系的变化

> AI 时代下，数据研发的最终目标不是「AI 替代人」，**而是让 AI 产生的数据和经验，能被 AI 持续消费**。

## P4 · AI Native: R2C，不只是 NL2SQL

**NL2DSL2SQL 是整个体系最核心的技术决策**。

R2C = Requirements To Consign · 从需求到交付的完整闭环。不只是写 Code，覆盖：技术方案 / 代码生成 / 数据验证 / 数据回刷 / 基线挂载。

| | NL2SQL（×） | NL2DSL2SQL（✓ 准确率 100%） |
|---|---|---|
| 流程 | 自然语言 → ??? 黑盒 → SQL ?! | 自然语言 → JSON DSL（√校验/√单测/√补常识）→ SQL ✓ |
| 缺点/优点 | 天花板低 / 出错难修 / 用户难干预 | 工程兜底 / 分段调试 / 全过程可审计 / 白盒化 |
| 出错处理 | 出错只能整段重写 | — |

**关键设计**：AI 调 DSL 的 check 接口 · 反馈驱动优化——AI 写错了，工程层告诉它哪里错，AI 自己修正。

### 四步走 · 先地基后建筑

- L02 流程层：规范化 Pipeline（研发范式）
- L03 平台层：服务化平台
- L04 智能层：多 Agent 协同

> 核心思想：用工程的确定性 接管（暂时）大模型的不确定性。

## P11 · 为什么是本地化的 AI Native

**从"每天清空记忆的实习生"到"越来越懂业务的老员工"**

### 中心化 AI Native 的缺点
01. Agent 依赖的 DSL 的迭代速度成为单点
02. Memory 无法轻易反哺到 Agent 的进化
03. SKILL 的出现让 AI Agent 的能力逐步平权
04. 每次 RAG 召回都要全量召回，精准率和效率都比较低

### 中心化 vs 本地化 AI Native

| | 中心化（实习生） | 本地化（老员工） |
|---|---|---|
| 记忆 | 每天清空 | 持续累积 |
| 经验 | 每次重学 | 可调用 |
| 错误 | 重复犯 | 不再犯 |
| 协同 | 从头交接 | 继承上下文 |

### ★ 最佳案例 · OpenClaw · 三 Agent 协作（SDD.md 文件协议）

🔍 **小需** · 需求分析 → 🛠️ **小模** · 建模 · DDL → 👓 **小码** · SQL & 自检

📝 记忆系统 · 踩坑沉淀 · 不再犯

成果：
- ⟨⟩ 0 手写代码
- 🔌 0 平台依赖
- ⏱ ETL 3 天 → 3 小时

主 Agent（总协调）：接收需求 → 判断类型 → 调度子 Agent → 审查产出 → 推进流程
- 小需 🔥 需求分析（覆盖 ①②③）
- 小模 🛠 数仓建模（覆盖 ④⑤）
- 小码 ⚡ SQL 开发（覆盖 ⑥⑦）

> 可继承的差异 = 可演化的组织 — 这是"从工具到员工"的关键技术差异

## P13 · 沉淀 Skill · 通往 Agentic

从 0 到 1 · 从 Agent · 从 范式 — 三种来源，统一 Skill 输出，为全面 Agentic 化打下坚实基础

### 方式 ① · 新 Skill：语义层 Skill
无 Agent / 无原范式可借鉴 · 围绕具体业务痛点从头建

流程：痛点定义 → 流程拆解 → SKILL.md → 迭代沉淀

- 需求驱动：从一个具体场景痛点出发，先跑通最小闭环，再扩展边界
- 渐进式披露：主 SKILL.md 仅路由 + references 按需加载，避免上下文爆炸
- HITL：关键节点强制人工确认 · 红线机制兜底，**不是建议**

### 方式 ② · 原有 Agent → Skill：verify-data
原本是独立验数 Agent · 沉淀为通用 Skill · 一句话触发评审级证据链

流程：基准发现 → SQL 生成 → 跨表对比 → 红线检查 → 钉钉发布 ✓

- SQL 模板：8 种常规验数模板 & 2 种高频出错模板
- 验证红线：不靠字段名猜映射 · 不跳基准表 · 不隐式聚合掩盖膨胀 · 不用 LIMIT 替代全量
- 踩坑沉淀：覆盖 5 类场景 — 新表 / 迭代 / 监控 / 业务质疑 / 口径迁移 · 效率 2-4h → 30min

### 方式 ③ · 原有范式 → Skill：copilot_req2sql
把 R2C 主流程的「需求澄清范式」抽离 · 最小可复用单元 · 24h 交付

流程：P1 需求澄清 ★ → P2 资源梳理 → P3 模型设计 ★ → P4 Data Agent 输入

- 渐进式披露：主 SKILL.md 200 行只放路由，细节 references/p1-p4.md 按阶段加载
- HITL：P1 口径确认、P3 模型 CR 强制人工节点 — 不是形式，是质量底线
- 模式可迁移：「需求 → 产出」类领域均可套用 · Data Agent 生码 准确率 50% → 80% · 多需求并行

> 不同来源 · 殊途同归 — Skill 化是 AI 能力沉淀的共同范式

       - 主键  anchor_id + ds

### AI 的边界 · SDD 降低沟通成本，不降低问题复杂度

CAN（AI 能做）：
- ✓ 上下文管理 — 跨长流程不丢线索
- ✓ 过程追溯 — 每一步都有 Spec 留痕
- ✓ Review 前移 — 写之前先对齐
- ✓ 模板化执行 — 标准化 SQL / DSL 生成

CAN'T（AI 不能做）：
- ✗ 数据建模决策 — 需要人的全局判断
- ✗ 指标口径判断 — 业务语义的最终拍板
- ✗ 复杂业务翻译 — 隐性规则没法穷举
- ✗ SQL 正确性保证 — 最终责任在人

- 业务域定制：点淘等业务有专属补充规范
- SQL 结构：CTE 分层 + UNION ALL + INSERT OVERWRITE
- 项目级 Rules：业务同学也能用 Agent 模式找表 + 生码

### 交互范式（Prompt 模板）

### 3 大核心原则

03. **追求实用**：50% → 80% 准确率即可，少量人工修改 + 人模协同

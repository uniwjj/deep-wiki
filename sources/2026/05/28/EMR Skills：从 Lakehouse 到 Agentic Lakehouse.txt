什么是EMR? 什么是Lakehouse?
阿里云EMR：基于开源框架的一站式大数据处理服务，是运行大数据任务的“操作系统”或“基础设施平台”，帮你搞定底层复杂性，让你专注于数据本
身。
Lakehouse：一份数据、多引擎协同。通过引入元数据层和事务机制，实现数仓级别的管理能力。EMR Serverless Spark + StarRocks + DLF（统一元数据）。

场景三：金融报表加速 — 告别 T+1，今天的
数据今天看
痛点

/starrocks-assistant 我有一张 MySQL 交易流水表，5000 万行/天，想做实时报表。

帮我设计表结构和入湖方案

• 监管报表、经营日报依赖 T+1 跑批，领导早会看到的是"昨天的昨天"
• MySQL 分库分表查不动聚合，BI 工具超时转圈
• 接一个新报表需求就要写 ETL + 建宽表，排期两周起步

Schema 设计建议：
表模型：Primary Key（支持 CDC 实时更新）
分区：PARTITION BY RANGE(txn_date) 按天
分桶：DISTRIBUTED BY HASH(user_id) BUCKETS 32
排序键：(user_id, txn_date, txn_time)

方案：StarRocks Skill 智能 Schema 设计 + 数据入湖选型
• 一句话建表：描述业务场景 → 自动推荐表模型、分区分桶策略
• 入湖方式自动选型：MySQL CDC → Flink CDC + Connector；文件 → Broker Load

• 物化视图替代宽表：无需手动 ETL，MV 自动增量刷新，报表实时可查

MySQL → Flink CDC → StarRocks Connector

场景四：MySQL → StarRocks 报表迁移
痛点
•团队有 20 张业务报表跑在 MySQL（RDS）上，覆盖订单、商品、客户、营销等核⼼域。随着数

据量增⻓，复杂聚合查询越来越慢。业务⽅要求做分析加速，同时保留 MySQL 作为业务写⼊源
•传统做法：逐表导出 MySQL DDL（10 分钟）→ ⼿动改写为 StarRocks 语法并调优分区分桶策
略（30 分钟 × 20张表）→ 编写数据迁移脚本（20 分钟）→ 全量导⼊并校验⼀致性（15 分
钟）。总计约 2-3 ⼩时，且 DDL 改写过程中极易遗漏 StarRocks 的分区裁剪、排序键、
Colocate Join 等优化点。

方案：StarRocks Skill 智能 生成StarRocks DDL+ 自动比对
•读取 MySQL Schema 后⾃动⽣成最优 StarRocks DDL；
•无需担心网络等配置问题，AI帮你搞定工程问题
• 根据业务特征，自动生成StarRocks 的分区裁剪、排序键、Colocate Join 等优化点

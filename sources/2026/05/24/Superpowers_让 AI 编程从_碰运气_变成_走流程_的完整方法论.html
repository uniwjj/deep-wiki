Superpowers：让 AI 编程从"碰运气"变成"走流程"的完整方法论

开场钩子：你有没有过这种经历？让 AI 写代码，它噼里啪啦写了几百行，跑起来没问题，用起来却一堆 bug。明明 AI 那么聪明，为什么写出来的东西总是差点意思？

一、一句话定位

Superpowers 是什么？

Superpowers = AI 编程的"交通规则"

就像交通规则不是让司机变慢，而是让所有人有序通行；
Superpowers 不是限制 AI，而是让 AI 编程从"随意变道"变成"有序行驶"。

一句话总结：

不是让 AI 更聪明，而是让 AI 更守规矩。

解决了什么问题：

痛点没有 Superpowers有 Superpowers
AI 写代码上来就写，写完发现方向错了先设计，再动手
上下文管理做着做着就忘了需求每个任务都是全新的开始
测试覆盖代码写完再说，测试能省则省先写测试，再写代码
代码质量靠感觉，"应该没问题"独立审查，有证据才能说完成
分支管理主分支直接改，改崩了没法回隔离工作区，搞砸了删掉重来

三、第一步 + 第二步：设计先行 + 环境隔离

3.1 头脑风暴（Brainstorming）：没有设计文档，就不许写代码

核心铁律（HARD-GATE）：

<HARD-GATE>
在呈现设计方案并获得你批准之前，绝对不能：
1. 调用任何实现技能
2. 写任何代码
3. 搭建任何项目结构
4. 做任何和实现相关的动作
</HARD-GATE>

深入浅出理解：

生活比喻：装修房子
❌ 没有 Superpowers：直接让你开始砌墙
✅ 有 Superpowers：先出设计方案，确认后再动手

为什么这么严？
因为"看起来简单"的项目，最容易出问题。
你以为"就改个配置"，结果改完才发现影响了一堆依赖。

9 步标准化流程：

步骤内容关键动作
1️⃣探索项目上下文AI 自动扫描现有代码和架构
2️⃣可选：视觉伴侣涉及 UI 时提供可视化工具
3️⃣苏格拉底式提问一次问一个问题，逐个澄清需求
4️⃣提出 2-3 个方案每个方案都有权衡分析
5️⃣分段展示设计按复杂度分段，逐段确认
6️⃣编写设计文档自动保存到 docs/superpowers/specs/
7️⃣规范自检检查有没有 TBD、TODO 等占位符
8️⃣等待你审查只有你确认了才会往下走
9️⃣调用写计划技能唯一合法的下一步

实际示例 - 优惠券核销功能：

AI：关于优惠券核销，我有几个问题：

Q1: 优惠券有几种类型？
   - 满减券（满100减20）
   - 折扣券（8折）
   - 固定金额券（减10元）

Q2: 一张券能用几次？
   - 一次性（用完就失效）
   - 可多次（每周可用一次）

Q3: 并发场景怎么处理？
   - 乐观锁（先到先得）
   - 悲观锁（排队处理）

核心设计原则：

1. 一次一个问题：绝不轰炸式提问
2. 多选题优先：减少歧义，加快速度
3. YAGNI 原则：不做多余功能
4. 模块化设计：拆成独立单元

3.2 Git 工作区隔离（Git Works）：保护主分支，给你一个安全的沙箱

核心铁律（FORBIDDEN）：

<FORBIDDEN>
1. 绝对不能在主分支直接做开发改动
2. 工作区目录必须被 Git 忽略
3. 开发前必须先验证基线测试
</FORBIDDEN>

深入浅出理解：

生活比喻：练车 vs 公路驾驶
❌ 没有 Superpowers：在公路上练车，改着改着撞上了
✅ 有 Superpowers：去驾校练车，搞砸了最多撞个假人

为什么这么严？
主分支是项目的"根"，一旦污染，整个项目都危险。
隔离工作区，就是给你一个可以随便折腾的沙箱。

4 步创建隔离工作区：

步骤内容说明
1️⃣目录选择优先使用 .worktrees/ 或 ~/.config/superpowers/worktrees/
2️⃣安全验证确保目录在 .gitignore 中
3️⃣创建工作树自动创建分支和独立目录
4️⃣基线验证安装依赖，运行测试，确保环境干净

隔离工作区示意图：

主分支 (main) - 永远干净
    │
    ├── .worktrees/feature-login/    ← 登录功能开发区
    │       └── 分支: feature-login
    │
    ├── .worktrees/feature-payment/  ← 支付功能开发区
    │       └── 分支: feature-payment
    │
    └── .worktrees/bugfix-crash/     ← Bug 修复区
            └── 分支: bugfix-crash

解决的问题：

✅ 主分支永远安全：搞砸了删掉就行
✅ 并行开发无压力：同时做多个功能
✅ 试错成本为零：实验性改动随便试
✅ 紧急切换无负担：随时切换上下文

四、第三步 + 第四步：任务拆分 + 多代理执行

4.1 编写计划（Writing Plans）：把大象切成小块

核心铁律（6 类禁止占位符）：

<FORBIDDEN>
1. TBD / To Be Determined：待确定
2. TODO：待完成
3. 下一步：... / 接下来：...
4. 后续会实现... / 之后会添加...
5. 我们将在后面的步骤中...
6. 任何模糊的、没有明确行动的描述
</FORBIDDEN>

深入浅出理解：

生活比喻：搬家
❌ 没有 Superpowers：给你一句"把东西搬过去"
   结果：搬过去才发现冰箱没拔电源，洗衣机还在运作
✅ 有 Superpowers：拆成步骤清单
   1. 关掉冰箱电源
   2. 排空洗衣机
   3. 打包易碎物品
   ...

为什么这么严？
AI 子代理的"记忆"有限，大任务会让它"失忆"。
拆成 2-5 分钟的小任务，每个都能轻松完成。

任务拆分示例 - TDD 开发流程：

原始：实现用户登录验证
拆分后：

任务 1.1: 编写空邮箱验证测试
  - 创建 tests/unit/test_login.py
  - 编写空邮箱场景的测试用例
  - 运行测试，确认失败（RED 阶段）

任务 1.2: 实现空邮箱验证
  - 在 src/auth/validators.py 添加邮箱非空判断
  - 运行测试，确认通过（GREEN 阶段）
  - 重构代码，确保符合 PEP8

任务 1.3: 提交本次变更
  - git add .
  - git commit -m "feat: 添加邮箱非空验证"

三大文件组织原则：

1. 增量修改：只改需要改的文件
2. 原子提交：每个小任务单独提交
3. 隔离工作区：不影响主分支

4.2 多代理开发（Subagent-driven Development）：每个任务都是全新的开始

核心铁律（FORBIDDEN）：

<FORBIDDEN>
1. 绝对不能用同一个代理做多个任务
2. 审查必须先规格审查，再质量审查
3. 子代理遇到问题必须主动求助
</FORBIDDEN>

深入浅出理解：

生活比喻：炒菜 vs 全席宴
❌ 没有 Superpowers：一个厨师从头做到尾
   问题：做到第10道菜，早期菜谱早忘了，味道全乱了
✅ 有 Superpowers：多个厨师，每道菜换新厨师
   每个厨师只做一道菜，专心又高效

为什么这么严？
单代理的上下文是累积的，做越多越容易"漂移"。
新代理 = 新鲜的思维 = 专注的执行。

多代理执行流程：

                    主代理（总调度）
                        │
            ┌───────────┼───────────┐
            │           │           │
            ▼           ▼           ▼
        子代理1     子代理2     子代理3
       (新上下文)   (新上下文)   (新上下文)
            │           │           │
            ▼           ▼           ▼
       规格审查1    规格审查2    规格审查3
            │           │           │
            ▼           ▼           ▼
       质量审查1    质量审查2    质量审查3
            │           │           │
            └───────────┼───────────┘
                        │
                        ▼
                    任务完成 ✅

两轮复查机制：

轮次审查内容核心问题
第一轮规格合规性我们有没有做对的事情？
第二轮代码质量我们有没有把事情做好？

解决的问题：

✅ 彻底解决上下文漂移：每个子代理都是全新开始
✅ 彻底解决代码跑偏：两轮审查把住关
✅ 出错定位超简单：每个任务独立，出了问题直接定位
✅ 效率大幅提升：独立任务可以并行执行

五、第五步 + 第六步：测试驱动 + 代码审查

5.1 TDD 测试驱动（Test-driven Development）：没有失败测试，就不许写代码

核心铁律（IRON LAW）：

<IRON LAW>
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

在你写出一个会失败的测试之前，绝对不许写一行生产代码。
</IRON LAW>

深入浅出理解：

生活比喻：装修验收标准
❌ 没有 Superpowers：先把墙砌好，再想怎么验收
✅ 有 Superpowers：先定验收标准，再按标准砌墙

为什么这么狠？
先写代码后写测试，测试只验证"你写的代码"对不对。
先写测试后写代码，测试验证的是"你要的行为"对不对。

RED-GREEN-REFACTOR 循环：

    ┌──────────────────────────────────────────────────┐
    │                                                  │
    │   【RED 红灯阶段】                                 │
    │   写测试 → 看着它失败                            │
    │   失败原因必须是"功能还没实现"                    │
    │                                                  │
    │   ↓                                              │
    │                                                  │
    │   【GREEN 绿灯阶段】                              │
    │   写最少的代码让测试通过                          │
    │   严格遵守 YAGNI：不需要的就不写                  │
    │                                                  │
    │   ↓                                              │
    │                                                  │
    │   【REFACTOR 重构阶段】                           │
    │   优化代码质量，保持测试绿色                      │
    │                                                  │
    └──────────────────────────────────────────────────┘

堵住偷懒的借口：

借口真相
太简单了，不用测简单的也会崩，测试只要 30 秒
回头再补测试后补的测试永远证明不了什么
我手动测过了手动测试今天测了明天就忘
都写一半了删掉浪费这是沉没成本谬误，不可信的代码才是浪费

解决的问题：

✅ 强制 AI 写测试：从根源解决测试覆盖率低
✅ 避免过度设计：只写测试要求的代码
✅ Bug 提前发现：在开发阶段抓住，不等到上线
✅ 回归测试保护网：后续改动有保障

5.2 代码审查（Code Review）：独立审查，破除盲区

核心铁律（IRON LAW）：

<IRON LAW>
1. 代码审查必须由独立代理执行
2. Critical 级别必须立即修复
3. 没有完成验证，不许宣称完成
</IRON LAW>

深入浅出理解：

生活比喻：文章自查 vs 别人审稿
❌ 没有 Superpowers：自己写完自己看，永远觉得完美
✅ 有 Superpowers：找没参与写作的人来审稿

为什么这么严？
人（包括 AI）都有"实现者盲区"：
你刚花两小时写的代码，脑子里全是自己的思路，
检查的不是代码，是自己的思路。

三级审查标准：

级别说明处理方式
Critical（严重）安全漏洞、数据泄露、权限缺失必须立刻修，不修不能往下走
Important（重要）边界条件漏了、逻辑错误、缺少日志必须修完才能进入下一个任务
Minor（次要）变量名不好、注释不够清楚记录下来，后续再处理

完成前 5 项验证清单：

□ 1. 所有测试是不是全绿？
□ 2. 代码审查是不是通过了？
□ 3. 有没有遗留的 TODO 和调试代码？
□ 4. 相关的文档更新了吗？
□ 5. 实现是不是和设计文档一致？

解决的问题：

✅ 破掉实现者盲区：独立审查者能看到你看不到的问题
✅ 把"应该没问题"变成"确认没问题"：所有完成都要有证据
✅ 再也不会留遗留问题：TODO、没更新的文档无处遁形
✅ 严重问题不会流到线上：Critical 问题在开发阶段就被拦住

六、第七步：完成分支（Finish Branch）- 优雅收尾

核心铁律（IRON LAW）：

<IRON LAW>
1. 收尾之前必须先跑完整测试
2. 高风险操作必须二次确认
3. 自定义技能也要用 TDD
</IRON LAW>

深入浅出理解：

生活比喻：餐厅打烊 vs 餐厅收尾
❌ 没有 Superpowers：客人走了就关门，餐具堆着明天再说
✅ 有 Superpowers：标准化收尾流程
   1. 确认账单结清（测试全过）
   2. 餐具分类清洗（代码归档）
   3. 桌椅归位（清理工作区）
   4. 写营业日志（提交记录）

为什么这么严？
收尾是最容易出乱子的环节。
合错分支、删错代码，一出问题前功尽弃。

四个标准化收尾选项：

选项适用场景执行流程
本地合并个人项目、小功能切回主分支 → 合并 → 测试 → 删除分支
创建 PR团队协作、需要审查推送 → 创建 PR → 保留分支待改
保留分支开发到一半，要做别的事保留分支和 worktree，继续开发
丢弃分支做砸了，要删掉重来输入 discard 确认 → 删除所有内容

自定义技能：把你的流程变成 AI 的纪律：

创建流程和 TDD 一模一样：

1. RED 阶段：模拟你偷懒的场景
   "测试"：如果我下班前赶时间，AI 会乱写提交信息吗？

2. GREEN 阶段：写技能堵住漏洞
   "实现"：要求提交信息必须有类型前缀（feat:/fix:/docs:）

3. REFACTOR 阶段：迭代优化
   "重构"：找到新的偷懒借口，堵住它

八、总结与行动建议

一句话总结

Superpowers 不是让 AI 更聪明，而是让 AI 更守规矩。

七步工作流快速回顾

1️⃣ Brainstorming    → 先设计再动手
2️⃣ Git Works        → 隔离环境保护主分支
3️⃣ Writing Plans    → 任务拆分 2-5 分钟
4️⃣ Subagent         → 每个任务全新代理
5️⃣ TDD             → 先写测试再写代码
6️⃣ Code Review     → 独立审查破盲区
7️⃣ Finish Branch   → 标准化收尾

立即可用的行动建议

□ 1. 今天：在你的 AI 编程工具上安装 Superpowers
□ 2. 本周：用 Superpowers 完成一个小功能
□ 3. 本月：尝试完整走一遍七步工作流
□ 4. 长期：创建团队专属的自定义技能

核心金句

纪律，释放能力。

不是限制 AI，而是让 AI 的能力真正发挥出来。
就像有纪律的天才，远比没纪律的天才要靠谱得多。

Claude Code + OpenSpec + Superpowers，AI协同开发实战详解（从入门到精通）

作者：小程故事多_80 发表：2026-03-29 19:01:21 来源：https://blog.csdn.net/u013970991/article/details/159598155 下载：2026-05-24

在AI编程普及的今天，很多开发者都有过这样的体验：用AI写代码速度飞快，可要么偏离需求返工不断，要么代码混乱无规范，要么思路跑偏没方向。其实问题不在于AI工具本身，而在于我们没有找到一套高效的协同逻辑，单独使用任何一款AI编程工具，都难以兼顾需求对齐、工程规范和高效执行。

而Claude Code + OpenSpec + Superpowers的组合，恰好解决了这个痛点。这三者不是简单的工具叠加，而是形成了“需求对齐→流程约束→代码执行”的闭环，让AI编程从“自由发挥”变成“规范高效”，既保留AI的开发速度，又兼顾工程质量。今天就结合实战案例，从工具协同逻辑、环境搭建、核心工作流、进阶技巧到避坑指南，手把手教你玩转这套AI协同开发方案，看完就能直接上手落地。

一、为什么要三者协同？打破单个工具的局限

在聊具体操作之前，我们先搞清楚一个核心问题：为什么单独用其中任何一个工具都不够？只有理解了单个工具的短板，才能明白三者协同的价值所在。

1.1 单个工具的“致命短板”，你大概率也遇到过

我们先逐个拆解，看看单独使用Claude Code、OpenSpec、Superpowers时，会遇到哪些实际问题，这些问题也是我们日常开发中最头疼的痛点。

首先是只用Claude Code的情况。作为Anthropic推出的AI编程工具，Claude Code的代码生成速度非常快，通常简单需求5分钟内就能给出代码，但最大的问题是“需求不对齐”。比如你跟它说“加个用户登录功能”，它很快就写好代码，可等你Review时才发现，你要的是JWT认证，它却用了Session，只能让它重新修改，返工成本极高。更麻烦的是，如果你没有明确说明细节，它很可能会按照自己的理解“自由发挥”，最后写出的代码和实际需求偏差很大。

然后是只用OpenSpec的情况。OpenSpec是一款需求规范工具，核心作用是生成标准化的需求文档，你只需要输入简单指令，它就能生成完整的spec文档，明确需求目标、范围和规范。但问题在于“有规范，无约束”，比如你用OpenSpec生成了登录功能的spec文档，让AI按照文档编码，可写到一半，AI就会偏离spec的要求，擅自修改逻辑，最后还是要人工干预，规范的价值根本没发挥出来。

最后是只用Superpowers的情况。Superpowers是Claude Code的插件，核心优势是“工程纪律”，能帮你进行头脑风暴、制定执行计划、强制执行TDD（测试驱动开发），但它的短板是“有纪律，无方向”。比如你让它做个登录功能，它会先头脑风暴、写计划、执行TDD，但这个计划的基础是什么？是它自己猜测的需求，没有明确的需求对齐，最后做出来的功能很可能不是你想要的，相当于“白忙活一场”。

总结下来就是：Claude Code擅长“执行”却缺“方向”，OpenSpec擅长“规范”却缺“约束”，Superpowers擅长“纪律”却缺“目标”。而三者协同，就能完美互补，形成一个闭环。

1.2 三者协同的核心优势：分工明确，闭环高效

三者协同的核心逻辑，就是“各司其职、相互配合”，每个工具都承担自己最擅长的角色，最终实现“需求不跑偏、开发有规范、执行有效率”的效果。具体分工如下：

OpenSpec：需求对齐 + 规范沉淀，相当于整个开发流程的“指南针”，是所有开发工作的“真理来源”（Source of Truth）。它负责把模糊的需求（比如老板说的“加个登录功能”）转化为清晰、可执行的规范文档，明确需求目标、范围、边界，确保AI和开发者对需求的理解完全一致，避免返工。

Superpowers：工程纪律 + 流程约束，相当于开发流程的“管理员”。它负责把OpenSpec的规范转化为可落地的执行计划，通过头脑风暴细化技术方案，制定分步任务，强制执行TDD和代码审查，约束AI的开发行为，避免AI自由发挥，确保开发过程符合工程规范。

Claude Code：代码执行 + 工具调用，相当于开发流程的“执行者”。它负责按照Superpowers制定的计划，高效生成代码、执行测试、修复bug，发挥自身代码生成的优势，提升开发效率，同时严格遵循OpenSpec的规范，确保代码符合需求。

三者协同的效应，远不止“1+1+1=3”，而是“1+1+1>3”。它能让AI编程从“无序”变得“有序”，从“低效返工”变得“高效落地”，尤其适合中型以上功能开发、多人协作项目和长期维护项目。

二、环境搭建：一步到位，确保三者无缝协同

要使用这套协同方案，首先需要完成环境搭建，包括安装Claude Code、OpenSpec、Superpowers，以及配置协同工作流。整个过程并不复杂，跟着步骤操作，新手也能快速完成，所有命令行都已整理好，直接复制粘贴即可执行。

2.1 安装Claude Code（核心执行工具）

Claude Code是这套方案的核心执行工具，支持macOS、Linux、WSL和Windows系统，不同系统的安装命令不同，具体如下：

  1. macOS / Linux / WSL系统：打开终端，输入以下命令，回车后等待安装完成即可：

curl -fsSL https://claude.ai/install.sh | bash

  1. Windows PowerShell系统：打开PowerShell（以管理员身份运行），输入以下命令，回车后等待安装完成：

irm https://claude.ai/install.ps1 | iex

  1. 验证安装是否成功：安装完成后，在终端（或PowerShell）中输入以下命令，若能输出版本号，说明安装成功：

claude --version

当前最新版本为2.1.41（2026年3月），如果输出版本号与这个一致，或者更高版本，都可以正常使用。

注意：Claude Code需要订阅才能使用，免费版不支持Claude Code功能，需要开通Claude Pro、Team或Enterprise订阅，订阅后才能正常调用代码生成、工具调用等功能。

2.2 安装OpenSpec（需求规范工具）

OpenSpec是基于Node.js开发的工具，安装前需要确保你的电脑已经安装了Node.js（建议版本16及以上），如果没有安装，可以先去Node.js官网下载安装，再进行OpenSpec的安装。

  1. 全局安装OpenSpec：打开终端（或PowerShell），输入以下命令，回车后等待安装完成：

npm install -g @fission-ai/openspec@latest

  1. 初始化项目：安装完成后，进入你的项目目录（如果没有项目，可以先创建一个，比如mkdir my-project && cd my-project），输入以下命令初始化OpenSpec：

cd your-project（将your-project替换为你的项目名称）

openspec init

  1. 选择AI工具：初始化过程中，会提示你选择AI工具，这里直接选择“Claude Code”，因为我们要实现三者协同，OpenSpec需要和Claude Code联动。

  2. 查看生成的目录结构：初始化完成后，你的项目目录中会新增一个openspec文件夹，里面包含了OpenSpec的核心文件，具体结构如下：

your-project/

├── openspec/

│ ├── specs/ # 当前系统规范（Source of Truth），存储所有需求规范

│ ├── changes/ # 进行中的变更，每个功能开发对应一个文件夹

│ │ └── archive/ # 已完成的变更归档，方便后续追溯

│ ├── AGENTS.md # OpenSpec 指令（AI读取，用于规范AI行为）

│ └── schemas/ # 自定义工作流schema（可选，用于自定义开发流程）

└── …（你的其他代码文件）

这个目录结构不需要手动修改，后续开发过程中，OpenSpec会自动维护这些文件夹和文件。

2.3 安装Superpowers（工程纪律工具）

Superpowers是Claude Code的插件，安装方式有两种，推荐使用第一种（Claude Code插件市场），操作更简单，兼容性更好。

方式1：Claude Code插件市场（推荐）

  1. 首先注册插件市场：打开终端，输入以下命令，回车执行：

/plugin marketplace add obra/superpowers-marketplace

  1. 安装Superpowers插件：注册完成后，输入以下命令，等待安装完成：

/plugin install superpowers@superpowers-marketplace

方式2：官方市场（备选，若方式1安装失败可使用）

输入以下命令，直接从官方市场安装：

/plugin install superpowers@claude-plugins-official

  1. 验证安装是否成功：安装完成后，在终端输入“claude”，启动Claude Code，若看到以下提示，说明Superpowers安装成功：

“You have Superpowers. RIGHT NOW, go read: @/…/getting-started/SKILL.md”

此时，Superpowers已经成功集成到Claude Code中，后续开发过程中会自动激活相关功能。

三、核心工作流：从需求到上线，一步一步落地（实战实操）

这套协同方案的核心工作流，分为5个阶段：OpenSpec需求对齐→Superpowers头脑风暴→Superpowers制定计划→Superpowers+Claude Code执行→OpenSpec归档。我们以“添加用户登录功能”为例，详细拆解每个阶段的操作步骤，所有命令和操作都可直接复制落地，新手也能轻松跟上。

3.1 阶段1：OpenSpec需求对齐——把模糊需求变成清晰规范

需求对齐是开发的第一步，也是最关键的一步，避免后续返工的核心就在于此。这个阶段的核心目标，是用OpenSpec把模糊的需求（比如“加个用户登录功能”）转化为清晰、可执行的规范文档，明确需求目标、范围、边界。

场景：老板说“加个用户登录功能”，没有给出更多细节，我们需要用OpenSpec梳理清楚需求。

步骤1：创建变更提案

  1. 打开终端，进入你的项目目录，输入“claude”启动Claude Code。

  2. 在Claude Code的交互界面中，输入以下指令，创建登录功能的变更提案：

/opsx:propose 用户登录功能

  1. 等待AI生成提案文档：输入指令后，OpenSpec会自动生成一个变更文件夹（openspec/changes/add-user-login/），里面包含4个核心文档，分别是：

openspec/changes/add-user-login/

├── proposal.md # 提案文档，说明为什么做、范围是什么

├── specs/ # 系统行为变更（Delta Spec），明确功能的具体行为

│ └── auth/

│ └── spec.md

├── design.md # 技术方案、架构决策，后续会由Superpowers细化

└── tasks.md # 实现清单（Checklist），后续会由Superpowers细化

其中，proposal.md是核心，它明确了需求的目标、范围、风险，避免AI和开发者理解偏差。以下是proposal.md的示例内容，也是AI自动生成的，我们可以在此基础上修改完善：

用户登录功能提案

背景

当前系统无认证机制，需要添加用户登录功能以保护敏感数据。

目标

  • 支持邮箱 + 密码登录

  • 颁发 JWT Token

  • 支持 Token 刷新

范围（In Scope）

  • 用户登录 API

  • JWT 颁发与验证

  • Token 刷新机制

范围外（Out of Scope）

  • 第三方登录（Google/GitHub）

  • 双因素认证（2FA）

  • 密码找回功能

风险

  • JWT 密钥管理需要安全存储

  • Token 过期策略需要权衡用户体验与安全性

这里有一个关键点：一定要明确写出“范围外（Out of Scope）”的内容，这样可以防止AI“过度帮助”，比如你只需要邮箱密码登录，AI就不会擅自添加第三方登录功能，避免做无用功。

步骤2：Review提案，修改完善

AI生成的提案的可能不是完全符合你的需求，需要我们进行Review和修改，修改方式有两种，可根据自己的习惯选择：

修改方式1：直接编辑文件（适合熟悉终端操作的开发者）

在终端中输入以下命令，打开proposal.md文件进行编辑（使用vim编辑器）：

vim openspec/changes/add-user-login/proposal.md

编辑完成后，按“Esc”键，输入“:wq”，回车保存并退出。

修改方式2：让AI修改（适合新手，更高效）

在Claude Code的交互界面中，直接输入修改指令，比如：

修改 proposal.md，增加"密码强度校验"需求

AI会自动修改proposal.md文件，添加密码强度校验的相关内容，修改完成后会提示你确认。

步骤3：确认提案，进入下一阶段

Review和修改完成后，在Claude Code的交互界面中输入以下指令，确认提案，进入后续的头脑风暴阶段：

提案确认，继续

3.2 阶段2：Superpowers Brainstorming——细化技术方案，解决核心疑问

提案确认后，Superpowers会自动激活brainstorming（头脑风暴）功能，它会以苏格拉底式提问的方式，向你确认技术方案的核心细节，帮你细化设计，避免后续开发中出现方向偏差。

这个阶段，AI会自动问你以下几个核心问题（不同需求会有差异，以下是登录功能的常见问题），你只需要逐一回答即可：

好的，我来帮你细化这个登录功能的设计。先确认几个问题：

  1. 用户存储：用户信息存在哪里？

  • 现有 PostgreSQL 数据库

  • 还是新的用户服务？

  1. 密码加密：使用什么算法？

  • bcrypt（推荐，但慢）

  • argon2（更安全，需要额外依赖）

  1. JWT 过期策略：

  • Access Token 多久过期？（15 分钟？1 小时？）

  • Refresh Token 多久过期？（7 天？30 天？）

  1. 错误处理：

  • 登录失败几次后锁定账户？

  • 是否需要防暴力破解机制？

你可以根据自己的项目实际情况，逐一回答这些问题。比如我们的回答可以是：

  1. 用户存储：现有PostgreSQL数据库（减少系统复杂度）；

  2. 密码加密：bcrypt（cost=12）（成熟稳定，团队熟悉）；

  3. JWT过期策略：Access Token 15分钟，Refresh Token 7天（平衡安全性与用户体验）；

  4. 错误处理：5次失败后锁定账户15分钟，添加IP级别限流（10次/分钟）。

回答完成后，Superpowers会自动将这些决策记录到design.md文件中，生成完整的技术方案设计文档。以下是生成的design.md示例内容：

登录功能设计文档

架构决策

决策 1：用户存储

  • 选择：现有 PostgreSQL 数据库

  • 理由：减少系统复杂度，避免数据同步问题

  • 表结构：新增 users 表

决策 2：密码加密

  • 选择：bcrypt（cost=12）

  • 理由：成熟稳定，团队熟悉，性能可接受

决策 3：JWT 策略

  • Access Token：15 分钟

  • Refresh Token：7 天

  • 理由：平衡安全性与用户体验

决策 4：安全防护

  • 失败锁定：5 次失败后锁定 15 分钟

  • 防暴力破解：IP 级别限流（10 次/分钟）

这个design.md文件会成为后续开发的技术依据，AI在生成代码时，会严格遵循这些决策，避免出现技术方案偏差。

3.3 阶段3：Superpowers Writing Plans——制定可落地的执行任务清单

头脑风暴完成后，Superpowers会自动进入writing-plans（制定计划）功能，将技术方案转化为分步执行的任务清单，每个任务都有明确的目标、验收标准和预计耗时，确保开发过程可追踪、可落地。

这个阶段，Superpowers会自动更新tasks.md文件，生成详细的任务清单。以下是登录功能的tasks.md示例内容，每个任务都清晰明确，可直接执行：

实现任务清单

任务 1：数据库迁移（预计 5 分钟）

  • 创建 users 表迁移文件

  • 字段：id, email, password_hash, created_at, updated_at

  • 添加唯一索引：email

  • 运行迁移并验证

任务 2：密码工具函数（预计 5 分钟）

  • 创建 src/utils/password.ts

  • 实现 hashPassword(password: string): Promise<string>

  • 实现 verifyPassword(password, hash): Promise<boolean>

  • 编写单元测试

任务 3：JWT 工具函数（预计 5 分钟）

  • 创建 src/utils/jwt.ts

  • 实现 generateToken(userId, email): string

  • 实现 verifyToken(token): Payload | null

  • 实现 refreshToken(oldToken): string | null

  • 编写单元测试

任务 4：登录 API（预计 10 分钟）

  • 创建 src/routes/auth.ts

  • 实现 POST /api/login 端点

  • 验证邮箱格式

  • 验证密码

  • 返回 JWT Token

  • 编写集成测试

任务 5：Token 刷新 API（预计 5 分钟）

  • 实现 POST /api/refresh 端点

  • 验证 Refresh Token

  • 颁发新 Access Token

  • 编写集成测试

任务 6：安全加固（预计 10 分钟）

  • 实现登录失败计数

  • 实现账户锁定逻辑

  • 实现 IP 限流中间件

  • 编写压力测试

这个任务清单有三个核心特点，也是我们制定开发计划的关键：

  1. 每个任务耗时短：单个任务预计2-5分钟，最多10分钟，避免任务过大导致难以执行；

  2. 验收标准明确：每个任务都有具体的执行内容，比如“创建src/utils/password.ts”“实现hashPassword函数”，完成与否一目了然；

  3. 包含测试要求：每个任务都要求编写单元测试、集成测试或压力测试，确保代码质量，符合TDD开发规范。

步骤：Review计划，确认执行

和提案一样，AI生成的任务清单也需要我们Review，检查任务顺序是否合理、是否有遗漏的内容。比如发现“任务3（JWT工具函数）”和“任务4（登录API）”顺序反了，应该先实现JWT工具函数，再实现登录API，就可以在Claude Code中输入以下指令修改：

任务 3 和任务 4 顺序反了，应该先实现 JWT 再实现登录 API

AI会自动调整任务顺序，修改完成后，我们在交互界面中输入以下指令，确认计划，进入执行阶段：

Plan 确认，开始执行

3.4 阶段4：Superpowers + Claude Code 执行——自动开发，自动测试，自动修复

计划确认后，Superpowers会自动激活subagent-driven-development（子代理驱动开发）功能，调用Claude Code执行任务，整个过程几乎不需要人工干预，AI会自动完成代码生成、测试、修复bug、代码审查和提交代码。

执行流程如下（以登录功能的6个任务为例），AI会实时在终端中展示执行进度：

[Task 1/6] 数据库迁移

├─ 创建子 Agent（专门负责执行这个任务的AI代理）

├─ 执行任务（创建users表迁移文件、添加字段和索引、运行迁移）

├─ 运行测试 → 通过 ✓（验证迁移是否成功，数据库是否新增users表）

├─ Code Review → 通过 ✓（检查迁移文件语法、字段设计是否合理）

└─ 提交代码（自动执行git add、git commit，提交信息为“完成数据库迁移任务”）

[Task 2/6] 密码工具函数

├─ 创建子 Agent

├─ 执行任务（创建password.ts文件、实现hashPassword和verifyPassword函数）

├─ 运行测试 → 失败 ✗（错误：bcrypt 未安装）

│ └─ 自动修复：安装 bcrypt（AI自动执行npm install bcrypt命令）

├─ 重新测试 → 通过 ✓

├─ Code Review → 通过 ✓（检查函数逻辑、命名规范、代码复杂度）

└─ 提交代码

…（自动继续执行后续任务，直到所有任务完成）

这个阶段有几个非常实用的核心特性，也是这套协同方案的优势所在，值得重点关注：

  1. TDD强制执行：AI会先写测试用例，再写代码，确保代码能够通过测试，从根源上保证代码质量，避免出现“代码能运行但测试不通过”的情况。

  2. 两阶段Review：每个任务完成后，都会进行两次审查，一次是Spec合规性检查（确认代码符合tasks.md和OpenSpec的规范），一次是代码质量检查（检查命名规范、代码结构、复杂度等），确保代码既符合需求，又具备可维护性。

  3. 自动修复：如果测试失败，AI会自动排查问题，尝试修复。比如上面的例子中，测试时发现bcrypt未安装，AI会自动执行npm install bcrypt命令，修复后重新测试，无需人工干预。

  4. 进度追踪：AI会实时更新tasks.md文件中的复选框，完成一个任务就勾选一个，你可以随时打开tasks.md文件，查看开发进度，一目了然。

整个执行过程，你只需要等待即可，AI会自动处理所有细节，遇到问题会自动修复，大大节省开发时间。比如登录功能的6个任务，预计耗时40分钟左右，AI可以全程自动完成，你只需要在最后验收即可。

3.5 阶段5：OpenSpec Archive——归档变更，沉淀规范

当所有任务完成后，我们需要将这次的变更归档，让OpenSpec的规范沉淀下来，方便后续开发时AI读取，避免重复工作，同时也便于项目追溯。

步骤：归档变更

在Claude Code的交互界面中，输入以下指令，完成变更归档：

/opsx:archive

归档后，OpenSpec会自动完成以下3件事，无需人工操作：

  1. 合并Delta Spec：将changes/add-user-login/specs/中的规范，合并到openspec/specs/中，成为系统的正式规范，后续开发新功能时，AI会自动读取这个规范，确保功能之间的兼容性。

  2. 移动文件夹：将changes/add-user-login/文件夹，移动到changes/archive/2026-03-27-add-user-login/（后面的日期会自动替换为你归档的日期），方便后续追溯这次变更的细节。

  3. 更新AGENTS.md：记录本次变更的摘要，包括需求目标、核心决策、完成情况，让团队成员和后续开发人员能够快速了解这次变更的内容。

归档完成后，我们可以打开openspec/specs/auth/spec.md文件，查看合并后的规范，这个规范会成为系统的“真理来源”，后续任何与认证相关的开发，都会遵循这个规范。以下是合并后的spec.md示例内容：

认证规范

目的

用户认证和会话管理。

需求

需求：用户登录

系统应在用户成功登录后颁发 JWT Token。

场景：有效凭证

  • GIVEN 用户有有效凭证

  • WHEN 用户提交登录表单

  • THEN 返回 JWT Token

  • AND 重定向到仪表盘

场景：无效凭证

  • GIVEN 无效凭证

  • WHEN 用户提交登录表单

  • THEN 显示错误消息

  • AND 不颁发 Token

需求：Token 刷新

系统应支持使用 Refresh Token 获取新的 Access Token。

场景：有效的 Refresh Token

  • GIVEN 用户有未过期的 Refresh Token

  • WHEN 调用 /api/refresh

  • THEN 返回新的 Access Token

需求：账户锁定

系统应在连续 5 次登录失败后锁定账户 15 分钟。

场景：多次失败

  • GIVEN 用户连续失败 5 次

  • WHEN 第 6 次尝试登录

  • THEN 返回"账户已锁定"错误

  • AND 15 分钟内不允许登录

到这里，从需求提出到功能上线的完整工作流就结束了。整个过程中，我们只需要做3件事：提出需求、Review提案和计划、确认归档，其余的代码生成、测试、修复、提交等工作，都由AI自动完成，既高效又规范。

四、进阶技巧：让AI协同开发更高效，适配复杂场景

掌握了核心工作流后，我们可以学习一些进阶技巧，适配更复杂的开发场景，比如并行开发多个功能、自定义工作流、验证规范与实现的一致性等，进一步提升开发效率。

4.1 并行开发多个功能，互不冲突

在实际开发中，我们经常需要同时开发多个功能，比如一边开发登录功能，一边开发暗黑模式。OpenSpec的Delta Spec机制支持并行开发，多个变更独立进行，不会相互冲突。

具体操作步骤如下：

  1. 打开终端1，启动Claude Code，开发登录功能：

claude

/opsx:propose 用户登录

然后按照核心工作流，逐步完成提案、头脑风暴、计划、执行等步骤。

  1. 打开终端2，启动Claude Code，开发暗黑模式：

claude

/opsx:propose 暗黑模式

同样按照核心工作流，逐步推进开发。

此时，openspec/changes/目录下会有两个进行中的变更文件夹，结构如下：

openspec/changes/

├── add-user-login/ # 变更 1（登录功能，进行中）

├── add-dark-mode/ # 变更 2（暗黑模式，进行中）

└── archive/ # 已完成归档的变更

两个变更独立进行，不会相互影响，因为每个变更的规范都是Delta Spec（增量规范），归档时OpenSpec会自动将两个变更的规范合并到openspec/specs/中，不会出现冲突。

归档时的操作也很简单，哪个功能先完成，就先执行归档指令：

登录功能完成后，在终端1中输入

/opsx:archive

暗黑模式完成后，在终端2中输入

/opsx:archive

这种方式非常适合多人协作开发，不同的开发者可以负责不同的功能，并行推进，大大提升项目整体进度。

4.2 自定义工作流Schema，适配个性化需求

默认的工作流是“proposal → specs → design → tasks → implement”，但在实际开发中，有些项目可能需要新增阶段，比如“Research（调研）”阶段，先进行市场分析、竞品对比、技术调研，再制定提案。这时，我们可以自定义工作流Schema，适配个性化需求。

具体操作步骤如下：

  1. 创建Schema文件：在openspec/schemas/目录下，创建一个新的文件夹（比如research-first），然后在该文件夹中创建schema.yaml文件：

mkdir openspec/schemas/research-first

vim openspec/schemas/research-first/schema.yaml

  1. 编写Schema内容：将以下内容复制粘贴到schema.yaml文件中，保存退出。这个Schema新增了Research阶段，并且规定了各个阶段的依赖关系（Research完成后才能做Proposal，Proposal完成后才能做specs和design等）：

name: research-first

artifacts:

  • id: research

generates: research.md

requires: []

  • id: proposal

generates: proposal.md

requires: [research]

  • id: specs

generates: specs/

requires: [proposal]

  • id: design

generates: design.md

requires: [proposal]

  • id: tasks

generates: tasks.md

requires: [specs, design]

  1. 激活Schema：在终端中输入以下命令，激活我们自定义的Schema：

openspec schema init research-first

  1. 使用自定义工作流：激活后，创建新的变更时，就会按照我们自定义的流程执行：

/opsx:new 新功能研究

第一步：Research（调研阶段）

/opsx:continue

AI 生成 research.md（包含市场分析、竞品对比、技术调研等内容）

第二步：Proposal（提案阶段）

/opsx:continue

AI 生成 proposal.md（基于调研结果制定提案）

… 依次继续后续阶段

这种方式可以适配不同项目的需求，比如有些项目需要严格的调研流程，有些项目需要新增测试阶段，都可以通过自定义Schema来实现。

4.3 验证实现与Spec的一致性，避免偏离规范

在开发过程中，我们可能会手动修改代码，或者AI在执行过程中出现微小偏差，导致代码实现与OpenSpec的规范不一致。这时，我们可以随时使用OpenSpec的验证功能，检查实现与规范的一致性，及时发现问题并修复。

具体操作步骤如下：

  1. 在Claude Code的交互界面中，输入以下指令，启动验证：

/opsx:verify

  1. 查看验证结果：AI会从三个维度进行验证，分别是完整性、正确性、一致性，然后输出详细的验证结果，示例如下：

验证结果：

[CRITICAL] 任务 4.2 未完成：登录 API 未实现邮箱格式验证

[WARNING] 任务 6.1 部分完成：登录失败计数实现，但未集成到 API

[SUGGESTION] 命名不一致：user_id vs userId

建议：修复 CRITICAL 问题后再归档

三个维度的验证内容说明：

  • 完整性：所有任务是否完成？所有需求是否有对应实现？比如上面的示例中，任务4.2未完成，属于严重问题（CRITICAL），必须修复。

  • 正确性：实现是否符合Spec的意图？边界情况是否处理？比如登录功能是否正确颁发JWT Token，无效凭证是否返回错误消息。

  • 一致性：代码是否符合design.md的决策？命名规范是否统一？比如上面的示例中，命名出现不一致，属于建议性问题（SUGGESTION），可以根据团队规范修改。

根据验证结果，我们可以针对性地修复问题，确保代码实现与规范完全一致，避免归档后出现隐患。

4.4 使用Git Worktrees隔离开发，降低风险

在开发过程中，如果直接在主分支（main）上开发，一旦出现问题，很难回滚，容易影响项目的稳定性。Superpowers的using-git-worktrees功能，可以自动创建隔离分支，隔离开发环境，降低开发风险。

具体操作步骤如下：

  1. 启动新会话：打开终端，输入“claude”启动Claude Code，然后输入以下指令，开始开发新功能（比如用户注册功能）：

帮我开发用户注册功能

  1. Superpowers自动创建隔离分支：输入指令后，Superpowers会自动执行以下操作，无需人工干预：

① 创建Git Worktree：执行git worktree add …/wt-user-register -b feature/user-register命令，创建一个隔离的工作目录（…/wt-user-register），并创建feature/user-register分支。这里需要注意，…/表示工作目录在当前项目目录的上级，避免与主分支工作目录混淆，也可以自定义路径，只需修改命令中的路径参数即可。

② 切换目录：自动切换到创建的隔离工作目录中，所有开发操作都在这个目录中进行，不影响主分支的代码。后续所有的代码生成、测试、提交，都会在这个隔离目录和对应分支中完成，主分支始终保持干净、稳定。

③ 验证测试基线：执行npm test命令，确保当前项目的测试用例全部通过，避免基于有问题的基线进行开发。如果测试失败，Superpowers会提示你先修复主分支的测试问题，再开始新功能开发，从根源上避免引入新的bug。

④ 开始开发：验证通过后，自动进入核心工作流，启动子代理，开始开发用户注册功能，全程自动执行提案、头脑风暴、计划制定、代码生成、测试修复等操作。

这种方式的优势非常明显：

  • 主分支不受影响：开发过程中出现任何问题，都只影响隔离分支，不会污染主分支的代码。比如开发过程中误删文件、引入严重bug，只需删除隔离工作目录和分支，主分支依然保持正常，无需回滚操作。

  • 支持并行开发：多个功能可以分别创建隔离分支，并行开发，互不干扰。比如一个开发者开发用户注册功能，另一个开发者开发用户编辑功能，各自在自己的隔离分支中开发，归档时再合并到主分支，大幅提升开发效率。

  • 便于回滚：如果开发失败，直接删除隔离分支和工作目录即可，不会影响主分支，降低开发风险。同时，隔离分支可以随时暂停开发，后续继续开发时，只需切换到对应的工作目录，即可恢复之前的开发状态。

  1. 开发完成后的操作：当用户注册功能开发完成后，在Claude Code的交互界面中输入以下指令，AI会提供多种后续操作选项：

/opsx:finish

AI 提供选项：

  1. Merge to main（合并到主分支）：直接将隔离分支的代码合并到主分支，适合单人开发、无需团队Review的场景。
  2. Create PR（创建Pull Request）：生成PR描述（包含功能说明、测试结果、规范验证情况），并推送分支到远程仓库，适合团队协作场景，需经过团队Code Review后再合并。
  3. Keep branch（保留分支）：不合并到主分支，保留隔离分支和工作目录，适合功能需要后续迭代优化的场景。
  4. Abandon（放弃开发）：删除隔离分支和工作目录，适合开发失败、无需保留代码的场景。

我们可以根据实际需求选择对应的操作，比如团队协作场景下，选择“2. Create PR”，AI会自动生成规范的PR描述，包含功能实现细节、测试覆盖率、规范验证结果等内容，推送分支到远程仓库后，团队成员可以基于PR进行Code Review，提出修改意见，AI会自动根据意见修复代码，直到PR通过。

补充说明：如果需要切换回主分支开发，只需在终端输入“cd 原项目目录”，即可退出隔离工作目录，回到主分支工作环境，操作简单便捷。

五、避坑指南：新手必看，避开AI协同开发的常见陷阱

在使用这套协同方案的过程中，新手很容易陷入一些误区，导致开发效率下降、代码规范失控，甚至出现项目风险。结合大量实战经验，整理了6个最常见的陷阱，以及对应的解决方法，帮你少走弯路。

5.1 陷阱1：跳过需求对齐，直接让AI写代码

很多开发者急于求成，拿到需求后不使用OpenSpec梳理规范，直接让Claude Code写代码，看似节省了时间，实则会导致后期大量返工——AI写的代码大概率偏离需求，甚至出现逻辑错误，最后需要反复修改，反而耗时更长。

解决方法：严格遵循“需求对齐→计划制定→代码执行”的流程，无论需求多简单，都要先用OpenSpec创建提案，明确需求目标、范围和边界，确认提案后再推进后续开发。哪怕是简单的“修改一个按钮样式”，也可以用OpenSpec快速生成简洁的规范，避免AI自由发挥。

5.2 陷阱2：不Review AI生成的提案和计划

AI生成的提案和计划虽然高效，但并非完美，可能存在需求遗漏、任务顺序不合理、技术方案不符合项目实际等问题。如果直接确认执行，会导致后续开发出现偏差，比如任务顺序颠倒、遗漏关键功能，甚至出现技术选型错误。

解决方法：AI生成提案和计划后，一定要花5-10分钟Review，重点检查3点：① 需求范围是否准确，是否有遗漏或多余的内容；② 技术方案是否符合项目现有架构，是否存在兼容性问题；③ 任务顺序是否合理，是否存在依赖关系错误。发现问题及时让AI修改，避免后期返工。

5.3 陷阱3：过度依赖AI，不关注代码质量

这套协同方案虽然能让AI自动完成代码生成、测试和修复，但不代表我们可以完全放手不管。有些AI生成的代码虽然能通过测试，但可能存在代码冗余、命名不规范、逻辑不够简洁等问题，长期下来会导致项目代码臃肿，难以维护。

解决方法：每个功能开发完成后，花少量时间Review核心代码（比如关键工具函数、API接口逻辑），结合OpenSpec的验证功能，检查代码是否符合规范、逻辑是否合理。同时，定期整理项目代码，优化冗余逻辑，确保代码的可维护性。

5.4 陷阱4：忽略配置安全，导致项目风险

环境配置阶段，很多新手会忽略Claude Code的权限配置，要么不限制AI的操作权限，要么配置错误，导致AI可能执行rm -rf *、sudo等危险命令，删除项目文件、修改系统配置，造成不可挽回的损失。

解决方法：严格按照前文2.4节的配置步骤，修改.claude/settings.json文件，限制AI的操作权限，只允许AI执行openspec、npm、git相关的安全命令。同时，定期检查配置文件，确保没有被意外修改，避免出现安全隐患。

5.5 陷阱5：并行开发时不注意规范冲突

虽然OpenSpec的Delta Spec机制支持并行开发，但如果多个功能涉及同一模块的规范修改，比如两个功能同时修改用户认证逻辑，归档时可能会出现规范冲突，导致后续开发出现问题。

解决方法：并行开发前，先检查各个功能的需求范围，确认是否涉及同一模块的规范修改。如果涉及，及时沟通，明确分工，避免同时修改同一规范；归档时，先归档一个功能，检查规范合并无问题后，再归档另一个功能，确保规范一致性。

5.6 陷阱6：不归档变更，规范无法沉淀

有些开发者开发完成功能后，忘记执行/opsx:archive指令归档变更，导致变更文件夹一直停留在openspec/changes/目录下，规范无法合并到系统正式规范中，后续开发新功能时，AI无法读取之前的规范，导致重复开发、规范不一致。

解决方法：每个功能开发完成后，立即执行/opsx:archive指令归档变更，确保规范沉淀到openspec/specs/中。同时，定期检查openspec/changes/目录，清理未归档的变更，避免目录混乱，便于项目追溯。

五、避坑指南：新手必看，避开AI协同开发的常见陷阱

在使用这套协同方案的过程中，新手很容易陷入一些误区，导致开发效率下降、代码规范失控，甚至出现项目风险。结合大量实战经验，整理了6个最常见的陷阱，以及对应的解决方法，帮你少走弯路。

5.1 陷阱1：跳过需求对齐，直接让AI写代码

很多开发者急于求成，拿到需求后不使用OpenSpec梳理规范，直接让Claude Code写代码，看似节省了时间，实则会导致后期大量返工——AI写的代码大概率偏离需求，甚至出现逻辑错误，最后需要反复修改，反而耗时更长。

解决方法：严格遵循“需求对齐→计划制定→代码执行”的流程，无论需求多简单，都要先用OpenSpec创建提案，明确需求目标、范围和边界，确认提案后再推进后续开发。哪怕是简单的“修改一个按钮样式”，也可以用OpenSpec快速生成简洁的规范，避免AI自由发挥。

5.2 陷阱2：不Review AI生成的提案和计划

AI生成的提案和计划虽然高效，但并非完美，可能存在需求遗漏、任务顺序不合理、技术方案不符合项目实际等问题。如果直接确认执行，会导致后续开发出现偏差，比如任务顺序颠倒、遗漏关键功能，甚至出现技术选型错误。

解决方法：AI生成提案和计划后，一定要花5-10分钟Review，重点检查3点：① 需求范围是否准确，是否有遗漏或多余的内容；② 技术方案是否符合项目现有架构，是否存在兼容性问题；③ 任务顺序是否合理，是否存在依赖关系错误。发现问题及时让AI修改，避免后期返工。

5.3 陷阱3：过度依赖AI，不关注代码质量

这套协同方案虽然能让AI自动完成代码生成、测试和修复，但不代表我们可以完全放手不管。有些AI生成的代码虽然能通过测试，但可能存在代码冗余、命名不规范、逻辑不够简洁等问题，长期下来会导致项目代码臃肿，难以维护。

解决方法：每个功能开发完成后，花少量时间Review核心代码（比如关键工具函数、API接口逻辑），结合OpenSpec的验证功能，检查代码是否符合规范、逻辑是否合理。同时，定期整理项目代码，优化冗余逻辑，确保代码的可维护性。

5.4 陷阱4：忽略配置安全，导致项目风险

环境配置阶段，很多新手会忽略Claude Code的权限配置，要么不限制AI的操作权限，要么配置错误，导致AI可能执行rm -rf *、sudo等危险命令，删除项目文件、修改系统配置，造成不可挽回的损失。

解决方法：严格按照前文2.4节的配置步骤，修改.claude/settings.json文件，限制AI的操作权限，只允许AI执行openspec、npm、git相关的安全命令。同时，定期检查配置文件，确保没有被意外修改，避免出现安全隐患。

5.5 陷阱5：并行开发时不注意规范冲突

虽然OpenSpec的Delta Spec机制支持并行开发，但如果多个功能涉及同一模块的规范修改，比如两个功能同时修改用户认证逻辑，归档时可能会出现规范冲突，导致后续开发出现问题。

解决方法：并行开发前，先检查各个功能的需求范围，确认是否涉及同一模块的规范修改。如果涉及，及时沟通，明确分工，避免同时修改同一规范；归档时，先归档一个功能，检查规范合并无问题后，再归档另一个功能，确保规范一致性。

5.6 陷阱6：不归档变更，规范无法沉淀

有些开发者开发完成功能后，忘记执行/opsx:archive指令归档变更，导致变更文件夹一直停留在openspec/changes/目录下，规范无法合并到系统正式规范中，后续开发新功能时，AI无法读取之前的规范，导致重复开发、规范不一致。

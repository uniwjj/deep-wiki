## Agent 时代，架构师到底应该学什么？

**文章信息**：
- **原文标题**：Agent 时代，架构师到底应该学什么，哪些能力半年后还站得住？
- **来源**：https://mp.weixin.qq.com/s/Litoqoj9WHmV1wCnxybdVQ
- **作者**：架构师（公众号 JiaGouX，主理人若飞）
- **发表时间**：未知

---

**一段话总结**：

文章不追 Agent 框架清单，而是从工程视角拆解 Agent 进入真实生产系统后不可绕过的七项系统能力：上下文管理、工具设计、评估体系、状态持久化、Harness 运行底座、权限与沙箱、多 Agent 边界。核心判断是：框架和模型会换，但"给模型设计一个可靠工作环境"的工程能力会留下来。作者建议团队从一个窄目标、可度量、可回滚的小闭环起步（单 Agent + 5-7 个工具 + 50 条评估样本 + trace），让失败模式拉着架构演进，而不是一开始就搭万能平台。

---

**作者判断**：

作者明确立场：架构师不用追每个新名词，更重要的能力是**分得出哪些会留下来，然后把它们做成团队资产**。他提出"半衰期"概念——模型封装层、CLI 参数、类 Devin 产品半衰期短；协议、状态、沙箱、评估、工具契约半衰期长。前者是工具选择，后者是系统能力，后者才值得沉淀。

作者反复强调几个实务判断：
- "在 Agent 这个领域，filter 比 feed 更重要"
- "Agent 产品的质量很难只按模型版本讨论，更准确的发布单元是模型加 Harness 的组合"
- "默认单 Agent 起步，仍然是更稳的选择"
- "我不太相信'先跑起来，等客户安全审计再补'那套打法"
- 对新框架的态度是"不是否定，只是先排序。能沉淀成系统能力的先做，只能增加表层复杂度的，晚一点也不亏"

---

**核心内容**：

**一、筛选比路线图更耐用**

Agent 领域表层变化太快——模型、上下文窗口、供应商 API 变动都会传导到工具 schema、缓存、超时和成本。作者看新东西用五个问题筛选：半年后它还重要吗？是包装层还是底层能力？能不能接进现有系统？解决什么失败模式？能不能被评估？核心关键词是"半衰期"：模型封装层和 CLI 参数半衰期短；协议、状态、沙箱、评估、工具契约半衰期长。

**二、上下文：当成运行时工作集去管**

作者把上下文分为五层：

- **模型窗口**：当前目标、关键约束、最近观察、下一步决策信息
- **会话状态**：任务计划、已完成动作、待办、错误修复、用户意图
- **文件/数据库**：大对象、日志、代码、历史文档、可回放事实
- **项目规范**：AGENTS.md、README、Runbook、团队约定、验收标准
- **工具层**：检索、分页、验证、写入、权限受控动作

每轮推理前先问：当前需要哪几块信息？哪些只要摘要？大对象是否留在文件/数据库？工具输出给 preview 还是完整展开？历史压缩后任务能否继续？

**三、工具：当成业务接口来设计**

五到十个命名清楚、边界清楚的工具，比二十个互相重叠的工具更稳。工具描述应写成小型接口文档：什么时候该用、什么时候不该用、参数填什么、返回里哪些是关键结果、失败后怎么修、危险动作有没有权限确认。可沉淀的资产不在"接了多少工具"，而在：命名干净、参数边界明确、错误消息可执行、返回格式有上限、大结果可分页、危险操作按权限分级、每次调用进 trace。协议层面，MCP 值得关注，它把工具、资源、能力边界拆开，让 Agent 不靠自定义胶水代码理解外部系统。

**四、评估：少靠体感说对不对**

评估应前置到上线前，哪怕只有五十条样本。评估集应从真实 trace 里长出来，每次失败补样本，每次改 prompt、换模型、改工具、调上下文压缩都跑一遍。评估的价值是把讨论从"感觉好像更聪明"拉回到具体问题：哪类任务退化？哪个工具传错参数？哪次压缩丢关键约束？作者列出不同业务值得看的信号：

- 编程 Agent：代码保留率、测试通过率、返工次数、回滚率
- 客服 Agent：同一问题是否继续追问、是否转人工
- 写作 Agent：段落保留率、改稿轮数
- 数据分析 Agent：SQL 是否被执行、图表是否进入报告
- 运维 Agent：告警是否恢复、误操作率、MTTR 变化

**五、Harness：模型外面那层运行底座**

Harness 不是薄薄的执行壳，而是模型和真实工作之间的运行底座，需处理：上下文组织、工具暴露、状态保存、失败重试、结果验证、权限收口、日志回放、模型切换、成本观察、旧规则清理。分工是：模型负责选下一步，Harness 负责验证、执行、捕获输出、决定反馈、检查点、调度子任务。

模型升级后，Harness 不但要加东西，有时也要删东西：旧 prompt、旧工具包装、旧规则可能是为老模型补短板加上的，模型变强后反而成负担。

**六、状态：让 Agent 跑完以后留下些什么**

状态应拆到窗口之外（文件系统、数据库、session 日志、checkpoint、memory），窗口只留当前推理层。作者对状态分类：

- 当前推理状态 → 模型窗口
- 任务进度 → session、task plan、checkpoint
- 可回放事实 → 文件、数据库、trace
- 团队经验 → AGENTS.md、Skill、Runbook、Checklist
- 用户偏好 → 小型记忆或用户画像

**七、多 Agent：先看隔离，再谈角色**

先画上下文边界和读写边界，再决定要不要拆。能独立调查、只读、结果可汇总的用 Subagent；需要频繁共享中间状态的留在主循环。Subagent 首先省的不是"人手"，而是主上下文的干净程度。默认单 Agent 起步，等上下文压力、串行工具延迟、任务异质性真的出现再上 orchestrator-subagent。

**八、权限和沙箱**

基本功：文件读写范围、命令白名单/审批、网络出口控制、密钥作用域、工具权限分级、危险动作二次确认、沙箱隔离、trace 和审计、回滚路径。安全不能后补。

**九、落地：从一个小闭环开始**

八步最小闭环：
1. 一个明确目标
2. 一个单 Agent 主循环
3. 三到七个边界清楚的工具
4. 一个窗口外状态层
5. 一个沙箱
6. 一套 trace
7. 五十条左右的初始评估样本
8. 一个能回滚的发布方式

第一批失败会告诉团队下一步该补什么：上下文污染就补分页/压缩/Subagent；工具乱用就改工具描述；结果不稳定就补评估样本；任务跑不长就把状态移到窗口外；权限风险高就收工具和沙箱；成本失控就看缓存命中/重试循环/工具返回大小。

**十、可以先放一放的东西**

- 包装层很厚的新框架——迁移成本高过收益
- 复杂记忆系统——多数团队先缺的是上下文分层和 trace，不是长期记忆
- 多 Agent 角色化搭建——先画边界再决定
- 公开 benchmark——替不了内部评估，真实业务里上下文更脏、约束更多
- 模型选型不必每周追——建议季度级重评估，锁定期内围绕当前模型把 Harness 跑稳

## LLM Wiki 六层架构

**文章信息**：
- **原文标题**：我把Karpathy的LLM Wiki改成了六层架构，更适合内容创作者
- **来源**：https://www.toutiao.com/article/7635079798767665698
- **作者**：BookNote
- **发表时间**：2026-05-02 10:35

---

**一段话总结**：

作者以 Karpathy 的 LLM Wiki（raw+wiki 两层架构）为基础，根据自身"内容创作者"的实战需求，将其改造为六层架构：raw（原始素材）、wiki（结构化知识）、index（导航）、memory（Agent 记忆卡片）、people（人物档案）、caogao（创作草稿区）。核心优化在于将高频信息压缩为 memory 记忆卡片（单篇文章 Token 消耗降一半）、人物档案按需调用，解决了原架构"规模一大就崩、Token 成本扛不住、只管写不管读"三个痛点。

---

**作者判断**：

Karpathy 架构本身没问题——它为研究者整理论文设计，不是为内容创作设计。作者作为需要写文章、做内容、跑业务的创作者，知识库需要同时承担素材库、方法论库、生产工具三重角色。核心主张：分层不是越多越好，而是要把高频使用的信息提到最近的地方（memory），把低频的信息降到按需调用的地方（people），让 AI 不用每次都从零推导。

---

**核心内容**：

**Karpathy 原架构的三个痛点**：
1. 规模一大就崩——超过 100 个来源后 wiki 层矛盾累积，无自动纠错
2. Token 成本扛不住——单次写作消耗高达 $17，知识库越大越贵
3. 只管写不管读——存入容易，高效提取使用难

**六层架构设计**：

| 层级 | 内容 | 用途 |
|------|------|------|
| raw/ | 123 篇文章，按 11 个主题分类 | 原始素材，只读 |
| wiki/ | 方法论、框架、案例提炼 | 结构化知识 |
| index/ | 主题索引、仪表盘、知识图谱 | 快速导航 |
| memory/ | 高度压缩的记忆卡片（几百字/条） | 原则/偏好/模板，AI 秒读 |
| people/ | 人物档案（Karpathy、Dan Koe 等） | 按需调用，平时不碰 |
| caogao/ | 创作草稿区 | 写作工作区 |

**关键设计理念**：
- memory 层最大改动：像"写作模型的 system prompt"，高频原则直接存卡片，不需要每次从 wiki 重新推导，单篇文章 Token 消耗降约一半
- people 层得意设计：人物档案单独拎出来，只在明确提到人名时才调用，避免写任何文章都被无关人物资料消耗 Token
- 工作流：写文章时 AI 先读 memory → 只在需要展开时翻 wiki 和 raw → 草稿进 caogao

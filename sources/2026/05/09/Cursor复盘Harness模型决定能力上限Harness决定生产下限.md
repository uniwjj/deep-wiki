## Cursor 复盘 Harness：模型决定能力上限，Harness 决定生产下限

**文章信息**：
- **原文标题**：Cursor 复盘 Harness：模型决定能力上限，Harness 决定生产下限（原标题较长，此为提炼）

- **作者**：架构师（公众号 JiaGouX，主理人若飞）
- **发表时间**：约 2026 年 4 月底至 5 月初

---

**一段话总结**：

文章基于 Cursor 发布的《Continually improving our agent harness》，回答了一个核心问题：当 Harness 开始承重，团队怎么持续运营它？核心论点——Agent 质量的发布单元不是单独的模型，而是"模型 + Harness"的组合。文章从上下文策略（静态→动态）、评估体系（离线 CursorBench + 线上 A/B + Keep Rate + 后续反应判读）、工具错误工程（按模型×工具切片建基线）、模型适配（每个模型需深度定制 Harness）、多 Agent 调度、Harness 运行闭环（十项朴素纪律）六个层面展开。最终结论：Agent 要从"智能体产品"拉回到"软件系统"，补上 SRE 式的评估、观测、回滚和演进能力。

---

**核心内容**：

**一、Harness 当线上系统看**

- Harness 日常在速度、智能、Token 成本三角中找平衡
- 核心难题：一次 Prompt 调整可能省 Token 但让长任务丢失关键上下文；工具描述压缩可能提升缓存但让某模型乱传参
- Cursor 做法：每次 Harness 改动先讲清假设 → 离线评测 → 线上反馈和错误切片 → 决定回滚、调优或清理

**二、上下文：从"提前塞满"到"按需拉取"**

- 2024 年末模型自己选上下文能力弱，Harness 大量兜底：lint 回灌、改读取请求、限制工具调用、预塞目录结构
- 现在保留的静态上下文极少（OS、git 状态、最近文件），更多能力转做动态上下文——通过 MCP 工具、活跃终端、历史对话按需拉取
- 核心转折：模型变强后，Harness 不只加护栏，也要拆护栏。每次模型升级做一次 dead weight 清理——哪些规则是给老模型补短板的？哪些工具包装可以退回通用接口？哪些静态上下文可以改动态？

**三、评估：不只能看跑分**

- 离线 CursorBench：基于真实 Cursor 会话构造，覆盖正确性、代码质量、效率和交互行为
- 线上实验叠两个代理指标：
  - Keep Rate：Agent 生成的代码改动过一段时间后保留率——被用户很快删除就是质量不够好的强信号
  - 后续反应判读：用模型读用户下一句话，语义判断是否满意
- 可按场景迁移：客服 → 是否二次追问/转人工；写作 → 段落保留率；数据分析 → SQL 是否执行/图表是否进报告

**四、工具错误工程**

- 工具失败不止影响当前轮——失败信息留在上下文里继续发酵，污染后续决策
- 错误分类：InvalidArguments / UnexpectedEnvironment / ProviderError / UserAborted / Timeout / 未知错误
- 未知错误默认当 Harness bug 处理，超阈值告警
- 不看完总错误率，按每个工具、每个模型分别建基线切片
- Agent 错误不是均匀分布的——某模型更容易把 patch 格式写错，某工具在大仓库更容易超时，千万不能只看平均值
- Cursor 每周跑一次配了 skill 的 Automation：让模型自己去查日志、找激增问题、创建工单、批量推动修复

**五、换模型不只是改 model id**

- 每个模型需要深度定制 Harness：OpenAI 模型习惯 patch 编辑文件，Anthropic 模型习惯字符串替换，给它不熟悉的形态就多消耗 reasoning token
- 新模型上线流程：从最近的现有模型 Harness 出发 → 离线评测 → 内部 dogfooding → 找困惑点 → 调 Prompt 和工具形态 → 迭代至可发布
- 发布单元是模型-Harness 组合
- 中途切模型本质是状态迁移：旧模型对话历史分布对新模型陌生，缓存也会未命中。建议复杂会话保持在同一个模型里，或通过 subagent 从 fresh context 开工
- 更稳的设计：给每个模型一套可版本化的 Harness 配置

**六、多 Agent 的实质是 Harness 的调度和隔离**

- 系统要知道调度哪个 Agent、怎么按它的优势描述任务、怎么整合结果
- 适合隔离：高输出任务（测试/日志/文档查找）、并行研究独立模块、需要专门工具权限的任务
- 不适合：频繁来回讨论、多阶段共享大量上下文、很小的目标修改
- 先做稳单 Agent 的上下文/工具/验证/错误/发布纪律，再把可隔离的拆出去

**七、第一版 Harness 的十项朴素纪律**

1. 先分任务类型，晚一点再分角色
2. 每类任务都要看结果有没有"留下"（代码是否留在仓库、文档是否进定稿、SQL 是否复用）
3. 离线回归 + 线上反馈一起看
4. 工具错误分类，按模型/工具/任务建基线
5. 上下文预算摆到台面上——窗口只载当前推理，不保存全部历史
6. 模型适配有版本——Prompt 版本、工具 schema 版本、上下文策略、错误基线、切换策略
7. 中途切模型按状态迁移处理
8. Subagent 描述当路由规则写
9. 模型升级后清旧补丁
10. 每次失败反向沉淀成 Harness 设计——约定写进 AGENTS.md，高风险加 hook 拦截，老是提前结束补验证回路

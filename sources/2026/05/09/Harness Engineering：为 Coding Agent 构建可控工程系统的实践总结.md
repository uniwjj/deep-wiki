## Harness Engineering 实践

**文章信息**
- **原文标题**：Harness Engineering：为 Coding Agent 构建可控工程系统的实践
- **来源**：<https://www.toutiao.com/article/7615634442426303028>
- **作者**：Phodal
- **发表时间**：2026-03-10 22:31

**一段话总结**：本文作者 Phodal 提出 Harness Engineering 的核心问题不在模型而在软件工程系统本身——今天的代码结构、开发工具、流程设计都是围绕人类开发者构建的，AI 不具备隐性知识。解决方案是将工程系统从 **Human-first system** 转变为 **Agent-aware system**，通过三层能力模型实现：**系统可读性**（将隐性知识显性化，逐步披露上下文，将工具暴露为 CLI/API）、**防御机制**（收敛 AI 行动空间——静态分析、自动化测试、架构验证不再是质量保障手段，而是系统边界的"物理定律"）、**自动化反馈回路**（AI 每次修改后立即获得反馈，从开发环境静态分析→CI 验证→线上监控形成闭环）。文中以 Stripe Agent Engineering 实践和开源项目 **Routa.js** 为例，展示了多 Agent 协作在同一代码库中遵循统一工程规范的真实模式。开源项目地址：https://github.com/phodal/routa

---

**核心事件**：作者提出 Harness Engineering 能力模型——系统可读性、防御机制、自动化反馈回路三层架构，并以 Stripe 和 Routa.js 为实践案例，论证 AI Coding 的突破往往来自工程系统而非模型能力。

### 问题本质：AI 写代码的速度远超团队控制系统复杂度的能力

AI 可以在几分钟内生成数千行代码，但这些代码不一定理解系统架构、历史设计决策或领域规则。缺乏工程约束时，AI 很容易绕过既有结构、引入新依赖关系、悄悄改变系统边界。

根本原因：问题不在模型，而在软件工程系统本身。今天的工程体系围绕人类开发者构建——假设开发者理解上下文、遵守约定、在评审中发现问题。但 AI 不具备这样的隐性知识。

核心转变：从 **Human-first system** → **Agent-aware system**。工程系统需要被重新设计，使其能被机器理解、调用和验证。

### Harness Engineering 能力模型

```
反馈回路 ▲
    │
系统可读性───防御机制───自动化反馈
```

三个能力逐层递进：系统首先具备可读性让 AI 理解结构；其次提供明确边界限制 AI 行为；最后持续向 AI 提供反馈使其修正行为。

### 能力一：系统可读性——让系统能被 AI 理解

关键转变：将隐性知识显性化。系统结构、架构原则、领域概念需被明确表达并以机器可读形式组织。

- **代码仓库变成可导航的知识地图**：AI 从高层架构文档进入，逐步探索模块结构、领域模型和实现细节
- **逐步披露上下文**：复杂系统信息量大，一次性全部提供反而增加推理不稳定性。根据任务逐步获取所需信息更有效
- **工具接口调整**：将系统能力暴露为 CLI 或 API，使 AI 可通过机器接口访问

### 能力二：防御机制——收敛 AI 的行动空间

AI 生成能力很强，缺乏约束时可能引入重复逻辑、绕过模块边界、创建隐式依赖。系统结构会逐渐腐化。

防御机制 = 系统的"物理定律"：AI 可以自由探索解决方案，但无法突破这些规则。

- 静态分析、自动化测试、架构验证不再只是质量保障手段，而是**系统边界的一部分**
- 通过自动化工具持续执行约束，确保系统在高频修改下保持结构稳定

### 能力三：自动化反馈——让系统持续学习

反馈系统为 AI 提供方向。AI 每完成一次修改都会立即获得反馈：
- 开发环境中的静态分析和测试 → 第一层信号
- 代码提交后 CI 系统验证 → 第二层信号
- 系统上线后监控数据和运行日志 → 长期反馈

当这些信号被统一收集并提供给 AI，软件开发从线性流程演变为**循环系统**——AI 在反馈驱动下不断调整，直到系统达到稳定状态。

### 实践案例

**Stripe Agent Engineering**：工程师通过 Slack 或 CLI 触发任务，Agent 生成实现代码并运行测试，CI 失败时自动进入修复循环，直到代码满足质量标准后生成 PR。人类角色更多是最终评审。

**Routa.js**（开源项目 https://github.com/phodal/routa）：多 Agent 协作平台，多种 AI Agent（Kiro、GitHub Copilot、Augment、Claude Code 等）在同一代码库协作，遵循统一工程规范：
- **可读性**：AGENTS.md 定义编码标准、测试策略、Git 纪律、Co-Author 格式；Specialist 配置明确各 Agent 角色和职责边界
- **防御机制**：双层 Git Hooks（pre-commit 快速检查 + pre-push 结构化错误反馈）；Lint 策略允许部分规则降级为警告以确保多 Agent 协作顺畅
- **反馈回路**：Issue Enricher 准备上下文提出解决方案；Copilot Complete Handler 将 Draft PR 转为 Ready for Review 触发审查；Kanban 系统触发 Agent 任务并要求提供可验证 artifacts

### 软件开发正在变成循环系统

Kief Morris 的观点：未来软件开发运行在两个循环中——"为什么要做"（业务目标，人类主导）和"如何实现"（代码生成/测试/修复/验证，AI 执行）。

人类开发者角色从编写代码转变为设计系统、制定规则、管理工程循环。**人类构建系统杠杆，AI 放大这些杠杆。**

### 结语

AI Coding 的未来不是完全自动化，而是 **Human 与 Agent 共同运行的软件工程系统**。Harness Engineering 正是这一系统的工程基础设施。

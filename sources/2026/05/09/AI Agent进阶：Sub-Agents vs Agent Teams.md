## AI Agent 进阶：Sub-Agents vs Agent Teams

**文章信息**：
- **原文标题**：AI Agent 进阶：Sub-Agents vs Agent Teams，选错架构全盘皆输
- **来源**：https://www.toutiao.com/article/7632524357541085748/
- **作者**：老猿视角
- **发表时间**：2026-04-27

---

**一段话总结**：

文章对比了多 Agent 系统中两种核心协作模式——Sub-Agent（子 Agent）和 Agent Team（Agent 团队）。Sub-Agent 的核心是隔离：每个 Agent 独立运行，有自己的 system prompt、工具集和隔离上下文，只做一件事并返回最终结果，三条铁律是不能互相通信、不能自建 Agent、信息流必须经过父级。Agent Team 的核心是协作：主导 Agent 分配任务，成员 Agent 执行，共享任务层跟踪进度，支持实时双向通信和动态适应。选型铁律：任务独立用 Sub-Agent，任务互相依赖用 Agent Team。作者指出最常见的错误是按角色拆分（Planner → Developer → Tester），应按上下文边界分解。最后给出 5 种实用模式：提示链、路由、并行化、编排-工作器、评估-优化，覆盖 90%+ 场景。

---

**作者判断**：

作者的核心观点是"围绕上下文边界设计，而不是围绕角色设计"。认为很多人第一反应是"加 Agent"，这是最常见也最致命的误区。真正的问题不是"要不要用多 Agent"，而是"这个任务需要哪种协作方式"。作者强调删除一个 Agent 往往比增加一个 Agent 更值钱，能设计出既高效又不失控的 Agent 架构的人才是真正的稀缺资源。

---

**核心内容**：

**一、Sub-Agent：隔离运行的专家（并行执行 + 严格隔离）**

每个 Sub-Agent 拥有：定义角色的系统提示、有限的工具集、完全隔离的上下文、单一边界清晰的任务。完成后只返回最终输出，不返回推理过程或中间步骤。

三条铁律：
1. Sub-Agent 之间**不能互相通信**
2. Sub-Agent 不能自己创建新的 Agent
3. 所有信息流都必须经过父级流转

适用场景：任务之间**完全独立**时——如同时审查安全漏洞、分析性能瓶颈、检测代码风格，三者互不依赖，并行跑完各自返回结论。

**二、Agent Team：持续协作的"项目组"（实时协作 + 上下文共享）**

一个典型 Agent Team 包含：一个主导 Agent（分配任务+综合结果）、多个团队成员 Agent（执行具体任务）、一个共享任务层（跟踪进度和依赖）。

核心能力：前端 Agent 改了 API 接口，后端 Agent 立刻感知并同步调整，不需要人工介入。

**核心区别一览**：

- **Sub-Agent**：侧重执行、无状态、单向/一次性交互、上下文隔离、父级主导
- **Agent Team**：侧重协作、持久化、双向/持续交互、上下文共享、对等通信

**选型铁律：任务独立用 Sub-Agent，任务互相依赖用 Agent Team。**

**三、大多数人踩的坑：按"角色"拆分**

常见做法：Planner Agent → Developer Agent → Tester Agent。问题：每一次交接都产生上下文丢失——实现者不知道规划者的完整思路，测试者不知道实现者的折衷决策，质量在每个边界上往下掉。

**更好的方法：按"上下文边界"分解。** 问自己：这个任务到底需要哪些信息？如果两个任务共享大量深层上下文，就放在同一个 Agent 里。只有当上下文可以被干净地切分时，才拆成不同的 Agent。

> 设计Agent系统的核心，不是划分角色，而是划分信息边界。

**四、真正有用的 5 种模式**

1. **提示链（Prompt Chaining）** — 顺序执行，每一步的输出是下一步的输入
2. **路由（Routing）** — 根据输入类型，把任务分发给最合适的 Agent
3. **并行化（Parallelization）** — 同时运行相互独立的任务
4. **编排-工作器（Orchestrator-Worker）** — 一个主 Agent 分解任务，派发给多个工作 Agent
5. **评估-优化（Evaluator-Optimizer）** — 一个 Agent 生成结果，另一个评估并反馈优化

这 5 种模式覆盖 90%+ 的多 Agent 场景。不要一上来就搞复杂团队协作，从最简单、最符合任务依赖关系的模式开始。

**五、什么时候不要用多 Agent 系统？**

✅ 应该用：需要上下文隔离、有真正并行的任务、需要高度专业化

❌ 应该避免：Agent 间过度依赖彼此实时输出（变成分布式死锁）、协调开销 > 并行收益、任务本身很简单一个 Agent 10 秒搞定

> 架构设计中，删除一个Agent往往比增加一个Agent更值钱。

**终极原则**：围绕上下文边界设计，而不是围绕角色设计。从最简单开始，只有在确实需要隔离、并行或专业化时，才引入多 Agent。

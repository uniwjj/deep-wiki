**核心事件**：LangChain 作者 Vivek Trivedy 发文系统定义了 Agent Harness 的概念与架构，引发开发者社区对"Prompt 工程过时了，现在只需学 Harness 工程"的热议。

### 什么是 Harness？

**Agent = Model + Harness。如果你不是模型，那就是 Harness。**

Harness 本质上是模型之外的一切：代码、配置、各种执行逻辑。模型只是能力的来源，只有通过 Harness 把状态、工具调用、反馈循环和约束机制串起来，它才真正变成一个 Agent。

Harness 的核心组件包括：
- **系统提示词**：定义模型的角色和目标
- **工具、技能、MCP**：模型可调用的外部能力
- **基础设施**：文件系统、沙箱、浏览器等运行环境
- **编排逻辑**：子 Agent、任务拆分、模型路由等
- **钩子/中间件**：压缩、续写、代码检查等确定性流程

用"模型 vs Harness"来划分系统的价值：逼你回答一件事——**模型负责什么？剩下的系统要补什么？**

### 为什么需要 Harness？

有些事我们希望 Agent 能做，但模型本身做不到。模型本质上只是一个**输入→输出的函数**，它不会：
- 在多轮交互中记住状态
- 执行代码
- 获取实时信息
- 操作环境（装依赖、跑程序）

即使是"聊天"这个最基本的能力，也需要 Harness 来维护对话历史、拼接上下文、循环接收输入输出。**你希望 Agent 表现出的能力，最终都要落在 Harness 上实现。**

核心思路不是"调教模型能不能做到"，而是**先想清楚你要它做到什么，再把这些能力一个个补到 Harness 里**。

### 文件系统：持久化存储和上下文管理

模型只能处理当前上下文窗口里的内容。没有文件系统时，用户只能不断复制粘贴喂信息。文件系统是最基础的 Harness 原语之一：

- Agent 有自己的工作空间，可读写数据、代码和文档
- 信息按需加载，而非一股脑塞进上下文
- 中间结果落盘，状态跨会话保留
- **文件本身就是协作接口**：人和多个 Agent 可围绕同一份内容协同

加上 Git 后更完整：记录每一步改动、出问题可回滚、可开分支做不同尝试。

### Bash + 代码执行：通用问题解决工具

传统 Agent 模式是"想一步→调一个工具→看结果→继续循环"，限制在于只能用提前给好的工具。更直接的做法是：**别给一堆工具，直接给它一台能干活的机器**。

有了 Bash + 代码执行能力后：
- 模型可自己写脚本解决问题
- 可临时"造工具"，不依赖预定义接口
- 可组合已有能力拼出新工作流

本质上是从"设计工具列表"升级为提供**通用执行环境**。

### 沙箱环境和工具：安全执行与工作验证

沙箱解决两个核心问题：

**安全性**：隔离执行环境、限制命令和权限、即使出错也被限制在沙箱内部。

**可扩展性**：按需创建环境、多任务并行执行、用完即销毁不留下状态污染。

开箱即用的默认工具包括：语言运行时、Git、测试工具、浏览器等。这些工具让 Agent 能**观察自己的工作结果**（看日志、跑测试、截图页面、检查输出），形成关键闭环：**写代码→运行→观察→修复→再运行**。

### 记忆与搜索：持续学习能力

模型的知识只来自训练权重和当前上下文，没有"记忆"。把新知识放进模型的唯一方式是**通过上下注入**。

常见做法是让 Harness 维护记忆文件（如 `AGENTS.md`）：Agent 运行中写入信息，下次启动时重新加载进上下文，文件更新则上下文随之更新——这是一种朴素的"学习方式"：**写下来→保存→下次继续用**。

对于模型"看不到"的实时信息（新发布的库版本、最新 API 变化等），需要搜索和外部知识获取工具（Web Search、Context7 等 MCP），把这些信息拉进上下文。

### 对抗上下文衰减（Context Rot）

上下文越长，模型表现往往越差——信息变多但有效比例下降、关键线索被淹没、推理能力不稳定。Harness 的核心职责之一就是把**上下文管理工程化**。

**三大手段**：

1. **压缩（Compaction）**：上下文快满时对已有对话做总结，保留关键信息，把细节移出上下文
2. **工具调用卸载（Tool Output Offloading）**：只保留开头+结尾（关键信号），完整内容写入文件系统，需要时再读取——**不让噪音占据上下文**
3. **技能（Skills）与延迟加载**：按需加载能力，先给最小必要信息，需要某个能力时再引入——**对工具和能力的懒加载机制**

### 长期自主执行

模型常见问题：容易提前结束、不擅长拆解复杂任务、跨上下文窗口后工作不连贯。Harness 要解决的核心问题：**如何让工作跨时间、跨上下文持续进行**。

**文件系统 + Git**：长任务产生大量中间结果，必须把工作外部化。文件系统记录当前状态，Git 记录历史和变化，新 Agent 可以快速接手。多 Agent 协作时，这套东西本质是**共享笔记本**。

**Ralph 循环**：拦截"我要结束"信号→重新给干净上下文→让 Agent 继续朝目标推进。关键：**上下文可以重置，但状态不能丢**。

**规划 + 自我验证**：把目标拆成步骤写进文件持续更新（每一步有"参照物"），每做完一步检查（跑测试、看日志、检查输出），失败则把错误信息喂回模型继续修——形成稳定闭环：**执行→检查→反馈→修正**。

### Harness 与模型的共同进化

Claude Code、Codex 等产品是模型和 Harness 同时演化的结果。模型在训练中学习使用 Harness 提供的工具和流程，形成反馈循环：Harness 提供原语→模型学习使用→训练结果反馈回下一代→在相同 Harness 中表现越来越好。

但有副作用：模型可能对特定工具"过拟合"，换了不同 Harness 环境性能可能下降。**最适合你任务的 Harness 不一定是训练时使用的那个。**

实证：Terminal Bench 2.0 测试中，Claude Code 中的 Opus 4.6 得分远低于其他 Harness 中的 Opus 4.6。LangChain 通过优化 Agent 运行环境（文档结构、验证回路、追踪系统），排名从全球第 30 升至第 5，得分从 52.8% 提升到 66.5%。

### 结语

随着模型变强，Harness 中部分功能可能被模型自身吸收。但这并不意味着 Harness 会变得不重要——就像提示工程今天仍有价值一样，**Harness Engineering 很可能仍对构建高效 Agent 起关键作用**。

比喻：Harness 是舞台和幕后控制系统，Agent 是舞台上的演员。无论演员多么出色，没有舞台和规则，他们也难以发挥全部能力。

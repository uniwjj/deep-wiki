## Harness 设计：构建长时间运行的应用开发总结

**文章信息**
- **原文标题**：Harness design for long-running application development
- **来源**：<https://www.anthropic.com/engineering/harness-design-long-running-apps>
- **作者**：Prithvi Rajasekaran（Anthropic Labs 团队）
- **发表时间**：未明确标注

**一段话总结**：本文作者 Prithvi Rajasekaran 从 Anthropic 之前的前端设计（Frontend Design）技能和长时间运行编码 Agent（Coding Agent）Harness 出发，借鉴生成对抗网络（GAN）的思路，设计了一套三 Agent 架构——**规划器（Planner）**、**生成器（Generator）**和**评估器（Evaluator）**——来解决两个核心问题：Agent 自我评估偏高（Self-Evaluation Bias）和长时间任务的连贯性退化（Context Coherence Degradation）。在前端设计实验中，评估器通过四维评分标准（设计质量、原创性、工艺、功能性）驱动生成器迭代 5-15 轮，最终产出从"安全平庸"跃升至"博物馆级空间体验"的设计。在全栈开发实验中，该 Harness 将一句简单提示扩展为 16 功能特性规划，构建的复古游戏制作工具和浏览器端数字音频工作站（DAW）在功能完整性上远超单 Agent 基线（Solo Agent）。文章还讨论了随模型能力提升（如 Opus 4.6）简化 Harness 的迭代过程，结论是：**Harness 的设计空间不会随模型变强而缩小，只会转移**——AI 工程师的工作是不断找到下一个有效的组合。

---

**核心事件**：作者构建了一套三 Agent Harness（规划器 + 生成器 + 评估器），让 Claude 能够在数小时的自主编码会话中产出高质量的全栈应用，解决了 Agent 自我评估偏高和长时间任务退化两大问题。

### 背景：为什么简单实现不够好

作者此前与同事通过 Harness 设计和提示工程显著提升了长时间运行 Agent 的性能，但都碰到了天花板。深入拆解后发现两个持久性问题：

**问题一：上下文退化（Context Degradation）与上下文焦虑（Context Anxiety）**。当上下文窗口（Context Window）接近填满时，模型的连贯性会下降。部分模型还会出现"上下文焦虑"——它们感觉快到上下文限制时会提前收尾工作。之前的解决方案是**上下文重置（Context Reset）**：清空上下文窗口，用结构化的交接工件（Handoff Artifact）携带状态到新的 Agent。这与**压缩（Compaction）**不同——压缩只是总结历史后继续运行同一个 Agent，上下文焦虑仍会持续。在 Sonnet 4.5 上测试时，仅靠压缩不足以支撑长任务表现，上下文重置成了必备手段。但随着 Opus 4.5 的发布，该模型自身的长上下文能力大幅改善，重置不再是必须。

**问题二：自我评估偏差（Self-Evaluation Bias）**。当 Agent 被要求评估自己的产出时，倾向给出高评价——即使产出质量明显平庸。这在主观任务（如设计）上尤为严重，因为没有类似软件测试的客观验证。即使在有可验证结果的任务中，Agent 的判断力也时常不足。**将做事的 Agent 和评判的 Agent 分离**成为解决此问题的关键杠杆。

### 前端设计：将主观质量变成可评分项

作者首先在前端设计领域实验。Claude 默认倾向于生成安全、可预测但视觉上平淡的布局。

关键洞察：
1. 美学虽然不能完全量化，但可以用编码了设计原则和偏好的评分标准来改进——"这个设计美吗？"难以一致回答，但"它是否遵循我们的设计原则？"可以给出具体的评分依据。
2. 将前端生成与前端评分分离，可以创造一个驱动生成器走向更强输出的反馈循环。

作者定义了**四维评分标准**：

- **设计质量（Design Quality）**：设计是否像一个有机整体？颜色、排版、布局、图像等元素是否组合出了独特的气质和身份？
- **原创性（Originality）**：是否有自定义决策的证据，还是模板布局、库默认值和 AI 生成套路？典型失败模式包括未修改的模板组件，以及"AI 味"十足的紫色渐变白色卡片等。
- **工艺（Craft）**：技术执行——排版层级、间距一致性、色彩和谐度、对比度。
- **功能性（Functionality）**：脱离美学的可用性——用户能否理解界面功能、找到主要操作并完成任务？

作者刻意**加大设计质量和原创性的权重**，因为 Claude 默认在工艺和功能性上表现已不错，但在设计质量和原创性上常产出"AI 水文"（AI Slop）。评分标准中"最好的设计是博物馆级别的"这类措辞，会将生成器推向特定视觉风格。

评估器配备了 **Playwright MCP**，可以自主导航页面、截图、仔细审视实现后再给出评估。生成器和评估器之间运行 5-15 轮迭代，每轮生成器根据评估反馈决定是继续优化当前方向，还是彻底转向全新美学。

**典型成果**：在为一个荷兰艺术博物馆生成网站的任务中，前 9 轮迭代产出的是干净的暗色落地页——符合预期但不出彩。第 10 轮时，生成器彻底推翻了原有方案，将网站重新构想为一个**空间体验**：用 CSS 透视渲染的 3D 房间、棋盘格地板、作品自由悬挂在墙上、通过门廊在不同展厅间导航。这是作者从未在单次生成中见过的创意飞跃。

### 扩展到全栈编码：三 Agent 架构

将 GAN 启发的模式应用到全栈开发，映射关系清晰：代码审查和 QA 在软件开发生命周期中扮演与设计评估器相同的结构性角色。

**架构详解**：

**规划器（Planner）**：之前的 Harness 要求用户提供详细规格说明。规划器将这个要求自动化——接收 1-4 句的简单提示，扩展为完整的产品规格。设计原则是**关注产品上下文和高层技术设计，而非详细技术实现**，因为如果规划器在技术细节上出错，错误会级联到下游实现。规划器还被要求寻找将 AI 特性编织进产品规格的机会。

**生成器（Generator）**：延续了之前 Harness 中"一次一个特性"的工作方式，以 **React + Vite + FastAPI + SQLite（后升级为 PostgreSQL）** 技术栈执行开发，使用 Git 进行版本控制。

**评估器（Evaluator）**：使用 Playwright MCP 像用户一样点击运行中的应用，测试 UI 特性、API 端点和数据库状态。评分维度包括：产品深度、功能性、视觉设计和代码质量。每项有硬性阈值，任何一项不达标即判定该轮不通过。

**冲刺合约（Sprint Contract）**机制：每轮冲刺前，生成器和评估器先就"完成的定义"达成一致——生成器提议要构建什么以及如何验证成功，评估器审查确保生成器在构建正确的东西。这弥合了用户故事（User Stories）与可测试实现之间的差距。

Agent 之间通过文件通信：一个 Agent 写文件，另一个读取并回复。

### 实验对比：Solo vs 完整 Harness

**提示词**：Create a 2D retro game maker with features including a level editor, sprite editor, entity behaviors, and a playable test mode.

| Harness 类型 | 运行时长 | 花费 |
|-------------|---------|------|
| Solo（单 Agent） | 20 分钟 | $9 |
| 完整 Harness | 6 小时 | $200 |

**Solo 结果**：初始看起来符合预期，但点击后问题暴露——布局浪费空间、工作流僵化、**核心游戏功能直接坏了**（实体出现在屏幕上但无响应输入）。深入代码发现实体定义和游戏运行时之间的连接断裂，界面上没有任何提示。

**Harness 结果**：规划器将一句提示扩展为跨 10 轮冲刺的 16 功能规格说明，并读取了前端设计技能来创建视觉设计语言。产出的应用在精细度和流畅度上立即显现出优势。最关键的区别在于**游戏模式能真正运行**——实体可移动、可操作。评估器在每轮冲刺中严格执行合约标准，捕获了大量具体 bug，包括：

- 矩形填充工具只在拖拽起止点放置瓦片而非填充区域
- 删除键处理逻辑要求 selection 和 selectedEntityId 同时成立，但点击实体只设置其中一个
- FastAPI 路由顺序错误导致 `reorder` 被匹配为 `frame_id`

### Harness 迭代：随模型能力精简

每个 Harness 组件都编码了一个关于"模型自身做不到什么"的假设，这些假设值得压力测试。核心原则来自 Anthropic 自己的博文 [Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)：**找到最简单的可行方案，只在必要时增加复杂度**。

**Opus 4.6 发布后**的精简方向："（Opus 4.6）规划更仔细，能更长时间维持 Agent 任务，在大型代码库中操作更可靠，拥有更好的代码审查和调试能力来自我纠错。"——这正是 Harness 之前需要补充的能力。

**移除冲刺结构**：Opus 4.6 的原生能力足以在不分解任务的情况下长时间连贯工作。保留规划器和评估器（各自继续提供明显价值），但评估器从每轮冲刺评估改为**单次终评**。

**核心洞察**：评估器的价值不是固定的——当任务处于模型可靠完成的边界内时，它是不必要的开销；当任务超出边界时，它仍然提供真实提升。**边界随模型进步外移，但永远存在**。

### 更新 Harness 的实验结果

**提示词**：Build a fully featured DAW in the browser using the Web Audio API.

| 阶段 | 时长 | 花费 |
|------|------|------|
| 规划器 | 4.7 分钟 | $0.46 |
| 构建（第 1 轮） | 2 小时 7 分钟 | $71.08 |
| QA（第 1 轮） | 8.8 分钟 | $3.24 |
| 构建（第 2 轮） | 1 小时 2 分钟 | $36.89 |
| QA（第 2 轮） | 6.8 分钟 | $3.09 |
| 构建（第 3 轮） | 10.9 分钟 | $5.88 |
| QA（第 3 轮） | 9.6 分钟 | $4.06 |
| **总计** | **3 小时 50 分钟** | **$124.70** |

生成器在没有冲刺分解的情况下连贯运行了超过两小时。QA Agent 仍然捕获了真实的功能缺失：剪辑无法在时间线上拖动、缺少乐器 UI 面板、效果可视化仅为数值滑块而非图形化显示。

最终产出是一个具备完整编排视图、混音器和传输控制的浏览器端音乐制作程序。通过提示，Agent 自主设置了调速和调性、铺设旋律、构建鼓轨、调整混音器电平并添加混响——从头到尾用工具驱动了一次完整的歌曲制作流程。

### 未来方向

随着模型持续改进，可以预期它们能处理更长时间、更复杂的任务。但 Harness 的设计空间不会缩小——只会转移。开发者应该：

1. 用实际构建的模型做实验，阅读其在真实问题上的轨迹（Traces）
2. 在复杂任务中探索任务分解和专门化 Agent
3. 新模型落地时重新审视 Harness——**剥离不再承重的组件，添加此前不可能实现的新能力**

**作者判断**：有趣 Harness 组合的空间不会随模型变强而缩小，只会转移——AI 工程师的工作就是不断找到下一个有效的组合。

## Agent Harness 综述：同一个模型，为什么做出来的 Agent 差这么远

**文章信息**：
- **原文标题**：Agent Harness 综述（原标题较长，此为提炼）
- **来源**：https://mp.weixin.qq.com/s/h49UiGERvz8BMkMW0_4Gwg
- **作者**：架构师（公众号 JiaGouX，主理人若飞）
- **发表时间**：约 2026 年 4-5 月

---

**一段话总结**：

文章基于 Akshay Pachaar 的《The Anatomy of an Agent Harness》，将 Harness 拆解为六个承重层（主循环、工具系统、上下文与记忆、状态与检查点、权限与错误护栏、验证与纠偏），沿一轮完整执行链路跑通各层的工程取舍。核心论点：Harness 是在模型和真实交付之间补上一套可运行、可恢复、可验证、可治理的软件系统。Prompt 管怎么表达，Context 管模型看到什么，Harness 管系统怎么跑。同一模型只换 Harness 结果可能差一个量级（LangChain 从 TerminalBench 前 30 名外拉到第 5）。文章还强调 AGENTS.md、Spec、Skills 也是 Harness 的一部分——把团队经验从人脑搬到系统层，缩小"必须靠模型临场发挥"的面积。

---

**核心内容**：

**一、三层概念关系**

| 层级 | 职责 | 类比 |
|------|------|------|
| Prompt Engineering | 怎么对模型说 | 指令 |
| Context Engineering | 让模型看到什么 | 工作台 |
| Harness Engineering | 系统怎么运行/持久化/验证/兜底 | 操作系统 |

核心类比：裸 LLM 是一颗没有 RAM、磁盘、I/O 的 CPU——上下文窗口是内存，外部数据库是磁盘，工具集是设备驱动，Harness 是让这台机器持续跑起来的调度执行校验保护机制。

**二、六个承重层**

**1. 主循环**：心脏。表面是 while loop，难点在谁控制循环、何时终止、出错后怎么回来。最先暴露的问题不是"模型没想明白"，而是循环失控——无限转圈、提前收尾、误把中间结果当最终结果。

**2. 工具系统**：手。不只注册函数名，还要管注册方式、参数校验、执行隔离、结果回写。Claude Code 分六类（文件操作/搜索/执行/Web/代码智能/子 Agent），OpenAI SDK 分函数/托管/MCP 工具。模型知道能调什么只是起点，能不能在合适时候带着正确参数调用、失败后恢复，才是 Harness 的事。

**3. 上下文与记忆**：该记什么、什么时候记、什么时候删。Claude Code 三层记忆——始终加载的轻量索引（~150 字符/条）、按需拉取的详细主题文件、仅搜索访问的原始会话记录。原则：记忆是线索不是真相，关键动作前必须回到真实文件确认。

**4. 状态与检查点**：长任务没有不行。系统需知道做到哪、失败从哪恢复、哪些中间产物值得保留。实现可以是结构化状态对象、git commit、进度文件或任务日志的组合。

**5. 权限、错误与安全**：Agent 无权限控制等于事故放大器。模型负责提出动作，系统决定能不能做、要不要确认、失败后重试/报错/终止。生产环境高风险动作必须有边界。

**6. 验证与纠偏**：Demo 和生产的分水岭。给模型外部反馈回路——测试、lint、类型检查、截图、端到端操作。Claude Code 创始人说：让模型验证自己的方式，质量提升 2-3 倍。没有验证，Harness 只是"更快地产出错结果"。

**三、一次完整循环（七步）**

组装输入（系统 prompt + 工具定义 + 记忆 + 会话状态 + 当前任务）→ 模型推理 → 分类输出（文本 or tool call）→ 执行工具（校验→权限→并发读/串行改/拒绝）→ 回写结果（Observation 含成败信息）→ 更新状态（历史/检查点/记忆/压缩触发）→ 决定是否继续（完成/预算耗尽/最大轮数/用户中断/安全拦截）

**四、为什么 2026 年都在谈 Harness**

两个信号：①同一模型只换外围基础设施，LangChain 从 TerminalBench 2.0 前 30 名外拉到第 5；搜索优化 Harness 本身拿到 76.4% 通过率。②长任务误差急速累积——10 步流程每步 99% 成功，全链路也只有 ~90%。

趋势：Vercel v0 删掉 80% 工具表现反而更好；Claude Code 懒加载实现 95% 上下文缩减。2024 卷 Prompt → 2025 补 Context → 2026 收到 Harness。

**五、Harness 设计取舍**

- 单/多 Agent：先把单 Agent 做通，再把明显超载的职责拆出去
- ReAct vs Plan-Execute：任务越长、代价越高、回滚越麻烦，越该把计划层显式拿出来
- 上下文管理：信号密度比窗口大小重要
- 验证交给谁：自评快但容易放过自己，外部验证在可实测的地方必须补
- Harness 厚薄：模型弱时补结构，模型强后删 dead weight。Manus 半年重建五次，每次做减法
- 工具数：超 10 个且职责重叠时调用质量明显下降

**六、AGENTS.md / Spec / Skills 也是 Harness**

- AGENTS.md：仓库默认规则——怎么读、标准入口、联动检查
- Spec：任务完成标准——交付什么、边界在哪、什么叫做完
- Skills：可复用操作规程——高频稳定流程、检查清单、脚手架
共同目标：把知识从聊天记录搬出来、把规则从口口相传搬出来、把经验从资深工程师脑子搬出来、把验证从"我觉得差不多"搬到系统动作里

**七、从最小可用开始搭**
1. 单 Agent 主链路先做稳（工具调用→回写→失败不跳过→结束条件明确）
2. 工具数量收紧（只暴露当前步骤最小工具集）
3. 记忆是提示不是事实（先做索引/摘要/决策记录，关键动作二次确认）
4. 验证尽早外移（测试/lint/类型检查/截图/API 响应）
5. 状态和恢复点显式写（假设必然失败，有 checkpoint 就是恢复问题）
6. 高风险动作从"能力"拆出（模型提议动作，系统决定边界）

核心金句：如果你不是模型，你就在做 Harness。

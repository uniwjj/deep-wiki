# 长时运行应用开发的 Harness 设计

**原文链接**: https://www.anthropic.com/engineering/harness-design-long-running-apps

**作者**: Prithvi Rajasekaran (Anthropic Labs 团队成员)

---

## 一句话总结

本文提出了一种受生成对抗网络（GANs）启发的三 Agent 架构（规划器 Planner、生成器 Generator、评估器 Evaluator），通过职责分离和迭代反馈循环，解决了长时运行 AI Agent 应用中的上下文漂移（Context Drift）和自我评估偏差（Self-evaluation Bias）问题，实现了复杂全栈应用的自主开发，实验表明该架构相比单 Agent 方案在功能完整性和代码质量上有显著提升。

## 完整翻译

### 引言

在过去几个月里，我一直在研究两个相互关联的问题：让 Claude 生成高质量的前端设计，以及让它在没有人工干预的情况下构建完整的应用程序。这项工作源于我们之前在**前端设计技能（frontend design skill）**和**长时运行编码 Agent Harness**上的努力，我和我的同事们通过提示工程（prompt engineering）和 Harness 设计显著提升了 Claude 的性能——但两者最终都遇到了瓶颈。

为了突破瓶颈，我寻找能够跨越两个截然不同领域（一个由主观品味定义，另一个由可验证的正确性和可用性定义）的新型 AI 工程方法。受**生成对抗网络（Generative Adversarial Networks, GANs）**启发，我设计了一个包含生成器（generator）和评估器（evaluator）的多 Agent 结构。构建一个能够可靠评分——且具有品味——的评估器，意味着首先要开发一套标准，将"这个设计好吗？"这样的主观判断转化为具体、可评分的术语。

然后我将这些技术应用于长时运行的自主编码，借鉴了我们早期 Harness 工作的两个教训：将构建分解为可处理的块，以及使用结构化产物（artifacts）在会话之间传递上下文。最终结果是一个**三 Agent 架构——规划器（planner）、生成器（generator）和评估器（evaluator）**——能够在多小时的自主编码会话中生成丰富的全栈应用程序。

---

### 为什么朴素实现会失败

我们之前已经证明，Harness 设计对长时运行 Agent 编码的有效性有重大影响。在早期的实验中，我们使用初始化 Agent（initializer agent）将产品规格分解为任务列表，编码 Agent（coding agent）一次实现一个功能，然后将产物传递给下一个会话以携带上下文。更广泛的开发者社区也得出了类似的见解，比如使用钩子或脚本让 Agent 保持连续迭代循环的"Ralph Wiggum"方法。

但仍有一些问题持续存在。对于更复杂的任务，Agent 仍然倾向于随着时间的推移而偏离轨道。在分解这个问题时，我们观察到 Agent 执行此类任务时存在两种常见的失败模式。

#### 失败模式 1：上下文漂移与上下文焦虑

首先是模型在冗长任务中随着上下文窗口填满而失去连贯性（参见我们关于**上下文工程（context engineering）**的文章）。一些模型还表现出"**上下文焦虑（context anxiety）**"，即当它们接近自己认为的上下文限制时，开始过早地结束工作。

**上下文重置（context resets）**——完全清空上下文窗口并启动一个新的 Agent，结合结构化交接（handoff）来携带前一个 Agent 的状态和下一步——解决了这两个问题。

这与**压缩（compaction）**不同，在压缩中，对话的较早部分被就地总结，以便同一个 Agent 可以在缩短的历史记录上继续。虽然压缩保持了连续性，但它没有给 Agent 一个干净的起点，这意味着上下文焦虑仍然可能持续。重置提供了一个干净的起点，代价是交接产物必须有足够的状态让下一个 Agent 干净地接手工作。

在我们早期的测试中，我们发现 Claude Sonnet 4.5 表现出足够强烈的上下文焦虑，仅靠压缩不足以实现强大的长任务性能，因此上下文重置成为 Harness 设计的关键。这解决了核心问题，但增加了编排复杂性、令牌开销和每次 Harness 运行的延迟。

#### 失败模式 2：自我评估偏差

第二个问题，我们之前没有解决，是**自我评估（self-evaluation）**。当被要求评估他们产生的工作时，Agent 倾向于自信地赞扬工作——即使对于人类观察者来说，质量明显平庸。这个问题在主观任务（如设计）中尤为明显，因为没有相当于可验证软件测试的二元检查。布局是否感觉精致或通用是一个判断问题，Agent 在评分自己的工作时可靠地偏向积极。

然而，即使在确实有可验证结果的任务上，Agent 有时仍会表现出阻碍其在完成任务时表现的糟糕判断。将执行工作的 Agent 与评判它的 Agent 分离，证明是解决这个问题的有力杠杆。这种分离本身并不能立即消除那种宽容；评估器仍然是一个倾向于对 LLM 生成的输出慷慨的 LLM。但调整一个独立的评估器使其持怀疑态度，比让生成器批评自己的工作要容易得多，一旦存在外部反馈，生成器就有具体的东西可以迭代。

### 前端设计：让主观质量可评分

我从前端设计开始实验，那里自我评估问题最为明显。没有任何干预的情况下，Claude 通常倾向于安全、可预测的布局，技术上功能正常但视觉上平淡无奇。

两个见解塑造了我为前端设计构建的 Harness。首先，虽然美学不能完全简化为一个分数——而且个人品味总是会有所不同——但可以通过编码设计原则和偏好的评分标准来改进。"这个设计漂亮吗？"很难一致地回答，但"这是否遵循我们良好设计的原则？"给了 Claude 具体可以评分的东西。其次，通过将前端生成与前端评分分离，我们可以创建一个反馈循环，推动生成器产生更强的输出。

考虑到这一点，我编写了四个评分标准，在提示中同时给生成器和评估器 Agent：

| 标准 | 描述 |
|-----|------|
| **设计质量（Design quality）** | 设计是否感觉像一个连贯的整体，而不是部分的集合？在这方面表现强劲意味着颜色、排版、布局、图像和其他细节结合起来创造独特的情绪和身份。 |
| **原创性（Originality）** | 是否有自定义决策的证据，还是模板布局、库默认值和 AI 生成的模式？人类设计师应该能识别出有意的创意选择。未修改的库存组件——或 AI 生成的明显迹象，如白色卡片上的紫色渐变——在这方面失败。 |
| **工艺（Craft）** | 技术执行：排版层次、间距一致性、色彩和谐、对比度比率。这是能力检查而非创造力检查。大多数合理的实现默认在这方面表现良好；失败意味着基础被破坏。 |
| **功能性（Functionality）** | 独立于美学的可用性。用户能否理解界面的功能，找到主要操作，并在不猜测的情况下完成任务？ |

我强调设计质量和原创性高于工艺和功能性。Claude 默认在工艺和功能性上得分很高，因为所需的技术能力往往自然来到模型。但在设计和原创性上，Claude 经常产生充其量是平淡的输出。标准明确惩罚高度通用的"AI 垃圾"模式，通过更重视设计和原创性，它推动模型承担更多的美学风险。

我使用**少样本示例（few-shot examples）**和详细的分数分解来校准评估器。这确保了评估器的判断与我的偏好一致，并减少了跨迭代的分数漂移。

我在 **Claude Agent SDK** 上构建了这个循环，使编排变得简单。生成器 Agent 首先基于用户提示创建 HTML/CSS/JS 前端。我给评估器 **Playwright MCP**，让它在评分每个标准并撰写详细评论之前直接与实时页面交互。在实践中，评估器会自行导航页面，截图并仔细研究实现，然后产生评估。该反馈流回生成器作为下一次迭代的输入。我每代运行 5 到 15 次迭代，每次迭代通常推动生成器朝着更独特的方向发展，因为它响应评估器的批评。因为评估器正在积极导航页面而不是对静态截图评分，每个周期都需要实际的挂钟时间。完整运行延伸到四个小时。我还指示生成器在每次评估后做出战略决策：如果分数趋势良好，则完善当前方向；如果方法不起作用，则转向完全不同的美学。

在运行中，评估器的评估在迭代中改进，然后趋于平稳，仍有剩余空间。一些生成是增量完善的。其他在迭代之间进行了急剧的美学转变。

标准的措辞以我没有完全预料到的方式引导生成器。包含像"最好的设计是博物馆级别的"这样的短语推动设计朝着特定的视觉收敛，表明与标准相关的提示直接塑造了输出的特征。

虽然分数通常在迭代中提高，但模式并不总是 cleanly 线性的。后来的实现作为一个整体往往更好，但我经常看到我喜欢中间迭代而不是最后一个的情况。实现复杂性也往往在轮次中增加，生成器响应评估器的反馈而寻求更雄心勃勃的解决方案。即使在第一次迭代中，输出也比完全没有提示的基线明显更好，表明标准和相关语言本身在评估器反馈导致进一步细化之前就将模型从通用默认值引开。

在一个显著的例子中，我提示模型为荷兰艺术博物馆创建一个网站。到第九次迭代，它产生了一个干净、深色主题的虚构博物馆落地页。页面视觉上很精致，但大体符合我的预期。然后，在第十个周期，它完全废弃了这种方法，将网站重新想象为空间体验：一个带有棋盘格地板的 3D 房间，用 CSS 透视渲染，艺术品以自由形式的位置挂在墙上，基于门的画廊房间之间的导航而不是滚动或点击。这是我以前从未在单次生成中见过的那种创造性飞跃。

### 扩展到全栈编码

有了这些发现，我将这种受 GAN 启发的模式应用于全栈开发。生成器-评估器循环自然地映射到软件开发生命周期，代码审查和 QA 在这里扮演着与设计评估器相同的结构角色。

#### 架构

在我们早期的**长时运行 Harness**中，我们通过初始化 Agent、一次处理一个功能的编码 Agent 和会话之间的上下文重置解决了连贯的多会话编码问题。上下文重置是一个关键解锁：Harness 使用了表现出上述"上下文焦虑"倾向的 Sonnet 4.5。创建一个在上下文重置中表现良好的 Harness 是保持模型专注于任务的关键。Opus 4.5 在很大程度上自行消除了这种行为，所以我能够完全从这个 Harness 中删除上下文重置。Agent 作为整个构建的一个连续会话运行，**Claude Agent SDK** 的自动压缩处理沿途的上下文增长。

为此，我在原始 Harness 的基础上构建了一个**三 Agent 系统**，每个 Agent 解决我在之前运行中观察到的特定差距。系统包含以下 Agent 角色：

**规划器（Planner）**：我们之前的长时运行 Harness 要求用户提供详细的前期规格。我想自动化这一步，所以创建了一个规划器 Agent，它接受简单的 1-4 句提示并将其扩展为完整的产品规格。我提示它在范围上要雄心勃勃，专注于产品上下文和高级技术设计而不是详细的技术实现。这种强调是因为担心如果规划器试图预先指定细粒度的技术细节并出错，规格中的错误会级联到下游实现中。约束要交付的产物并让 Agent 在工作时找出路径似乎更明智。我还要求规划器在其产品规格中寻找编织 AI 功能的机会。

**生成器（Generator）**：早期 Harness 中一次处理一个功能的方法对范围管理很有效。我在这里应用了类似的模型，指示生成器以冲刺（sprints）方式工作，从规格中一次挑选一个功能。每个冲刺使用 React、Vite、FastAPI 和 SQLite（后来的 PostgreSQL）技术栈实现应用程序，生成器被指示在每个冲刺结束时自我评估其工作，然后交给 QA。它还有用于版本控制的 git。

**评估器（Evaluator）**：早期 Harness 的应用程序通常看起来令人印象深刻，但在你实际尝试使用时仍有真正的错误。为了捕捉这些，评估器使用 **Playwright MCP** 像用户一样点击运行的应用程序，测试 UI 功能、API 端点和数据库状态。然后它根据发现的错误和一套以前端实验为模型的标准对每个冲刺进行评分，这里 adapted 以涵盖产品深度、功能性、视觉设计和代码质量。每个标准都有一个硬阈值，如果任何一个低于它，冲刺就失败，生成器会得到关于出错原因的详细反馈。

在每个冲刺之前，生成器和评估器协商一个**冲刺合同（sprint contract）**：在编写任何代码之前，就那一块工作的"完成"样子达成一致。这存在是因为产品规格是故意高层次的，我想要一个步骤来弥合用户故事和可测试实现之间的差距。生成器提议它将构建什么以及如何验证成功，评估器审查该提议以确保生成器在构建正确的东西。两者迭代直到达成一致。

通信通过文件处理：一个 Agent 写入文件，另一个 Agent 读取它并在该文件内响应或用前一个 Agent 会读取的新文件响应。然后生成器根据商定的合同构建，然后将工作交给 QA。这使工作忠实于规格，而不会过早过度指定实现。

#### 运行 Harness

对于 Harness 的第一个版本，我使用了 **Claude Opus 4.5**，针对完整 Harness 和单 Agent 系统运行用户提示进行比较。我使用 Opus 4.5 是因为这是我开始这些实验时我们最好的编码模型。

我编写了以下提示来生成一个复古视频游戏制作器：

> 创建一个 2D 复古游戏制作器，功能包括关卡编辑器、精灵编辑器、实体行为和可玩的测试模式。

下表显示了 Harness 类型、运行时长和总成本：

| Harness | 时长 | 成本 |
|---------|------|------|
| 单 Agent | 20 分钟 | $9 |
| 完整 Harness | 6 小时 | $200 |

Harness 贵了 20 多倍，但输出质量的差异立即显而易见。

我期望一个界面，我可以在其中构建关卡及其组成部分（精灵、实体、瓦片布局），然后点击播放来实际玩关卡。我首先打开单 Agent 运行的输出，初始应用程序似乎符合这些预期。

然而，当我点击时，问题开始出现。布局浪费了空间，固定高度的面板让大部分视口空着。工作流程很僵化。尝试填充关卡首先提示我创建精灵和实体，但 UI 中没有任何东西引导我走向那个顺序。更重要的是，实际游戏坏了。我的实体出现在屏幕上，但没有任何东西响应输入。深入代码发现实体定义和游戏运行时之间的连接断了，没有表面迹象表明问题在哪里。

评估完单 Agent 运行后，我将注意力转向 Harness 运行。这个运行从相同的一句话提示开始，但规划步骤将提示扩展为跨越十个冲刺的 16 功能规格。它远远超出了单 Agent 运行尝试的范围。除了核心编辑器和游戏模式外，规格还要求精灵动画系统、行为模板、音效和音乐、AI 辅助精灵生成器和关卡设计师，以及带有可分享链接的游戏导出。我给规划器访问我们的**前端设计技能**的权限，它阅读并使用它为应用程序创建视觉设计语言作为规格的一部分。对于每个冲刺，生成器和评估器协商定义冲刺具体实现细节和将被测试以验证完成的可测试行为的合同。

应用程序立即显示出比单 Agent 运行更多的精致和流畅。画布使用了完整的视口，面板大小合理，界面具有一致的视觉身份，跟踪规格中的设计方向。单 Agent 运行中的一些笨拙仍然存在——工作流程仍然没有明确表明你应该在尝试填充关卡之前构建精灵和实体，我不得不通过摸索来弄清楚。这读起来像是基础模型产品直觉中的差距，而不是 Harness 旨在解决的问题，尽管它确实表明 Harness 内针对性迭代可以进一步提高输出质量的地方。

在编辑器中工作时，新运行相对于单 Agent 的优势变得更加明显。精灵编辑器更丰富、功能更完整，有更干净的工具调色板、更好的颜色选择器和更可用的缩放控制。

因为我要求规划器在其规格中编织 AI 功能，应用程序还带有内置的 Claude 集成，让我可以通过提示生成游戏的不同部分。这显著加快了工作流程。

最大的差异在游戏模式中。我实际上能够移动我的实体并玩游戏。物理有一些粗糙的边缘——我的角色跳上平台但最终与它重叠，这在直觉上感觉不对——但核心东西起作用了，这是单 Agent 运行没有做到的。在移动一会儿后，我遇到了 AI 游戏关卡构建的一些限制。有一堵大墙我无法跳过，所以我被困住了。这表明 Harness 可以处理以进一步改进应用程序的一些常识性改进和边缘情况。

阅读日志，很明显评估器使实现与规格保持一致。每个冲刺，它都遍历冲刺合同的测试标准，通过 Playwright 运行应用程序，针对任何偏离预期行为的行为提交错误。合同是细粒度的——仅冲刺 3 就有 27 个涵盖关卡编辑器的标准——评估器的发现足够具体，无需额外调查即可采取行动。下表显示了我们的评估器识别的一些问题示例：

| 合同标准 | 评估器发现 |
|---------|-----------|
| 矩形填充工具允许点击拖动用选定的瓦片填充矩形区域 | **失败** — 工具只在拖动起点/终点放置瓦片，而不是填充区域。fillRectangle 函数存在但在 mouseUp 上没有正确触发。 |
| 用户可以选择并删除放置的实体生成点 | **失败** — LevelEditor.tsx:892 的删除键处理程序需要同时设置 selection 和 selectedEntityId，但点击实体只设置 selectedEntityId。条件应该是 selection \|\| (selectedEntityId && activeLayer === 'entity')。 |
| 用户可以通过 API 重新排序动画帧 | **失败** — PUT /frames/reorder 路由定义在 /{frame_id} 路由之后。FastAPI 将 'reorder' 匹配为 frame_id 整数并返回 422："unable to parse string as an integer"。 |

让评估器达到这个水平需要工作。开箱即用，Claude 是一个糟糕的 QA Agent。在早期运行中，我看到它识别合法问题，然后说服自己决定它们不是大问题并批准工作 anyway。它还倾向于表面测试，而不是探测边缘情况，所以更微妙的错误经常溜过。调整循环是阅读评估器的日志，找到其判断与我的判断分歧的例子，并更新 QA 的提示以解决这些问题。在评估器以我找到的合理方式评分之前，这花了这个开发循环的几轮。即便如此，Harness 输出显示了模型 QA 能力的极限：小布局问题、在某些地方感觉不直观的交互，以及评估器没有彻底测试的更深层嵌套功能中未发现的错误。显然还有更多的验证空间可以通过进一步调整来捕捉。但与单 Agent 运行相比，其中应用程序的核心功能根本不起作用，提升是显而易见的。

### 迭代 Harness

第一组 Harness 结果令人鼓舞，但它也很笨重、缓慢且昂贵。合乎逻辑的下一步是找到在不降低性能的情况下简化 Harness 的方法。这部分是常识，部分是一个更普遍原则的函数：Harness 中的每个组件都编码了一个关于模型自己不能做什么的假设，这些假设值得压力测试，既因为它们可能不正确，也因为随着模型改进它们可能很快过时。我们的博客文章**构建有效 Agent**将底层思想框架化为"找到最简单的解决方案，只在需要时增加复杂性"，对于任何维护 Agent Harness 的人来说，这是一个一致出现的模式。

在我第一次尝试简化时，我大幅削减了 Harness 并尝试了一些创造性的新想法，但我无法复制原始的性能。也很难判断 Harness 设计的哪些部分实际上是负载承载的，以什么方式。基于这种经验，我转向了一种更有条理的方法，一次移除一个组件并审查它对最终结果的影响。

当我经历这些迭代周期时，我们还发布了 **Opus 4.6**，这为进一步减少 Harness 复杂性提供了动力。有很好的理由预期 4.6 需要比 4.5 更少的脚手架。从我们发布的博客："[Opus 4.6] 计划更仔细，维持 Agent 任务更长时间，可以在更大的代码库中更可靠地运行，并有更好的代码审查和调试技能来捕捉自己的错误。"它在长上下文检索方面也大幅改进。这些都是 Harness 旨在补充的能力。

#### 移除冲刺构造

我首先完全移除了冲刺构造。冲刺结构有助于将工作分解为模型可以连贯处理的块。鉴于 Opus 4.6 的改进，有很好的理由相信模型可以在没有这种分解的情况下原生处理工作。

我保留了规划器和评估器，因为每个都继续增加明显的价值。没有规划器，生成器范围不足：给定原始提示，它会开始构建而不先规划工作，最终创建比规划器功能更少的应用程序。

随着冲刺构造的移除，我将评估器移到运行结束时的单次通过，而不是每个冲刺评分。因为模型更有能力，它改变了评估器对某些运行的负载承载程度，其有用性取决于任务相对于模型自己可靠能做的事情的位置。在 4.5 上，那个边界很近：我们的构建处于生成器自己做得好能力的边缘，评估器在整个构建中捕捉有意义的问题。在 4.6 上，模型的原始能力增加了，所以边界向外移动。过去需要评估器检查才能连贯实现的任务现在通常处于生成器自己处理得很好的范围内，对于那个边界内的任务，评估器成为不必要的开销。但对于仍处于生成器能力边缘的构建部分，评估器继续提供真正的提升。

实际含义是，评估器不是一个固定的 yes 或 no 决定。当任务处于当前模型自己可靠能做的事情之外时，它值得成本。

除了结构简化，我还添加了提示来改进 Harness 如何将 AI 功能构建到每个应用程序中，特别是让生成器构建一个可以通过工具驱动应用程序自身功能的正确 Agent。这需要真正的迭代，因为相关知识足够新，Claude 的训练数据覆盖得很薄。但经过足够的调整，生成器正确地构建了 Agent。

#### 更新 Harness 的结果

为了测试更新后的 Harness，我使用以下提示生成了一个**数字音频工作站（Digital Audio Workstation, DAW）**，一个用于作曲、录音和混音的音乐制作程序：

> 使用 Web Audio API 在浏览器中构建一个功能齐全的 DAW。

运行仍然漫长且昂贵，大约 4 小时和 $124 的令牌成本。

大部分时间用于构建器，它在没有 Opus 4.5 需要的冲刺分解的情况下连贯运行了两个多小时。

| Agent & 阶段 | 时长 | 成本 |
|-------------|------|------|
| 规划器 | 4.7 分钟 | $0.46 |
| 构建（第 1 轮） | 2 小时 7 分钟 | $71.08 |
| QA（第 1 轮） | 8.8 分钟 | $3.24 |
| 构建（第 2 轮） | 1 小时 2 分钟 | $36.89 |
| QA（第 2 轮） | 6.8 分钟 | $3.09 |
| 构建（第 3 轮） | 10.9 分钟 | $5.88 |
| QA（第 3 轮） | 9.6 分钟 | $4.06 |
| **V2 Harness 总计** | **3 小时 50 分钟** | **$124.70** |

与之前的 Harness 一样，规划器将一行提示扩展为完整规格。从日志中，我可以看到生成器模型在规划应用程序和 Agent 设计、连接 Agent 并在交给 QA 之前测试它方面做得很好。

也就是说，QA Agent 仍然捕捉到真正的差距。在第一轮反馈中，它指出：

> 这是一个具有出色设计保真度、可靠 AI Agent 和良好后端的强大应用程序。主要的失败点是功能完整性——虽然应用程序看起来令人印象深刻且 AI 集成工作良好，但几个核心 DAW 功能只是显示而没有交互深度：剪辑不能在时间线上拖动/移动，没有乐器 UI 面板（合成器旋钮、鼓垫），也没有视觉效果编辑器（EQ 曲线、压缩器仪表）。这些不是边缘情况——它们是使 DAW 可用的核心交互，规格明确要求它们。

在第二轮反馈中，它再次捕捉到几个功能性差距：

> 剩余差距：
> - 音频录制仍然是仅存根（按钮切换但没有麦克风捕获）
> - 剪辑通过边缘拖动调整大小和剪辑分割未实现
> - 效果可视化是数字滑块，不是图形（没有 EQ 曲线）

当任其自行处理时，生成器仍然容易遗漏细节或存根功能，QA 在捕捉那些最后的细节问题供生成器修复方面继续增加价值。

基于提示，我期望一个程序，我可以在其中创建旋律、和声和鼓点模式，将它们安排成歌曲，并在整个过程中从集成的 Agent 获得帮助。

应用程序远非专业的音乐制作程序，Agent 的歌曲创作技能显然还有很多工作要做。此外，Claude 实际上听不到，这使得 QA 反馈循环在音乐品味方面效果较差。

但最终应用程序拥有一个功能性音乐制作程序的所有核心部分：浏览器中运行的编曲视图、混音器和传输。除此之外，我能够通过提示完全组合一个简短的歌曲片段：Agent 设置速度和调性，铺设旋律，构建鼓轨，调整混音电平，并添加混响。歌曲创作的核心原语存在，Agent 可以自主驱动它们，使用工具从头到尾创建一个简单的制作。你可能说它还不是完美的——但它正在接近。

### 未来展望

随着模型继续改进，我们可以大致期望它们能够工作更长时间，处理更复杂的任务。在某些情况下，这意味着围绕模型的脚手架随着时间的推移变得不那么重要，开发者可以等待下一个模型，看到某些问题自行解决。另一方面，模型越好，就有越多的空间来开发能够实现模型基线之外复杂任务的 Harness。

考虑到这一点，这项工作的一些教训值得继续发扬。试验你正在构建的模型，阅读它在现实问题上的跟踪，并调整其性能以实现你想要的结果，这总是一个好的做法。在处理更复杂的任务时，有时可以通过分解任务并将专门的 Agent 应用于问题的每个方面来获得提升。当新模型落地时，通常最好重新审视 Harness，剥离不再对性能负载承载的部分，并添加新的部分以实现以前可能无法实现的更大能力。

从这项工作中，我的信念是，随着模型改进，有趣的 Harness 组合空间不会缩小。相反，它会移动，对于 AI 工程师来说，有趣的工作是不断找到下一个新颖的组合。

## 核心内容提炼

### 一、问题域界定

| 挑战 | 表现 | 技术根源 |
|-----|------|---------|
| **上下文漂移（Context Drift）** | 长任务中模型失去连贯性 | 上下文窗口填满，早期信息丢失 |
| **上下文焦虑（Context Anxiety）** | 模型过早结束工作 | 模型接近感知上下文限制时的防御行为 |
| **自我评估偏差（Self-evaluation Bias）** | Agent 过度积极地评价自己的工作 | 生成器与评估器未分离，缺乏外部反馈 |
| **主观质量评估困难** | 设计等主观任务难以量化评分 | 缺乏可验证的二元检查机制 |

### 二、核心方法论：三 Agent 架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                        三 Agent 架构                                 │
├─────────────────┬─────────────────────┬─────────────────────────────┤
│    规划器        │      生成器          │         评估器              │
│   (Planner)     │    (Generator)      │      (Evaluator)            │
├─────────────────┼─────────────────────┼─────────────────────────────┤
│ • 接收用户提示   │ • 按冲刺实现功能     │ • 通过 Playwright 测试应用   │
│ • 扩展为完整规格 │ • 使用 React/Vite   │ • 评分设计/功能/代码质量     │
│ • 寻找 AI 功能   │ • 自我评估后交 QA   │ • 识别 bug 和差距            │
│   集成机会      │ • 根据反馈迭代       │ • 提供具体修复建议            │
└─────────────────┴─────────────────────┴─────────────────────────────┘
                              ↓
                    ┌─────────────────┐
                    │   冲刺合同协商    │
                    │ (Sprint Contract)│
                    └─────────────────┘
```

**关键设计原则**：

1. **职责分离（Separation of Concerns）**
   - 规划器专注于"做什么"（产品规格）
   - 生成器专注于"怎么做"（技术实现）
   - 评估器专注于"做得如何"（质量验证）

2. **迭代反馈循环（Iterative Feedback Loop）**
   - 生成器 → 评估器 → 生成器的闭环
   - 外部评估比自我评估更可靠
   - 调整评估器持怀疑态度比让生成器自我批评更容易

3. **上下文管理策略**
   - **上下文重置（Context Reset）**：完全清空上下文，结构化交接
   - **压缩（Compaction）**：就地总结早期对话
   - 选择依据：模型是否表现出上下文焦虑

4. **可评分的主观质量**
   - 将"这个设计好吗？"转化为具体标准
   - 四个维度：设计质量、原创性、工艺、功能性
   - 使用少样本示例校准评估器

### 三、实施指南

**前端设计 Harness**：

| 组件 | 配置 |
|-----|------|
| 生成器 | 创建 HTML/CSS/JS 前端 |
| 评估器 | Playwright MCP 交互 + 四标准评分 |
| 迭代次数 | 5-15 次 |
| 运行时长 | 最长 4 小时 |
| 关键策略 | 每次迭代后决定：完善方向或转向新美学 |

**全栈编码 Harness V1**：

| Agent | 职责 | 技术栈 |
|-------|------|--------|
| 规划器 | 1-4 句提示 → 完整规格 | - |
| 生成器 | 冲刺开发，一次一个功能 | React, Vite, FastAPI, SQLite/PostgreSQL |
| 评估器 | Playwright 测试 + 合同验证 | 硬阈值评分 |

**全栈编码 Harness V2（简化版）**：

| 变更 | 原因 | 效果 |
|-----|------|------|
| 移除冲刺构造 | Opus 4.6 能力增强 | 连续 2+ 小时连贯运行 |
| 评估器移至最后 | 任务在模型能力边界内 | 减少开销 |
| 保留规划器 | 防止生成器范围不足 | 确保功能完整性 |

**成本对比**：

| 方案 | 时长 | 成本 | 质量 |
|-----|------|------|------|
| 单 Agent | 20 分钟 | $9 | 核心功能损坏 |
| V1 Harness | 6 小时 | $200 | 功能完整，有 bug |
| V2 Harness | 3.8 小时 | $125 | 功能完整，迭代修复 |

- **提示**：使用 Web Audio API 在浏览器中构建功能齐全的 DAW
- **结果**：浏览器中运行的编曲视图、混音器和传输

### 五、核心观点与金句

> "Separating the agent doing the work from the agent judging it proves to be a strong lever to address this issue."
> 
> **（将执行工作的 Agent 与评判它的 Agent 分离，证明是解决这个问题的有力杠杆。）**

> "Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions are worth stress testing."
> 
> **（Harness 中的每个组件都编码了一个关于模型自己不能做什么的假设，这些假设值得压力测试。）**

> "The space of interesting harness combinations doesn't shrink as models improve. Instead, it moves."
> 
> **（随着模型改进，有趣的 Harness 组合空间不会缩小。相反，它会移动。）**

> "Find the simplest solution possible, and only increase complexity when needed."
> 
> **（找到最简单的解决方案，只在需要时增加复杂性。）**

**核心哲学**：
- **Harness 设计是模型能力的补充而非替代**：随着模型改进，重新评估每个组件的必要性
- **评估器不是固定开销**：当任务处于模型能力边界之外时，它值得成本
- **主观质量可工程化**：通过明确标准和迭代反馈，将主观判断转化为可评分指标
- **持续迭代**：试验模型、阅读跟踪、调整性能是 AI 工程师的持续工作

从提示词魔法到流程标准：Superpowers 解决的核心问题

AI 编程工具现在遍地都是。Claude Code、Cursor、Codex——选哪个都差不多。但有一个通用毛病：你把需求扔进去，它三秒甩出一堆代码。能跑。但边界条件呢？单元测试呢？安全性呢？没人管。

传统 AI 编程：给提示词 → AI 生成代码 → 人工检查修改。每一步的质量全靠提示词设计水平和 review 的细致程度。

Superpowers 的思路转了 180 度：流程大于提示。定义标准化软件工程流程 → AI 按流程执行 → 自动验证质量 → 继续优化改进。

杀手锏在于「强制执行」——不是你提醒 AI 先写测试再写代码，而是 TDD 技能在 AI 准备动手时自动介入，强迫走完红-绿-重构循环。人不需要时刻盯着，AI 自己管自己。

14 个工程化技能拆解：从需求到上线完整流水线

完整跑一遍就知道它为什么叫「超能力」了。这 14 个技能覆盖软件开发的全部关键节点，下面挑最核心的 6 个逐个拆。

brainstorming 头脑风暴技能：苏格拉底式需求澄清

在任何代码出现之前，brainstorming 技能强制 AI 进入提问模式。你说「做个用户认证系统」，它按标准模板追问：用户规模多大？支持哪些登录方式？要不要 2FA？密码策略怎么定？session 还是 JWT？

这个技能单独用就值回时间成本。很多项目失败不是因为代码烂，而是需求一开始就没搞对。AI 用苏格拉底式提问把模糊想法变成结构化设计文档，这个过程在传统开发中通常发生在开会吵了半小时之后。

编写计划和执行计划技能：大任务切成 2-5 分钟小块

编写计划技能把需求拆成结构化清单，每个子任务标注文件路径、代码意图、验证步骤。执行计划技能按清单顺序执行，每完成一步自动验证。

实测：让 AI 写带分页和权限过滤的 REST API。没装 Superpowers 时它一口气写完，路由参数搞错一个，我花 15 分钟定位 bug。装上后自动拆成数据模型、查询逻辑、路由层、测试 4 个子任务，第二个子任务跑单元测试时就发现了参数缺失并修正，全程不需要我介入。

子代理并行驱动开发：多核 AI 同时干活


dispatching-parallel-agents 技能把独立子任务分发给多个并行子代理。比如开发电商系统，子代理 1 搞用户模块，子代理 2 搞商品模块，子代理 3 搞订单模块。三个子代理同时写代码，主代理最后整合。

对大型项目效率提升是倍数量级的。但有一个坑：子代理之间的依赖关系必须在编写计划阶段提前定义清楚，否则会出现 API 接口不一致。这套工具对工程纪律的要求比传统 AI 编程更高——不是更省心，而是更规范。

TDD 测试驱动开发技能：红-绿-重构自动循环

这是 Superpowers 的灵魂技能，也是代码通过率从 58% 拉到 91% 的核心原因。

标准 TDD 三步走：红灯（写失败的测试）→ 绿灯（最少代码让测试通过）→ 重构（优化代码结构）。Superpowers 全程自动执行这三步，不需要手动干预。

差距的根源：AI 自己写的测试发现了自己写的 bug，人工修复变成了 AI 自修。这组对比值得截图，下次选 AI 编程工具时直接翻出来看。

调试技能：科学化诊断替代盲猜修复

遇到 bug 时，调试技能让 AI 走五步系统化诊断流程：复现问题 → 提出假设 → 逐一验证 → 找到根因 → 修复验证。

我故意埋了一个 N+1 查询的性能问题。没装 Superpowers 时 AI 直接开始改代码，猜了两个方向全错。装上后它先用 profiling 工具确认查询次数异常，定位到关联查询未使用 eager loading，一处修改解决。从「盲修」到「诊断」，效率差距肉眼可见。

代码审查技能：提交前自动拦截质量问题

receiving-code-review 技能在代码写完后自动扫描：检查规范遵守、安全漏洞、性能隐患。所有问题在提交前被拦截。

思路跟生产环境 CI Pipeline 完全一致，区别在于 Superpowers 把它前置到了开发阶段——问题在代码还热着的时候就被发现，而不是等推到 CI 流水线再报错。

Superpowers 三种最有效的使用场景

新项目从零开始

这是 Superpowers 的最佳战场。brainstorming 澄清需求 → 编写计划拆任务 → TDD 执行开发 → 代码审查收尾，整套跑下来，从需求到可上线代码比手动 + AI 辅助的模式至少快 3 倍。

关键数字：8 接口 REST 项目，全程47 分钟，测试覆盖率96%，零人工调试。

接手别人的遗留代码仓库

调试技能在这种场景下最有用。你不熟悉代码库，AI 也不熟悉——传统模式下 AI 会「猜」逻辑，猜错概率很高。调试技能强制它先理解再动手，第一步就是分析代码结构和调用关系。

我在一个有 2300 行的遗留 Node.js 项目上实测：加一个新功能。没装 Superpowers：AI 改了 3 个文件，改坏了 1 个老功能。装上后：AI 先画调用图，确认影响范围，再动代码，零回归问题。

团队协作开发

子代理并行能力在团队场景下是核武器级别。编写计划产出的任务清单可以直接发配给多个开发者，每人用一个子代理按计划执行。任务拆分标准统一、代码风格一致、测试覆盖相同——传统 Code Review 流程根本做不到这种一致性。

Superpowers 的局限和适用边界

不是万能药，几个实在的限制需要讲清楚。

平台兼容性：Superpowers 目前只支持 Claude Code 作为主驱动引擎。虽然官方写着兼容 Codex 和 OpenCode，实战中 Claude Code 的集成度最高，其他环境或多或少有技能触发不稳定的问题。

学习成本：14 个技能不是「装了就变强」。你需要理解每个技能的触发时机和适用场景。比如 brainstorming 技能在改一个小 bug 时也会跳出来提问，反而拖节奏。知道什么时候开、什么时候关，才算真正会用。

Token 消耗：完整跑通 brainstorming + 编写计划 + TDD + 代码审查这套流程，Token 消耗大约是直接让 AI 写代码的 3-4 倍。对个人开发者每月多花几十块钱，但对频繁使用的小团队，API 费用翻倍需要认真算账。

Superpowers 代表的不只是更好用，而是 AI 编程工程化的拐点

Superpowers 的火爆不是偶然。它标记了一个分水岭：AI 编程从「让 AI 写代码」进化到「让 AI 按工程标准交付软件」。

2023 年 AI 编程是代码补全。2024 年是对话式编程。2025 到 2026 年是 Agent 自主编程。但自主不代表质量——能自己干活和能按规范干活是两码事。Superpowers 做的事，就是给 AI 的自主能力套上软件工程的纪律约束。

这个趋势一旦确立，未来所有 AI 编程助手都会内置类似的流程引擎。到时候比的不再是代码生成速度，而是工程流程规不规范。

## Agent Skills 实战：把 QA 测试用例（Test Cases）写成 Skill

**文章信息**：
- **原文标题**：Agent Skills 实战：把 QA 测试用例（Test Cases）写成 Skill
- **来源**：https://www.toutiao.com/article/7625207208619360787/
- **作者**：优美葡萄HtTxcX
- **发表时间**：2026-04-05 17:44

---

**一段话总结**：

文章演示如何将 QA 测试用例的书写规范写成一条 Agent Skill（`.cursor/skills/test-case/SKILL.md`）。团队写测试用例格式不统一（有人不写前置条件、步骤不可执行、预期结果含糊、无法追溯需求），Skill 解决的是：用户说"写测试用例"时，助手按统一模板输出——必填字段（用例 ID、标题、关联需求、前置条件、测试步骤、预期结果、优先级）+ 可选字段（测试类型、测试数据、备注、执行结果）。Skill 定义了三种输出模式：生成模式（从需求直接生成用例）、补全模式（补充已有用例缺项）、评审模式（检查用例是否可执行/可验证/可追溯，输出必改/建议/可选清单）。配套提供 reference.md（完整模板和书写规范）和可选 scripts（检查关键字段完整性）。与测试计划 Skill 分工：测试计划回答"测什么、怎么安排"，测试用例回答"每条怎么测、怎么判定通过"。

---

**作者判断**：

作者认为实战套路是"把团队规范清单化→写成 SKILL→加 reference（+ scripts）→触发与协作"，重点在于把测试计划里的策略落成可执行、可验证、可追溯的逐条 case，补上从需求到执行之间最关键的一层。

---

**核心内容**：

**问题背景**：团队测试用例格式不统一导致评审难聚焦、执行易漏测。把用例模板和书写规范写进 Skill，助手在触发时按同一套结构输出。

**用例必填字段**（7 项）：用例 ID、标题（格式：模块-场景-预期）、关联需求（PRD/用户故事/接口）、前置条件（无则明确写"无"）、测试步骤（按顺序编号，一步一动作）、预期结果（可观察、可判断通过失败）、优先级（P0/P1/P2）。

**用例可选字段**：测试类型（冒烟/功能/回归/异常/边界/兼容性）、测试数据、备注、执行结果字段（结果/执行人/日期/缺陷单号）。

**三种输出模式**：
1. **生成模式** — 从需求/流程直接生成一组测试用例
2. **补全模式** — 对已有用例补充缺失字段、异常场景、边界 case
3. **评审模式** — 检查已有用例的完整性，输出必改/建议/可选清单

**书写规范要点**：
- 标题避免笼统（如"登录测试"），改为具体场景（"登录-输入正确账号密码后成功进入首页"）
- 步骤必须可执行（"输入已注册邮箱与正确密码，点击登录按钮"），避免"进行登录操作"
- 预期结果必须可验证（"接口返回 200 且 body 含 token"），避免"功能正常"
- 每条用例至少关联一个需求 ID，避免孤立 case
- 多个 case 仅在输入不同时应明确区分差异，不要只换标题

**目录结构**：
```
test-case/
├── SKILL.md          # 技能行为规范（核心）
├── reference.md      # 可选：完整模板、命名规范、覆盖维度、优先级定义
└── scripts/          # 可选：检查用例字段完整性的 shell 脚本
    └── check-testcase-fields.sh
```

**与其他 Skill 的协作**：
- 测试计划（Test Plan）负责范围、策略、准入准出；测试用例负责逐条验证动作与预期
- 测试用例定义"测什么、怎么判定"→ 测试报告总结"测得怎么样、能不能过"
- 一般测试用例 Skill 不依赖 MCP 即可先用起来；如需从 PRD/Jira 拉需求可配合 MCP

**团队协作**：技能放在 `.cursor/skills/` 随仓库提交，新技能走 PR review（检查 description 覆盖常见说法、字段定义与团队模板一致、步骤/预期规范是否清晰）。

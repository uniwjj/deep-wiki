Apache Iceberg v3:数据湖仓的分水岭时刻

先说正事

著名数据湖表格式 Apache Iceberg 出 v3 了。

规范在 2025 年上半年由社区正式批准。一年之内,四大湖仓引擎集体跟进:Snowflake 于 2026 年 3 月宣布 v3 公开预览,Databricks 在 2026 年 4 月紧随其后,Google BigQuery、Dremio、AWS(EMR、Glue、S3 Tables)同步落地。

v3 带来了五个关键特性:删除向量(行级更新快 10 倍)、行级血缘(CDC 变成表的原生能力)、VARIANT 类型(原生支持半结构化数据)、默认列值(Schema 变更从小时级变秒级)、纳秒时间戳与地理空间类型。更关键的是,它让 Iceberg 真正成为了行业公共标准。

要理解这件事的分量,得从数据湖本身讲起。


一、数据湖:从”散装文件”到”可靠的表”

传统数据仓库(Oracle、Teradata)贵、封闭、规整。数据进来前必须定好表结构,想换工具查询几乎不可能。

2010 年前后新思路出现:先把数据原样扔进便宜存储(HDFS、S3、OSS),想查再说。这就是数据湖。好处是便宜、开放、灵活,但致命缺陷是——它只是一堆散装文件,不是一张表。没法可靠更新、没法保证并发写入一致、没法回到历史版本。

于是业界开始给数据湖”加账本”——表格式(Table Format)。有了它,散装文件才变成真正的表。

二、四大数据湖格式:各有心思

Apache Hudi(Uber,2016):深度绑定 Hadoop 生态,对 Hive、HDFS 兼容最好。侧重老 Hadoop 栈平滑升级、upsert 场景。代价是概念繁多、学习曲线陡。

Delta Lake(Databricks):和 Spark 无缝衔接,在 Databricks 平台上调得极好。侧重 Spark 生态和企业级可靠性。长期是 Databricks 的”自家孩子”。

Apache Paimon(Flink 社区,2022):原名 Flink Table Store,由阿里 Flink 团队主导。侧重流批一体、实时离线一体——把流处理的实时性和湖仓的低成本结合起来,CDC 入湖丝滑是它的招牌。国内采用度非常高,字节、阿里、小米、shopee 都在大规模使用。

Apache Iceberg(Netflix,2018):侧重中立、开放、跨引擎。Iceberg 从设计之初就不绑定任何引擎或云厂商,想做的是整个行业的公共标准。

Hudi 偏 Hadoop、Delta 偏 Spark、Paimon 偏 Flink——它们都有”娘家”。Iceberg 没有娘家,这正是它最大的特点。

围绕这些格式,行业出现了几个理念:湖仓一体(Databricks 主推,用一份数据同时服务 BI 和 ML)、流式湖仓(Paimon 的主场,分钟级甚至秒级延迟)、开放湖仓(Iceberg 阵营,强调不被厂商锁定)。Delta 讲一体化体验、Paimon 讲实时、Hudi 讲兼容老架构,而 Iceberg 讲的是标准——在一个碎片化的行业里,“标准”是最有长期价值的叙事。

三、国内市场怎么看?

国内数据湖的路线和海外有明显差异。

Paimon 在国内是第一梯队。阿里云推出 EMR Serverless StarRocks + Paimon 的组合,主打实时湖仓;字节跳动内部大规模使用 Paimon 处理 Flink CDC 数据;湖仓一体的实时场景几乎是 Paimon 的主场。

Iceberg 在国内也有强势布局:

• 腾讯云 DLC(数据湖计算)、EMR 均以 Iceberg 为核心表格式

• 华为云 MRS、DLI 全面支持 Iceberg

• 阿里云 MaxCompute、EMR 同时支持 Iceberg 和 Paimon,两条腿走路

• 网易数帆(Amoro 项目)深度参与 Iceberg 社区,贡献了自适应合并等特性

• 字节跳动(ByteHouse、LAS)、小米(Talos)、B 站都有大规模 Iceberg 生产实践

Hudi 在国内主要活跃在仍以 Hadoop 为底座的传统企业,以及一些早期湖仓改造的互联网公司。Delta Lake 在国内采用度最低,因为 Databricks 在中国区没有直接落地。

简单说:海外是 Databricks(Delta)+ Snowflake(Iceberg)双雄格局,国内是 Paimon 主打实时、Iceberg 主打通用,两强并立。而 Iceberg v3 的行级血缘、删除向量、VARIANT 直接对标了 Paimon 擅长的实时增量场景——这意味着国内市场的竞争,在 v3 之后会进入新阶段。

四、Iceberg 的来路:v1、v2、v3

• v1(2018):打地基。统一元数据、快照、schema 演化。让”散装文件”第一次有了数据库表的可靠语义

• v2(2021):引入行级删除(positional delete + equality delete)。update/delete 第一次成为可能,但实现方式不够优雅——频繁更新会产生大量小删除文件,拖垮读性能

• v3(2025):补齐最后几块拼图


五、v3 到底带来了什么(核心部分)

1. 删除向量(Deletion Vectors):行级更新终于不再是性能灾难

v2 的问题:行级删除是巨大进步,但有严重瓶颈——每次删除生成独立小文件,查询时引擎要把这些小文件和原始数据文件逐一匹配、合并、过滤。

频繁更新的表(订单状态、CDC 实时同步),删除文件会像雪花一样越积越多。一张被频繁更新的大表可能积累成千上万个小删除文件,每次查询都要先”扫雪”。写入越频繁的表,读取越慢——恰恰是最需要实时分析的场景,性能最差。

v3 的解法:引入二进制删除向量,根本性的架构变革。

核心思路:给每个数据文件附加一个位图(Bitmap),每一位对应数据文件中的一行,0 表示有效行、1 表示已删除行。底层采用 Roaring Bitmap——专门为稀疏整数集设计的高效压缩算法。

这意味着:

• 不再有成千上万的小删除文件。所有删除信息被压缩进一个紧凑位图,附着在对应数据文件旁(通常存储在 .puffin 文件中)

• 查询时不再需要”扫雪”。引擎读数据文件时同时读对应的删除向量,做一次位运算就能跳过被删除的行,开销几乎可忽略

• 写入放大大幅降低。修改几行不再需要重写整个数据文件,只需更新位图

Databricks 官方数据:删除向量的数据处理速度比传统 Copy-on-Write 快 10 倍。更关键的是,v3 强制要求引擎为每个数据文件维护单一删除向量(不允许多个删除文件散落),从根本上杜绝”小文件地狱”的回归。同时 v2 的 position delete files 在 v3 中被标记为 deprecated。

2. 行级血缘(Row Lineage):CDC 从”手工活”变成表的固有属性

v2 的问题:“这张表从上次查询以来,哪些行变了?”——这个看似简单的问题,过去回答起来非常痛苦。你需要自己维护增量标记:ETL 里加时间戳字段、维护 watermark、做全表比对,或者依赖外部 CDC 工具(Debezium、Maxwell、Flink CDC)。这些方案要么重、要么脆、要么贵。这是困扰数据工程界多年的基础设施级痛点——而这恰恰是 Paimon 在国内市场最大的卖点之一。

v3 的解法:在规范层面引入行级血缘。每一行数据天然携带两个元数据字段:

• _row_id:全局唯一行标识符,伴随行的整个生命周期

• _last_updated_sequence_number:记录该行最后一次被添加或修改的提交版本

这两个字段不是业务字段,而是 Iceberg 表格式规范本身的一部分,所有写入引擎必须维护它们。

“哪些行变了”从此变成简单的元数据查询。比较两次快照的序列号差异,就能精确找到所有新增、修改、删除的行——不需要全表扫描、不需要外部 CDC 工具、不需要手工维护水位线。

Databricks 博客里有一句杀伤力极强的话:“Together, row lineage and deletion vectors make CDC a native property of the table itself.”

场景意义:

• 增量刷新物化视图变得异常简单——只需重算变化行涉及的聚合,而不是重跑整张源表。PB 级数据下,这可能是”30 分钟”和”10 秒”的差距

• AI 工程的数据质量管理——精确追踪某条训练数据的来源和变更历史,对模型可审计性和合规是关键能力

• 对标 Paimon 的实时场景——Iceberg 第一次在规范层面具备了”流式湖仓”的基础能力

3. VARIANT 类型:半结构化数据终于不再是二等公民

v2 的问题:现实中一半以上的数据是半结构化的——API 响应、IoT 传感器、用户行为日志、webhook。字段不固定、层级不确定、schema 随时可能变。处理这类数据只有两条路:

• 展平成固定列:一张几百列的超宽表,大量 NULL,上游加字段就要跑一次 schema evolution

• 存成字符串:列式存储的一切优势全部丧失,查询性能惨不忍睹

v3 的解法:引入原生的 VARIANT 数据类型。

VARIANT 使用高效的二进制编码格式,在保持模式灵活性的同时支持列式引擎的核心优化:

• 谓词下推:WHERE payload.event_type = 'purchase' 可以在文件扫描阶段就完成过滤,不需要把整个 JSON 读进内存再解析

• 切碎优化(Shredding):高频访问字段可被”切碎”为独立列式存储,实现接近原生列的读取性能;不常访问的字段仍以紧凑二进制存储

• 零 Schema 迁移:新字段出现时不需要任何 ALTER TABLE,直接写、直接查


CREATE TABLE events (

event_id BIGINT,

ts TIMESTAMP_NS,

payload VARIANT

) USING iceberg

TBLPROPERTIES ('format-version' = '3');


SELECT payload:user_id, payload:action, payload:metadata:device_type

FROM events

WHERE payload:action = 'purchase'

AND ts > current_timestamp() - INTERVAL 1 HOUR;



策略从此变成:原始数据直接入湖,查询时再提取需要的字段。对 AI 工程师来说,这意味着可以用 SQL 直接查 LLM 推理日志、Agent 工具调用记录、模型 A/B 测试的完整 trace,不需要先”规整”成关系型表。

4. 默认列值:一秒完成十亿行的 Schema 变更

以前给一张十亿行的表加一列并指定默认值,需要做回填——重写所有数据文件。一张 PB 级表可能意味着几小时计算和服务中断风险。

v3 的默认列值让这变成纯元数据操作:


ALTER TABLE orders ADD COLUMN priority STRING DEFAULT 'normal';



执行时间亚秒级。只修改表的元数据文件,没碰任何数据文件。查询引擎读取没有 priority 列的旧数据文件时,会自动从表模式查找默认值并即时填充。

(注意:Databricks 当前版本对默认列值还未完全支持。特性在规范里,引擎实现程度不一。)

5. 纳秒级时间戳:精度的最后一公里

v3 新增 timestamp_ns 和 timestamptz_ns,将精度从微秒提升到纳秒。看似细节,对特定场景意义重大:

• 高频交易:金融订单簿事件间隔在纳秒级,微秒精度会导致事件顺序丢失

• IoT 传感器:工业振动传感器、电力监控采样率可达百万次/秒

• 分布式系统调试:微服务架构中,纳秒级时间戳是重建因果关系、排查竞态条件的基础

以前这些场景不得不用 BIGINT 存纳秒时间戳,丧失了时间类型的语义。现在,时间就是时间,精度到纳秒。

6. 跨格式互操作:格式战争的终局

比技术细节更重要的是生态响应。Databricks 在博客中公开宣布:v3 的删除向量、行级血缘、VARIANT 与 Delta Lake 实现了同一数据层的互操作。通过 UniForm,客户只需写入一次 Delta Lake,即可以 Iceberg 格式从 Snowflake、BigQuery、Redshift、Athena、Trino 等引擎读取。

这等于 Databricks 公开承认:“你用 Iceberg 还是 Delta,已经不重要了。”


六、结语

回头看这条脉络:

数据湖最初是为了摆脱昂贵封闭的数据仓库;四大格式各自从自家母体出发——Hudi 背着 Hadoop 的包袱、Delta 绑着 Spark 的战车、Paimon 扛着 Flink 的实时大旗,而 Iceberg 从一开始就朝着”做全行业标准”的方向走。v1 打地基,v2 打开行级更新的大门,v3 让它真正具备了对标传统数据库的能力。

v3 带来的不只是性能提升,而是范式转变:删除向量让行级更新快 10 倍,行级血缘让 CDC 变成表的原生能力,VARIANT 让半结构化数据不再是二等公民,默认列值让 Schema 变更从小时变秒,纳秒时间戳和地理空间类型填补了最后的能力空白。更重要的是,Databricks、Snowflake、Google、Dremio、AWS、Starburst 同时押注 v3 并打通跨格式互操作,格式战争实质结束。

在国内,Paimon 凭借 Flink 生态和实时场景继续领跑,但 v3 让 Iceberg 第一次在实时能力上有了正面回应。腾讯云、华为云、阿里云、字节、网易、B 站、小米对 Iceberg 的持续投入,意味着国内市场在 v3 时代会出现Paimon 主打实时、Iceberg 主打通用标准的新竞争格局。

数据湖和数据仓库之间的那堵柏林墙,在 Iceberg v3 这里,正式倒塌了。​​​​​​​​​​​​​​​​

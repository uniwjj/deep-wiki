Understanding Gravitino's Core Object Model: Metalake, Catalog, Schema, and Beyond

上一篇我们讲了 Gravitino 的四层架构。这一篇，我们把镜头拉近到核心对象模型——Metalake、Catalog、Schema、Table、Fileset、Topic、Model。理解了这些对象，你就能用 Gravitino 做几乎所有元数据管理工作。

一、全景：对象模型层级树

Gravitino 的对象模型是一个严格四层树形结构，从上到下分别是 Metalake → Catalog → Schema → 叶节点对象。先看一张全景图：

关键特征：

• 层级固定：Metalake → Catalog → Schema → Table/Fileset/Topic/Model，不能跳级

• 同级类型混搭：一个 Schema 下可以同时有 Table、Fileset、Topic、Model

• 统一 API：无论底层是 MySQL 还是 Kafka，操作方式完全相同

二、Metalake：元数据湖（顶层容器）

Metalake 是 Gravitino 对象模型的最顶层容器。你可以把它理解为一个「元数据租户」——每个团队、每个业务线拥有一个 Metalake，里面的所有元数据互不可见。

Metalake 的核心属性只有三个：

// 创建 Metalake（REST API） POST /api/metalakes { "name": "data_corp", // 名称（全局唯一） "comment": "数据平台团队", // 描述（可选） "properties": {} // 自定义属性（可选） } // 响应 { "name": "data_corp", "comment": "数据平台团队", "audit": { "creator": "admin", "createTime": "2026-04-25T10:00:00Z" }, "id": 1 }

使用建议：一个组织通常创建 1 个 Metalake（小团队）或按业务线创建多个（大组织）。Metalake 之间的元数据完全隔离，RBAC 权限也是独立生效的。

三、Catalog：数据源（核心抽象）

Catalog 是 Gravitino 最重要的抽象——每个 Catalog 代表一种数据源。一个 Metalake 下可以挂多个 Catalog，每个 Catalog 对应不同的数据源类型。

创建 Catalog 时需要指定两个关键字段：

• type：数据源类型，包括 relational（关系型）、lakehouse（数据湖）、messaging（消息队列）、file（文件系统）等

• provider：具体实现，如 mysql、hive、kafka 等

// 创建 MySQL Catalog POST /api/metalakes/data_corp/catalogs { "name": "mysql_prod", "type": "relational", "provider": "mysql", "comment": "生产 MySQL 集群", "properties": { "jdbc-url": "jdbc:mysql://prod-mysql:3306", "jdbc-user": "admin", "jdbc-password": "***", "jdbc-database": "*" // * 表示暴露所有 database } } // 创建 Kafka Catalog { "name": "kafka_prod", "type": "messaging", "provider": "kafka", "properties": { "bootstrap.servers": "kafka-broker1:9092,kafka-broker2:9092" } }

创建成功后，Catalog 下的 Schema 和 Table 就会自动从底层数据源同步过来——你不需要手动一个个创建，Gravitino 的 Connector 会直接操作底层系统获取元数据。

四、Schema：命名空间

Schema 是 Catalog 下的逻辑分组，对应关系型数据库中的「database」或「namespace」。它不需要手动创建——当你连接一个 MySQL Catalog 时，MySQL 中的所有 database 会自动以 Schema 的形式出现在 Gravitino 中。

Schema 本身很简单，主要就是名称和注释。它的真正价值在于作为叶节点对象的容器。

// 列出 Catalog 下的所有 Schema GET /api/metalakes/data_corp/catalogs/mysql_prod/schemas // 响应（自动从 MySQL 同步） { "schemas": [ { "name": "ecommerce", "comment": null }, { "name": "inventory", "comment": null }, { "name": "analytics", "comment": "分析库" } ] }

五、五种叶节点对象

Schema 之下有五种叶节点对象，每种对应不同类型的元数据：

5.1 Table（关系型表）

Table 是最常见的叶节点对象，描述关系型数据的表结构。包含列定义（名称、类型、注释）、分区信息、表属性等。

// 获取表结构 GET /api/.../schemas/ecommerce/tables/users // 响应（自动从 MySQL INFORMATION_SCHEMA 同步） { "name": "users", "comment": "用户表", "columns": [ { "name": "id", "type": "BIGINT", "nullable": false, "comment": "用户ID" }, { "name": "name", "type": "VARCHAR(100)", "nullable": false, "comment": "用户名" }, { "name": "email", "type": "VARCHAR(200)", "nullable": true, "comment": "邮箱" } ] }

Gravitino 的 Table API 不会改变底层的表数据，只负责元数据的读取和管理。你通过 Gravitino 读取一个 Iceberg 表的列定义，实际上是由 Iceberg Connector 调用底层 Iceberg REST Catalog 获取的。

5.2 Fileset（文件集）

Fileset 是 Gravitino 对非结构化/半结构化文件的元数据抽象。它将底层存储系统（S3、HDFS、GCS）中的文件目录映射为一个逻辑名称，纳入统一的元数据管理。

典型场景：

• ETL 原始数据：日志文件、CSV、JSON 等原始数据目录的元数据管理

• ML 训练数据集：图片、音频、特征向量等训练素材目录

• 数据湖 Staging 层：增量数据的暂存目录

• 报表导出：BI 报表的输出目录

// 创建 Fileset POST /api/.../schemas/data_lake/filesets { "name": "raw_logs", "comment": "ETL 原始日志文件", "properties": { "location": "s3://data-bucket/raw-logs/", "storage-location": "s3://data-bucket/raw-logs/" } } // 核心价值：用逻辑名称 catalog.schema.raw_logs // 代替物理路径 s3://data-bucket/raw-logs/ // 享受统一的 RBAC、标签和审计

5.3 Topic（消息主题）

Topic 对象管理 Kafka 等消息队列的元数据。在没有 Gravitino 之前，Kafka Topic 的管理通常需要单独的 Admin 工具，和关系型元数据完全割裂。有了 Topic 对象，Kafka 的元数据也能纳入统一管理。

// 列出 Kafka Catalog 下的 Topic GET /api/.../catalogs/kafka_prod/schemas/events/topics // 响应（自动从 Kafka Cluster 同步） { "topics": [ { "name": "order_created", "comment": "订单创建事件" }, { "name": "user_registered", "comment": "用户注册事件" } ] }

5.4 Model 和 ModelVersion（ML 模型）

这是 Gravitino v1.0 引入的新对象，面向 AI/ML 场景。Model 对象描述一个机器学习模型，ModelVersion 描述模型的具体版本。

// Model 对象结构 { "name": "recommendation_v2", "comment": "推荐模型 v2", "properties": { "algorithm": "deepfm", "training_dataset": "s3://ml-datasets/recommendation/" } } // ModelVersion 对象 { "version": 2, "uri": "s3://ml-models/recommendation/v2/model.pt", "alias": ["production", "champion"], "comment": "AUC: 0.847, 上线测试通过" } // 一个 Model 下可以有多个 ModelVersion // 支持别名（alias）标记版本状态：staging / production / archived

Model + ModelVersion 的设计灵感来自 MLflow，但 Gravitino 的优势在于模型元数据和数据元数据在同一个 Metalake 里——你可以直接追踪「这个模型用了哪些表的数据训练的」，实现真正的 Data + AI 统一治理。

六、对象继承：标签和策略如何向下传递

Gravitino 的对象模型有一个非常实用的设计：标签（Tag）和策略（Policy）支持继承。当你在 Metalake 级别打上「PII」标签，其下所有 Catalog、Schema、Table 都会自动继承这个标签。

继承规则：Metalake 上的标签/策略会向下级联到所有 Catalog → Schema → Table。Catalog 级别的标签会级联到其下所有 Schema → Table。但如果子对象显式设置了不同的标签，以子对象为准（子覆盖父）。

这意味着你只需要在 Metalake 层面打一次标签，就不用逐个 Catalog、逐个 Table 去标记了。当安全团队要求「所有包含用户信息的表都标记 PII 并自动脱敏」，这个工作变得非常简单。

七、小结

Gravitino 的核心对象模型用一套统一的四层命名空间，把关系型数据库、数据湖、消息队列、文件系统和 ML 模型这五种完全不同的元数据类型纳入了同一个管理体系。

• Metalake：顶层容器，提供多租户隔离

• Catalog：对应一种数据源，通过 type + provider + properties 描述连接信息

• Schema：命名空间，自动从底层数据源同步

• Table / Fileset / Topic / Model：四种叶节点对象，覆盖关系型数据、文件、消息流和 AI 模型

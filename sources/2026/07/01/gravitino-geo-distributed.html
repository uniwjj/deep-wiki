Apache Gravitino: A High-Performance, Geo-Distributed, Federated Metadata Lake

上一篇文章我们对比了 DataHub、Amundsen、Gravitino 三大开源元数据平台，得出的结论是：三者的架构哲学截然不同。从今天起，我们正式进入 Phase 2：Gravitino 入门篇。本文是 Gravitino 系列的第一篇，帮你彻底搞清楚 Gravitino 到底是什么、为什么它与传统元数据平台有本质区别。

先给 Gravitino 一个清晰的定位

What is Apache Gravitino — And What It Is Not

Apache Gravitino 的官方定义是：一个高性能、地理分布式、联邦元数据湖（Metadata Lake）。2025年3月从 Apache 孵化器毕业成为顶级项目，当前最新版本为 v1.2.0。

但光看定义太抽象了。我们用一个类比来理解：

DataHub / Amundsen 像是一个图书馆的卡片索引柜——它自己建一套目录，定期去书架上抄书名，但书架上的书变了，索引柜不一定知道。

Gravitino 像是图书馆的总管理处——它直接连接到每个书架（数据源），实时知道书在哪、谁在读、谁可以读。不需要抄一份副本，因为它本身就在管理。

直接管理 vs 爬取式：两种范式的根本差异

Direct Management vs Crawling: The Fundamental Paradigm Shift

传统元数据平台（DataHub、Amundsen 等）采用的是爬取式（Crawl-Based）架构：

• 通过 Ingestion Framework 主动去各个数据源「抄」元数据

• 元数据存储在自己的数据库中（副本）

• 数据源发生变更后，需要重新爬取才能同步

• 反向操作（通过平台修改数据源）需要额外实现写回逻辑

Gravitino 采用的是直接管理式（Direct Management）架构：

• 通过 Connector 直接连接底层数据源

• 不存储元数据副本，直接操作底层系统的元数据

• 在 Gravitino 中的任何变更，立刻反映到底层系统

• 底层系统的变更，Gravitino 也能实时感知

核心区别一句话：传统平台是元数据的「观察者」（读一份副本），Gravitino 是元数据的「管理者」（直接控制）。

四层架构：Gravitino 的内部设计

Four-Layer Architecture: How Gravitino Works Internally

Gravitino 的架构分为四个层次，从上到下分别是功能层、接口层、核心对象模型和连接层。

功能层（Functionality Layer）是 Gravitino 的大脑，负责所有元数据操作的实现：

• 元数据 CRUD：创建、读取、更新、删除元数据对象

• 访问控制（RBAC）：基于角色的权限管理

• 标签 & 策略引擎：自动化数据治理

• 数据发现 & 审计日志：追踪谁在什么时候做了什么

接口层（Interface Layer）是用户和引擎与 Gravitino 交互的窗口。REST API 是目前最成熟的接口，Java SDK 和 Python SDK 则为开发者提供了编程式访问能力。特别值得一提的是 MCP Server（v1.2.0 新增），它是 AI Agent 访问企业元数据的桥梁，我们会在 Phase 6 详细展开。

核心对象模型（Core Object Model）定义了 Gravitino 的统一命名空间：Metalake → Catalog → Schema → Table。这个模型下一篇文章会深入讲解，这里先有个概念。

连接层（Connection Layer）是 Gravitino 的手脚，通过一系列 Connector 直接连接到底层数据源。这是 Gravitino「直接管理」能力的基础。

Connector 生态：一套接口，连接所有数据源

Connector Ecosystem: One Interface, All Data Sources

Gravitino 的 Connector 生态覆盖了大数据领域的方方面面：

关系型数据库：通过 JDBC Connector 统一连接 MySQL、PostgreSQL、Doris、OceanBase、ClickHouse 等。企业里的 OLTP 系统元数据，可以用一套 API 统一管理。

数据湖格式：这是当前最热门的方向。Iceberg、Hudi、Paimon 三大湖仓格式各有优势，Gravitino 为每个都提供了专用 Connector。比如 Iceberg REST Catalog Service，可以直接替代传统的 Hive Metastore。

文件系统：通过 Fileset Connector，给 HDFS、S3 等文件系统上的无结构/半结构化文件赋予元数据——ML 训练数据集、日志文件、ETL 原始数据，都可以纳入统一管理。

消息队列：Kafka Topic 的元数据管理，包括分区、副本、配置等。

AI 模型：Model Catalog 是 Gravitino 的独有特色，将 ML 模型元数据纳入统一管理，实现 Data + AI 资产的一体化。

在计算引擎方面，Gravitino 已原生集成 Trino、Apache Spark、Apache Flink、Daft 四大引擎。引擎不需要知道底层有哪些数据源，只需要连接 Gravitino，就能透明访问所有数据。

版本演进：从孵化到 ASF 顶级项目

From Incubation to Apache Top-Level Project

Gravitino 的发展速度相当快。2023年6月由 Datastrato 公司开源，2025年3月即从 Apache 孵化器毕业，成为顶级项目（TLP）。我们用一条时间线来回顾关键里程碑：

几个关键节点值得关注：

• v0.1.0（2023.06）：项目首次开源，基础架构确立

• v0.3.0（2023.11）：引入 Fileset 和 Model Catalog，开始支持 AI 资产

• v0.5.0（2024.04）：Kafka Connector、Hudi Connector、策略系统上线

• v0.6.0（2024.06）：地理分布式架构、标签系统、审计日志

• v1.0.0（2025.03）：成为 Apache 顶级项目，GA 正式版发布

• v1.2.0（2026.02）：最新版，MCP Server、Web UI v2、Paimon Connector

不到两年时间从 v0.1.0 到 v1.0.0，100+ 开源贡献者参与，10+ Connector 覆盖主流数据源——这个发展速度在 Apache 项目中相当少见，说明社区和行业对这个方向有强烈需求。

Gravitino 的四大核心价值

Four Core Values of Gravitino

综合上面的分析，Gravitino 的核心价值可以归纳为四个维度：

1. 单一真相来源（SSOT）：不存副本，直接管理底层数据源的元数据。不存在同步延迟和数据不一致的问题。

2. Data + AI 资产统一管理：表格、文件、Topic、ML 模型——所有资产类型在一个平台管理。不再需要分别维护数据目录和模型注册中心。

3. 集中式安全管控：RBAC 权限模型、标签系统、策略引擎三合一。在一个地方配置安全策略，自动应用到所有数据源。

4. 地理分布式架构：支持跨区域、跨云部署。不同地区的 Gravitino 实例可以互相发现，提供全域元数据视图。这对跨国企业或多云架构尤其重要。

适用场景：如果你的企业在建设多云数据湖或湖仓一体架构，需要跨 Hive/Iceberg/Kafka/S3 的统一元数据控制，或者正在做 Data + AI 融合，Gravitino 值得重点关注。

一、引言：当“变更”成为第一性原理

我们习惯了用“静态表”看待数据。

Hive 表是一堆文件，
Iceberg 表是带元数据的快照，
Hudi 表是支持更新的日志合并结构。

但现实世界的数据，从来不是“静态集合”，而是 持续流动的变更事件流（Change Stream）。

用户资料在更新，订单状态在流转，库存数量在波动——这些都不是“写入一次就结束”的操作，而是无限延长的 changelog 序列。

可我们的存储系统却长期滞后： - 要么只能追加（Append-only），无法高效更新； - 要么更新代价高昂（MOR 模式慢如蜗牛）； - 要么依赖 Kafka 中转，造成架构冗余。

直到 Apache Paimon 出现。

它不是又一个“兼容 Hive 的数据湖”，也不是“优化版 Hudi”。
它是 第一个把“变更流”作为核心抽象的存储格式 —— 我称之为：流原生存储（Stream-Native Storage）。

核心思想：一张表，既是结果，也是过程；既是批处理对象，也是流式消息源。

二、重新定义“表”：Paimon 的本质是什么？

Apache Paimon 是一种面向“变更流”的持久化存储系统，它让“表”本身成为一个可订阅、可回放、可聚合的事件源。

换句话说：

传统视角Paimon 视角
表 = 数据集合表 = 数据 + 变更历史
查询 = 读取当前状态查询 = 读取状态快照 或 消费变更流
写入 = 插入/覆盖写入 = 提交 changelog 记录

它的底层哲学接近 Event Sourcing（事件溯源）： - 每次更新不是“改数据”，而是“记一笔账”； - 最终状态由所有变更合并得出； - 历史任意时刻的状态都可重建。

这才是 Paimon 真正的创新所在 —— 它把数据库的 changelog 能力，原生集成到了数据湖中。

三、架构设计：为什么说它是“LSM-Inspired”而非“LSM-Based”？

Paimon 常被宣传为“基于 LSM 树”，但这容易引发误解。

❌ 误解澄清：

  • 它 没有内存表（MemTable）；
  • 它 不使用 WAL（Write-Ahead Log）；
  • 它 不在同一文件内做多层合并。

✅ 实际机制：LSM-style 的文件级合并策略

Paimon 的更新能力来源于其独特的 Sorted Run + Compaction 模型，灵感来自 LSM，但实现于分布式文件系统之上。

工作流程如下：

  1. 写入阶段：Flink 将 changelog（+I, -U, +U, -D）以 Parquet/ORC 文件形式写入 data/ 目录；
  2. 组织方式：按主键哈希分 bucket，每个 bucket 内部按主键排序（Sorted Run）；
  3. 合并阶段：后台 Compaction Job 异步合并多个小文件，按主键去重、保留最新版本；
  4. 读取阶段：查询时通过最新快照获取所有有效文件，跳过已被覆盖的旧记录。
这种“文件级 LSM”设计，既保留了 LSM 的高效更新特性，又适配了对象存储的不可变性。

四、表类型全景：从“静态表”到“动态数据体”的进化

Paimon 不只是一个“表格式”，而是一个多模态数据实体管理平台，支持多种数据组织方式：

类型特性典型场景
主键表支持 upsert/delete，后台合并保留最新用户画像、维度表、CDC 同步
Append-only 表仅追加写入，类似 Hive日志流、事件数据
物化表自动维护聚合结果，Flink 实时刷新实时指标、宽表构建
虚拟表（View）仅保存查询逻辑，跨引擎共享统一语义层、指标定义
格式表（Normal Table）映射外部 CSV/Parquet 目录，无需转换湖仓融合、存量数据接入
对象表（Object Table）管理 OSS/S3 上的非结构化文件元数据图片、文档、音视频分析

▶️ 深度融合场景：多模态协同工作流

-- 1. 映射原始日志目录（格式表）
CREATE TABLE raw_logs WITH ('type'='format-table', 'path'='s3://logs/');

-- 2. 创建用户画像主键表
CREATE TABLE user_profile (
    user_id BIGINT PRIMARY KEY,
    last_login TIMESTAMP
) WITH ('merge-engine'='deduplicate');

-- 3. 定义虚拟视图，供多引擎使用
CREATE VIEW user_active_7d AS 
SELECT user_id FROM user_profile 
WHERE last_login >= CURRENT_DATE - INTERVAL '7' DAY;

-- 4. 物化每日订单聚合
CREATE MATERIALIZED TABLE daily_order_sum AS
SELECT DATE(order_time), SUM(amount) FROM orders GROUP BY 1;
✅ 一套系统，完成“接入、更新、聚合、共享”全链路。

五、流批统一的本质：不是“都能读”，而是“语义一致”

很多文章说“Paimon 支持流批读取”，但这只是表象。

真正的价值在于：无论你是批处理还是流处理，看到的是同一份语义一致的数据视图。

统一读取模型

模式SQL 示例返回内容
批读取SELECT * FROM t当前快照的完整状态
流读取SELECT * FROM t /*+ OPTIONS('scan.mode'='latest') */自上次以来的所有变更（含 rowkind）

关键点： - 批读取 ≠ 全量扫描：它读的是最新快照，性能接近 Iceberg； - 流读取 ≠ 持续拉取：它基于快照增量拉取，避免重复消费； - rowkind 字段：暴露变更类型（+I/-U/+U/-D），实现精确 CDC 语义。

换句话说：Paimon 把“表”变成了一个 可随机访问的 changelog 存储 —— 这才是“流批一体”的终极形态。

▶️ 增强能力：Lookup Joins 与 First Row 去重


1. Lookup Joins：流式维度关联利器

在 Flink 流作业中，常需关联维度表。Paimon 原生支持 Lookup Join，通过主键索引快速点查：

SELECT o.order_id, o.user_id, p.level 
FROM orders_stream AS o
JOIN user_profile FOR SYSTEM_TIME AS OF o.proc_time AS p
ON o.user_id = p.user_id;
  • ✅ 自动缓存热点数据；
  • ✅ 支持 TTL 过期；
  • ✅ 比 Hudi MOR Lookup 性能提升 3~5 倍。

2. First Row 合并引擎：窗口去重最佳实践

对于“只保留第一条记录”的场景（如首次登录），Paimon 提供 first-row 合并语义：

CREATE TABLE user_first_login (
    user_id BIGINT PRIMARY KEY,
    first_login_time TIMESTAMP(3)
) WITH ('merge-engine' = 'first-row');
  • ✅ 写入时自动忽略后续更新；
  • ✅ 配合 CDC 使用，实现“首次事件捕获”。

六、CDC 入湖：从“链路拼接”到“直写即服务”

⛓ 传统 CDC 架构痛点

MySQL → Debezium → Kafka → Flink → Iceberg/Hudi

问题显而易见： - 组件多，延迟高（>30s 常见）； - Kafka 成本高，运维复杂； - 偏移量管理易出错，数据丢失难追溯。

Paimon 解法：Flink CDC 直写 Paimon

env.fromSource(mySqlSource, WatermarkStrategy.noWatermarks(), "MySQL-CDC")
   .sinkTo(PaimonSink.forRowData(ROW(BIGINT, STRING, INT))
           .setTableIdentifier("catalog.db.users")
           .build());

优势： - ✅ 端到端延迟 < 1s - ✅ 零中间件依赖 - ✅ 自动处理 schema 演化 - ✅ 支持 exactly-once 语义

更进一步：
Paimon 可作为 统一变更枢纽（Change Hub），供多个下游消费：

  • Flink 流任务：订阅用户资料变更，触发推荐系统；
  • Spark 批任务：每日拉取增量，构建画像宽表；
  • Trino 查询：查看最新用户状态；
  • AI 工作流：监听新增图片，调用模型打标。
它不再是“一个表”，而是一个 变更分发中心。

七、Compaction 机制详解：合并如何工作？


Compaction 是 Paimon 高效更新的核心，其流程如下：

# 1. 写入产生多个小文件（Sorted Runs）
data/
  bucket-0/001.parquet  # user:1001 → name:A
  bucket-0/002.parquet  # user:1001 → name:B
  bucket-0/003.parquet  # user:1001 → name:C

# 2. Compaction 合并为一个文件
data/
  bucket-0/merged-001.parquet  # user:1001 → name:C（保留最新）

# 3. 更新快照元数据，标记旧文件可删除
  • 支持多种合并策略：deduplicate, aggregation, partial-update, first-row
  • 可配置触发条件：file-num, size, time
  • 支持批模式独立运行，避免阻塞写入

最佳实践：生产环境应 分离写入与 compaction 任务：

# 写入 Job
write-only: true
compaction.enabled: false

# 单独启动 Compaction Job
execution-mode: batch
compaction.trigger-mode: file-num
类似数据库中的“归档进程”，避免合并操作干扰主写入流。

八、元数据与可观测性：从系统表到 REST API

Paimon 不仅是一个存储格式，更是一个可治理、可扩展的数据运行时。

✅ 系统表：数据治理的显微镜

Paimon 提供了丰富的 $ 开头系统表，用于监控与诊断：

系统表用途
$snapshots查看快照历史、提交时间
$files分析文件分布、小文件问题
$audit_log获取完整 changelog（rowkind）
$consumers查看消费者进度
$tags, $branches支持数据版本标记（类似 Git）

实战示例：诊断写入延迟

-- 检查快照提交间隔
SELECT 
    snapshot_id,
    commit_time,
    TIMESTAMPDIFF(SECOND, commit_time, LEAD(commit_time) OVER (ORDER BY snapshot_id DESC)) AS delay_sec
FROM users$snapshots 
ORDER BY snapshot_id DESC LIMIT 10;

若 delay_sec > 60，说明写入卡顿；再查 $files 判断是否 compaction 未生效。

✅ RESTCatalog 与 REST API：跨语言生态集成


Paimon 支持 RESTCatalog，通过 HTTP 接口管理元数据，实现多语言接入：

# 配置 REST Catalog
catalog:
  type: rest
  uri: http://paimon-rest:8080
  • ✅ 支持 Python、Go、Rust 等语言通过 REST API 创建表、提交快照；
  • ✅ 与 DataHub、Amundsen 等元数据平台集成；
  • ✅ 支持身份认证与访问控制（JWT/OAuth）；
是构建 统一数据平台（Unified Data Platform） 的关键组件。

九、复杂类型与类型演化：为现代数据建模而生

Paimon 不仅支持 ARRAY<MAP<STRING, ROW<...>>> 这类嵌套结构，还支持：

  • 字段描述：ROW<name STRING '姓名', age INT '年龄'>
  • 类型演化：添加字段、修改精度、重命名（需开启 schema.change-mode）
  • 主键包含复杂类型：实验性支持（如 ARRAY 作为主键分桶字段）
特别适合处理 JSON 扁平化前的原始数据，减少 ETL 成本。

Paimon 的野心不止于“表格式”。

十二、结语：Paimon 正在改写数据架构的底层逻辑

Paimon 的真正意义，不在于它多快或多省，而在于它让我们重新思考：什么是“数据”？

在过去，数据是“结果”；
在未来，数据是“过程”。

Paimon 把“变更”作为一等公民，让表既能存储状态，也能传播变化。

它不是 Iceberg 的对手，而是 下一代数据架构的探路者。

如果你正在构建： - 实时数仓 - 动态维度系统 - 流式特征平台 - 湖仓一体架构

那么，请认真对待 Paimon ——
因为它可能就是你一直在等的那个 “统一数据入口”。

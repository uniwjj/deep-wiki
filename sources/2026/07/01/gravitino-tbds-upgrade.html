01

TBDS 产品技术架构升级

首先简要介绍一下腾讯云 TBDS 产品。

1. 腾讯云 TBDS 产品形态

TBDS（Tencent Big Data Suite）是基于腾讯多年大数据实践，面向数据全生命周期，对外提供的安全、可靠、易用的一站式、高性能、企业级大数据平台，具备开箱即用等特性。
TBDS 产品具备两种产品形态：一种是面向传统 Hadoop 生态的架构，另一种是基于 Lakehouse 数据架构打造的云湖仓产品架构。
TBDS 通过一套平台实现了两种产品形态的统一，包括在资源体系、数据体系、用户权限体系以及运维管控体系等方面的全面统一，可确保新老架构之间的丝滑演进。

2. TBDS 全新一代云湖仓产品架构，类数仓体验

TBDS 全新一代云湖仓产品架构是基于 Lakehouse 数据架构打造的全场景湖仓产品，类数仓体验，具备计算隔离性强、存算按需伸缩、全托管智能调优等众多高级特性，通过统一元数据的支持，可与多家厂商的数据系统实现无缝互访，数据无需搬迁，从而可以保护客户现有投资，降低用户整体使用成本。
新架构核心主要包含存储、计算和公共服务三部分。存储方面支持 HDFS 和符合 S3 标准的对象存储；计算方面支持批处理 Spark、流处理 Flink、交互式分析 Trino，以及最新扩展支持的 StarRocks。

3. 新老架构演进面临的挑战

在新老架构演进的过程中，主要会面临架构、数据和资源三个层面的挑战：
  • 架构层面：存算一体架构存在资源分配相对固定，长期运营容易导致资源利用不均等问题，难以适应企业多样化、高变业务需求。而存算分离后可实现计算插件化、存储多样化以及存算按需伸缩。
  • 数据层面：数据孤岛造成数据共享难，使得企业难以全面准确的掌握数据全貌，从而制约了企业数据价值挖掘与创新发展，致使创新协同受限。统一数据视图后可实现安全便捷的数据共享并促进价值创新。
  • 资源层面：池化向云化演进时面临资源利用率低、管理复杂等问题，比如传统大数据主要还是基于 Yarn 来池化资源， 对于在离线混部以及资源的动态调配等方面存在诸多局限性。而基于云化资源来实现统一资源池可以有效提升资源的使用。

4. 存算分离架构：存算独立扩展，成本效益最大化

首先是架构层面。
一个简化的存算分离架构的基本实现，核心在于将元数据独立出来管理，同时构建无状态、可插拔的计算层，以及可扩展、可共享的存储层。然而，这仅仅完成了存算分离架构的初级形态。想要达到符合生产要求的完整架构，还需进行多方面的优化工作和兼容性调整。
存算一体相比存算分离主要优势体现在性能方面。尽管存算一体通过计算与存储的紧密耦合实现了低延迟的本地数据访问，但存算分离可以借助网络并发策略和计算下推机制，在一定程度上抵消这部分带来的损失。更进一步还可以通过优化元数据管理、引入缓存加速逻辑以及提升引擎服务（如向量化、RSS 技术）等优化策略，来进一步提升整体性能及稳定性，最终实现综合收益最大化。
上图右侧列举了该架构带来的诸多优势，比如存算按需伸缩、故障域隔离、扩容无需数据重分布以及安全便捷的数据共享。尤其可扩展性设计方面，使得 TBDS 新架构能够兼容多家厂商数据系统，从而有效保护客户现有投资。

5. 多种因素带来的数据孤岛挑战

其次是数据层面。
数据孤岛可以说是老生常谈的问题了。由于组织、技术、安全和管理等因素的存在，导致数据难以共享和互联互通。尤其是在业务以及技术创新发展的驱动下，比如多云、多负载场景，会进一步加剧数据孤岛的存在，从而导致数据平台的价值发挥受限。
鉴于组织壁垒、安全限制等诸多复杂且难以调和的因素存在，企业数据孤岛实际上是很难在短时间内彻底消除的，那么如何有效避免或缓解数据孤岛的影响呢？关键在于推动元数据的统一，这是实现数据共享最大化的核心所在。

6. 直管与托管模式的选择与设计

统一元数据服务一般存在两种设计模式：直管模式和托管模式。但无论采用哪种模式，一般对外都是提供统一的元数据访问方式。
直管模式就是直连真实的数据源，无需爬取元数据，可确保数据的实时性和一致性，比较适合引擎在线使用。
托管模式就是将真实的数据源的元数据同步至托管存储，再对外提供统一的元数据服务，由于同步的存在，可能会引入不一致性，一般适合数据治理和迁移升级使用。
因为适用场景不同，在企业实际落地过程中，这两种模式一般都会同时存在，所以 TBDS 基于 Gravitino 进行了元数据服务的改造，让 TBDS 同时支持这两种模式，这样可以更好地适用新老业务并跑以及架构平滑升级。

7. 统一数据管控面，实现全方位湖仓管理

同时，在新架构里面还将数据权限、数据调优等相关能力融入到统一元数据服务当中，从而实现全方位便捷的湖仓管理。
众所周知，传统大数据系统普遍采用 Ranger 来实现统一的权限管理，然而 Ranger 的权限管理机制是围绕引擎服务独立管理的，这意味着每类引擎服务都需要单独管理其策略服务，这样多套策略很容易出现不一致，且管理成本较高，还可能引发安全风险。
如果权限策略管理跟随数据而非引擎服务，就不存这样的问题，还可以实现所有引擎无差别的访问数据。只要遵循这一原则，无论当前采用何种引擎、或是未来扩展新引擎，其行为上都将保持高度一致，这样就可以大大降低了数据管理成本。
除了权限问题，小文件治理也是数据湖管理经常遇到的问题，大量的小文件会影响整体的性能和稳定性。通过将数据治理优化跟元数据服务相结合，可以针对不同的数据读写模型实现智能化调优，从而大大简化数据湖的使用和维护。

8. 全新资源管理模型：统一资源池+多层级资源隔离+存算动态伸缩

最后是资源层面。
TBDS 新架构采用了全新的资源管理模型，将云化资源和传统 Yarn 管理的资源进行了统一抽象，并以资源组方式对外提供统一服务。
传统模式下 Yarn 使用队列来管理资源，而云化场景下 TBDS 则抽象出虚拟计算 VC 来管理资源。原先是将作业提交至 Yarn 队列上运行，而云化场景下是将作业提交至 VC 上运行。此种设计确保了用户端体验的一致性，使得用户无需深入了解云化底层信息。
同时，新的资源管理模型还实现了跨异构集群资源的统一调度，涵盖存储和存算。存储方面可以无缝对接不同厂商数据存储系统，如 HDFS 存储和符合 S3 标准的对象存储；计算方面则支持对接多个独立的 Kubernetes（k8s）集群，甚至是不同版本的 k8s 集群。这一特性打破了存算单集群或联邦集群的限制，实现了资源层面的平行扩展、不受限制。
此外，在资源层面还提供了多层级的资源隔离机制和基于在离线混部时负载感知的动态伸缩能力，在满足业务的同时可以精细化控制资源成本，从而为客户提供有效地成本优化手段，比如应对潮汐效应明显的业务场景。

9. ALL in One 一站式全场景数据分析

前面主要是从架构、数据、资源三个层面简要介绍了新老架构演进时遇到的问题及解法，本质上还是在提升大数据系统全场景下的综合表现。
TBDS 新架构通过单一产品形态实现了实时场景、离线场景、交互场景的全覆盖，配合数据开发治理平台 WeData、腾讯云 BI 等相关产品可实现全链路产品方案覆盖，一站式满足企业客户的各种需求。
02

Apache Gravitino 统一元数据湖

1. 数据被地域分割

现在越来越多的用户开始在多云上开展自己的业务，使用公有云上大量的 SaaS 服务和在私有云上自己部署的大数据系统。一个企业从单云向多云演进，不可避免地会导致数据被分割在多个地方，特别是在出海业务日益增长的今天，很多业务是在海外进行扩展的，受到各个国家地区的数据合规的要求，数据被分割在不同的地区，按照数据合规的要求，数据是不能够离线进行传输的。所以在多云多区域的情况下，数据孤岛的问题变得更加严重。

2. 数据被组织分割

在同一个企业内部，其实数据也是被分割在不同的部门或者是团队之中的。有个典型的例子，比如数据团队往往管理着结构化表格数据，而 AI 团队往往管理着像视频、图片、文件这些非结构化的数据，由于现在没有一个很好的系统能够同时管理结构化和非结构化的数据，所以导致数据在两个部门之间很难互相的流通，共同使用，这样我们就很难去开展新的业务。特别是在新的大模型时代，需要一种统一的方式去访问所有数据。

3. 数据被冰山隐藏

另外，现在很多数据，其实我们能够看见的都只是冰山一角，其下还隐藏着更大量的数据。比如商业数据、个人信息，还有从第三方购买的一些数据。由于这些数据缺乏有效的数据分类、数据发现和生命周期管理的能力，所以大量的数据被隐藏在冰山之下，并没有被有效利用。

4. 元数据湖– 统一的数据/AI 目录

Gravitino 的目标就是打破数据孤岛，构建下一代的数据目录，推进下一代数据架构的升级。Gravitino 将所有的元数据分为四个大类：第一个大类是以 Hive 的 Metastore 为标准的；第二大类是 MySQL 和 Oracle 这种内置了元数据服务的；第三大类像 Kafka 这种流式数据，通过 Schema Registry 进行元数据管理的这种系统；第四大类就是机器学习和大模型的模型的元数据。
Gravitino 统一管理好这四种元数据之后，向上可以为数据引擎和 AI 框架提供统一的元数据接入。让 Trino、Spark、PyTorch、TensorFlow 能够统一接入 Gravitino 统一管理好的元数据，就可以对 Gravitino下管理的所有底层数据源进行统一的数据处理。这样，使用 Gravitino 即可构建出一个数据的统一视图，从而进行统一的数据访问和治理。

5. Apache Gravitino 统一元数据湖

Gravitino 是一个可以在多云之上进行分布式部署的系统。我们在访问任何一个 Gravitino 的节点时，都可以连接到所有分散在各个云、各个 region 里面的数据，包括数据库、数据仓库文件，还有流数据和模型文件。通过 Gravitino 可以对分散在各个云、各个 region 的数据进行统一的访问和统一的管理。

6. Apache Gravitino 核心架构

上图展示了 Gravitino 的核心架构。Gravitino 的核心其实是定义了一套标准的元数据模型和 API 接口，可以使用 catalog.db.table 这种数据库三段式的方式去访问所有数据，不仅仅是 Table 的数据，还包括 Fileset 文件的数据模型数据，或者是一些流数据。Gravitino 向上提供统一的 RestAPI 和 Iceberg 的 API 接口，业务侧用引擎就可以统一进行数据处理和数据管理。
Gravitino 还提供了元数据的独立存储，可以扩展底层数据源额外的一些元数据信息，来丰富其元数据。向下还提供了各种各样的 Connection 连接到底层的数据源。包括 Tabular 表数据，文件模型，以及消息队列等等，使用 Gravitino 可以管理底层所有的数据源包，并且让各种引擎都可以无缝连接到这些数据。

7. Apache Gravitino 统一数据访问

Gravitino 统一管理了元数据之后，又统一了数据的访问。我们将数据的访问分为两大类：第一种是 Tabular 表格数据，第二种是 Non-tabular 的数据，比如文件、音视频这种非结构化的数据。
针对 Tabular 的数据，我们主要是为 Spark、Flink 和 Trino 提供了各式各样的 Gravitino 的 connector，让这些数据引擎可以通过这些 connector 直接连接到 Gravitino 里来，然后对这些元数据进行统一的数据加工处理。
对于非结构化的数据，Gravitino 提供了虚拟化的文件系统和 Fileset 的 Catalog，这样 PyTorch 和 TensorFlow 这些机器学习框架就可以通过这些虚拟化的文件系统，直接统一访问 S3、HDFS 中的非结构化数据。
Gravitino 通过接入 Tabular 和非 Tabular 的数据，就可以对 data 和 AI 团队和引擎提供统一的支持。

8. Apache Gravitino 统一权限

Gravitino 对底层的数据源还提供了统一的权限管理。各种各样的数据源，其权限模型和权限管理方式方法都是不一样的。大数据系统主要是使用 Ranger 进行权限管理，MySQL 和 Oracle 有自己的权限模型和操作方式，SaaS 服务也有自己的权限管理模式。权限模型的不同也是导致数据难以使用的一个痛点问题。
Gravitino 统一定义了一套自己的权限模型和REST接口，无论底层是什么样的数据源，都可以使用统一的 API 接口，把权限 push 到底层的数据源里去。这样就解决了之前不同数据源权限授权的难题，让用户使用数据变得更加简单。

9. 使用 Apache Gravitino 构建元数据湖

我们通过 Gravitino 可以构建出自己的元数据湖。前面讲到 Gravitino 可以对表数据的数据库还有文件进行底层数据源的管理。比如对表数据，提供了 JDBC 的 Catalog，可以管理 PG、MySQL 等数据库。数据湖支持了 Iceberg、Hudi 和 Paimon 这样的数据和格式，同时也支持 S3 和 HDFS 这些文件系统。另外通过 connector 已经支持了 Spark、Flink 这些数据引擎。另外在 AI 的机器学习框架上，支持 TensorFlow、PyTorch，这些都可以通过 Gravitino 的 Python 客户端进行连接使用。
在对数据源和引擎提供支持的基础上，业务层只需要对底层的数据源进行一次元数据的定义，就可以在各个引擎中无缝使用，可以做数据发现、数据 matrix、数据血缘，以及数据 pipeline 的处理等等。
所以，Gravitino 提供了一个通用的 API 和接口，在元数据的层面上实现了 single source of truth，把所有的数据构建成一个完整的知识图谱，让业务和引擎都能够无缝地使用。

10. 元数据驱动的数据智能

这是数据处理链路的完整链路图。最左边是数据源，我们从数据源中摄取数据，存储到 HDFS 数据库里面，再进行数据的清洗和转换，然后再把数据入湖入仓。基于数据湖和数据仓库里的数据就可以去做BI的商业智能，或者是大模型的 RAG 系统，还可以构建 AIGC 应用。
通过 Gravitino 可以很好地管理这些数据的元数据。我们认为元数据是数据的大脑，引擎和存储则是数据的腿和胳膊。Gravitino 在管理好整个数据链路的所有元数据之后，可以利用大模型的能力，让整个数据链路得到提升。通过语言数据去驱动数据的智能，实现整个数据架构的升级。

11. Apache Gravitino 社区开发者和用户

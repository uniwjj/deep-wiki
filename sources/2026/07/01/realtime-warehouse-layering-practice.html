一、前言：为什么实时数仓要“分层”？

在传统离线数仓体系中，我们早已熟悉了 ODS → DW → ADS 的三层结构。
但当业务对“实时分析”、“秒级看板”、“在线决策”提出要求时，离线分层模式就显得力不从心。

于是——实时数仓分层模型（ODS → DWD → DWS → ADS） 应运而生，它不仅借鉴了离线架构的精华，还融合了流式计算的灵活性，真正实现了 “T+0” 的数据驱动业务能力。

二、实时数仓的整体架构

一个完整的实时数仓通常包括以下组件：

  • 数据采集层（Kafka / Canal / Flink CDC）
    捕获业务数据库的变更日志或消息流。

  • 实时计算层（Flink / Spark Streaming）
    对实时数据进行清洗、聚合、关联、维表补充等处理。

  • 存储层（HBase / Kafka / Elasticsearch / StarRocks）
    存储不同粒度和用途的实时数据。

  • 应用层（BI 报表 / 实时看板 / 推荐系统）
    展示或消费实时结果。

三、分层设计详解

1️⃣ ODS 层（Operational Data Store）

定义：
实时数据的“落地缓冲层”，主要负责接入并原样保存源系统变更数据。

特点：

  • 不做复杂逻辑处理；

  • 一般使用 Kafka Topic 或 HDFS 临时存储；

  • 按业务系统区分，如 ods_order, ods_user。

设计建议：

  • 数据保留周期：一般 1~3 天；

  • 命名规范：ods_业务系统_表名_rt；

  • 示例：ods_trade_order_rt。

2️⃣ DWD 层（Data Warehouse Detail）

定义：
数据明细层，负责数据清洗、规范化、补全维度字段，是 实时数仓最关键的层级。

典型操作：

  • 过滤脏数据；

  • 统一时间字段格式；

  • 维表 Join（异步维表或广播维表）；

  • 提取业务主键，构建事实流。

示例：

-- 订单明细 DWD
CREATE TABLE dwd_trade_order_rt AS
SELECT
  order_id,
  user_id,
  product_id,
  price,
  quantity,
  order_time,
  province_id,
  city_id
FROM ods_trade_order_rt
WHERE order_status = 'paid';

输出场景：

  • 为 DWS 层提供统一的“事实数据流”。

3️⃣ DWS 层（Data Warehouse Service）

定义：
服务层（或宽表层），主要负责指标聚合与主题宽表构建。

典型操作：

  • 以时间窗口（Window）计算统计指标；

  • 按主题领域（如用户、订单、流量）汇总；

  • 支撑实时指标监控、看板展示。

示例：

-- 每小时订单金额汇总
CREATE TABLE dws_order_hour_rt AS
SELECT
  TUMBLE_START(order_time, INTERVAL '1' HOUR) AS window_start,
  SUM(price * quantity) AS total_amount,
  COUNT(DISTINCT user_id) AS unique_users
FROM dwd_trade_order_rt
GROUP BY TUMBLE(order_time, INTERVAL '1' HOUR);

存储：

  • HBase / ClickHouse / StarRocks 等高并发查询引擎。

4️⃣ ADS 层（Application Data Service）

定义：
应用服务层，面向报表与接口输出，是“业务方”最直接接触的数据。

主要功能：

  • 支撑实时 BI、监控告警、推荐算法；

  • 通过 REST API 或 SQL 查询对外提供服务。

示例：

  • 实时 GMV（总交易额）

  • 实时活跃用户数

  • 实时转化率监控

设计建议：

  • 数据结构应贴近业务需求；

  • 支持高并发访问（QPS > 1000）；

  • 常用落地：ES、Redis、StarRocks。

四、实时数仓与离线数仓的结合

对比项离线数仓实时数仓
延迟T+1秒级
技术栈Hive / SparkFlink / Kafka
存储HDFS / HiveKafka / HBase / StarRocks
场景报表统计实时看板 / 告警系统
数据一致性批处理保障精确一次 (Exactly Once)

在企业级项目中，通常采用 Lambda 架构（离线 + 实时）或 Kappa 架构（纯实时流）：

  • 小规模实时业务可用 Kappa；

  • 大规模报表系统推荐 Lambda，保障数据一致性。

五、落地案例：旅游行业实时数仓

背景：

某省文旅数据中心需要实时监控景区客流与门票收入。

方案：

  1. ODS 层：Flink CDC 采集景区售票系统 MySQL；

  2. DWD 层：清洗异常订单、统一时间字段；

  3. DWS 层：按小时聚合景区客流量、销售额；

  4. ADS 层：落地到 StarRocks 实时看板展示。

最终实现了：

  • 实时监控游客入园量；

  • 景区高峰预警；

  • 财务实时收入追踪。

六、总结

实时数仓分层设计的核心在于：稳定的数据链路 + 清晰的分层边界 + 高效的实时计算引擎。
只有合理划分层级，数据流才能稳定、可追踪、可扩展。

💡结语：未来的趋势

未来的数据体系，将朝着 “全实时化、批流融合化、智能调度化” 方向演进。
掌握分层设计思维，是成为大数据专家的“入场券”。

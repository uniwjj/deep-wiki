Gravitino Ranger 授权配置指南

1. 接口逻辑链路梳理

1.1 REST API 层

文件: server/src/main/java/org/apache/gravitino/server/web/rest/RoleOperations\.java

HTTP 方法路径功能
GET/metalakes/\{metalake\}/roles列出所有角色名称
GET/metalakes/\{metalake\}/roles/\{role\}获取指定角色详情
POST/metalakes/\{metalake\}/roles创建新角色
DELETE/metalakes/\{metalake\}/roles/\{role\}删除角色

关键逻辑:

  • 所有请求通过 Utils\.doAs\(\) 进行用户身份验证

  • 创建角色时会验证：

    • 权限名称不重复 (checkDuplicatedNamePrivilege)

    • 权限与对象类型匹配 (checkPrivilege)

    • 元数据对象存在性 (checkMetadataObject)

1.2 核心管理层

文件: core/src/main/java/org/apache/gravitino/authorization/AccessControlManager\.java

作为统一入口，协调三个核心组件：

AccessControlManager
├── RoleManager        # 角色 CRUD 操作
├── UserGroupManager   # 用户/组管理
└── PermissionManager  # 权限授予/撤销

1.3 RoleManager 核心逻辑

文件: core/src/main/java/org/apache/gravitino/authorization/RoleManager\.java

createRole 流程:

  1. 验证 Metalake 存在

  2. 构建 RoleEntity 并存储到 EntityStore

  3. 调用 Ranger 插件同步角色 (callAuthorizationPluginForSecurableObjects)

deleteRole 流程:

  1. 先从 EntityStore 获取角色信息

  2. 调用 Ranger 插件删除角色

  3. 从 EntityStore 删除记录

1.4 Ranger 授权插件集成

文件: authorizations/authorization\-ranger/src/main/java/org/apache/gravitino/authorization/ranger/RangerAuthorizationPlugin\.java

关键方法:

Gravitino 操作Ranger 插件回调
createRoleonRoleCreated\(\) → 创建 Ranger Role + Policy
deleteRoleonRoleDeleted\(\) → 删除 Ranger Role + Policy
grantPrivilegeToRoleonRoleUpdated\(\) → 更新 Ranger Policy Items
grantRolesToUseronGrantedRolesToUser\(\) → 添加用户到 Ranger Role
revokeRolesFromUseronRevokedRolesFromUser\(\) → 从 Ranger Role 移除用户

Ranger 数据模型映射:

Gravitino Role        →  Ranger Role
Gravitino Privilege   →  Ranger PolicyItemAccess
Gravitino SecurableObject →  Ranger Policy Resource
Gravitino User/Group  →  Ranger VXUser/VXGroup

1.5 关键设计模式

回调机制 (AuthorizationPluginCallbackWrapper)

在 PermissionManager 中，权限变更操作分两阶段执行：

  1. 先在 EntityStore 中更新数据

  2. 通过回调同步到 Ranger

// PermissionManager 中的典型模式
store.update(..., roleEntity -> {
    // 1. 构建新的 SecurableObjects
    // 2. 设置回调
    authorizationCallbackWrapper.setCallback(() -> 
        AuthorizationUtils.callAuthorizationPluginForMetadataObject(...)
    );
    return updatedRoleEntity;
});
// 3. 事务提交后执行回调
authorizationCallbackWrapper.execute();
批量操作支持

  • grantPrivilegeToRoles\(\) / revokePrivilegesFromRoles\(\) 支持批量处理

  • 使用 BatchAuthorizationPluginCallbackWrapper 累积变更，一次性同步到 Ranger

2. 多集群多 Ranger 配置方案

2.1 架构设计原理

Gravitino 的授权插件是按 Catalog 维度加载的，每个 Catalog 可以独立配置连接不同的 Ranger 服务。

2.5 角色与权限同步流程

当创建 Role 时，Gravitino 会：

1. 用户调用 POST /api/metalakes/{metalake}/roles
   └─> 2. RoleOperations.createRole()
       └─> 3. AccessControlManager.createRole()
           └─> 4. RoleManager.createRole()
               │
               ├─> 4a. 存储 Role 到 EntityStore (MySQL/PostgreSQL)
               │
               └─> 4b. callAuthorizationPluginForSecurableObjects()
                   │
                   └─> 遍历 SecurableObjects 找到对应 Catalog
                       └─> 调用该 Catalog 的 AuthorizationPlugin
                           └─> RangerAuthorizationPlugin.onRoleCreated()
                               │
                               ├─> 在对应 Ranger 服务创建 Role
                               └─> 在对应 Ranger 服务创建/更新 Policy

4. RangerAuthorizationPlugin 核心代码

4.1 插件入口类

文件: authorizations/authorization\-ranger/src/main/java/org/apache/gravitino/authorization/ranger/RangerAuthorization\.java

public class RangerAuthorization extends BaseAuthorization<RangerAuthorization> {
  @Override
  public String shortName() {
    return "ranger";
  }

  @Override
  public AuthorizationPlugin newPlugin(
      String metalake, String catalogProvider, Map<String, String> properties) {
    String serviceType = properties.get(RangerAuthorizationProperties.RANGER_SERVICE_TYPE).toUpperCase();
    switch (serviceType) {
      case "HADOOPSQL":
        return new RangerAuthorizationHadoopSQLPlugin(metalake, properties);
      case "HDFS":
        return new RangerAuthorizationHDFSPlugin(metalake, properties);
      default:
        throw new IllegalArgumentException("Unsupported service type: " + serviceType);
    }
  }
}

5. 注意事项

5.1 Ranger Service 配置

  • 预先创建: 确保每个 Ranger Admin 中已创建对应的 Service

  • 自动创建: 可配置 authorization\.ranger\.service\.create\-if\-absent=true 让 Gravitino 自动创建

5.2 用户/组同步

  • Gravitino 中的用户/组名需要与 Ranger 中的一致，否则授权会失败

  • 首次使用时，Gravitino 会自动在 Ranger 中创建不存在的用户/组

5.3 网络连通性

  • Gravitino Server 必须能访问所有 Ranger Admin 服务

  • 如有防火墙，需开放对应端口（默认 6080）

5.4 权限冲突处理

  • 如果多个 Catalog 配置了相同的 Ranger Service Name，可能导致权限冲突

  • 建议为每个 Catalog 使用唯一的 authorization\.ranger\.service\.name

5.5 Kerberos 认证

生产环境建议使用 Kerberos，配置示例：

authorization.ranger.auth.type=kerberos
authorization.ranger.username=gravitino/_HOST@EXAMPLE.COM
authorization.ranger.password=/etc/security/keytabs/gravitino.keytab
authorization.ranger.hadoop.security.authentication=kerberos
authorization.ranger.hadoop.rpc.protection=privacy

5.6 日志排查

遇到问题时，检查以下日志：

# Gravitino Server 日志
tail -f $GRAVITINO_HOME/logs/gravitino-server.log

# Ranger Admin 日志
tail -f /var/log/ranger/admin/ranger_admin.log

Q2: 多集群场景下，如何隔离权限？

  • 为每个集群创建独立的 Catalog

  • 每个 Catalog 配置不同的 authorization\.ranger\.service\.name

  • 创建 Role 时，通过 securableObjects 指定作用范围

Q3: 如何批量更新多个 Ranger 的权限？

使用批量操作接口：

// PermissionManager 中的批量方法
grantPrivilegeToRoles(
    String metalake,
    MetadataObject object,
    List<Pair<String, Set<Privilege>>> batchRoles,
    boolean notSyncRanger)

Q4: Ranger 服务不可用时，Gravitino 能否继续工作？

  • 可以。Gravitino 会将角色信息存储在本地 EntityStore 中

  • 但同步到 Ranger 的操作会失败，并记录警告日志

  • 建议配置监控告警，及时发现 Ranger 服务异常

01

引言：多模态洪流下的湖仓新挑战

随着多模态数据与 AI 应用的爆发式增长，传统以结构化数据为核心的大数据体系正面临前所未有的挑战。一方面，文本、图像、音视频、向量等异构数据分散在不同的存储系统（如 Iceberg、Hudi、对象存储、向量数据库）中，形成了难以逾越的“元数据孤岛”，不仅阻碍了数据的统一视图，也加剧了数据栈的不一致性。

另一方面，现有的技术栈在应对多模态工作负载时显得力不从心：

  • 计算效率低下：传统计算引擎（如 Spark）缺少为多模态数据设计的原生 UDF 和分布式处理能力，而一些 Python 框架则难以规模化扩展。

  • 存储性能瓶颈：以 Parquet 为代表的列存格式，虽擅长批量扫描，但在 AI 应用典型的高频随机访问和向量检索场景下，其固有的 Row Group 设计、元数据开销和 I/O 放大问题成为了性能枷锁。

为了应对这些挑战，业界正在探索一套新的技术范式。其中，由 Gravitino、Daft 和 Lance 组成的“三件套”正成为一个备受关注的解决方案，分别从元数据治理、多模态计算和底层存储三个层面，为构建现代多模态湖仓提供了坚实的基础。

02

Gravitino：AI-First 的统一元数据目录

Gravitino 并非又一个元数据管理系统，而是一个专为 AI 时代设计的“目录的目录”（Catalog of Catalogs）。它的核心价值在于为上层应用提供一个统一的数据视图，打破底层异构数据源带来的壁垒。

1. 统一视图与联邦查询

Gravitino 能够统一管理包括数据湖表（Iceberg/Hudi）、非结构化数据（通过 Fileset 概念）、特征数据、模型元数据乃至 Lance 向量数据在内的全生命周期 AI 元数据。通过提供统一的 REST API，它极大地降低了 AI/Python 生态接入的门槛。

更重要的是，Gravitino 提供了强大的联邦查询能力。这意味着用户可以在同一个查询中，无缝地融合来自 Hive 的传统结构化数据与来自 Lance 的多模态 AI 数据，实现跨数据源的洞察分析，真正打破了 BI 与 AI 的数据边界。

2. 不只是元数据：智能的数据优化与治理

除了元数据管理，Gravitino 还扮演着“元数据驱动的行动系统”（Metadata Action System）的角色。它能够基于用户定义的策略（Policy）和系统采集的统计信息（Statistics），自动执行数据优化任务，例如：

  • TTL (Time-To-Live)：自动清理过期数据。

  • Compaction：合并小文件，优化存储和查询性能。

  • 数据迁移与压缩：根据数据冷热程度，自动将其迁移至成本更低的存储层或进行压缩。

这种将治理能力与元数据深度融合的设计，使得数据湖的管理从被动响应转变为主动优化。

03

Lance Namespace：SPEC 优先的开放生态哲学

Lance 社区在设计其元数据管理层 Namespace 时，借鉴了 Iceberg 的成功经验，采取了“SPEC 优先，实现其次”的哲学。

与某些实现即标准的项目不同，Lance 首先定义了一套与语言和计算引擎无关的清晰规范（SPEC）。这种做法极大地激发了开源生态的活力，允许多个社区、多种语言（如 Rust、Java、Python）和不同计算引擎（如 Spark、Daft、Trino）的实现能够齐头并进，避免了单一语言实现遥遥领先而其他生态严重滞后的“生态绑架”问题。

正是通过这套开放的 SPEC，Lance Namespace 成功地在各种计算引擎和元数据服务之间架起了一座桥梁，为构建其强大的AI和BI生态奠定了基础。

04

Gravitino 与 Lance 的双向奔赴：两种接入路径

为了充分发挥二者的协同效应，Gravitino 提供了两种接入 Lance 的路径，各有侧重，满足不同场景的需求：

1. 路径一：通过 Gravitino 统一 Table API 接入

  • 核心优势：这种方式能最大化发挥 Gravitino 的核心能力。所有对 Lance 的操作都通过 Gravitino 的 REST API 进行，使得 Lance 数据集能够无缝融入 Gravitino 的统一视图和联邦查询体系中。更重要的是，可以对 Lance 数据集应用 Gravitino 强大的数据治理能力，如统一的权限控制、审计、血缘追踪和数据优化。

  • 适用场景：需要将 Lance 与其他数据源进行联邦分析，或希望在一个统一的平台上对所有数据资产（包括 Lance）实施标准化、企业级安全治理的场景。

2. 路径二：将 Gravitino作为 Lance 的 Catalog 实现

  • 核心优势：对于已经在使用 Lance 并希望快速引入元数据管理能力的用户而言，这条路径提供了一个低门槛的选项。用户可以直接使用原生的 Lance Client，只需将 Catalog 指向 Gravitino 即可。这样做的好处是能够继续使用 Lance 的一些原生高级特性，如时间旅行（Time Travel），同时又能利用 Gravitino 提供的元数据持久化和管理能力。

  • 适用场景：已有大量基于 Lance 原生 API 构建的应用，希望在不改变上层代码的情况下，快速引入一个稳定、可靠的外部元数据管理服务的场景。

通过这两条路径，Gravitino 和 Lance 实现了灵活而紧密的集成，让用户可以根据自身的技术栈成熟度和业务需求，选择最适合的“姿势”来拥抱多模态湖仓。

05

企业级安全治理：为多模态数据保驾护航

当数据变得统一和易于访问时，安全治理就显得尤为重要。Gravitino 提供了一套完整的、可低成本应用于 Lance 等多模态数据源的企业级安全框架，覆盖了认证、鉴权和审计三大核心环节。

1. 灵活的认证机制

Gravitino 支持多种认证方式，以适应不同的企业环境：

  • OAuth2：与现代云服务和 SaaS 应用无缝集成。

  • Kerberos：兼容现有大数据生态的用户认证体系。

  • Simple 认证：用于开发和弱安全场景，仅通过用户名识别，无需校验密码。

  • 可扩展插件：允许用户自定义认证逻辑，满足特殊的安全需求。

2. 精细而高效的 RBAC 鉴权

Gravitino 的鉴权体系基于角色访问控制（RBAC），并引入了两个极具特色的设计，实现了便利性与严谨性的平衡：

  • 权限继承（Privilege Inheritance）：Gravitino 将数据对象组织成一个树状结构（Catalog -> Schema -> Table）。管理员可以直接在一个较高的层级（如 Schema）上授权，该权限会自动被其下的所有子对象（如所有 Table）继承。这极大地简化了授权操作，避免了逐表授权的繁琐，使得按业务域或数据域进行批量授权成为可能。

  • Deny 优先（Deny-First）：为了处理权限继承带来的例外情况，Gravitino 采用了 Deny 优先的规则。例如，管理员可以将整个 Schema 的读取权限授予某个团队，但同时对其中一两张高度敏感的表明确设置 Deny 规则。这样，用户在访问时，即使拥有上层目录的权限，也会因为明确的 Deny 而被拒绝，确保了敏感数据的安全。

这套“继承赋权、Deny 兜底”的机制，在保证高效管理的同时，也提供了金融级别的安全严谨性。

06

总结与未来展望

由 Gravitino、Daft 和 Lance 构成的“三件套”为解决多模态数据时代的挑战提供了一套开源、协同且高效的解决方案：

  • Gravitino 提供了统一的数据视图和强大的治理能力，打破了元数据孤岛。

  • Daft 提供了专为多模态数据设计的分布式计算能力。

  • Lance 解决了底层存储的性能瓶颈，尤其是在随机访问和向量检索方面。

这套组合不仅避免了拼凑多个单一功能系统带来的高昂成本和复杂性，也有效规避了厂商锁定的风险。

展望未来，Gravitino 社区将继续深化与 Daft、Lance 等生态的集成，推动 Lance REST Namespace 在更多生产环境的落地，并与社区共同打磨其使用体验，共同塑造 AI/BI 一体化的多模态湖仓新范式。

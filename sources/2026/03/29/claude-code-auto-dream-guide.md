# Claude Code 自动梦境（Auto Dream）：Anthropic 的全新记忆功能

**原文链接**: https://claudefa.st/blog/guide/mechanics/auto-dream

---

## 一句话总结

Claude Code 自动梦境（Auto Dream）功能通过四阶段流程（定向→收集信号→整合→修剪索引）自动整理记忆文件，将相对日期转换为绝对日期、删除过时信息、解决矛盾条目，解决了自动记忆（Auto Memory）随时间积累导致的信号噪声比下降问题，类似于人类 REM 睡眠对短期记忆的巩固过程。

---

## 完整翻译

Claude Code 自动梦境（Auto Dream）整合记忆文件，修剪过时的笔记并合并洞察。就像你的 AI Agent 的 REM 睡眠。

Anthropic 悄悄地向 Claude Code 发布了一个他们尚未正式宣布的功能。它叫做自动梦境（Auto Dream），它解决了自动记忆（Auto Memory）的最大问题：随时间衰减。

### 问题：自动记忆的困境

自动记忆（Auto Memory）是一个突破。Claude 终于开始自己记录关于你项目的笔记。但在 20+ 次会话之后，这些笔记变成了一团糟。矛盾的条目堆积如山。像"昨天"这样的相对日期失去了意义。过时的调试解决方案引用的文件已经不存在了。本应帮助 Claude 记住的笔记本反而变成了让它困惑的噪音。

**快速检查**：检查自动梦境是否在你的设置上处于激活状态。在任何 Claude Code 会话中运行 `/memory`。在选择器中查找 "Auto-dream: on"。如果你看到了，Claude 已经在后台会话之间整合你的记忆文件了。

如果自动梦境可用且已开启，你就设置好了。Claude 会定期审查你项目中的每个记忆文件，修剪过时的内容，解决矛盾，并重新组织其余内容。如果你还没有看到这个开关，请继续阅读。你可以手动触发它。

### 命名背后的隐喻

这个命名是刻意的，而且这个隐喻出奇地准确。

白天，你的大脑吸收原始感官输入并将其存储为短期记忆。在 REM 睡眠期间，你的大脑回放当天的事件，加强重要的连接，丢弃不重要的内容，并将所有内容组织成长期记忆。没有得到足够 REM 睡眠的人难以形成持久的记忆。信息进来了，但从未被巩固。

**自动记忆（Auto Memory）**是 Claude 的白天大脑。它在工作时做笔记：调试模式、构建命令、架构决策、你的偏好。每次会话都会添加更多条目。但如果没有整合步骤，这些笔记会以同样的方式积累，就像未巩固的短期记忆一样。矛盾持续存在。过时的条目挥之不去。每次会话信号噪声比都会下降。

**自动梦境（Auto Dream）**是 REM 睡眠周期。它审查自动记忆收集的内容，加强仍然相关的内容，移除过时的内容，并将其余内容重新组织成干净、索引的主题文件。没有自动梦境的 Claude Code 本质上是睡眠不足的。它不断添加随机笔记而从不清理。

### 自动梦境的四阶段流程

当自动梦境运行时，它遵循一个结构化的四阶段流程。每个阶段都有特定的目的，它们共同将分散的会话笔记转化为有组织的项目知识。

#### 阶段 1：定向（Orientation）

Claude 读取当前记忆目录并清点存在的内容。它打开 MEMORY.md（索引文件），扫描主题文件列表，并构建当前记忆状态的心理地图。

这个阶段回答："我已经知道什么，它是如何组织的？"

#### 阶段 2：收集信号（Gather Signal）

这是自动梦境搜索你的会话记录（Claude 本地为每次会话存储的 JSONL 文件）的地方。它不会从头到尾阅读每个记录。对于拥有数百次会话的项目来说，那将过于昂贵。相反，它狭隘地搜索特定模式：

- **用户纠正**：你告诉 Claude 它错了或重定向其方法的时刻
- **显式保存**：你说"记住这个"或"保存到记忆"的时候
- **重复主题**：在多个会话中出现的模式
- **重要决策**：架构选择、工具选择、工作流程变更

这种狭隘的搜索是故意的。详尽地阅读 500+ 会话记录会消耗大量令牌和时间，收益递减。有针对性的 grep 风格搜索有效地提取最高价值的信号。

#### 阶段 3：整合（Consolidation）

核心阶段。Claude 将新信息合并到现有的主题文件中，并执行关键维护：

- **将相对日期转换为绝对日期**。"昨天我们决定使用 Redis"变成"在 2026-03-15 我们决定使用 Redis"。这可以防止随着时间推移记忆老化而产生的时间混淆。
- **删除矛盾的事实**。如果你在三周前从 Express 切换到 Fastify，旧的"API 使用 Express"条目会被移除。
- **移除过时的记忆**。关于在重构期间被删除的文件的调试笔记没有用处。它们会被修剪。
- **合并重叠的条目**。如果三个不同的会话记录了相同的构建命令怪癖，它们会整合成一个干净的条目。

#### 阶段 4：修剪和索引（Prune and Index）

最后阶段专注于 MEMORY.md 索引文件。这个文件保持在 200 行以下，因为这是启动时加载的截止点。第 4 阶段更新 MEMORY.md 以准确反映所有主题文件的当前状态：

- 移除指向不再存在的主题文件的指针
- 添加指向整合期间创建的新主题文件的链接
- 解决索引与实际文件内容之间的矛盾
- 按相关性和新近度重新排序条目

在整合期间不需要更改的文件保持不动。自动梦境不会在每次运行时重写所有内容。它是外科手术式的。

以下是驱动自动梦境的实际系统提示。这是当梦境周期启动时 Claude 收到的内容：

> 关于这个提示有几点值得注意。它明确告诉 Claude 要狭隘地使用 grep 而不是阅读完整的记录。它强制执行 MEMORY.md 的 200 行限制。它指示 Claude 在源头解决矛盾，而不是仅仅标记它们。这个提示对记忆应该如何结构化有明确的观点，这就是为什么输出始终干净的原因。

### 触发条件

在整合周期触发之前，必须同时满足两个条件：

- 自上次整合以来已过去 24 小时
- 自上次整合以来已发生超过 5 次会话

两个条件都是必需的。如果你有一次持续两天的长会话，自动梦境不会触发（会话不够）。如果你在两个小时内运行了 10 次快速会话，它也不会触发（时间不够）。这种双重门防止在轻度使用的项目上不必要的整合运行，同时确保活跃项目得到定期清理。

作为背景，一次整合可能有多重：在一个观察到的案例中，自动梦境在大约 8-9 分钟内整合了来自 913 次会话的记忆。不微不足道，但它在后台运行，所以你不会注意到成本。

### 安全约束

自动梦境运行时带有有意义的约束以防止意外。

**项目代码的只读模式**。在梦境周期期间，Claude 只能写入记忆文件。它不能修改你的源代码、配置、测试或任何其他项目文件。梦境过程被沙盒化到记忆目录。

**锁文件防止并发运行**。如果两个 Claude Code 实例在同一个项目上打开，只有一个可以运行自动梦境。锁文件确保没有两个整合过程同时运行，防止记忆文件中的合并冲突。

**后台执行**。你可以在自动梦境运行时继续在 Claude Code 中工作。它不会阻塞你的会话或要求你等待。整合在单独的进程中发生。

### 手动触发

你不必等待自动触发。如果你知道你的记忆文件需要清理（例如，在重大重构之后），你可以强制进行整合。

`/dream` 命令存在但尚未向所有人推出。与此同时，你可以通过直接告诉 Claude 来手动触发它：

> "运行自动梦境周期" 或 "整合我的记忆文件"

Claude 识别意图并运行相同的四阶段流程。这在你想要干净记忆而不等待下一个自动周期的大项目变更之后很有用。

### 前后对比

以下是典型的记忆目录在自动梦境运行前后的样子。

**之前（混乱，30+ 会话的积累）**：
- 随机笔记文件堆积
- MEMORY.md 超过 200 行
- 矛盾条目并存
- 相对日期造成混淆

**之后（已整合）**：
- random-notes.md 文件消失了。其内容要么合并到相关的主题文件中，要么作为过时的被修剪。
- MEMORY.md 从 280 行减少到 142 行，回到 200 行启动阈值以下。
- 矛盾已解决。
- 相对日期已转换。

### 自动记忆 vs 自动梦境

这两个功能是互补的，而不是竞争的。将它们视为单一记忆系统的收集层和维护层。

| 方面 | 自动记忆（Auto Memory） | 自动梦境（Auto Dream） |
|-----|------------------------|------------------------|
| 角色 | 收集新信息 | 整合现有信息 |
| 何时运行 | 每次会话期间 | 定期（24小时 + 5次会话） |
| 做什么 | 写入关于发现模式的笔记 | 修剪、合并和重新组织笔记 |
| 触发 | 自动、连续 | 自动或手动 |
| 输出 | 记忆文件中的新条目 | 清理、重新组织的记忆文件 |
| 跳过风险 | 错过捕获有用信息 | 记忆质量随时间下降 |

没有自动梦境的自动记忆是一个从不整理笔记本的笔记记录者。没有自动记忆的自动梦境没有什么可整合的。你希望两者都运行。

### Claude Code 的四种记忆机制

有了自动梦境，Claude Code 现在有四种不同的记忆机制。以下是它们如何组合在一起：

| 方面 | CLAUDE.md | 自动记忆（Auto Memory） | 会话记忆（Session Memory） | 自动梦境（Auto Dream） |
|-----|-----------|------------------------|---------------------------|------------------------|
| 谁写入 | 你 | Claude（每次会话） | Claude（自动） | Claude（定期） |
| 目的 | 指令和规则 | 项目模式和学习 | 会话摘要 | 记忆整合 |
| 何时运行 | 你手动编辑 | 每次会话期间 | 后台，每约 5K 令牌 | 每 24小时 + 5次会话 |
| 范围 | 每个项目或全局 | 每个项目 | 每次会话 | 每个项目 |
| 启动时加载 | 完整文件 | MEMORY.md 的前 200 行 | 相关的过去会话 | 不适用（在会话之间运行） |
| 存储位置 | ./CLAUDE.md | ~/.claude/projects/<project>/memory/ | ~/.claude/projects/<project>/<session>/session-memory/ | 与自动记忆相同 |
| 最适合 | 标准、架构、命令 | 构建模式、调试、偏好 | 会话之间的连续性 | 保持记忆干净和准确 |
| 人类类比 | 说明书 | 白天做笔记 | 短期会话回忆 | REM 睡眠整合 |

最强的设置使用全部四种。CLAUDE.md 提供你控制的权威规则。自动记忆捕获 Claude 在工作时学到的东西。会话记忆保持会话连续性。自动梦境保持积累的知识干净、最新和有组织。

### 最佳实践

**让它在大多数项目上自动运行**。默认触发条件（24小时 + 5次会话）对活跃开发很有效。你不需要监督这个。

**重大重构后强制进行梦境**。如果你重命名了一半的代码库、迁移了框架或更改了 API 结构，请触发手动梦境周期。在它们被整合之前，旧的记忆条目会比清晰造成更多混淆。

**偶尔审查输出**。在梦境周期运行后，浏览你的 MEMORY.md。自动梦境很好，但并不完美。你可能会发现你不同意的整合选择。这些文件是纯 markdown。自由编辑它们。

**结合上下文管理实践**。自动梦境保持你的记忆文件干净，但你仍然需要管理你的活跃会话上下文。干净的记忆文件意味着 Claude 在启动时加载更少的噪音，这为你的上下文窗口留出更多空间用于实际工作。

### 结论

自动梦境是将 Claude 的记忆从不断增长的笔记堆转变为积极维护的知识库的功能。自动记忆捕获原材料。自动梦境将其塑造成有用的东西。结果是一个不仅随时间记住更多，而且记得更好的 Claude。

## 核心内容提炼

### 一、问题域界定

| 挑战 | 表现 | 根源 |
|-----|------|------|
| 自动记忆衰减 | 20+ 会话后笔记混乱 | 缺乏整合机制 |
| 矛盾条目堆积 | 新旧信息冲突 | 没有自动清理 |
| 相对日期失效 | "昨天"失去意义 | 时间参照系漂移 |
| 过时解决方案 | 引用已删除文件 | 缺乏过期检测 |

### 二、核心方法论：四阶段整合流程

```
┌─────────────────────────────────────────────────────────────┐
│                     Auto Dream 流程                         │
├─────────────┬─────────────┬─────────────┬───────────────────┤
│  阶段 1     │   阶段 2    │   阶段 3    │     阶段 4        │
│  定向       │  收集信号   │   整合      │   修剪和索引      │
├─────────────┼─────────────┼─────────────┼───────────────────┤
│ 读取        │ 搜索会话    │ 合并信息    │ 更新 MEMORY.md    │
│ MEMORY.md   │ 记录(JSONL) │ 转换日期    │ 移除无效指针      │
│ 扫描主题    │ 查找模式    │ 删除矛盾    │ 添加新链接        │
│ 文件列表    │ 提取信号    │ 修剪过时    │ 重新排序条目      │
└─────────────┴─────────────┴─────────────┴───────────────────┘
```

**关键设计原则**：
1. **定向优先**：先了解现有记忆结构，再决定如何整合
2. **精准搜索**：使用 grep 风格搜索而非全量阅读，控制成本
3. **日期标准化**：所有相对日期转换为绝对日期（如 2026-03-15）
4. **矛盾解决**：在源头解决，而非仅仅标记
5. **外科手术式**：只修改需要变更的文件，不全部重写

### 三、实施指南

**自动触发条件**（必须同时满足）：
- 自上次整合 ≥ 24 小时
- 自上次整合 ≥ 5 次会话

**手动触发方式**：
```
/dream 命令（尚未全面推出）
或
直接告诉 Claude："运行自动梦境周期" / "整合我的记忆文件"
```

**最佳实践**：

| 场景 | 建议操作 |
|-----|---------|
| 日常开发 | 让自动梦境自动运行 |
| 重大重构后 | 手动触发梦境周期 |
| 定期维护 | 偶尔审查 MEMORY.md 输出 |
| 上下文管理 | 结合 CLAUDE.md 使用 |

### 四、四种记忆机制对比

| 机制 | 写入者 | 运行时机 | 作用 | 存储位置 |
|-----|-------|---------|------|---------|
| CLAUDE.md | 用户 | 手动编辑 | 权威规则和标准 | `./CLAUDE.md` |
| Auto Memory | Claude | 每次会话 | 捕获学习到的模式 | `~/.claude/projects/<project>/memory/` |
| Session Memory | Claude | 后台自动 | 会话连续性 | `~/.claude/projects/<project>/<session>/session-memory/` |
| Auto Dream | Claude | 24h+5会话 | 记忆整合与清理 | 同 Auto Memory |

**性能数据**：
- 整合 913 次会话约需 8-9 分钟
- 在后台运行，不阻塞用户会话
- MEMORY.md 从 280 行减少到 142 行（示例）

### 五、核心观点与金句

> "Auto Memory without Auto Dream is a note-taker who never organizes their notebook. Auto Dream without Auto Memory has nothing to consolidate. You want both running."
> 
> **（没有 Auto Dream 的 Auto Memory 是一个从不整理笔记本的笔记记录者。没有 Auto Memory 的 Auto Dream 没有什么可整合的。你希望两者都运行。）**

> "Claude Code without Auto Dream was essentially sleep-deprived. It kept adding random notes without ever cleaning up."
> 
> **（没有 Auto Dream 的 Claude Code 本质上是睡眠不足的。它不断添加随机笔记而从不清理。）**

**核心哲学**：
- 记忆系统需要**收集**和**维护**两个层面协同工作
- 类比人类记忆：白天学习（Auto Memory）+ 夜间巩固（Auto Dream）
- 干净的记忆文件 = 更少的启动噪音 = 更多的上下文窗口用于实际工作

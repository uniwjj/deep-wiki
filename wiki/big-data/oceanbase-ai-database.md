# OceanBase 湖库一体 AI 数据库

2026 年 6 月 29 日 OceanBase Hours 线上发布会，OceanBase 面向湖库一体的 AI 数据库正式发布，**Lakebase**、**DataStudio**、**DataPilot** 三大 AI 产品家族同步亮相。

## 核心定位

> "这不是传统数据库的功能升级，而是 OceanBase 面向 AI 时代的一次基础设施重建。" —— 杨冰（OceanBase CEO）

OceanBase AI 数据库以**湖库一体**为核心架构，面向 Agent 和多模态数据设计：

- **一体化**：多模态数据、离线与在线计算，统一在同一强一致底座之上
- **多模态**：结构化、半结构化、非结构化数据统一管理，支持标量/全文/向量混合搜索
- **Agent 友好**：记忆、上下文、隔离、分支、回滚与规模化运行能力作为原生能力
- **开放**：存储与计算保持开放，支持 S3/Iceberg，不被锁定在专有系统

## 时代背景：Agent 成为新的数据使用者

### 两组关键数据

| 趋势 | 数据 |
|------|------|
| Agent 成为软件交互主体 | Gartner 预测 2028 年 1/3 企业软件交互由 Agent 完成 |
| 非结构化数据占比 | 全球超 80% 的数据是非结构化数据 |

### Agent 对数据库的三个核心需求

| 需求 | 挑战 | OceanBase 应对 |
|------|------|--------------|
| **上下文** | 底层数据分散在多套割裂系统中，Agent 难以获得完整一致的业务上下文 | 湖库一体统一底座，结构化与非结构化融合 |
| **规模** | 海量 Agent 轻应用需低成本共存、相互隔离、按需调度资源 | 弹性架构 + 逻辑表，支撑千万级应用 |
| **进化** | Agent 持续迭代，失败路径不能污染生产数据 | 数据分支 + 隔离环境 + 快速回滚 |

蚂蚁灵光典型案例：承载 **3000 万个闪应用**，每个应用仅数百行数据，99% 处于休眠状态，但需随时秒级响应。Schema 动态变化，同时需要完整 SQL 计算能力——传统数据库从未面对过的新型负载。

### 三股变革力量

1. **数据形态之变**：客服录音、产品图像、会议记录等非结构化数据从"不可计算"变为战略资产
2. **数据流动之变**：从线性割裂链路 → 数据闭环（在线供给上下文 → 推理结果反哺数据库）
3. **数据交互之变**：从 SQL 工程师查询 → 自然语言成为 Agent 与数据交互的全新入口

## 产品家族

| 产品 | 定位 | 核心能力 |
|------|------|---------|
| **OceanBase Lakebase** | 底层核心引擎 | 湖库一体、多模态数据管理、AI 列、Agent 友好的实时上下文工程 |
| **OceanBase DataStudio** | 数据生产与治理工作台 | 数据接入、加工、任务编排、语义建模、数据治理、Agent 协作 |
| **OceanBase DataPilot** | 业务智能 Agent | 自然语言完成分析报告、数据看板、可信答案生成，统一企业业务智能入口 |

三者构成闭环：Lakebase 解决数据底座 → DataStudio 解决数据生产与治理 → DataPilot 解决业务人员直接使用数据智能。

## Lakebase 核心引擎特性

### 多模表

不同数据形态进入同一张表的语义之下，用户看到的仍是一张表，但表背后承载结构化、半结构化、非结构化数据，统一检索、计算和调用。

### AI 列

把模型能力引入数据处理链路——基于原始数据生成摘要、标签、特征、向量或其他语义结果，让模型理解能力以"列"的形式进入数据库。非结构化数据不再只是"存下来的文件"，而是可搜索、可计算、可治理、可被 Agent 安全调用的数据资产。

### Agent 友好的上下文工程

- **数据分支**：像代码仓库一样快速创建数据分支，隔离环境中试验，验证通过后合并，失败直接回滚或释放
- **逻辑库**：面向海量 Agent 应用的逻辑表，每个应用 Schema 与数据映射为可查询可计算的逻辑表，避免"一应用一物理表"的元数据膨胀
- **资源隔离**：海量应用低成本共存、相互隔离、按需调度
- **标准化接口**：API/MCP 接口与工具集，降低 Agent 集成门槛

### 开放架构

基于开放式存储格式与可扩展计算架构，支持 S3 兼容对象存储、Iceberg 开放表格式，可对接 Spark、Ray 等计算引擎。

## 四条工程底线

| 底线 | 传统要求 | AI 时代升级 |
|------|---------|-----------|
| **一致性** | 高标准 | **生死线**——Agent 直接替人决策（风控/内容安全），数据错一条=业务事故 |
| **扩展性** | 把一个库做大 | **让一百万个库低成本共存**——海量 Agent 应用经济共存，闲置近乎零成本 |
| **可靠性** | 有人兜底 | **智能体的生命线**——Agent 7×24h 无人值守，金融级高可用保障业务连续 |
| **实时性** | 隔夜跑批 | **毫秒级在线决策**——Agent 判断发生在毫秒间，数据底座必须跟上 |

## 真实场景实践

### 蚂蚁灵光（3000 万闪应用）

- **挑战**：海量应用各自独立数据空间，Schema 动态变化，需要完整 SQL 计算能力
- **方案**：逻辑表将每个应用 Schema 与数据映射为可查询可计算的逻辑表，标准 SQL 完成过滤/聚合/Join
- **成果**：大量闪应用在统一数据库基础设施上低成本、安全运行

### 智慧驾驶

自动驾驶模型持续迭代依赖视频、图像、传感器数据和结构化标签。Lakebase 通过多模态数据管理与离在线协同处理，连接存储、加工、检索与模型使用链路。

### 证券行业

数据来源分散、格式异构、权限复杂。Lakebase 作为多模态数据处理与服务中枢，统一连接异构数据源，解析研报、制度文件等非结构化内容，通过 API/MCP 向上层 Agent 提供智能研报、合规管理和业务分析服务。

## TCO 降低 30%-50%

相较于多系统组合方案，OceanBase AI 数据库通过减少数据冗余、重复计算和系统数量，使整体 TCO 降低 30%-50%。收益不仅来自少部署几套系统，更来自数据、元数据、权限和工作负载在统一架构中的协同。

# OceanBase 湖库一体 AI 数据库

2026 年 6 月 29 日 OceanBase Hours 线上发布会，正式发布湖库一体 AI 数据库，**Lakebase**（核心引擎）、**DataStudio**（治理工作台）、**DataPilot**（业务 Agent）三大产品亮相。

## 核心命题

> "AI 数据库解决的，不是'数据如何存储'，而是'Agent 如何安全、准确并持续地使用数据'。" —— 杨冰

数据库的角色正在发生根本转变——从"人的存储与查询工具"升级为"Agent 的数据底座"。OceanBase 这次不是功能叠加，而是**架构重建**。

## 为什么必须重建

两股力量同时改变了数据库的需求方和需求内容：

**需求方变了**：Gartner 预测 2028 年 1/3 企业软件交互由 Agent 完成。Agent 成为新的数据使用者——7×24h 持续调用、直接替人决策（风控审核/内容安全等环节已是 Agent 拍板）、无人值守。传统数据库是给"人写 SQL、人看报表"设计的，不是给 Agent 用的。

**需求内容变了**：全球 80%+ 数据是非结构化的（客服录音、产品图像、会议记录、工程图纸）。这些过去"不可计算"的数据，在 AI 眼中是战略资产，需要和结构化数据在同一个底座上被统一管理、治理和调用。

还有**数据交互方式的变革**：从 SQL 查询变为自然语言交互，数据库需要理解业务语义，而不只是做存储和检索。

杨冰的总结：**从"通用聪明"到"懂企业业务"，靠的不是更大的模型，而是高质量、实时且可信的数据与上下文**。如果底层数据还分散在多套割裂系统中，Agent 拿不到完整一致的业务上下文，模型再强也没用。

## 产品家族：一条完整闭环

| 产品 | 定位 | 解决的问题 |
|------|------|-----------|
| **Lakebase** | 底层核心引擎 | 数据怎么存、怎么算——多模态数据统一管理、AI 计算嵌入、Agent 上下文工程 |
| **DataStudio** | 数据生产与治理工作台 | 数据怎么生产、怎么治理——接入→加工→编排→语义建模→Agent 协作 |
| **DataPilot** | 业务智能 Agent | 业务人员怎么直接使用数据智能——自然语言生成分析报告、看板、可信答案 |

三者闭环：Lakebase（底座）→ DataStudio（生产治理）→ DataPilot（业务入口）。

## Lakebase 核心引擎：三个关键设计

### 多模表

结构化、半结构化、非结构化数据进入**同一张表的语义之下**。用户看到的仍是一张表，表背后承载多种数据形态，在统一治理体系中被检索、计算和调用。消除多系统间的数据搬运和割裂。

### AI 列

模型能力以"列"的形式嵌入数据处理链路。原始数据进入 → 自动生成摘要、标签、特征、向量等语义结果。非结构化数据不再只是"存下来的文件"，变成**可搜索、可计算、可治理、可被 Agent 安全调用**的数据资产。

### Agent 友好的上下文工程

这不是外挂能力，是原生设计：

| 能力 | 解决什么问题 |
|------|------------|
| **数据分支** | Agent 进化需要试错——像 Git 一样创建数据分支，隔离试验，验证通过合并，失败直接回滚/释放。失败路径不能污染生产数据 |
| **逻辑库** | 千万个 Agent 应用各自需要独立数据空间，但每个才几百行数据——逻辑表映射避免"一应用一物理表"的元数据膨胀，支持标准 SQL |
| **资源隔离** | 海量应用低成本共存，99% 休眠时近乎不占资源，唤醒时秒级响应 |
| **MCP/API 接口** | 标准化集成入口，降低 Agent 接入门槛 |

架构保持开放：S3 兼容对象存储、Iceberg 开放表格式、可对接 Spark/Ray 等计算引擎。

## 四条工程底线：从高要求升级为生存条件

这四条是 OceanBase 15 年金融级积累在 AI 时代的延伸：

| 底线 | 传统定义 | AI 时代重定义 |
|------|---------|-------------|
| **一致性** | 高标准 | **生死线**——Agent 直接替人决策，数据错一条不是指标偏差，是业务事故 |
| **扩展性** | 把一个库做大 | **让一百万个库低成本共存**——海量 Agent 应用需各自独立空间，按需调度、闲置近乎零成本 |
| **可靠性** | 有人值班兜底 | **Agent 的生命线**——7×24h 无人值守，金融级高可用保障业务连续性 |
| **实时性** | 隔夜跑批 | **毫秒级决策**——Agent 判断发生在毫秒间，数据底座跟不上就等于没底座 |

## 真实验证：三个场景、同一结论

| 场景 | 数据形态 | 核心挑战 | 方案 |
|------|---------|---------|------|
| **蚂蚁灵光** | 3000 万闪应用，每个数百行 | Schema 动态变化、海量小数据空间、秒级响应 | 逻辑表映射 + 统一 SQL 计算 |
| **智慧驾驶** | 视频+图像+传感器+标签 | 多模态混合、模型持续迭代、离在线协同 | 多模态管理 + 统一加工检索链路 |
| **证券行业** | 异构数据源+研报+制度文件 | 格式异构、权限复杂、合规严格 | 多模态中枢 + MCP 向上层 Agent 服务 |

三种完全不同的数据形态和业务需求，共同指向同一个结论：**不拼装多系统后，TCO 降低 30%-50%**。收益不仅来自少部署几套系统，更来自数据、元数据、权限和工作负载在统一架构中的协同。

## 15 年底气

OceanBase 从分布式事务数据库起步，在金融核心系统经历 15 年大规模实战检验：

- 服务 4000+ 客户，60%+ 将其作为核心系统
- 连续三年中国分布式数据库本地部署市场份额第一
- 迄今唯一同时在 TPC-C、TPC-H 两项基准测试登顶的数据库
- 打磨环境：支付宝 AI 支付、蚂蚁阿福、灵光、淘宝 AI 购物助理、通义千问、高德、飞猪

每一次架构演进（从支撑交易 → 统一交易与分析 → 处理多模态数据 → 存算分离与开放生态）都在消除数据系统的工程复杂性。

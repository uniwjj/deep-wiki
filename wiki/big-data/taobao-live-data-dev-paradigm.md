# 淘宝直播：AI 驱动的数据研发范式升级

2026-05-28 阿里云大数据智能体"虾"聊日分享，由淘宝直播数据团队介绍其联合 [[dataworks-data-agent|DataWorks Data Agent]] 实践 AI 驱动数据研发范式升级的完整路径，主题为"构建数据领域的数字员工"。

- 演讲人：**丁源**｜阿里巴巴淘天集团高级数据专家、淘宝直播数据研发负责人
- 录音补充原话：
  - NL2DSL2SQL：现场强调"我们不是做完整的 NL2SQL 过程，而是中间加了一层 NL to DSL to SQL……所有代码生成过程都是白盒的，每个过程可以分段调试和审计"
  - R2C：现场定义"R2C 流程，不是拿到一句话给个 SQL 就完了，而是从需求沟通开始到最终需求交付的全流程：需求沟通、技术方案设计、代码生成、运维挂基线、数据回刷、数据验证、最后上线"
  - 中心化转本地化的 4 个原因：①Agent 迭代速度成单点能力 ②Memory 记忆冲突无法像代码合并 ③Skill 让 Agent 变成 Skill 实现 AI 平权 ④中心化召回是全量的，无法个性化（淘宝直播 RAG 量极大）
  - verify-data Skill 已在阿里云公众号发布

### P4 · R2C 与 NL2DSL2SQL

P4 明确否定单纯 NL2SQL。NL2SQL 路径是自然语言 → 黑盒 → SQL，问题是天花板低、出错难修、用户难干预；NL2DSL2SQL 路径是自然语言 → JSON DSL → SQL，JSON DSL 具备校验、单测、补常识能力，带来工程兜底、分段调试、全过程可审计和白盒化。底部四步走是标准化数仓 → 规范化 Pipeline（研发范式）→ 服务化平台 → 多 Agent 协同。

### P5 · 一年成果

P5 将成果分为两条生码路径：DSL 生码资产维护成本高，但已有 75% AI 代码占比和 100% 数据准确率；DataWorks Data Agent 生码无维护成本、上手快，文档清晰临时取数可 24h 交付，复杂需求生码准确率 50% → 80%。剪刀差曲线显示 AI 自动代码上升、手写代码下降，并在 25Q4 交叉。直播 360 交易模块案例包含 56 个口径、6 个基础模型、2 个融合任务、2 个同步任务、1500+ 行代码，全部由 AI 产出。

### P10 · Multi-Agent 协同

P10 将 Multi-Agent 定义为"有明确角色分工的工厂流水线"，不是 Agent 互相聊天。澄清阶段不写代码，只用元数据反复校验需求，包含调元数据校准、口径确认 + 黑话翻译、场景模板沉淀，输出双方认可的技术方案；执行阶段输入完整技术方案，完成 DSL 生成、SQL 翻译、任务发布、验数、上线。四角色是需求分析 Agent、Coordinator + Planner、Research Team、Reporter，每个节点都可人工 confirm。

### P11 · 本地化 AI Native

P11 解释为什么要本地化：中心化 AI Native 的缺点是 DSL 迭代速度成为单点、Memory 无法轻易反哺 Agent 进化、Skill 让 AI Agent 能力逐步平权、每次 RAG 召回都要全量召回而精准率和效率低。中心化像"每天清空记忆的实习生"，本地化像"越来越懂业务的老员工"。OpenClaw 最佳案例用 SDD.md 文件协议串联小需、小模、小码，实现 0 手写代码、0 平台依赖、ETL 3 天 → 3 小时。

### P12 · AI 能力层

P12 是章节页，定义直播数据团队的 AI 能力层由 Skill、SDD、AI Coding 三个组件串联：Skill 是能力封装，SDD 是共享方法论，AI Coding 是输出协议。

### P13 · 沉淀 Skill

P13 强调 Skill 化有三种来源但统一输出：从 0 到 1 的新 Skill、从原有 Agent 沉淀为 Skill、从原有范式抽象为 Skill。语义层 Skill 走痛点定义 → 流程拆解 → SKILL.md → 迭代沉淀；`verify-data` 走基准发现 → SQL 生成 → 跨表对比 → 红线检查 → 钉钉发布；`copilot_req2sql` 把 R2C 需求澄清范式抽成最小可复用单元，覆盖 P1 需求澄清、P2 资源梳理、P3 模型设计、P4 Data Agent 输入。

### P14 · SDD 共享方法论

P14 强调 Skill 和 Agent 都基于同一套规约工作，审计前置而不是事后 Review。四阶段 Spec 链是 P1 需求澄清、P2 资源梳理、P3 模型设计、P4 Data Agent 输入；`SKILL.md` 主文件约 200 行，只做路由，`references/p1.md` 到 `p4.md` 按阶段加载。AI 能做上下文管理、过程追溯、Review 前移、模板化执行；AI 不能替代数据建模决策、指标口径判断、复杂业务翻译和 SQL 正确性保证。

### P15 · AI Coding 双协议

P15 把两条生码协议画成同一个 SQL 出口：路径 A 是 DSL 生码全自动，P3 模型设计 / P4 Prompt 输出作为上游，Agent 进入 R2C 主流程，生成 JSON DSL，再由服务化平台 DSL2SQL，最终产出 SQL，准确率 100%；路径 B 是 DataWorks Data Agent 生码半自动，Skill `copilot_req2sql` 产出结构化 Prompt，经人工粘贴给 DataWorks Data Agent，再产出 SQL，准确率 50% → 80%。底部示例给出 `trade.dwd.json` DSL 与 `copilot_req2sql.prompt` SQL。

### P16 · DataWorks Data Agent 深度实践

P16 明确 Data Agent 补位 R2C 覆盖盲区：次抛、临时取数、历史任务、CDM 层。核心指标是 72% 代码采纳率、24.63% 代码渗透率、2.91 平均对话轮次、52% 月均使用团队占比。研发规范 Rules 总是应用，包括全局规范、业务域定制、SQL 结构、项目级 Rules；交互范式用 Prompt 模板把模糊业务语义翻译成精准数据语义。底部总结瓶颈从代码研发转移到需求澄清。

### P17 · 基建层

P17 是章节页，把数据研发基石定义为 CDM、知识库、工程平台三大基础设施，分别对应数仓建模、双引擎召回、4 模块编排。

### P18 · CDM 双产物 + AI 自动建模引擎

P18 把 CDM 定义为双产物 + 一个引擎。OneData 体系侧包括基础数仓 + 主题数仓，关系有引用、聚合、组合、拓展，范式是 OneData 维度建模和命名一致性；分层骨架是 ODS 操作层、DWD 明细层、DWS 汇总层、ADS 应用层。AI Friendly 交付侧包括资产平台 + 高质量 RAG 语料，标准是命名、枚举、口径无二义性，节奏是持续保鲜、喂得饱 AI。AI 自动建模引擎基于 SPEC、OneData 理论基础、业务建模知识库，并驱动上方双产物。

### P19 · 知识层双引擎

P19 将知识层拆成 LightRag 与业务 WIKI。LightRag 是数仓模型的 GraphRag 检索，内容形态是表、字段、指标、维度、实体关系图谱，价值是"AI 找得到表"；业务 WIKI 承载业务黑话翻译、口径文档、枚举值字典、query 改写，价值是"AI 听得懂人话"。底部强调 R2C 能做到 24h 交付的底层逻辑是二者互补；早期只有 LightRag 时，需求理解压力都堆在 Skill 自己身上。

### P20 · 工程平台

P20 定义自研直播数据服务化平台是一站式原子能力，为数据同学提效，也为 AI 调用提供工程底座。四模块包括：语义层管理（模型/口径、指标/维度、枚举字典、版本管理）、DSL 生码（代码生成、SQL 预校验、DDL 变更、任务发布）、研发后链路（发布、验数、回刷、DQC、上线、基线）、中间件 MCP / 代理（DataWorks、Holo、Lindorm、ADB、Dataphin、Flink 实时生码、图治平台、自动校验/回刷）。右侧 Pipeline 是 `__start__ → super_router → 360 / data_card → require_feedback → coordinator → planner → human_feedback → research_team(api/model/sync) → reporter / loop_feedback → __end__`。

### P21 · 总结与未来规划

P21 总结从 0 到 1 完成数据研发 R2C 全流程：需求 → Spec → 代码 → 交付。未来规划是以 ChatBI 为切入点的数据消费 / 分析探索，示例问题为"上月直播 GMV?"，输出是 AI 回答 + 数据可视化，示例数值为 1.2 亿、上升 18%。底部判断是 R2C 闭环只是起点，ChatBI 是下一个"从 0 到 1"。

## 核心思想

> **用工程的确定性，接管（暂时）大模型的不确定性。**

AI 时代数据研发不只是工具和效率的升级，而是**生产力的质变和生产模式的突破**。最终目标不是"AI 替代人"，**而是让 AI 产生的数据和经验，能被 AI 持续消费**。

## 三层架构

```
应用层：多 Agent 协同 + 中心化 AI Native + 本地化 AI Native    （用得好 AI）
   ↓
AI 能力层：Skill + SDD + AI Coding                            （管得住 AI）
   ↓
基建层：CDM + 知识库 + 工程平台                                （喂得饱 AI）
```

### AI 角色演进

| 阶段 | 角色 |
|------|------|
| 早期 | 经验执行者 |
| 当下 | 经验生产者 |
| 未来 | SuperAgent / HarnessAgent |

## AI 时代的五大挑战

| 挑战 | 应对 |
|------|------|
| **准确率是红线** | 100% 准确才能上线，人 review 跟不上 AI 输出速度，红线必须硬约束 |
| **知识库是标配** | 传统经验在员工脑子里，AI 时代必须全部线上化，领域模型体系化构建异常重要 |
| **消费场景复杂** | 不只给下游 ADS 同学用，AI 也要吃数据——命名/枚举/口径无二义性成硬约束 |
| **协同模式重构** | 人干需求 → 多 Agent 并行，人退出执行环节、只守关键决策点 |
| **生产关系重塑** | 生产力质变必将带来生产模式和生产关系的变化 |

## R2C：从需求到交付的完整闭环

**R2C = Requirements To Consign**，从需求到交付的完整闭环。不只是写 Code，覆盖**技术方案 / 代码生成 / 数据验证 / 数据回刷 / 基线挂载**全流程。

### NL2DSL2SQL：最核心的技术决策

| 对比 | NL2SQL | NL2DSL2SQL（R2C 选择） |
|------|--------|----------------------|
| 中间态 | ??? 黑盒 | JSON DSL（结构化中间态） |
| 天花板 | 低 | 高 |
| 出错处理 | 整段重写 | 工程兜底 / 分段调试 |
| 用户干预 | 难 | 全过程可审计 / 白盒化 |
| 准确性 | 不稳定 | **100%** |

JSON DSL 作为结构化中间产物，是压住 AI 幻觉的"符号锚点"。**关键设计：AI 调 DSL 的 check 接口，反馈驱动优化**——AI 写错了，工程层告诉它哪里错，AI 自己修正。

### 四步走 · 先地基后建筑

```
L01 数据层（标准化数仓）
 → L02 流程层（规范化 Pipeline · 研发范式）
 → L03 平台层（服务化平台）
 → L04 智能层（多 Agent 协同）
```

## Agent 应用层：中心化 vs 本地化 AI Native

| 维度 | 中心化 AI Native（现阶段） | 本地化 AI Native（未来演进） |
|------|---------------------------|----------------------------|
| 记忆 | 每天清空 | 持续累积 |
| 经验 | 每次重学 | 可调用 |
| 错误 | 重复犯 | 不再犯 |
| 协同 | 从头交接 | 继承上下文 |

**两种用户体验范式**：一种偏统一入口，一种偏定制工作流。

中心化 AI Native 缺点：
1. Agent 依赖的 DSL 迭代速度成为单点
2. Memory 无法轻易反哺到 Agent 进化
3. SKILL 的出现让 AI Agent 能力逐步平权
4. 每次 RAG 召回都要全量召回，精准率和效率都比较低

**最佳本地化案例：OpenClaw 三 Agent 协作（SDD.md 文件协议）**
- 小需 · 需求分析 → 小模 · 建模 · DDL → 小码 · SQL & 自检
- 记忆系统 · 踩坑沉淀 · 不再犯
- **0 手写代码 · 0 平台依赖 · ETL 3 天 → 3 小时**

主 Agent 负责总协调：接收需求 → 判断类型 → 调度子 Agent → 审查产出 → 推进流程。三个子 Agent 分别覆盖需求、建模和 SQL 开发：
- **小需**：需求分析，覆盖需求理解、口径澄清、资源初筛
- **小模**：数仓建模，覆盖模型设计、DDL、建模规范
- **小码**：SQL 开发，覆盖 SQL 生成、自检、验证反馈

> 可继承的差异 = 可演化的组织——这是"从工具到员工"的关键技术差异。

## Multi-Agent 协同（工厂流水线）

有明确角色分工的工厂流水线，**不是 Agent 互相聊天**。

### 两阶段 + 中间锚点

```
阶段① 澄清阶段（不写代码，只用元数据反复校验需求）
   ├─ 元数据校准
   ├─ 口径确认 + 黑话翻译
   ├─ 场景模板沉淀
   └─ 输出：双方认可的技术方案 ──┐
                                 ▼
                       【技术方案 · 中间锚点 ✓ 已签收】
                                 │
阶段② 执行阶段（动手，但已经没有歧义）  ◀──┘
   ├─ 输入：完整需求技术方案
   ├─ DSL 生成 → SQL 翻译
   ├─ 任务发布 + 验数 + 上线
   └─ 全程 AI 驱动，人只检查确认
```

### 4 角色分工

```
需求分析 Agent (360 / data_card 等)
   → Coordinator + Planner（HITL 检查点）
   → Research Team（并行调度 · model/sync/api Agent · 每节点 HITL）
   → Reporter（总结存档）
```

> **HITL 不是降级方案，是设计目标——人保留在关键决策点，其他全部交给 Agent。**

## AI 能力层 = Skill + SDD + AI Coding

```
Skill（能力封装） → SDD（共享方法论） → AI Coding（输出协议）
```

### 沉淀 Skill · 通往 Agentic

三种来源，统一 Skill 输出：

| 方式 | 示例 | 说明 |
|------|------|------|
| **① 新 Skill** | 语义层 Skill | 无 Agent / 无原范式可借鉴 · 围绕具体业务痛点从头建 · 痛点定义 → 流程拆解 → SKILL.md → 迭代沉淀 |
| **② 原有 Agent → Skill** | `verify-data` | 独立验数 Agent 沉淀为通用 Skill · 一句话触发评审级证据链 · 8 种常规验数模板 + 2 种高频出错模板 · 覆盖新表/迭代/监控/业务质疑/口径迁移 5 类场景 · **效率 2-4h → 30min** |
| **③ 原有范式 → Skill** | `copilot_req2sql` | 把 R2C 主流程的"需求澄清范式"抽离 · 最小可复用单元 · 24h 交付 · P1 需求澄清 → P2 资源梳理 → P3 模型设计 → P4 Data Agent 输入 |

关键设计原则：
- **渐进式披露**：主 SKILL.md 仅路由 + references 按需加载，避免上下文爆炸
- **HITL 节点**：关键节点强制人工确认 · 红线机制兜底，**不是建议**
- **验证红线**：不靠字段名猜映射 · 不跳基准表 · 不隐式聚合掩盖膨胀 · 不用 LIMIT 替代全量

> 不同来源 · **殊途同归**——Skill 化是 AI 能力沉淀的共同范式。

## SDD · 共享方法论

Skill 和 Agent 都基于**同一套规约**工作——**审计前置，不是事后 Review**。

### 4 阶段 Spec 链

```
P1 需求澄清 (★ HITL)  →  P2 资源梳理  →  P3 模型设计 (★ Human CR)  →  P4 Data Agent 输入
   p1.md                 p2.md            p3.md                       p4.md
```

文件组织：
```
SKILL.md（主文件 ~200 行）
└── references/
    ├── p1.md   仅 P1 加载
    ├── p2.md   仅 P2 加载
    ├── p3.md   仅 P3 加载
    └── p4.md   仅 P4 加载
```
按阶段按需加载，避免长上下文迷失。

P1 需求澄清示例强调强制 HITL：例如"近 7 天打赏 TOP10 主播"必须显式写出口径，`gmv = 礼物 GMV`，排除测试，窗口为 `[T-7, T-1]` 且不含当日，并记录确认人和确认日期。P3 模型设计示例强调强制 Human CR：目标表如 `dws_ctlive_anchor_gmv_di`，主键 `anchor_id + ds`，上游引用 `dwd_ctlive_gift_di`，聚合逻辑为 `SUM gift_gmv`，通过后才进入 P4。

### AI 的边界（SDD 降低沟通成本，不降低问题复杂度）

| ✓ AI 能做 | ✗ AI 不能做 |
|---------|------------|
| 上下文管理——跨长流程不丢线索 | 数据建模决策——需要人的全局判断 |
| 过程追溯——每一步都有 Spec 留痕 | 指标口径判断——业务语义的最终拍板 |
| Review 前移——写之前先对齐 | 复杂业务翻译——隐性规则没法穷举 |
| 模板化执行——标准化 SQL/DSL 生成 | SQL 正确性保证——最终责任在人 |

## AI Coding · 双协议

DSL 生码 + DataWorks Data Agent 生码**双路径**·都从 SDD 取上游（P3 → DSL，P4 → Data Agent）。

| 路径 | 上游 | 链路 | 准确率 | 适用场景 |
|------|------|------|--------|---------|
| **A · DSL 全自动** | SDD P3 | Agent (R2C 主流程) → JSON DSL → 服务化平台 → SQL | **100%** | R2C 主路径 · 标准化需求 |
| **B · Data Agent 半自动** | SDD P4 | Skill (`copilot_req2sql`) → 结构化 Prompt → 人工粘贴 → DataWorks Data Agent → SQL | **50% → 80%** | 兜底临时取数 / 次抛 / CDM 层 |

> **一条 DSL 全自动通往 R2C 主路，一条 Data Agent 半自动兜底临时取数**——主从配合，覆盖全场景。

路径 A 的 DSL 示例以 `trade.dwd.json` 表达模型、分层、来源和字段，例如 `model = dwd_ctlive_trade_di`、`layer = DWD`、`sources = ods_pay / dim_user`，字段包括 `gmv` 和 `order_cnt`。路径 B 的 Prompt 示例把"近 7 天直播打赏 GMV TOP10 主播"翻译为 Data Agent 可消费的结构化 SQL 需求，明确表 `dws_ctlive_anchor_gmv_di`、维度 `anchor_id`、聚合字段 `gift_gmv`、时间窗口和排序方式。

## DataWorks Data Agent 深度实践

补位 R2C 覆盖盲区（次抛 / 临时取数 / 历史任务 / CDM 层）·**50% → 80% 准确率 + 少量人工修改 = 24h 交付**。

### 关键指标

| 指标 | 数据 | 说明 |
|------|------|------|
| 代码采纳率 | **72%** | 采纳代码对话数 / 全部对话数 |
| 代码渗透率 | **24.63%** | Data Agent 生码 / 总提交代码 |
| 平均对话轮次 | **2.91** | 单任务交互成本低 |
| 月均使用团队占比 | **52%** | 3 月生码 > 1w 行 |

### 研发规范（Rules · 总是应用）

| 类型 | 内容 |
|------|------|
| **全局规范** | 表 / 字段 / 主题域 / 业务过程 / 命名 / 缩写 / 指标体系 |
| **业务域定制** | 点淘等业务有专属补充规范 |
| **SQL 结构** | CTE 分层 + UNION ALL + INSERT OVERWRITE |
| **项目级 Rules** | 业务同学也能用 Agent 模式找表 + 生码 |

### 交互范式（Prompt 模板）

```
# 目标：表名 + 注释 + 生命周期
# 主要步骤：按需填写
# 二次加工字段：按需填写
# 特殊要求：防膨胀 / 排除逻辑
# 全局过滤逻辑：按需填写
# 相关 CTE 参考代码：按需填写
```

把模糊业务语义 → 精准数据语义翻译。

### 3 大核心原则

1. **精准提示**：Prompt 质量 = 代码质量，不直接复制 Aone 原文
2. **严格 CR**：LLM 幻觉不可避（虚假表/字段/规范），必须逐行 CR
3. **追求实用**：50% → 80% 准确率即可，少量人工修改 + 人模协同

> 瓶颈从**代码研发**转移到**需求澄清**——更清晰的需求 = 更高的 AI 提效。

## 基建层 = CDM + 知识库 + 工程平台

支撑 AI Native 数据研发的三大基础设施。

### CDM · 双产物 + AI 自动建模引擎

| 产物 | 内容 |
|------|------|
| **① OneData 体系**（分层建模） | 基础数仓 + 主题数仓 · 4 种关系（引用/聚合/组合/拓展）· OneData 维度建模 · 命名一致性 |
| **② AI Friendly 交付**（喂得饱 AI） | 资产平台 + 高质量 RAG 语料 · 命名/枚举/口径无二义性 · 持续保鲜 |

分层骨架：ODS（操作层 · 原始入仓）/ DWD（明细层 · 30+ 事实）/ DWS（汇总层 · 50+ 矩阵）/ ADS（应用层 · 直接服务业务）。

**AI 自动建模引擎**：基于 SPEC + OneData 理论基础 + 业务建模知识库——这是 AI 时代快速扩展数仓的核心机制。

### 知识层 · 双引擎

LightRag 结构化召回 + 业务 WIKI 业务语义——两者职责互补，缺一不可。

| 引擎 | LightRag | 业务 WIKI |
|------|----------|----------|
| 角色 | 结构化召回 | 业务语义 |
| 内容 | 表/字段/指标/维度的实体关系网络（GraphRag） | 业务黑话翻译 + 口径文档 + 枚举字典 |
| 形态 | 实体关系图谱 · 表/字段/指标/维度 | 业务黑话翻译 / 口径文档 / 枚举值字典 / query 改写 |
| 价值 | "AI 找得到表" | "AI 听得懂人话" |

LightRag 的图谱节点覆盖业务过程和业务对象，P19 图中标注实体关系图谱为 `t*` 业务过程 / `b*` 业务对象；局部可辨识节点包括 `acvt_redpacket_pay`、`acvt_redpacket_award`、`read`、`sns`、`confirm_excl_rfd_lock` 等。业务 WIKI 的 `llm-wiki-v2` 目录结构包含 `log`、`pre`、`raw`、`wiki`，其中 `raw` 下又有 `archive`、`pending`、`ready`、`apis` 等状态目录，用于承载业务黑话、口径、枚举和 query 改写语料。

> 这是 R2C 能做到 24h 交付的底层逻辑——早期只有 LightRag 时，需求理解的所有压力都堆在 Skill 自己身上。

### 工程平台 · 工程原子能力与 Pipeline 编排

自研直播数据服务化平台——一站式原子能力，为数据同学提效，也为 AI 调用提供工程底座。**服务化平台只提供原子能力，不编排 pipeline——pipeline 由 AI Native 来编排。**

| 模块 | 阶段 | 能力 |
|------|------|------|
| ① **语义层管理** | 研发前 | 模型 / 口径 / 指标 / 维度 / 枚举字典 / 版本管理 |
| ② **DSL 生码** | 研发中 | 代码生成 / SQL 预校验 / DDL 变更 / 任务发布 |

## 总结 & 未来规划

- **R2C 全流程已跑通**：需求 → Spec → 代码 → 交付
- **未来规划**：以 **ChatBI** 为切入点的数据消费 / 分析探索

典型 ChatBI 目标体验是用户直接问"上月直播 GMV?"，AI 返回答案并生成可视化，例如"1.2 亿，环比 ↑ 18%"。这意味着 R2C 闭环从数据研发继续延伸到数据消费和分析探索。

> R2C 闭环只是起点——ChatBI 是下一个"从 0 到 1"。

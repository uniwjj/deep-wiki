# 本体化语义层

## 一句话定义

本体化语义层是在传统数仓、OLAP、指标平台之上建立的一层**业务语义控制面**——把"表里的数据"翻译成"业务世界里的对象和变化"，让 Agent 在一个已被企业定义、审核、发布和约束过的语义空间内工作。

## 为什么需要这一层

### 传统数据基建的结构性缺口

过去十余年数据基建的主线围绕"数据如何生产和被人消费"设计：

```
离线数仓 → 实时数仓 → 湖仓一体 → 指标平台 → OLAP → 数据治理 → 血缘/质量
```

这些建设的共同特征：核心对象是**表、字段、分区、任务、指标、宽表、物化视图**——全是数据生产体系的抽象。

当 Agent 进入经营分析时，它面对的是另一类问题：

- 这个客户处于什么生命周期阶段？
- 这个订单为什么毛利异常？
- 门店下滑和库存、活动、人员、区域是什么关系？
- 结论能不能给区域负责人看？建议能不能触发后续动作？

这些问题不能只靠宽表摊平字段、OLAP 加速查询来解决。Agent 还需要知道**业务对象之间的关系、事件之间的因果线索、指标和对象的归属、权限和动作的边界**。

### AI 经营 vs 传统 BI 的关键差异

| 维度 | 传统 BI | AI 经营管理 |
|------|---------|------------|
| 核心交互 | 人看数据 | Agent 理解、推理、建议 |
| 问题类型 | "是多少" | "为什么变化"、"影响范围"、"下一步该查哪里" |
| 系统角色 | 查询系统 | 经营协作系统 |
| 输出 | 报表、图表、下钻 | 归因、影响分析、行动建议、证据链 |
| 使用者 | 人类分析师 | Agent + 人类 |

## 六类核心能力

本体化语义层至少承担六类建模能力：

### 1. 业务对象建模

明确企业中哪些对象值得建模：客户、订单、商品、门店、合同、商机、项目、设备、工单、库存、供应商、员工、费用、预算。

每个对象需定义：主键和跨系统身份映射、来源系统和主事实表、关键属性、生命周期状态、关联的核心指标、高频经营问题。

### 2. 关系建模

对象之间的关系不能靠临时 JOIN 猜。需明确定义：关系类型（一对一/一对多/多对多/时间相关）、依赖字段或中间表、哪些路径适合分析 vs 仅适合明细追踪、哪些 JOIN 会带来口径膨胀或重复计算、哪些关系需要权限或租户隔离。

### 3. 事件和状态建模

经营分析关心变化——下单、支付、退款、发货、签收、投诉、审批、调价、补货、逾期等都是事件。对象处于什么状态，直接影响指标解释和后续动作。

### 4. 指标和规则建模

指标不仅要定义公式，还要挂到**对象、事件和关系**上：

- 指标归属对象（销售额挂在订单？门店？商品？区域？）
- 指标适用业务场景
- 指标粒度和时间窗口
- 指标依赖的事件或状态
- 指标的互斥口径和近似口径
- 指标异常时常见的归因方向

### 5. 权限和边界建模

Agent 场景下权限远超表和字段层：

- 用户能否看到某类经营对象？
- 用户能否看到对象之间的关系？
- 用户能否获得 AI 的归因解释？
- 用户能否让 Agent 给出行动建议？
- 用户能否触发后续任务或写回流程？

语义权限治理要管的不只是"查到了什么"，还有"理解了什么、解释了什么、建议了什么"。

### 6. 证据和版本建模

AI 给出的答案必须可追溯到：数据事实、语义版本、解释路径、运行证据。语义资产需要版本管理、发布审核、回归测试、回滚机制——从"文档资产"升级为"可发布、可回归、可审计的工程资产"。

## 与传统数据基建的关系

**不是替代，而是补位：**

```
┌─────────────────────────────────────────────┐
│              本体化语义层                    │
│  对象 · 关系 · 事件 · 状态 · 指标 · 权限 · 证据  │
├─────────────────────────────────────────────┤
│  OLAP / 指标平台 / 数据服务                   │
├─────────────────────────────────────────────┤
│  离线数仓 / 实时数仓 / 湖仓一体                │
└─────────────────────────────────────────────┘
```

- **数仓**：事实底座——负责数据生产、加工、性能、稳定供给
- **OLAP**：执行底座——快速查询和交互分析
- **本体化语义层**：语义底座——AI 理解企业经营世界的控制面

## 为什么不能只靠 RAG + Prompt + SQL Agent

| 方案 | 局限性 |
|------|--------|
| **RAG** | 召回文档相关 ≠ 语义正确；不解决跨部门口径差异 |
| **Prompt** | 难以承担版本责任——无法回答"这次回答基于哪个语义版本？" |
| **SQL Agent** | SQL 能跑通只能说明语法正确，不能证明业务含义正确；跨对象 JOIN 会迅速放大歧义 |

正确分工：RAG 辅助召回业务说明，Prompt 组织交互和解释，SQL Agent 承担工具调用——但生产级 AI 经营管理需要一个**更硬的语义控制面**，确保从自然语言到 SQL 的路径中间有业务语义计划、权限约束、执行证据和回归验证。

## 落地路径：七项准备工作

数据负责人不应直接跳到产品选型（买什么平台？要不要知识图谱？），而应先做好以下准备：

### 1. 从表清单升级到业务对象清单

选一条业务线（不要全企业铺开），对每个对象整理：名称和业务定义、主键和跨系统映射、来源系统和主事实表、关键属性、生命周期状态、关联指标、高频经营问题。

### 2. 把指标从公式升级成"对象上的经营含义"

给每个指标补充：归属对象、适用场景、粒度/时间窗口、依赖事件/状态、互斥口径、是否允许跨部门复用、异常时的归因方向。

### 3. 梳理对象关系，别让 JOIN 路径全靠模型猜

显式整理：业务关系类型、字段/中间表依赖、分析路径 vs 明细追踪路径、口径膨胀风险、权限/租户隔离需求。

### 4. 建经营问题集，别只建 SQL 样例库

每个经营问题补充：所属场景、涉及对象、涉及指标、归因路径、必须避开的误解、需返回的证据、适用角色和权限边界。这可以演变为语义层测试集、Agent 回归集、业务验收集。

### 5. 把权限从数据权限升级到语义权限

从"能看到哪些表/字段"升级到"能看到哪些对象、关系、指标、归因解释、行动建议"。

### 6. 给语义资产建立版本、审核和回滚意识

语义模型有版本、发布要审核、高频问题要回归、错误回答可追踪到语义路径、口径变更可解释影响范围、发布后可回滚。

### 7. 让业务专家进入建模流程，不要幻想完全自动建模

模型负责加速候选生成、文档整理、差异发现和问题聚类；人负责定义边界、审核口径、签署发布、处理争议。**AI 能降低语义建模的体力成本，但不能取消语义责任。**

## Data Infra 团队角色的重新定义

未来优秀的数据负责人需要同时管理三类资产：

| 资产类型 | 内容 |
|----------|------|
| **数据资产** | 表、字段、任务、分层模型、质量、血缘、成本、性能 |
| **语义资产** | 对象、关系、事件、状态、指标、规则、权限、问题集、解释路径 |
| **Agent 可用资产** | 工具、Skill、语义计划、问题回归、回答证据、运行 Trace、异常复盘 |

数据团队从"数据供给者"升级为"AI 经营基座建设者"——不只把数据算出来，还把企业经营世界建成 Agent 可理解、验证和治理的结构。

## 实施建议

1. 先选一个结构化数据清晰、经营问题频繁的场景切入（如销售经营分析、门店分析、客户增长分析）
2. 围绕场景准备好五类资产：核心业务对象、核心指标口径、核心对象关系、高频经营问题、权限和证据边界
3. 第一个场景跑通后再复用方法论扩展
4. **不要急着替换数仓**——保留离线数仓、实时数仓、湖仓一体的事实供给能力，让沉淀下来的结构化数据资产长出可被 Agent 使用的业务语义控制面

# Superpowers 工作流与场景实战

Superpowers 的基础工作流是一个完整的 7 步流程，覆盖从需求到交付的全链路。不同场景可按需裁剪。

## 完整 7 步工作流

```
brainstorming → using-git-worktrees → writing-plans → subagent-driven-development → test-driven-development → requesting-code-review → finishing-a-development-branch
```

### 第 1 步：头脑风暴

Agent 不直接写代码，而是先退后一步。

**硬性门槛**：呈现设计方案并获得批准前，绝不调用实现技能、不写任何代码、不搭建项目结构。

**9 步标准化流程**：

| 步骤 | 内容 | 说明 |
|------|------|------|
| 1 | 探索项目上下文 | 自动扫描现有代码和架构 |
| 2 | 视觉伴侣（可选） | 涉及 UI 时提供可视化参考 |
| 3 | 苏格拉底式提问 | 一次一个问题，逐个澄清需求 |
| 4 | 提出 2-3 个方案 | 每个方案都有权衡分析 |
| 5 | 分段展示设计 | 按复杂度分段，逐段确认 |
| 6 | 编写设计文档 | 自动保存到 `docs/superpowers/specs/` |
| 7 | 规范自检 | 检查有没有 TBD、TODO 等占位符 |
| 8 | 等待用户审查 | 只有用户确认了才会往下走 |
| 9 | 调用写计划技能 | 唯一合法的下一步 |

**设计原则**：一次一个问题（不轰炸式提问）、多选题优先（减少歧义）、YAGNI 原则（不做多余功能）、模块化设计（拆成独立单元）。

### 第 2 步：Git Worktree 隔离

用 Git Worktree 创建独立工作目录，和主分支完全隔离。

**铁律**：绝不在主分支直接开发、目录必须被 `.gitignore`、开发前先验证基线测试。

**4 步创建**：

| 步骤 | 内容 | 说明 |
|------|------|------|
| 1 | 目录选择 | 优先 `.worktrees/` 或 `~/.config/superpowers/worktrees/` |
| 2 | 安全验证 | 确保目录在 `.gitignore` 中 |
| 3 | 创建 worktree | 自动创建分支和独立目录 |
| 4 | 基线验证 | 安装依赖、运行测试，确保环境干净 |

核心价值：可随时丢弃的沙箱，改坏了直接删掉重来，主分支毫发无损。

### 第 3 步：任务拆分

将设计文档拆成 2-5 分钟粒度的小任务。每个任务包含：精确的文件路径、完整代码、验证步骤。

**6 类禁止的占位符**：

1. TBD / To Be Determined
2. TODO / 待完成
3. "下一步... / 接下来..."
4. "后续会实现... / 之后会添加..."
5. "我们将在后面的步骤中..."
6. 任何模糊、没有明确行动的描述

**三大文件组织原则**：增量修改（只改需要改的文件）、原子提交（每个小任务单独提交）、隔离工作区（不影响主分支）。

### 第 4 步：子代理驱动开发

每个任务派发一个独立子代理执行。核心思想是**上下文隔离**——每个子代理只看当前任务所需信息，不被历史上下文污染。

**铁律**：绝不能用同一个代理做多个任务，审查必须先规格审查再质量审查，子代理遇到问题必须主动求助。

**两轮复查机制**：

| 轮次 | 审查内容 | 核心问题 |
|------|---------|---------|
| 第一轮 | 规格合规性 | 我们有没有做对的事情？ |
| 第二轮 | 代码质量 | 我们有没有把事情做好？ |

不通过打回重做。

### 第 5 步：TDD 测试驱动

标准的 RED → GREEN → REFACTOR 循环：

- **RED**：先写会失败的测试，失败原因必须是"功能还没实现"
- **GREEN**：写刚好让测试通过的代码（越简单越好，YAGNI）
- **REFACTOR**：重构优化，保持测试绿色

**堵住 4 种偷懒借口**：

| 借口 | 真相 |
|------|------|
| 太简单了，不用测 | 简单的也会崩，测试只要 30 秒 |
| 回头再补测试 | 后补的测试永远证明不了什么 |
| 我手动测过了 | 手动测试今天测了明天就忘 |
| 都写一半了删掉浪费 | 沉没成本谬误，不可信的代码才是浪费 |

### 第 6 步：代码审查

派发独立审查子代理，破除"实现者盲区"。

**三级审查标准**：

| 级别 | 说明 | 处理方式 |
|------|------|---------|
| **Critical** | 安全漏洞、数据泄露、权限缺失 | 必须立刻修，不修不往下 |
| **Important** | 边界条件遗漏、逻辑错误、缺日志 | 修完才能进下个任务 |
| **Minor** | 变量名不好、注释不够清楚 | 记录下来，后续处理 |

**5 项完成验证清单**：测试全绿、审查通过、无遗留 TODO/调试代码、文档已更新、实现与设计一致。

### 第 7 步：收尾

验证所有测试通过后，提供四个标准化选项：

| 选项 | 适用场景 | 操作 |
|------|---------|------|
| 本地合并 | 个人项目、小功能 | 切回主分支→合并→测试→删分支 |
| 创建 PR | 团队协作、需审查 | 推送→创建 PR→保留分支待改 |
| 保留分支 | 开发到一半需切换 | 保留分支和 worktree |
| 丢弃分支 | 做砸了需重来 | 输入 discard 确认→删除所有内容 |

高危操作（合并、删除）必须有二次确认。

## 强制触发机制

工作流能跑起来的关键设计：**如果存在适用技能，Agent 必须使用，没有选择余地**。这是通过提示词中的心理学技巧实现的——权威性（声明"技能是强制性的"）、承诺（让 Agent 主动宣布使用）、社会证明（描述"始终"会发生什么）。

## 场景 1：新项目从零开始

走完整 7 步流程。新项目的风险不在"写不出来"，而在"写偏了"——写完发现架构不对、技术栈选错、需求理解偏差，返工成本远高于前期设计成本。

### 场景要点

- **头脑风暴阶段别急**：多花时间在需求澄清上，省下的时间远比这里多
- **任务拆分的粒度**：太大或太小都影响子代理效率
- **Worktree 创建后先跑测试基线**：确认环境没问题再开始

## 场景 2：老项目加新功能

步骤和场景 1 相同，但每步重点截然不同。核心原则三条：

1. 遵循现有代码模式和架构
2. 不提议无关的重构
3. 新代码要和项目风格一致

### 场景要点

- **头脑风暴阶段**：先摸底项目结构（框架、模板引擎、已有实现），然后才提方案——从"梳理需求"变为"理解现状 + 评估影响范围"
- **任务拆分阶段**：特别关注和现有代码的集成点（路由位置、复用中间件、格式一致性）
- **子代理开发阶段**：除了常规审查，还要确认没有破坏已有功能
- **代码审查阶段**：重点偏向"新代码是否和项目风格一致"和"有没有引入不必要的依赖"
- **不要跳过 worktree**：老项目更需要可随时丢弃的沙箱，因为改动可能影响已有业务逻辑

## 场景 3：发现并修复 Bug

走精简 3 步：`systematic-debugging → test-driven-development → verification-before-completion`。

不需要完整 7 步——Bug 目标明确、范围可控，不需要头脑风暴梳理需求，通常在当前分支上直接修。

### 系统化调试四阶段

**阶段一：根因调查**——不做任何修改，只收集信息。检查日志、复现 bug、查看相关代码。目标是找到真正原因，不是表象。例如"点击导出按钮没反应"，表象是按钮失效，根因可能是 API 返回 500 错误而前端未处理错误状态。

**阶段二：模式分析**——分析 bug 模式：偶发还是必现？和特定输入相关还是环境相关？特定用户还是所有用户？这直接影响修复策略。

**阶段三：假设与测试**——基于发现形成修复假设，写自动化测试验证。这里的"测试"不是随便跑一下，而是能复现 bug 的自动化测试。

**阶段四：实施修复**——确认假设正确后动手改代码，跑完整测试套件确认无新问题。

### 为什么要先写失败测试

先写能复现 bug 的失败测试，然后再修。两个好处：

1. 确认真的找到了根因，不是碰巧改对了
2. 测试留在套件里，防止同一 bug 再次出现

失败测试本质是**诊断证明**——证明 bug 存在，也证明修复有效。

### 修复后验证

verification-before-completion 技能要求提供**新鲜的验证证据**：所有测试通过（新旧都过）、手动验证 bug 已修复、相关功能无回归。"新鲜"是关键——不是"昨天跑过"，而是"刚刚验证过"。

## 工作流选择指南

| 场景 | 推荐工作流 | 步数 |
|------|----------|------|
| 新项目 | 完整 7 步 | 7 |
| 老项目加功能 | 完整 7 步（侧重现有代码兼容） | 7 |
| 修 Bug | 精简 3 步 | 3 |
| 多个不相关独立任务 | dispatching-parallel-agents 并行 | 按需 |

Superpowers 适合有一定复杂度的开发任务。改个配置文件、写个简单脚本，用它反而重了。

## 安装与验证

各平台一行命令安装：

| 平台 | 安装命令 |
|------|---------|
| Claude Code（官方推荐） | `/plugins official` |
| Cursor | `/addplugin superpowers` |
| Gemini CLI | `gemini extensions install [仓库地址]` |
| GitHub Copilot CLI | 注册市场后执行安装 |

**验证安装**：开新会话让 AI 做功能，预期它先问需求、做设计，而非直接写代码。

## 自定义技能：用 TDD 方式约束 AI

创建自定义技能的流程和写代码一样走 TDD：

1. **RED**：模拟你偷懒的场景——"如果赶时间，AI 会乱写提交信息吗？"
2. **GREEN**：写技能堵住漏洞——要求提交信息必须有类型前缀（feat:/fix:/docs:）
3. **REFACTOR**：迭代优化——找到新的偷懒借口，堵住它

# Google 上下文工程白皮书

> 主题：打造有状态、有记忆的 AI Agent
> 核心主张：超越模型堆砌，转向精巧系统设计

## Context Engineering vs Prompt Engineering

| 维度 | Prompt 工程 | Context 工程 |
|------|-----------|------------|
| 关注点 | **静态提示词** | **动态组装** |
| 类比 | 优化菜谱 | 备料与组装——提供恰到好处的信息 |
| 构成 | 系统指令 | 系统指令 + 工具定义 + Few-Shot + RAG 知识 + 长期记忆 + 子 Agent 输出 + 会话历史 + 工作内存 |

上下文工程将多源信息**动态组合**为每次模型调用的完整输入载荷。

### 核心挑战

- 有限上下文窗口
- Token 成本随上下文线性增长
- 注意力衰减（lost-in-the-middle）
- → 解决方案：动态裁剪、压缩、摘要

## 会话架构（Session）

Session 是短期对话上下文的**"工作台"**，目的：保持对话连贯性。

### 核心组件

| 组件 | 内容 | 存储形式 |
|------|------|---------|
| **Events（事件序列）** | 用户输入 → AI 回复 → 工具调用 → 工具结果 | 事件流 |
| **State（状态/工作内存）** | 临时结构化数据（当前任务、中间结果） | 便笺 |

### 框架实现差异

| 特性 | Google ADK | LangChain 等 |
|------|-----------|-------------|
| Event 管理 | 独立 Event 列表 | 合并到大状态对象 |
| State 管理 | 独立 state 对象 | 大状态对象（含对话和中间信息） |
| 结构 | **事件与状态分离** | 事件与状态混在一起 |

### 多 Agent 会话共享

**模式一：统一共享会话历史**
- 所有 Agent 维护单一日志，事件带 Agent 标签
- 一致的"真相来源"，适合紧密协作
- 示例：Google ADK 多代理委派

**模式二：独立私有会话历史**
- 每个 Agent 私有日志，类似黑箱工具
- 适合松散耦合，需设计消息协议
- 示例：Agent-as-a-Tool 模式

> 不论哪种模式，调用 LLM 时应**从历史中挑选相关片段作为上下文，而非整个日志**。

## 记忆系统（Memory）

### Session vs Memory

| Session（短期） | Memory（长期） |
|----------------|--------------|
| 眼前的**工作桌** | 分类有序的**档案柜** |
| 存储原始交互数据 | 经过加工的**高价值精炼知识点** |
| 原始对话流 | 结构化事实、偏好、见解 |

### Memory vs RAG

| 维度 | RAG | Memory |
|------|-----|--------|
| 角色 | 图书管理员——查外部世界知识 | 私人助理——熟知服务对象 |
| 面向 | 公共事实、通用知识 | 用户和交互、个性化上下文 |
| 数据 | 静态外部数据 | 动态用户特定数据 |
| 示例 | "巴黎人口?" | "我喜欢泳池酒店" |

**二者协同**：权威知识 + 贴心服务 = 真正聪明的 AI

### 记忆种类

| 类型 | 知道什么 | 回答 | 本质 |
|------|---------|------|------|
| **陈述性记忆** | 事实和事件 | "是什么" | 事实/数据 |
| **程序性记忆** | 技能和流程 | "怎么做" | 脚本/范例 |

陈述性记忆下分：
- **语义记忆**：世界知识
- **情景记忆**：用户的个人信息

### 记忆范围

| 范围 | 针对 | 用途 | 数据 |
|------|------|------|------|
| **用户级** | 特定用户 ID | 跨会话个性化 | 个人档案 |
| **会话级** | 单次会话 | 压缩长对话、存加工精华 | 当前会话精华 |
| **应用级/全局** | 整个应用 | 通用知识库、程序性知识 | 需严格审核与匿名化 |

### 记忆存储架构

| 方案 | 机制 | 适用 |
|------|------|------|
| **向量数据库** | 语义向量 + 相似度检索 | 非结构化自然语言、模糊记忆 |
| **知识图谱** | 实体-关系图谱遍历 | 结构化关系性查询与推理 |
| **混合存储** | 语义搜索 + 图谱遍历 | 模糊匹配 + 精确推理（Mem0、Cognee） |

### 多模态记忆

- 常见做法：多模态源 → **文本描述**存储（统一格式，全文搜索）
- 高级做法：直接存储原始图像/音频 → 需专用模型，复杂度高

### 记忆生成：LLM 驱动的 ETL

```
1. 摄取（Ingestion）→ 原始对话历史
2. 提取与过滤 → LLM 作为"筛子"，基于 Schema 提取有意义信息
3. 整合（Consolidation）→ 与已有记忆比对，合并/更新/冲突处理
4. 存储入库 → 向量数据库或知识图谱持久化
```

**后台异步执行**，AI 自动沉淀知识，无需人工干预。

### 记忆溯源与信任管理

| 信任层级 | 来源 | 特性 |
|---------|------|------|
| **高可信** | 预加载数据（CRM、内部资料） | 权威 |
| **中可信** | 用户明确提供 | 可靠 |
| **低可信** | 工具调用结果 | 易陈旧，作短期缓存 |
| **缺乏佐证** | 无印证 | 视为低分 |

**动态置信度**：时间衰减（长期未用降低置信度）+ 定期修剪（移除低信记忆）

**推理中的信任**：记忆带高分/低分标签 → 模型衡量证据 → 高分信赖、低分质疑/忽略 → 防范"记忆中毒"

## 上下文窗口管理

### 三种策略

| 策略 | 机制 | 特点 |
|------|------|------|
| **固定窗口** | 保留最近 N 轮 | 简单，丢弃久远内容 |
| **Token 截断** | 精确计算 Token，保留至上限 | 比轮次更精细 |
| **递归摘要** | LLM 总结旧内容，后台异步 | 智能保留关键信息 |

触发机制：阈值达标、闲置期、主题/任务完成时。

## 生产架构

### 解耦与异步

```
主应用（即时响应）
   ↓ 非阻塞调用
独立记忆服务 + 异步队列
   ↓ LLM 提取/整合
持久化存储
```

优势：降低延迟、隔离故障、灵活处理。

### 并发与一致性

- **队列削峰**：消息队列暂存洪峰，匀速处理
- **并发冲突控制**：事务/乐观锁，确保单写
- **失败重试**：指数退避 + 死信队列
- **多地域**：多主同步/强一致性复制

### 数据隔离与安全

1. 严格隔离：用户 A 记忆 ⇎ 用户 B 记忆
2. 用户控制：允许关闭/删除所有数据
3. 写入前校验：脱敏/去识别/防注入/审查有害内容
4. 防泄露：共享记忆彻底匿名化

## 与 Claude Code 对比

| 维度 | Claude Code | Google 白皮书 |
|------|-----------|-------------|
| Session | 只追加 JSONL 转录 + 独立 state | Event 列表 + 独立 state（ADK 模式） |
| Memory | 文件级 Markdown + Sonnet 侧查询 | 向量数据库 + 知识图谱混合 |
| 信任 | 时间过期警告 | 分层信任 + 动态置信度 + 修剪 |
| 多 Agent | 共享 + 独立（subagent sidechain） | 统一共享日志 vs 独立私有历史 |

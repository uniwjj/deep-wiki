# GLM-5.1 Harness 实战

## 核心命题

如果模型本身够强，Harness 可以拆掉一半——模型自身在同一个上下文中完成「规划-生成-评估」闭环。

## 实验：知识图谱全栈系统

| 指标 | 数据 |
|------|------|
| 模型 | GLM-5.1 |
| 工作时长 | **2 小时 47 分钟** |
| Token 消耗 | 74.7 万 |
| 自主规划 | 8 个大阶段，细化到可执行步骤 |
| 技术栈 | Flask + React + Neo4j + SQLite |

**需求**：仅一句话——"做一个知识图谱抽取及可视化系统。"

## 执行过程

1. 模型先调研技术栈、设计整体架构（约 18 分钟规划）
2. 拆为 8 阶段：后端基础 → 模式引擎 → 抽取 API → 本体 CRUD → RDF 导出 → 前端脚手架 → 图谱可视化 → 集成测试

## 自我修正（核心亮点）

不只修代码 bug，还能发现**设计级问题**：

| 场景 | 问题 | 自修正 |
|------|------|--------|
| 组件联动断链 | GraphContainer 未暴露点击事件 | 自行修复状态传递 |
| 中英文符号 | 分句逻辑硬编码 `.`，中文失效 | 改用正则统一匹配 |
| 路径导入 | import 路径错误 | 自行定位修复 |
| API 路由 | 路由配置冲突 | 自行修正 |
| 数据库连接池 | 连接泄漏 | 自行修复 |

> 整个过程就像一个老练的工程师：写代码、跑测试、看日志、定位根因、修 bug、验证通过。

## 使用建议

1. 必须说清楚技术栈和边界，否则可能给出奇怪方案
2. 跨语言、跨工具链的全栈任务反而是强项
3. 不要让它写简单脚本，"太浪费了"

## 核心洞察

> "GLM-5.1 已不是写代码的工具，而是能独立交付完整功能的虚拟工程师。"

**反直觉发现**：跨语言全栈任务是强项，简单单文件脚本反而浪费。

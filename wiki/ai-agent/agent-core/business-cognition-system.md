# 可演进的业务认知系统

> 核心主张：业务认知不是一次性写进 Context 的一段文本，而是一套**可构建、可组织、可加载、可演进**的系统。

## 为什么需要系统化业务认知

很多关于 [[data-agent-enterprise-practice|Data Agent]] 的讨论容易停留在"模型会不会生成 SQL"的表层问题上。但真正决定 Data Agent 能否从"回答问题"走向"参与分析"的，是**业务认知的深度**。

没有系统化的业务认知：
- Agent 无法理解业务概念本身
- 无法识别问题所处的场景
- 无法掌握对应的分析思路
- 只能做"自动化问数"，不能真正承担分析任务

> "很多时候，一个系统之所以让人觉得'会说，但不够懂业务'，并不是因为它没有语言能力，而是因为它缺少足够深的领域认知和场景理解能力。"

## 三类知识来源

```
显性知识抽取 ──→ 人工整理校正 ──→ Agent 反馈沉淀
    │                │                │
    ▼                ▼                ▼
  文档/代码      归纳/消歧/补充    成功案例/失败案例
  SQL逻辑                          /用户修正
```

### 1. 显性知识抽取
从文档、代码库、SQL 逻辑中自动提取：
- 业务术语及其定义
- 业务规则和约束条件
- 分析思路和分析路径

### 2. 人工整理和校正
真正高价值的知识往往需要人来做：
- 归纳分散在多个来源中的知识
- 消解不同来源之间的矛盾和歧义
- 补充隐性知识（只存在于专家头脑中的内容）

### 3. Agent 反馈沉淀
从 Agent 实际运行中持续收集：
- **成功案例**：哪些分析路径有效，可作为参考模板
- **失败案例**：哪些路径走不通，暴露了什么知识缺口
- **用户修正**：用户对结果的修正反馈，直接指示了认知偏差

## 两种知识组织形式

| 类型 | 适用内容 | 组织方式 | 加载策略 |
|------|----------|----------|----------|
| **召回知识库** | 定义、规则、短知识片段 | Chunking + 向量化 | 按需语义召回 |
| **文件系统** | 领域文档、长分析路径、结构化 Know-How | 分层级组织、可搜索 | 渐进式加载 |

**设计原则**：不把所有知识放进一个统一容器，而是根据知识形态选择合适的存储和召回方式。

## 动态加载机制

不同于静态 Prompt 模式，认知系统采用**渐进式、按需加载**：

```
用户提问
  ↓
意图理解与路由
  ↓
判断需要哪类知识
  ↓
┌─ 短知识/规则 → 从召回知识库语义检索
└─ 长分析路径 → 从文件系统层级定位
  ↓
动态注入分析流程
```

关键特性：
- **不一次性注入全部知识**（避免上下文噪声和推理偏差）
- **根据意图路由选择知识形式**（定义用向量召回，长路径用文件系统）
- **随分析推进持续激活新知识**（上下文动态管理）

这与 [[data-agent-enterprise-practice|Data Agent 的上下文管理]] 高度耦合——不仅要动态加载知识，还要动态卸载不再相关的知识。

## 反馈回流闭环

业务认知系统最重要的特性是**持续演进**，而非一次性构建：

```
分析执行 → 结果验证 → 识别缺口/修正 → 沉淀反哺 → 认知更新
    ↑                                                    ↓
    └──────────── 下一次分析受益于更新的认知 ─────────────────┘
```

反哺内容：
- **知识缺口**：Agent 未能找到对应知识的场景
- **用户修正**：用户主动纠正的结果
- **资产问题**：发现的数据口径不一致或语义歧义

> "业务认知不是静态存在的，而是随着提问、理解、分析和修正不断被激活和增强的。"

## 当前局限与演进方向

### 现状：Human-in-the-loop

- **反馈可快速进入构建流程**，但认知沉淀必须经人工审核（产品经理 + 数仓同学）
- **不支持热更新**：知识变更必须经过人工干预和审核调整
- **审核标准**：判断反馈是否足够稳定、是否适合作为通用认知沉淀

### 方向：Human-on-the-loop

- 人从"深度参与执行" → "负责监督、审核和关键决策"
- 更多结构化、可验证的工作交给 Agent 自主完成
- 知识保鲜机制从"纯人工审核" → "自动校验 + 人工把关"

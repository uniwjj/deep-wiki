# AI Agent 测评框架

## 为什么需要测评

传统软件测试依赖确定性（同一输入 → 同一输出），但 AI Agent 的自主性引入了三个新难题：

| 难题 | 表现 |
|------|------|
| **非确定性** | 同一 prompt 多次执行结果不同，"跑通一次"不代表"稳定能跑" |
| **黑盒化** | 模型升级、Prompt 微调、工具链变化导致行为漂移，肉眼难以察觉 |
| **错误级联放大** | 几十步工具调用，前序小偏差沿链路逐级放大，最终结论完全偏离 |

没有测评体系的后果：主观判断替代量化决策、旧场景悄悄退化、人工验证成本指数增长、模型不敢升级、缺乏效率基线、正确结果可能来自错误过程（碰巧答对）。

## 核心公式

```
Eval（评估）= Agent 输入 → 执行 → 捕获执行过程（Trace + 产物） → 一组检查规则 → 可对比的分数
```

**Trace（执行轨迹）**：Agent 执行过程中产生的结构化日志，记录每步的工具调用、参数、返回值和思考过程，类比程序调试中的"调用栈记录"。

测评目标是建立**可重复、可量化、可持续演进**的评估闭环——每次变更都能快速跑出可比较的分数，用数据代替直觉，用全量代替抽查。

## 三类评分器

Agent 输出既有可程序化验证的硬指标（文件存在、调用正确），也有需语义理解的软指标（推理合理性、建议质量）。单一手段无法兼顾，必须三类组合使用。**选择优先级：确定性评分器 > Rubric 评分器 > 人工评分器**。

### 确定性评分器

快速、客观、可复现，负责所有"能用代码判断"的事：

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| 工具调用检查 | 是否调用指定工具、参数是否正确 | 过程验证 |
| 产物检查 | 文件是否存在、内容是否符合预期 | 结果验证 |
| 关键词匹配 | 响应中是否包含/不包含特定内容 | 结果验证 |
| 执行指标 | 工具调用次数、token 消耗是否在阈值内 | 效率/成本验证 |
| 基线对比 | 本次执行过程和结果与基线快照逐项对比 | 回归验证 |

特点：毫秒级/免费、100% 稳定、作为**硬门禁**。

### Rubric 评分器（模型评分）

灵活、可扩展，负责"代码搞不定但能结构化描述"的场景。由另一个固定版本的 LLM 按评分标准（Prompt + JSON Schema）打分：

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| LLM 评判 | 按评分标准判断回答质量、推理过程、幻觉检测 | 回答质量、语气、规范遵循度 |

特点：秒级/API 费用、有抖动需降噪、作为**分级门禁**（error 硬、warning 软）。

评分示例（评估过程质量与结果正确性）：
```yaml
rubric:
  observation_points:
    - 回答是否基于项目规范/知识库而非通用知识
    - 推理过程是否清晰连贯
    - 是否存在幻觉或编造内容
scoring:
  process_score: 0-100   # 过程分
  result_score: 0-100    # 结果分
  is_false_positive: bool # 虚假成功（结果对但过程错）
```

### 人工评分器

最昂贵，仅在六个必要场景投入：

1. **校准 LLM 评委**（最核心）——抽样 100–200 条与 LLM 打分对齐，一致率 ≥ 85% 才算可用
2. **主观任务打分**——同理心、论证严谨度
3. **诊断通过率异常**——0% 或 100% 通常是"任务/评分器坏了"而非模型弱
4. **建立 Ground Truth**——新套件上线的前 20–50 个参考解
5. **Trace 采样审查**——每周固定抽样读轨迹，找隐藏失败模式
6. **高风险兜底**——医疗/金融/安全场景 100% 人工复核

经验法则：专家时间应 60% 以上花在"校准 LLM 评委"和"诊断异常"上。

### 三类评分器对比

| 维度 | 确定性评分器 | Rubric 评分器 | 人工评分器 |
|------|------------|-------------|-----------|
| 谁来评 | 脚本（Bash/Python） | 大模型（固定版本） | 领域专家 |
| 规则在哪 | 代码里 | Prompt + JSON Schema | 人脑 + 标注指南 |
| 成本 | 毫秒级/免费 | 秒级/API 费用 | 分钟–小时级/人力 |
| 稳定性 | 100% | 有抖动 | 取决于标注员 |
| 覆盖能力 | 已知硬指标 | 未知软指标 | 主观 + 边缘场景 |
| 门禁角色 | 硬门禁 | 分级门禁 | 不进门禁，采样审查 |

## 五大评估维度

从"做对了吗"到"好用吗"逐层递进：

| 维度 | 核心问题 | 主要评委 | 优先级 |
|------|---------|---------|--------|
| 1. 功能正确性 | 做对了吗？ | 代码（确定性） | P0 |
| 2. 过程质量 | 过程合理吗？ | Rubric + 人工 | P1 |
| 3. 效率与成本 | 划算吗？ | 代码（确定性） | P1 |
| 4. 鲁棒性与安全 | 靠谱吗？ | 代码 + 人工 | P0 |
| 5. 体验与对齐 | 好用吗？ | Rubric + 人工 | P2 |

### 1. 功能正确性（Functional Correctness）— P0

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| 结果正确性 | 代码比对、单元测试、数据库校验 | pass@1 / pass^k |
| 任务完成度 | 子目标打点 | 完成率 % |
| 指令遵循度 | JSON Schema 校验、正则、字段检查 | 遵循率 % |
| 工具调用正确性 | 调用日志断言 | 调用准确率 % |

### 2. 过程质量（Process Quality）— P1

最能体现"智能"水平，是 **Rubric 评委的主战场**：

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| 推理合理性 | Rubric 评委 | 合理性评分 1–5 |
| 步骤最优性 | 实际步数 vs 参考解法 | 步数比 |
| 信息完整性 | Rubric 按要点核查 | 要点覆盖率 % |
| 上下文利用率 | Rubric + 人工抽查 | 利用率评分 |
| 自我纠错能力 | Trace 分析 | 纠错成功率 |

### 3. 效率与成本（Efficiency & Cost）— P1

被很多团队忽视但极其重要——pass@1 高但 token 花 10 倍的方案在生产上不可接受：

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| Token 消耗 | API 统计 | avg / p95 tokens |
| 工具调用次数 | Trace 统计 | avg / p95 calls |
| 端到端延迟 | 时间戳 | p50 / p95 / p99 latency |
| 失败重试率 | Trace 统计 | 重试率 % |
| 单次任务成本 | Token × 单价 | ¥/task |

### 4. 鲁棒性与安全（Robustness & Safety）— P0

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| 一致性/稳定性 | 多次试验（k=5/10） | pass^k |
| 异常恢复 | 故障注入测试 | 恢复成功率 |
| 抗对抗/抗注入 | 红队用例集 | 抗攻击率 |
| 幻觉率 | 事实核查（代码或人工） | 幻觉率 % |
| 越权/越界风险 | 权限断言 | 越权次数 |
| 合规性 | 正则 + 人工抽查 | 违规率 |
| 拒绝合理性 | Rubric + 人工 | 误拒 % / 漏拒 % |

**pass^k vs pass@k**：pass^k 是 k 次都成功的概率（衡量稳定性），pass@k 是 k 次中至少 1 次成功的概率（衡量峰值能力）。

### 5. 体验与对齐（Experience & Alignment）— P2

早期用 Rubric + 人工抽样，成熟后引入线上 A/B + 用户反馈闭环。

| 子维度 | 评测方法 | 典型指标 |
|--------|---------|---------|
| 语气风格 | Rubric 评委 | 风格评分 |
| 回复清晰度 | Rubric + 人工 | 清晰度评分 |
| 主动澄清 | Rubric | 澄清率 % |
| 同理心 | 人工 + Rubric | 同理心评分 |
| 可解释性 | 人工抽查 | 可解释评分 |
| 用户满意度 | 线上 👍/👎、NPS | CSAT / NPS |

## 不同类型 Agent 的测评侧重

| Agent 类型 | 测评侧重 | 典型检查项 |
|-----------|---------|-----------|
| 知识库问答 | 准确性、幻觉检测、引用溯源 | 是否基于知识库内容；是否正确引用来源 |
| 代码编写 | 产物正确性、可运行性 | 是否编译/运行通过；代码风格是否符合规范 |
| 功能工具 | 过程合规性、工具调用正确性 | 是否按预期步骤调用正确工具；参数是否正确 |
| 问题定位 | 推理链路、根因准确性 | 推理是否逻辑清晰；是否定位到真实根因 |

差异体现在"用什么评分器"和"检查什么"，而非流程本身。

## 评分规则

### 通用评分公式

采用**负分制**：初始总分 100 分，每有一个不符合预期的轨迹或结果扣对应分数，扣至 0 分。达标标准为 **80 分**（推荐值）。

| 评分维度 | 计分项 | 扣分 |
|---------|--------|------|
| 结果 | 任务结果与预期不符 | -100 |
| 过程 | 步骤不遵循预设工作流 | -10/步 |
| 过程 | 中间步骤输出结果不一致 | -10/步 |
| 过程 | 调用工具/知识库不一致 | -10/项 |
| 效率 | 超出基准时长 | -10/分钟 |
| 稳定性 | 多次试验一致性（N=5） | 取平均分（均达标时） |

# Agent 长短期记忆系统

**来源**：微信公众号「小林coding」

Agent 的记忆系统分为两层：短期记忆管当下任务不翻车，长期记忆管跨任务攒经验。存储方式、粒度设计、使用场景完全不同。

## 为什么需要记忆系统？

没有记忆的 Agent 每次对话都是全新开始——不记得用户偏好、不记得之前达成什么约定、无法在多次任务之间建立连贯性。

## 两层架构

| 维度 | 短期记忆 | 长期记忆 |
|------|---------|---------|
| 本质 | Context window 中的 messages 列表 | 向量数据库（Vector DB）+ Embedding |
| 生命周期 | 任务结束时清空 | 跨任务持久保存 |
| 作用时机 | 任务执行过程中全程使用 | 任务开始前检索，任务结束后写入 |
| 类比 | 工作台 | 档案库 |

## 短期记忆

短期记忆就是每次调 LLM 时传入的 messages 列表——LLM 当前的「工作台」，上面摆着：
- 用户的指令
- LLM 的思考过程
- 工具调用的结果
- 每一步的中间状态

实现上就是维护一个列表，每次追加 `{role, content}`。每次调 LLM 传**完整历史**而非仅最新一条。

**核心代价**：messages 随任务进展越来越长，context window 装满后会截断早期内容，Agent 开始「遗忘」。

## 长期记忆

### 核心工具

| 概念 | 说明 |
|------|------|
| **Embedding** | 将文字转为数字向量，语义相近的向量在高维空间中距离近 |
| **向量数据库** | 专用存储向量的数据库，核心能力是相似度检索 |

### 写入流程

```
信息 → Embedding模型 → 向量 + 原文 → 向量数据库
```

### 读取流程

```
查询 → Embedding模型 → 查询向量 → 向量数据库相似度检索 → Top-K 原文
```

「语义相似 = 向量距离近」的特性让你不需要精确关键词匹配，语义相近就能检索出相关记忆。

## 粒度设计

粒度直接影响检索质量：

| 粒度 | 问题 |
|------|------|
| 太细（每句话一条） | 碎片化，检索命中不全 |
| 太粗（整次对话一条） | 噪音多，LLM 难以找到有用信息 |

**推荐粒度**：
- 按「**一次完整交互**」存——用户请求 + Agent 处理结果
- 按「**一个独立知识点/事件**」存——如「用户偏好：Python，简洁风格，英文注释」

## 两层配合流程

```
任务开始前：检索长期记忆 → 注入 system prompt
任务执行中：短期记忆维持全过程状态
任务结束后：将重要结论写入长期记忆
```

**短期记忆**保证当前任务的连贯性，**长期记忆**保证跨任务的积累。两层配合让 Agent 不会在中途失忆，也能越用越了解用户。

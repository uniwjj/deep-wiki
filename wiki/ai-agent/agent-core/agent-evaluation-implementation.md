# AI Agent 测评落地实践

> 理论框架参见 [[agent-evaluation-framework]]，本文聚焦工程化落地——从用例设计到 CI 集成的完整实操链路。

## 一、用例集设计：四大场景

用例集从**触发 → 核心逻辑 → 产物质量 → 异常容错**层层递进：

| 场景 | 关注点 | 正向用例 | 负向用例 |
|------|--------|---------|---------|
| ① 触发 | 该触发吗？ | 各种合法表述都能触发 | 相似但不相关的 prompt 不误触发 |
| ② 核心逻辑 | 过程对吗？ | 工具调用序列、参数、顺序正确 | 缺步骤、调错工具、参数错误 |
| ③ 产物质量 | 产物好吗？ | 响应准确、文件完整、格式正确 | 内容缺失、格式错误、幻觉/编造 |
| ④ 异常容错 | 出错扛得住？ | 优雅降级、明确提示 | 崩溃、死循环、静默失败 |

### 核心逻辑用例设计三步法

核心逻辑是用例量最大的场景，数量通常是其他三类之和的 **2–3 倍**。

1. **梳理分支流程图**：画出 Skill 的所有核心分支路径
2. **每条分支至少 1 个用例**：正向覆盖 + 关键分支负向
3. **补充组合和边界**：分支间组合、阈值边缘、跨分支冲突

经验法则：如果 Skill 有 N 条核心分支路径，至少需要 N 个正向用例。分支过多时优先覆盖高频和易出错分支。

## 二、基线管理

### 什么是用例基线

用例基线是单个用例执行 1 次后，经人工确认的**预期过程和预期结果的快照**。回答"正确的执行应该长什么样？"

| 类别 | 内容 |
|------|------|
| **预期过程** | 思维链（CoT）、工具调用序列（名称+参数+返回值）、中间产物、完整 Trace |
| **预期结果** | 最终响应文本、输出文件、输出报告、结构化数据 |

### 基线建立流程

```
用例设计（定义 prompt + 检查规则）
  → 执行 1 次
  → 人工审核（过程是否合理、结果是否准确）
  → 不可接受 → 调整 Skill 后重新执行
  → 确认 OK → 快照成为基线
```

关键：用例设计阶段只需定义 prompt 和检查规则，**不需要手写预期过程和结果**——先跑出来，确认后就是预期。

### 基线在评分中的作用

基线同时服务于两种评分器：

- **确定性评分器**：工具调用序列是否一致、产物文件是否存在、token 数是否超标
- **Rubric 评分器**：基线作为参考答案（Reference Answer），模型对比语义差异按维度打分

| 对比维度 | 确定性判定 | Rubric 判定 |
|---------|-----------|------------|
| 过程对比 | 关键步骤缺失/顺序偏离 → 过程退化 | 思维链合理性、步骤最优性 |
| 结果对比 | 关键内容缺失/产物不一致 → 结果退化 | 结论准确性、表述完整性、建议质量 |
| 效率对比 | 超出基线一定比例（>50%）→ 效率退化 | — |

### 基线更新时机

| 触发条件 | 操作 |
|---------|------|
| Agent/Skill 逻辑变更 | 重新执行 1 次并确认新基线 |
| 模型版本升级 | 重新执行 1 次并确认新基线 |
| 用例本身修改 | 重新执行 1 次并确认新基线 |

## 三、工程化落地

### 3.1 前置依赖：Trace 输出能力

过程评测的前提是被测 Agent 能输出结构化的执行轨迹。要求：

- 输出结构化 Trace（JSONL/JSON），含工具调用、思维链、中间产物
- 至少包含：每步工具名称、入参、返回值、时间戳
- 字段命名和结构在版本间保持稳定

**CodeBuddy-Code 的 `-p` 模式示例**：
```jsonl
{"type":"tool_call","name":"read_file","params":{"path":"src/main.ts"},"timestamp":"..."}
{"type":"tool_result","name":"read_file","result":"...","timestamp":"..."}
{"type":"thinking","content":"需要修改 handleRequest 函数...","timestamp":"..."}
{"type":"tool_call","name":"edit_file","params":{"path":"src/main.ts","changes":"..."},"timestamp":"..."}
```

如果 Agent 不支持结构化 Trace：有日志但非结构化则编写解析器（成本高、易碎）；仅有最终输出则只能做结果评测；建议在 Agent 设计阶段就将 Trace 作为标准能力纳入。

### 3.2 环境隔离

每次测评在隔离环境中执行，避免状态污染：
- Clone 仓库到临时目录
- 每个用例执行前重置环境（`git checkout . && git clean -fd`）
- 测评产物（Trace、日志、报告）统一归档

### 3.3 稳定性评估

通过多轮执行（N 次）检测 AI 的非确定性。核心原则：**只要 N 次中有 1 次不通过，就说明该用例存在稳定性风险。**

| Agent 类型 | 容忍阈值 | 说明 |
|-----------|---------|------|
| 关键决策类（问题定位、故障诊断） | **0%**（N/N 全部通过） | 任何一次失败都可能导致误判 |
| 辅助分析类（性能分析、报告生成） | **≤ 10%** | 偶发偏差可接受，需持续观察 |
| 创意生成类（代码建议、文案撰写） | **≤ 40%** | 输出多样性本身是特性 |

N=5 时的结果解读：

| 结果 | 含义 | 行动 |
|------|------|------|
| ✓✓✓✓✓ | 全部通过 | 稳定可信赖 |
| ✓✓✓✓✗ | 存在幻觉 | 根据类型判断是否可接受 |
| ✓✗✓✗✓ | 高幻觉率 | 深入排查 prompt/评分器/Agent 逻辑 |
| ✗✗✗✗✗ | 稳定失败 | 先检查任务定义或基线是否有问题 |

> 调试技巧：0% pass^N 通常说明**任务定义有问题**，而非 Agent 能力不行——先检查用例描述歧义、评分器配置错误、成功标准不合理。

### 3.4 触发时机

| 场景 | 触发方式 | 说明 |
|------|---------|------|
| Agent/Skill 提示词变更 | CI 自动触发 | PR 合入前回归验证 |
| 模型版本升级 | 手动触发 | 验证新模型兼容性 |
| 定期巡检 | 定时触发 | 周期性全量回归，发现潜在退化 |

### 3.5 能力测评 vs 回归测评

| 类型 | 核心问题 | 期望通过率 | 作用 | 维护频率 |
|------|---------|-----------|------|---------|
| **能力测评** | "Agent 能把什么做好？" | 起步 20–50%，逐步提升 | 目标山峰，指导优化方向 | 主动拓展、频繁迭代 |
| **回归测评** | "原有能力是否还在？" | 接近 100% | 防御漂移、保住既有阵地 | 只增不减，CI 每次运行 |

生命周期转化：能力测评通过率稳定 100% 后 → 毕业到回归套件。回归用例集只增不减（除非用例真的过时）。Anthropic 推荐：确定性任务通过率需达 100%，非确定性任务推荐 ≥ 95%。

### 3.6 测评报告结构

报告应包含：
- **全局概览**：通过率、平均分、模型版本、总 Token、总费用
- **用例列表**：按任务分组、得分、Trial 分布直方图
- **单用例详情**：Prompt、基线信息、多 Trial 评分明细、稳定性评分
- **成本统计**：每次评测的 token 消耗和费用

## 四、TPerf 性能 AI 分析 Agent 案例

TPerf AI 分析 Agent 属于**功能工具类型**，部署在 Knot 智能体平台，通过 MCP 调用 TPerf-API 获取性能数据，经决策树判断资源瓶颈和压测有效性，输出结构化分析报告。

### 评分标准（定制化）

在通用负分制框架基础上，TPerf 做了领域适配：

**确定性评测维度**：

| 维度 | 关注点 | 最大扣分 | 方法 |
|------|--------|---------|------|
| 操作步骤 | 工具调用流程与基线一致？ | -10 | LCS 算法对齐，差异/多余/缺少/顺序错各 -2 |
| 效率 | 耗时和 Token 是否合理？ | -10 | 耗时以 300s 为基准每超 10% -1；Token 类似 |
| 稳定性 | 多次分析结论一致？ | — | N 次均达标取平均分，任一不达标则 0 分 |

**模型评分维度**：

| 维度 | 关注点 | 最大扣分 | 方法 |
|------|--------|---------|------|
| 关键判定 | 压测有效性和性能结果判定一致？ | -80 | 不一致直接 -80 |
| 章节内容 | CPU/内存/网络/磁盘/业务指标/根因/建议等 | -5/项，上限 -80 | LLM 逐项对比基线报告 |

**综合评分公式**：
```
trial_score = max(0, 100 + 步骤扣分 + 效率扣分 + 报告扣分)
trial_pass  = (trial_score ≥ 80)
case_score  = avg(trial_score) 若全部通过，否则 0
```

### 用例集覆盖

按性能瓶颈场景分组，覆盖 9 大类 30+ 个用例，每个用例对应真实 TPerf 压测记录：

| 场景分组 | 用例数 | 典型场景 |
|---------|--------|---------|
| CPU 瓶颈 | 6 | 中断绑核异常、RSS 哈希异常、大象流 |
| 网卡瓶颈 | 5 | RSS 队列不均、硬件丢包 |
| 磁盘瓶颈 | 4 | IO Util > 80%、await > 10ms |
| 磁盘 IO | 2 | 高负载 + iowait 高 |
| 内存 | 1 | 内存占用 > 85% |
| TCP 队列 | 3 | Accept 队列溢出 |
| Nginx | 1 | 代码问题导致核间不均衡 |
| 配置问题 | 1 | 压测压力不足 |
| 正常场景 | 5 | 无异常的基线验证 |

### 基线管理实践

基线以 Knot 会话标识（session_id + message_id）保存，评测时通过 API 动态获取：
- 触发分析 → 获取 session_id/message_id → 人工审核 → 确认后写入用例配置
- 运行时从 Knot API 实时获取基线的报告、步骤、耗时、Token
- 基线配置随代码仓库版本化，变更可通过 Git 追溯

### 步骤评分算法

使用**最长公共子序列（LCS）**对齐基线与 Trial 的步骤。对齐前归一化处理：时间戳替换为占位符、搜索工具忽略查询参数、写入工具忽略内容保留路径、连续读文件组内不校验顺序。

### 工程架构

按职责分层：

| 层级 | 说明 |
|------|------|
| API 封装层 | 封装 Knot 智能体 API 和 TPerf 平台 API |
| 配置层 | 全局配置（模型/超时/评分）、凭证、模型定价 |
| 用例定义层 | 按场景分组的 YAML 用例文件 |
| 采集层 | 基线数据采集器 |
| 评分层 | 三个独立评分器：步骤/效率/报告 |
| 执行层 | 用例执行器（触发 Agent、收集结果、编排评分） |
| 解析层 | AG-UI 事件流解析器 |

支持 pytest marker 按模型筛选用例，每个用例 × 每个模型生成独立参数化测试。7 个模型对比测评，全量 30+ 用例约 3–5 小时。

### 模型对比报告

横向对比多模型在全量用例集上的表现：

| 指标 | 含义 |
|------|------|
| pass@1 | 跑一次评分达标的概率（峰值能力） |
| pass^5 | 连续 5 次都达标的概率（稳定性） |
| 用例通过率 | 跑 5 次时所有用例通过的占比 |
| 得分 | 多次试验平均分 |
| Token/费用/耗时 | 效率维度 |

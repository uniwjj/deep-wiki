# Data Agent 企业级实践

> 核心主张：Data Agent 不是一个"回答系统"，而是一个**能力系统**——Coding Agent 已率先验证了 Agent 稳定可用的工程闭环，Data Agent 正在承接这套闭环红利并完成领域化补齐。

## 三个前置判断

### 判断一：Data Agent 承接 Coding Agent 验证的工程闭环

Coding Agent 的突破不是模型能力的自然外溢，而是**模型能力与工程系统的第一次完整协同**。像 [[agent-harness-overview|Claude Code]] 这类框架把任务拆解、环境交互、上下文管理、反馈修正组织成了稳定可用的工程闭环。

Data Agent 与 Coding Agent 的关键差异：
- **上下文管理更敏感**：长周期分析会持续产生中间结果、假设和临时结论，全部累积会引发幻觉。Data Agent 不仅要具备**上下文动态加载**能力，还必须具备**上下文动态卸载**能力
- **工具系统更复杂**：仅能执行 SQL 的 Agent 只是"自动化问数工具"。真正稳定的分析需要同时具备取数、语法校验、资产业务语义召回、结果校验、知识加载等能力

### 判断二：企业需要的是可组合能力，而非孤立产品

企业内部的能力供给方式正在发生**范式转变**：从"构建独立入口"转向"沉淀可被调度的能力单元"。Data Agent 在企业内部需要适配的是**完整工作链路**（分析→运营→通知→报告），而非静态问答界面。

能力体系的四层结构（由外到内）：

| 层级 | 定位 | 沉淀价值 |
|------|------|----------|
| **交互层** | CLI / MCP / API，同时面向人和 Agent | 通用，非差异化 |
| **编排与框架层** | 任务拆解、路由、调度、工作流组织 | 通用框架快速演进 |
| **可复用能力层** | 分析框架、校验能力、知识召回、路由能力 | ⭐ 长期沉淀重点 |
| **数据资产与业务认知** | 语义模型、业务规则、口径定义 | ⭐ 企业最独特、最难被吸收 |

**企业级 Data Agent 的差异化价值应主要沉淀在可复用能力层及数据资产与业务认知之中。**

### 判断三：数据资产决定下限，业务认知决定上限

- **数据资产 → 效果下限**：取数错误未被识别，比"不回答"风险更大。需要的是整套数据资产能力（资产定位、口径定义、口径约束一致性、链路稳定性），而非单点 SQL 能力
- **业务认知 → 效果上限**：只有取数没有认知 = 问数工具。业务认知让 Agent 从"回答问题"走向"参与分析"

## 四层技术架构

```
┌─────────────────────────────────────────┐
│              渠道层 (Channel)            │
│  Skill形态（龙虾/Coding工具）| 独立入口（MCP）│
├─────────────────────────────────────────┤
│          Agent 核心流程层 (Core)         │
│     规划 → 分析 → 反思迭代（闭环）        │
├─────────────────────────────────────────┤
│            认知层 (Cognition)            │
│    业务术语 | 分析场景 | 角色差异 | 用户偏好  │
├─────────────────────────────────────────┤
│             语义层 (Semantic)            │
│   元信息增强 | 语义模型 | 指标管理 | 标签管理 │
└─────────────────────────────────────────┘
```

设计原则：**两层能力+一个闭环**
- **认知能力**：解决"懂不懂业务"
- **数据能力**：解决"能不能取对数据"
- **闭环流程**：规划→分析→反思迭代，失败模式和知识缺口持续回流反哺

## 四个关键实践

### 1. 增强元信息（Enhanced Metadata）

**问题**：一个主题域下有几千到上万张离线资产表，仅靠表名+字段名+备注，Agent 推理必败。大量字段看起来相似但口径定义、过滤逻辑完全不同。

**方案**：在离资产最近的地方补充语义：
- 数据采样 → 统计 Profiling → 歧义消解 → 置信度打标 → 专家审核
- 知识在数据研发和治理链路里自然"长"出来，不完全依赖额外人工维护
- 通过 [[agent-mcp-protocol|MCP]] 对外暴露，成为 Agent 可直接消费的语义服务
- 目标：资产从"可浏览、可检索" → "可理解、可推理、可解释"

### 2. 混合语义方案（Hybrid Semantic Layer）

不是单一方案，而是**路由策略**：

| 路径 | 方案 | 适用场景 | 优势 | 劣势 |
|------|------|----------|------|------|
| NL → DAL → SQL | Chat BI | BI 报表数据集 | 效率高、准确度高、用户友好 | 覆盖有限、自由度低 |
| NL → Model → SQL | Data Model（主力） | 核心业务分析洞察 | 口径一致、版本化管理 | 初期建设成本高 |
| NL → SQL | Table Metadata + 知识增强 | 探索式分析、长尾问题 | 覆盖面广、冷启动快 | 准确度不稳定、口径难统一 |

参考建模思路：MetricFlow、Cube.dev——通过度量、指标、维度和视图组织统一语义。

**系统先判断问题类型，再选择最适合的路径。**

### 3. 建立业务认知（[[business-cognition-system|可演进认知系统]]）

业务认知是一套**可构建、可组织、可加载、可演进**的系统，详见 [[business-cognition-system|业务认知系统]]。

核心要点：
- 三类知识来源：显性知识抽取 → 人工整理校正 → Agent 反馈沉淀
- 两类组织形式：召回知识库（向量化短知识）+ 文件系统（体系化长文档）
- **动态加载**：不一次性注入全部知识，按意图路由后选择合适知识形式
- **持续演进**：随着提问、理解、分析和修正不断被激活和增强

### 4. 用 Coding Agent 构建 Data Agent（[[harness-engineering-practice|Harness Engineering]]）

将 Agent 日志分析、Bug 收集、功能需求纳入需求管理系统，通过 MCP 暴露给 Coding Agent，让它参与 Data Agent 的持续优化。

**多 Agent 协同模式**：
- **Main Agent**：整体规划和分析
- **Reviewer Agent**：持续 Review
- **Executor Agent**：具体研发动作（提示词优化、技能优化、元信息增强、知识加工）
- **Eval Agent**：评测和效果验证

当前阶段：**Human in the loop**（人深度参与执行）→ 目标：**Human on the loop**（人负责监督审核和关键决策）

## 企业信用场景的特殊挑战

**三维复杂度**叠加：

| 维度 | 来源 |
|------|------|
| **用户空间** | 不同角色（业务/运营/产品/技术）有不同分析习惯和认知背景 |
| **场景空间** | 企业信用涉及多种分析场景，每种有不同指标、口径和分析路径 |
| **资产空间** | 海量离线表+复杂字段语义，歧义多、口径差异大 |

**结论**：企业级 Data Agent 不能靠通用问答壳子跑通，必须围绕真实分析过程来组织。

## 未来方向

1. **数据平台 Agent 友好化**：平台天然支持 MCP、Client、API 等 Agent 可消费接口
2. **去中心化资产管理**：从大一统平台走向分域管理，知识在靠近业务域处沉淀
3. **智能协同治理**：Agent 承担更多自动发现、自动归因、自动回流的治理职责
4. **持续深挖垂直场景**：探索 Agent 在复杂企业系统中的自演进能力

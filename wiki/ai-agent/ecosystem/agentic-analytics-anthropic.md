# Anthropic 自服务数据分析：用 Claude 构建 Agentic 分析平台

## 概述

Anthropic 利用 Claude 将 **95% 的业务分析查询自动化**，整体准确率达 **~95%**，使数据科学团队得以专注于因果建模、预测和机器学习等战略工作。

核心洞见：分析 Agent 的复杂度不在于代码生成，而在于**数据歧义**——能否将用户问题映射到数据模型中的特定、最新实体，并知晓正确的使用方式。

## 三大失败模式

| 模式 | 描述 | 对策层 |
|------|------|--------|
| **概念-实体歧义** | Agent 在数百万字段中无法选出正确字段来回答问题 | 数据基础 + 真相源 |
| **数据过时** | 数据源、业务定义、模式持续变化；Agent 知识腐化，返回微妙的错误答案 | 维护 + 验证 |
| **检索失败** | 正确信息在数据模型中但 Agent 找不到 | Skills |

## Agentic 分析架构

```
┌─────────────────────────────────────────┐
│            验证层 (Validation)           │
│  离线评估 · 消融实验 · 在线验证          │
├─────────────────────────────────────────┤
│           Skills 层 (Skills)             │
│  知识路由 · 分析手册 · 对抗审查          │
├─────────────────────────────────────────┤
│         真相源层 (Sources of Truth)      │
│  语义层 · 血缘图 · 查询语料 · 业务上下文  │
├─────────────────────────────────────────┤
│        数据基础层 (Data Foundations)     │
│  权威数据集 · 元数据治理 · 制品同仓       │
└─────────────────────────────────────────┘
```

### 数据基础层

**关键转变**：数据模型的最终用户不再是数据专家，而是代表各类用户行动的 Agent——最终用户自己无法验证底层正确性。

最佳实践：
- **创建权威数据集**（canonical datasets）：将概念映射到唯一治理答案，积极淘汰近似重复
- **工具化执行标准**：通过结构化路由、CI 审查和强制要求来落实治理
- **制品同仓**（colocation）：建模代码、语义层、参考文档、看板定义在同一仓库，CI 保护跨层完整性
- **元数据即一等公民**：列/表描述、指标定义、粒度文档、值域、血缘、所有权需与转换逻辑同等级别维护

### 真相源层

按信任度降序：

1. **语义层**（最优先）：编译后的度量与维度定义。Agent 通过 Skill 指令**必须在结构上强制优先使用语义层**。失败教训：让 LLM 从原始表自动生成指标定义会产生看似合理但编码了歧义的定义，在评测中净负面——推荐用 Claude 生成文档，但由人主导定义
2. **血缘与转换图**：当语义层无覆盖时，血缘和表排名让 Agent 推理上游模型关系
3. **查询语料**：历史 SQL。实践中直接让 Agent 检索数千条历史查询对准确率提升不到 1%。有效做法是将其蒸馏为结构化领域参考文档和分析模式
4. **业务上下文**：最被低估的一层——公司知识图谱（文档、路线图、决策记录、组织架构），让 Agent 解决歧义引用并提出更好的澄清问题

### Skills 层

Skills 是 Agent 的**程序性知识**——查询顺序、歧义导航、分析成品形态。在 Claude Code 中，Skill 是 Agent 按需读取的 Markdown 文件夹。

**效果**：无 Skills 时准确率不超过 21%，加入后整体 ~95%，特定领域 ~99%。

最佳实践：
- **成对 Skill**：*知识 Skill* 作为顶层路由；*分析手册 Skill* 编码资深分析师的工作流程（澄清问题→寻源→查询→对抗审查）
- **为 LLM 检索而写的参考文档**：描述表的粒度/范围/排除项、常见陷阱、显式路由触发条件，但不写易过时的固定配方
- **Skills 维护是一等公民**：离线准确率从 ~95% 在约一个月内漂移至 ~65%。将 Skill Markdown 文件与转换模型同仓，修改模型的 PR 同时更新对应文档。约 90% 的数据模型 PR 现在包含 Skill 变更
- **跨界面一致性**：同一 Skill 在 Slack、IDE、看板工具、独立 Agent 会话中给出相同答案，通过自动同步到插件市场、云存储和 MCP 资源实现

### 验证层

#### 离线评估

- **基于看板的评估**：Claude 自动生成（人类验证），覆盖最常见问题
- **长尾评估**：喂入业务上下文，让 Claude 生成领域内其他合理问题
- 持续采集 Stakeholder 在线程中纠正 Agent 的时刻作为候选评估用例

关键做法：
- 将真实答案锚定到快照日期，防止数据漂移
- 每次运行存入数仓（Skill 版本、git SHA、模型 ID、通过/失败、Token 数）
- 按领域设置闸门阈值（~90%）后才能对外发布
- 离线评估准确率应接近 100%

#### 消融实验

每个架构决策通过固定评估集、仅变更一个组件来验证。关键发现：

- **原始 SQL 语料直接检索几乎无用**：给 Agent 直接 grep 访问所有看板/转换/笔记本 SQL 对准确率影响不到 1 个百分点——瓶颈是结构而非信息可及性
- **文档过度优化适得其反**：连续三轮净负面迭代
- **廉价模型替代对抗审查不可行**：损失大部分准确率增益，实际无提速

#### 在线验证

- **对抗审查**：用 Claude Skill 挑战所有基础假设，准确率 +6%，代价是 Token +32%、延迟 +72%
- **溯源页脚**（provenance footer）：来源层级、数据新鲜度、模型负责人——帮助消费者判断可信度
- **数据质量检查**：确保引用字段是最新的、完整的、无异常的
- **被动监控**：语义层解决率、纠正性语言占比，每周审查
- **主动修正采集**：定时 Agent 扫描 Slack 中的纠正语言，自动起草参考文档修正、开 PR 并标记领域负责人

**尚未完全解决的问题**：静默失败——答案错误但看起来合理且被无异议使用。缓解措施包括 provenence footer 和领导层报告的人工签批。

## 启动建议

从零开始时，几个权威数据集、几十个离线评估用例、一个精简的知识 Skill 就能捕获大部分价值。

需对齐的核心问题：
- 当前正确性 vs 未来模型能力提升的权衡
- 业务复杂度预期变化
- 目标受众的技术水平
- 为提升准确率愿意付出的成本与延迟
- 访问控制与内部数据隐私的容忍度

最大的收益来自于逐一解决三大失败模式：将歧义压缩为单一治理答案、让答案易于发现、在任一环节腐化时标记出来。
